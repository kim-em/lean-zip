# Progress: Track E — Zstd multi-frame and skippable frame support

**Date**: 2026-03-02T05:29Z
**Session**: 712f3d96 (feature)
**Issue**: #454

## What was accomplished

- Added `skipSkippableFrame` function to `Zip/Native/ZstdFrame.lean` — parses
  skippable frame headers (magic 0x184D2A50–0x184D2A5F, 4-byte LE size) and
  returns position after the frame data.
- Rewrote `decompressZstd` from single-frame to multi-frame loop: iterates
  through concatenated frames, dispatches Zstd frames to `decompressFrame` and
  skippable frames to `skipSkippableFrame`, rejects unrecognized magic numbers.
- Updated existing test 22 (was: reject skippable frame → now: skip and return
  empty output).
- Added 9 new tests (45–53) covering: `skipSkippableFrame` valid/truncated/bad
  magic, multi-frame concatenation, skippable + Zstd frame interleaving (both
  orderings), trailing garbage detection, and empty input.
- Used hand-crafted minimal raw-block Zstd frame in trailing garbage test to
  avoid dependency on FFI compression format producing RLE blocks.

## Decisions

- Empty input and skippable-only input return empty ByteArray (not an error).
  This matches the RFC: "A Zstd compressed file can contain multiple Zstd frames"
  — zero is a valid count.
- Trailing bytes that don't form a valid frame magic produce an error, ensuring
  corrupted inputs are detected.
