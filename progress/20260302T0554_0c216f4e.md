# Track E: Interleaved FSE sequence decode loop for Zstd

**Session**: 0c216f4e
**Issue**: #469
**Type**: feature

## What was done

1. Added `decodSequences` function to `Zip/Native/ZstdFrame.lean` (~85 lines):
   - Decodes interleaved FSE sequences from a backward bitstream (RFC 8878 section 4.1.1)
   - Initializes three FSE state machines (litLen, offset, matchLen) from the bitstream
   - Per-sequence loop: symbol lookup (offset, matchLen, litLen), extra bits reading
     (offset, matchLen, litLen), then state update (litLen, matchLen, offset — reversed order)
   - Last sequence skips state update bits (standard FSE behavior)
   - Uses `decodeLitLenValue`, `decodeMatchLenValue`, `decodeOffsetValue` for extra bits
   - Returns `Array ZstdSequence`

2. Added 3 tests in `ZipTest/ZstdNative.lean` (~108 lines):
   - Test 74: Single sequence with minimal FSE tables (accuracyLog=1), verifies
     litLen=0, matchLen=3, offset=3
   - Test 75: Two sequences with state updates and extra bits, hand-computed
     backward bitstream across 2 bytes
   - Test 76: Zero sequences returns empty array

3. Added `Inhabited` to `ZstdSequence` deriving clause (needed for `[i]!` indexing)

4. Added `import Zip.Native.Fse` to ZstdFrame.lean (required for `FseTable`,
   `FseCell`, `BackwardBitReader` types)

## Decisions

- Used explicit `BackwardBitReader.readBits br` instead of dot notation for mutable
  variables (Lean 4 limitation with dot notation on `mut` bindings)
- Cached cell field values (`numBits.toNat`, `newState.toNat`) in `let` bindings before
  `if` blocks to help type inference
- Hand-constructed backward bitstreams for tests rather than using real compressed data,
  since predefined FSE tables (#437) aren't merged yet
