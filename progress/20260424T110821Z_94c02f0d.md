## Decisions and key observations

- **Why 24, not 13.** The issue was filed 2026-04-24T10:23; the
  sweep completed after a rebase onto master that pulled in PR
  #1793 (duplicate-ZIP64-extra-block fixture). By sweep time the
  live tree carried 24 instances — the 13 originally flagged,
  five introduced by PRs #1785/#1788 (landed between issue filing
  and claim), three new from PR #1793's own rows/bullet, and
  three prose `this PR's CD-parse guard` references on the
  oversized-*.zip corpus rows the planner's by-hand count missed.
  The verification requirement — linter clean and
  `grep -nE '(#TBD|this PR)' SECURITY_INVENTORY.md` returns no
  matches — drives to zero regardless of the original count.

- **Why pass (d) is soft.** A stale placeholder is a documentation
  drift (the inventory still points at the right file, just with
  a missing PR cross-ref). Upgrading to a hard failure would tie
  CD-touching feature PRs to a post-merge sweep step before they
  can land — the exact sequencing problem that created the drift
  class in the first place. The soft warning is noisy enough to
  catch the drift in the next review cycle without blocking the
  feature worker, which is the ROI ranking the paired reviews
  converged on.

- **Opt-out reuse.** The `<!-- drift-detector: ... -->` marker
  already present for pass (c) was threaded through pass (d) at
  trivial cost (three lines). The three uses currently present
  in the inventory (Zlib/Gzip/RawDeflate whole-buffer rows at
  :480-:482) would not be affected by pass (d) in any case
  because they do not contain `#TBD` / `this PR`; the marker's
  presence on pass (d) is insurance against a future legitimate
  prose use.

- **Rebase recovery.** The session started on `agent/94c02f0d` at
  `e645805`; master advanced by two commits (`d0be969`, `00bbba4`
  = PR #1793) during the work. The first attempt produced a
  SECURITY_INVENTORY.md conflict because the #1793 landing shifted
  line numbers and introduced three new `this PR` placeholders
  that overlapped my sweep region. Resolution: soft-reset to
  `origin/master`, restore all files *except* my linter script
  edits and the progress entry from master, then re-run the
  sweep on the fresh tree. This produced a clean, conflict-free
  commit on top of current master.
