## What was accomplished

Fourth dimension of the CD/LH consistency triad (now tetrad): extended
`Archive.readEntryData` to reject a new ZIP-smuggling vector where the
CD and LH carry different filenames at the same `nameLen`. A re-parsing
extractor would decode CD name `hello.txt` for path-traversal checks
but display LH name `evil.html` — or vice versa.

Three code changes in `Zip/Archive.lean`:

- `Archive.Entry`: new `pathBytes : ByteArray := default` field,
  populated from the existing `nameBytes` in `parseCentralDir`. Dropped
  `Repr` from the deriving clause (no `Repr ByteArray` instance in
  stdlib and no call-sites use `Repr Entry` — verified by grep).
- `readEntryData`: after the flags-consistency check and before the
  data-descriptor-gated size checks, compare `nameAndExtra.extract 0
  nameLen == entry.pathBytes` at the byte level. Bytes-level comparison
  is essential — the smuggling vector is precisely the case where two
  byte sequences decode to the same `String` under CD's declared
  encoding but diverge under a different encoding choice at the LH.

The check is **not** gated on bit 3 (data-descriptor), matching the
flags check — the name field is independent of bit 3.

Error message: `"zip: name mismatch between CD and local header for
{label}"` (no raw bytes interpolated; they may contain non-printable
characters).
