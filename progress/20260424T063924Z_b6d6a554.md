# Track E: method 0 (stored) compressedSize == uncompressedSize consistency check

Session: b6d6a554 (feature)
Date: 2026-04-24 (UTC)
Issue: #1763
Branch: agent/b6d6a554

## Summary

Added a CD-parse stored-method size-invariant check to `parseCentralDir`
in `Zip/Archive.lean`: when a CD entry has `method == 0` (stored), the
(ZIP64-resolved) `compressedSize` and `uncompressedSize` must be equal,
per APPNOTE §4.4.5 *"The file is stored (no compression)"*. Landed the
122-byte `cd-stored-size-mismatch.zip` regression fixture (both CD and
LH advertise `compSize=6, uncompSize=7` with `method=0`) and wired it
into `ZipTest/ZipFixtures.lean` alongside the per-entry disk-number
rejection block.

## Precedence shift on three pre-existing fixtures

Three existing malformed fixtures now fire the new CD-parse check
before their historical attribution points, because they all carry
`method=0` with ZIP64-resolved `compSize ≠ uncompSize`:

- `testdata/zip/malformed/oversized-compressed-size.zip`
  (previously `"local data span"`)
- `testdata/zip/malformed/oversized-zip64-compressed-size.zip`
  (previously `"local data span"`)
- `testdata/zip/malformed/oversized-zip64-uncompressed-size.zip`
  (previously `"truncated ZIP64 local extra field"`)

Their expected substrings were updated to `"stored-method size mismatch"`
and comments in both `ZipTest/ZipFixtures.lean` and
`SECURITY_INVENTORY.md` explain the shift. The fixtures themselves are
unchanged (byte-identical) and continue to provide regression coverage,
now attributed to the earlier CD-parse guard rather than the later
`readEntryData` checks. Future fixtures that need to exercise the
`"local data span"` or `"truncated ZIP64 local extra field"` checks
should use `method=8` (deflate) so the stored-method check does not
fire — this is a documented follow-up opportunity in the
`error-wording-catalogue` skill's precedence note.
