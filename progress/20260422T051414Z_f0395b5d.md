## 1. Input-distribution coverage (deliverable §1)

### What the harness does

`sizeClasses := #[0, 1, 16, 512, 8192, 65536]` cycled per iteration
via `pick sizeClasses (xorshift64 state)`. For each iteration the
same-sized random `ByteArray` is fed through every whole-buffer
entry point and (after chunking) the streaming path.

### Boundary coverage

| Boundary | Hit by which size? | Notes |
|---|---|---|
| Empty input | 0 | header path, early-return |
| Single byte | 1 | below zlib 2-byte header, below gzip 10-byte header |
| Partial header / tiny block | 16 | covers the 2-byte zlib + 10-byte gzip header prefixes |
| Small non-trivial body | 512 | |
| Sub-DEFLATE-window | 8192 | ¼ of the 32 KiB sliding window |
| DEFLATE-window crossing | 65536 | 2× the 32 KiB window, exercises ring-buffer wrap on native |

Missing exact-boundary points — 2 (exact zlib header), 10 (exact
gzip header), 32768 (exact DEFLATE window), and powers-of-two
between (256, 1024, 16384). With random-byte inputs these
near-boundary probes would produce the same "reject at byte 1 or 2"
outcome as neighbouring sizes, so the added coverage is marginal.

### The deeper limitation — *honest observation*

All inputs are random bytes. DEFLATE has strict syntactic
constraints at the first ~20 bytes (zlib header CMF/FLG, gzip
magic, DEFLATE block header bits). For random input, the
overwhelming majority of iterations fail at the first syntactic
check — the rest of the `ByteArray` never reaches the LZ77 state
machine. Consequently:

- The 65536 input size does **not** meaningfully exercise the
  32 KiB window wrap for the decoder, because random bytes rarely
  produce a valid DEFLATE stream that reaches 32 KiB of decoded
  output.
- Where the large input *does* buy coverage is the
  **allocation / bounds-checking path**: allocator grow loops in
  zlib (`inflate_fast` output buffer resize), the `realloc`
  growth in `c/zlib_ffi.c`, and the Lean-side output-cap guard
  (`defaultMaxOutput = 4 MiB`).

This is a limitation of random-byte fuzzing, not specifically of
the size set. Structure-aware fuzzing (libFuzzer / AFL++ with a
corpus seeded from `testdata/`) would fix it — already flagged as
out-of-scope follow-up in PR #1602's "Follow-ups" section.

### Is 64 KiB enough?

For the *random-byte regime* this harness operates in: yes. A
larger cap (256 KiB or 1 MiB) would increase per-iteration CPU
cost without broadening decoder-state-machine coverage, because
almost every iteration still errors at header parsing. The ~10 ms
smoke-test cost claim would hold only if the large-size class
stays a rare selection — which the uniform `pick` distribution
doesn't guarantee (1/6 iterations × 65 KiB = non-trivial).

Measured now: `LEAN_ZIP_FUZZ_SEED=42 scripts/fuzz-inflate.sh 3`
completed 29 700 iterations (run 1) and 29 799 iterations (run 2)
in 3 s, so per-iteration cost on a mixed size-class distribution
averages ~100 μs on this host. The 100-iteration smoke-test cost
extrapolates to ~10 ms, matching the PR's claim.

### Recommended-size-addition analysis

Adding 2 and 32768 to `sizeClasses` would:
- 2: cover the exact zlib header length (CMF+FLG boundary).
- 32768: cover the exact DEFLATE window size.

Expected marginal benefit is low. Not recommended.

### Verdict — §1

**keep as-is.** The size set is correctly chosen for the
random-byte regime. The 64 KiB cap is right for smoke-test cost.
The real coverage gap (LZ77 state-machine reachability under
random bytes) is architectural — a corpus-seeded / coverage-guided
harness is the future work, not adding more size-class points.
Record this limitation in the progress entry of any follow-up
planner issue that extends fuzz coverage.

## 2. Seed-diversity and iteration-count audit (deliverable §2)

### What the harness does

Smoke path: `runFuzz 0xdeadbeef 100` from `ZipTest/FuzzInflate.lean`'s
`tests`. Fixed seed, fixed iteration count.

Stand-alone exe: `LEAN_ZIP_FUZZ_SEED` (default `0xdeadbeefcafef00d`)
and `LEAN_ZIP_FUZZ_SECONDS` (default 30) — both environment-driven.

### Back-of-envelope bug-catch probability

For a bug with empirical hit rate *p*, a run of *N* iterations has
catch probability `1 - (1-p)^N`. For *p = 1/1000* and *N = 100*:

    1 - 0.999^100 ≈ 0.0952 ≈ 9.5%

**The issue's 50%-chance target needs N = 693, not 100.** At
~100 μs per iteration (§1 measurement), N=693 would add ~70 ms to
`lake exe test`, still trivial. N=1000 would add ~100 ms.

### Fixed seed vs. rotating seed

Argument for *fixed seed*: reproducibility. A failing iteration
at index K is reproducible by anyone running the same build.

Argument for *rotating seed*: a 100-iteration run with a fixed
seed covers 100 specific inputs forever. A 100-iteration run
with a daily-rotating seed covers 100 × N-days distinct inputs
over time, at the cost of today's failure not being
reproducible from yesterday's seed alone.

The issue's framing — "one seed × 100 iterations exercises
enough of the input-space" — points at the trade-off but doesn't
quantify it. Quantification:

- **Fixed seed, N=100**: 100 distinct inputs, all reproducible.
- **Fixed seed, N=1000**: 1000 distinct inputs, all reproducible,
  ~100 ms smoke-test overhead.
- **Daily-rotated seed, N=100**: 100 × D inputs over D days. CI
  failure on day D is reproducible by reading the seed from the
  CI log (already printed:
  `[fuzz-inflate] seconds=3 seed=42 ...` — but `tests` in
  `FuzzInflate.lean` doesn't print the seed; see below).

### Recommendation hierarchy (ordered by cost / benefit)

1. **Bump smoke-test N from 100 → 1000** in
   `ZipTest.FuzzInflate.tests`. Cost: ~100 ms per `lake exe test`.
   Benefit: catch probability for 1-in-1k bug rises from
   9.5% → 63% per run. This is the cheapest meaningful
   improvement.
2. **Print the seed in the smoke-test output line**
   (`IO.println s!"    seed=0x{seed...} {n} iterations completed"`)
   so a CI failure log tells the triage engineer which seed to
   reproduce from.
3. **Rotate the smoke seed daily** (e.g. `seed ← mix 0xdeadbeef
   (← IO.getEnv "CI_RUN_DAY").getD (day-since-epoch)`). Only
   worthwhile after (1) and (2); adds "find the log" complexity.

### Stand-alone exe — 30 s default

30 s × ~100 μs per iteration = ~300 000 iterations. At a 1-in-1k
bug, a single run has effectively 100% catch probability. The
default is fine.

The 7-day exit criterion assumes one CI run per day (7 runs) at
30 s each. `7 × 300 000 ≈ 2.1M` iterations covered. For a
1-in-1M bug, cumulative catch probability rises to ~88%. That's
a meaningful coverage signal.

120 s or 300 s defaults would not materially change the coverage
level — they'd just push "1-in-1M" down to "1-in-10M" per run,
which is well past the point of diminishing returns for random
bytes against a decoder.

### Verdict — §2

**recommended change: bump smoke N to 1000; print seed in the
smoke output line.** (Low-friction. Daily-rotated seed is a
further nice-to-have but not urgent.) The 30 s default for the
stand-alone exe is fine; 120 s / 300 s is not materially better.

Neither recommendation is *blocking* for starting the 7-day
fuzzer-clean window — the PR's smoke test already costs ~10 ms
and works. These are follow-up improvements.

## 3. Exception-handling scope audit (deliverable §3)

### What the harness does

```lean
private def tryFFI (label : String) (action : IO ByteArray) : IO Unit := do
  try
    let _ ← action
  catch e =>
    let _ := s!"[fuzz {label}] {e}"
    pure ()
```

Catches any exception the `IO` action raises — regardless of
`IO.Error` variant. The progress entry's rationale was that
panics and sanitizer traps bypass the Lean `catch` anyway, so
the variant distinction "does not help".

### What the FFI actually throws today

Only `IO.Error.userError`, via `lean_mk_io_user_error` — confirmed
by `grep -n "lean_mk_io" c/zlib_ffi.c` (only one hit:
`lean_mk_io_result_mk_error(lean_mk_io_user_error(...))`). Every
zlib-FFI error path funnels through that helper. No other
`IO.Error` variant is produced by the FFI today.

Other `IO.Error` variants that *could* show up if the FFI
surface changes:

- `IO.Error.unexpectedEof` — if a future streaming surface uses
  the Lean runtime's EOF detection (e.g. reading from a file
  handle).
- `IO.Error.hardwareFault`, `IO.Error.runtimeError` — produced
  by the Lean runtime on structural failures (rarely from FFI,
  but reachable).
- A future `IO.Error` case Lean adds upstream.

### What Lean's `catch` does and does not catch

- *Catches*: every value of type `IO.Error` (the only exception
  type in `IO`).
- *Does not catch*: `panic!` / `panicWithPosWithDecl` / kernel
  `unreachable!`. These call `lean_panic_fn`, which prints to
  stderr and calls `lean_internal_panic`, which in normal
  (release) builds calls `abort(3)`. Process-level abort
  bypasses the Lean `IO` machinery — exit status is
  `128 + SIGABRT (6) = 134` on Linux.
- *Does not catch*: sanitizer traps (ASan, UBSan) — these invoke
  the sanitizer runtime's `__asan_report_error` path which
  `_exit`s (or aborts under `abort_on_error=1`, which
  `scripts/sanitize-ffi.sh` already sets).
- *Does not catch*: `SIGSEGV` / `SIGBUS` from a memory fault in
  the FFI with sanitizers off — this terminates the process
  with exit 139 / 138.

So the progress entry's claim holds: **panics and sanitizer traps
exit the process non-zero regardless of the Lean `catch`.** The
harness correctly treats non-zero exit as failure.

### The forward-compatibility concern

If a future `c/zlib_ffi.c` change introduces (say) an
`IO.Error.runtimeError` to indicate *structural* corruption
that the current FFI would abort on — meaning: a real bug
signalling but not crashing — the current `tryFFI` would silently
swallow it. The fuzz run would pass, missing a regression.

The narrower variant would make this fail loudly:

```lean
catch e =>
  match e with
  | .userError _ => pure ()
  | _ => throw e  -- re-raise anything non-userError
```

Cost: one extra pattern match per caught exception.
Benefit: forward-safety against a future FFI change that reports
failure via a new variant.

### Recommendation

**Narrow to `IO.Error.userError` only.** This is a small refinement
that matches the current FFI contract (only `userError` is thrown)
and flags the future-change case as a loud test failure instead of
silent success. The progress entry's simplicity argument is
reasonable for the current FFI, but the code is one `match` away
from being robust against the future.

Alternative reading: the fuzz harness is specifically for
"decoder did not crash the process". A non-userError `IO.Error`
reaching the catch would mean "decoder reported a problem via the
wrong variant" — arguably that's a *contract bug in the FFI*, not
a *bug in the fuzzer*, and flagging it via the fuzzer is a side
benefit rather than the primary concern. Either way, narrowing
costs very little and adds coverage.

### Verdict — §3

**recommended change: narrow `tryFFI` to match `.userError` only.**
Not blocking; the current broad catch is safe for today's FFI.

## 4. Stream chunk-size coverage (deliverable §4)

### What the harness does

```lean
private def chunkSizes : Array Nat := #[1, 7, 31, 127]
```

`splitIntoChunks data state` loops `while pos < data.size`,
picking a chunk size per iteration via `pick chunkSizes` then
taking `data.extract pos (min data.size (pos + size))`.

### Sampling distribution

`pick arr s := arr[s.toNat % arr.size]!`. For `chunkSizes.size = 4`
and xorshift64-produced `s`, the distribution of `s.toNat % 4` is
approximately uniform — xorshift64's low-order bits are not
cryptographically uniform, but for `% 4` at this scale the bias
is negligible. **Uniform over `{1, 7, 31, 127}`.**

### Boundary coverage

| Chunk size | What it stresses |
|---|---|
| 1 | Byte-at-a-time `push`; every Huffman code crosses a chunk boundary; max pressure on `inflateReset` / internal state machine |
| 7 | Odd, doesn't align with zlib's 64-byte internal work chunks |
| 31 | 2⁵ − 1, won't align with 32-byte DEFLATE block-header or Huffman-code lengths |
| 127 | 2⁷ − 1, spans multiple short Huffman codes but won't align with a byte/word boundary |

All four sizes are primes or 2^n − 1. That's a sensible
"adversarial-to-alignment" choice: any bug that manifests only
when a chunk happens to end on a block boundary or byte-pair
boundary will *not* hide behind these four sizes.

### What's missing

Larger chunks (256, 1024, 4096) would let multiple Huffman codes
pack into one chunk, exercising the "happy path" where `push`
returns more bytes per call. The current set skews adversarial —
every chunk crosses at least one code boundary — which is the
*correct* bias for a bug-hunting fuzzer. Happy-path coverage is
what the regression tests in `ZipTest/Gzip.lean` and
`ZipTest/RawDeflate.lean` are for.

*Observation, not recommendation:* adding a single larger chunk
(e.g. 1024) would marginally increase coverage of the
non-adversarial path at trivial cost, but again the regression
tests already do this.

### Whole-input / single-chunk edge case

With `sizeClasses` including 0, `splitIntoChunks` is called on
an empty `ByteArray` → while loop never enters → empty
`chunks : Array ByteArray`. `tryStreaming #[]` loops zero times
and calls `state.finish` once. This exercises the empty-stream
edge case, which is good.

For input size 1, any chunk size in `{1, 7, 31, 127}` produces a
single-chunk stream (`[1 byte]`). That's the one-byte-push edge
case. Good.

### Verdict — §4

**keep as-is.** The adversarial-bias is correct; the uniform
distribution over primes-and-Mersennes is the right shape for
stressing chunk-boundary state. No blocking issues.

## 5. Reproducibility and CI-readiness (deliverable §5)

### Determinism of `runFuzz` (fixed iteration count)

`runFuzz seed iterations` uses no clock reads, no `IO.rand`,
pure `xorshift64`. Given the same `(seed, iterations)`, byte
output is identical across runs. **Byte-deterministic.**

### Determinism of `runFuzzUntil` (wall-clock budget)

`runFuzzUntil seed deadlineMs` checks `IO.monoMsNow` per
iteration. The *sequence of inputs-per-iteration-index* is
deterministic (same xorshift state machine), but the *iteration
count* is wall-clock bounded. Two 3 s runs with the same seed
will share the first `N = min(count1, count2)` iterations but
may differ after.

**Empirical check (this session):**

```
$ LEAN_ZIP_FUZZ_SEED=42 scripts/fuzz-inflate.sh 3
...
[fuzz-inflate] iterations=29700 elapsed=3000ms
[fuzz-inflate] OK

$ LEAN_ZIP_FUZZ_SEED=42 scripts/fuzz-inflate.sh 3
...
[fuzz-inflate] iterations=29799 elapsed=3000ms
[fuzz-inflate] OK
```

Iteration counts differ by 99 across two back-to-back 3 s runs
on the same host. **The issue's claim ("should be identical")
does not hold empirically.** On a noisy CI host the spread will
be larger.

This does *not* compromise bug-triage reproducibility: a crash
at iteration K under a given seed can always be reproduced by
running `runFuzz seed (K+1)` (fixed iteration count) — but it
means the tolerated iteration-count drift needs to be
documented.

### Recommendation for the 7-day CI window

When CI starts running the fuzz harness, log both:

- `seed` (so a failing run is reproducible)
- `iterations` (so "how much got covered" is visible in the
  rollup)

`ZipFuzzInflate.lean` already prints both:

```
[fuzz-inflate] seconds=30 seed=... deadline=...
[fuzz-inflate] iterations=... elapsed=...ms
```

So the CI rollup can just grep these lines. No further code
change needed.

### Non-zero exit on uncaught exception

`ZipFuzzInflate.lean`'s `main` has signature `IO Unit` with no
top-level `try`. `runFuzzUntil` calls `oneIteration`, which
calls the three `tryX` helpers — each wraps its decoder call in
`try { ... } catch _ => pure ()`. So the ways `main` can exit
non-zero are:

1. A `panic!` / `abort` inside the FFI or runtime — process
   exits with `SIGABRT` (signal 6, exit 134 on Linux).
2. A sanitizer trap (ASan / UBSan) — exits via sanitizer's
   `_exit` path (non-zero).
3. An exception thrown *outside* the `tryX` blocks — e.g.
   `IO.monoMsNow` failing (never observed), or an exception
   from the outer `for` loop (shouldn't happen).

Case 3 would surface as `uncaught exception: ...` printed by
Lean's default top-level handler, exit 1. **Confirmed correct:
Lean's default `main` handler prints the `IO.Error` and exits 1.**

### Nix-shell-fallback path

`scripts/fuzz-inflate.sh` lines 64–70:

```bash
if ! command -v pkg-config >/dev/null || ! pkg-config --exists zlib 2>/dev/null; then
  if [[ -z "${IN_NIX_SHELL:-}" && -f shell.nix ]] && command -v nix-shell >/dev/null; then
    echo "[fuzz-inflate] re-executing under nix-shell (zlib not found on host)"
    exec nix-shell --run "bash $(pwd)/scripts/fuzz-inflate.sh $*"
  fi
fi
```

Falls through to the plain `lake build` path if neither
pkg-config-finds-zlib nor nix-shell-available holds. On bare
Ubuntu with `libz-dev` installed, `pkg-config --exists zlib`
returns true — no re-exec, direct `lake build`. This works.

On a host with neither libz-dev nor nix, the script would fail
at `lake build` (zlib not found by the linker). That is the
correct failure — the wrapper doesn't need to synthesise a
zlib.

One gap: **the script does not check for `$IN_NIX_SHELL` before
running** — it relies on the `pkg-config` probe instead. On a
host where `pkg-config` is installed but zlib is missing and
`nix-shell` is not on `$PATH`, the script exits non-zero with a
confusing error (`pkg-config not found zlib` printed, then
`lake build` fails with linker error). Not blocking, but a
one-line improvement would be to print a clearer "install
libz-dev or enter nix-shell" message before attempting
`lake build`. Recommended for a follow-up.

### Verdict — §5

**recommended change (documentation only): correct the issue's
byte-deterministic claim to "byte-deterministic given a fixed
iteration count; wall-clock-budgeted runs may drift by O(1%)
iterations across runs".** The harness itself is CI-ready; the
claim in the issue body needs clarifying when the 7-day
criterion is codified into `PLAN.md`.

Not blocking the 7-day window. Reproducibility of *failing
iterations* is preserved by the seed + iteration-index logging
that `ZipFuzzInflate.lean` already prints.

## Is the harness good enough to gate the 7-day criterion?

**Yes.** None of the five deliverables surfaced a blocking
defect. The harness is:

- Deterministic under fixed iteration count (for triage).
- Cheap enough that CI can run it per-push (~30 s × 1/push).
- Robust to decoder exceptions (all caught; process-level
  crashes bypass the catch and exit non-zero).
- Covers every inflate entry point — FFI whole-buffer
  (3 paths), native whole-buffer (4 paths), streaming (1 path).

The three recommended changes (smoke N=1000, print seed,
narrow `tryFFI`) are *improvements*, not *prerequisites*. The
7-day window can start counting as soon as CI is wired to run
the harness on each master push; the improvements can land
during the window without resetting the clock, because they
don't change the set of decoder entry points covered.
