## Decisions

- **Location: end of *Tar Parser/Extractor* section, top-level
  bullet.** The issue body offered two options: (a) under
  *Tar Parser/Extractor* (where the post-#2405 Recent-wins / parent
  Reproducer Corpus row already live) or (b) as a top-level
  paired-review subsection mirroring the cadence-file shape used by
  PR #2398 / PR #2403. The previous tar-fixture-closure paired-reviews
  (#1882, #1884, #1913, #1940, #1947, #1951, #1962, #1963, #1986)
  were progress-file-only — they did not edit
  `SECURITY_INVENTORY.md` — so there is no in-inventory precedent
  for tar-fixture-closure paired-review placement. I followed the
  newer cadence (PR #2398, PR #2403, PR #2403) which places
  paired-reviews as top-level bullets *under the relevant TCB
  section*. PR #2405 is a Tar Parser/Extractor change, so its
  paired-review naturally belongs under that section. The bullet
  sits as the last entry under the section's body, before
  `#### Symlink/hardlink extraction policy`, so it does not
  visually break the bulleted-list flow of the section.
- **Length: ~125 lines, comparable to PR #2403 (136 lines).** The
  issue body suggests *~50–80 lines of prose*; PR #2403's immediate
  sibling came in at ~136 lines. I aimed for the same density
  (six audit dimensions × short paragraph each). The dimensions
  listed in the issue body all required substantive content — the
  family-closure-fidelity dimension in particular benefits from
  spelling out arm-vs-fixture-vs-slot accounting explicitly because
  the *3/3* claim mixes two different counting schemes (1 keyBytes
  arm + 1 valueBytes arm = 2 *guard arms*; 3 *fixtures* spread across
  the keyBytes arm and the value-side filesystem-reaching slots).
  Trimming further would have lost audit content.
- **No follow-up issue minted.** The audit confirms all five
  dimensions land cleanly. The one observation worth recording —
  that `applyPaxOverrides`'s value-side `"uname"` / `"gname"`
  overrides exist as defense-in-depth parallels of UStar
  `uname` / `gname`, but are excluded from the *2-slot* count by
  the inventory's *filesystem-reaching* framing — is logged in the
  *Family-closure fidelity* sub-bullet as a scoping convention, not
  a closure gap. Whether to extend the value-side family with
  per-slot defense-in-depth fixtures is a separate scoping decision
  for a future planner, not a follow-up arising from this PR's
  closure claim.
