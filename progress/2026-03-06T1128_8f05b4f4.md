#### Signature gaps blocking future composition

**Gap 1 ÔÇö ValidSequence predicate mismatch (MAJOR)**:
`ValidSequence` constrains `seq.offset` (the raw offset field), but
`executeSequences.loop` checks `(resolveOffset seq.offset history litLen).1`
(the resolved offset). For repeat offset codes 1ÔÇô3, the resolved offset is a
history entry unrelated to `seq.offset`. A theorem "if all sequences satisfy
ValidSequence, then executeSequences succeeds" cannot be proven with the
current predicate because `seq.offset ÔëĄ outputSoFar + seq.literalLength`
with `seq.offset Ôłł {1,2,3}` says nothing about whether the resolved history
entry is within bounds. The predicate needs to either track resolved offsets
or include a history-aware bound.

**Gap 2 ÔÇö ValidOffsetHistory preservation (addressed by issue #692)**:
No theorem proves `ValidOffsetHistory history Ôćĺ ValidOffsetHistory
(resolveOffset ...).2`. The shifted value theorems give new history entries
but don't assemble positivity. This is exactly what #692 plans to deliver.

**Gap 3 ÔÇö Incomplete positivity for litLen = 0**:
`resolveOffset_positive_litLen_pos` requires `litLen > 0`. For `litLen = 0`:
- rawOffset Ôłł {1,2}: resolved offset is a history entry (positive by
  ValidOffsetHistory), but no composed theorem states this
- rawOffset = 3: `history[0]! - 1` can be 0 (documented correctly in
  docstring) ÔÇö this is a genuine edge case, not a spec gap

A connector lemma like `resolveOffset_positive_shifted12` for
`rawOffset Ôłł {1,2}` with `litLen = 0` would complete coverage.

**Gap 4 ÔÇö No overlapping copyMatch content theorem**:
`copyMatch_getElem_ge_nonoverlap` requires `offset Ôëą length`. Zstd commonly
uses overlapping matches (RLE patterns where offset < length). A full
`executeSequences` content correctness proof needs the overlapping case. This
is more of a future need than a blocking issue for the current loop invariant.
