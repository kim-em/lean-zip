## What landed

Second sibling-class fixture in the silent-skip family, extending the
ten per-typeflag arms (PRs #1555/#2413/#2417/#2422/#2425/#2428/#2431/
#2434/#2437/#2439) and the first sibling-class fixture
`tar-mixed-skipped.tar` (PR #2449). All eleven existing silent-skip
fixtures use `size == 0` for the skipped entry, so
`skipEntryData input 0` is a no-op except for the trailing 512-byte
header alignment — none of them exercises the data-advance arithmetic
of `skipEntryData input e.size` for non-zero `e.size`.

The new `tar-skipped-payload.tar` is a three-entry archive:

- `before.txt` — typeflag `'0'`, payload `"BEFORE\n"` (size 7)
- `fifo-entry` — typeflag `'6'` (POSIX UStar FIFO), declared
  `size == 512`, all-zero 512-byte payload, silently skipped
- `after.txt` — typeflag `'0'`, payload `"AFTER\n"` (size 6)

Total geometry: 6 × 512 = 3072 bytes. SHA-256:
`2d32429ac507e5b321ba71b179e289878320989ed0d4fa05aeb752bdcddf92a4`.

## Issue spec inconsistency

The issue's *Deliverables §1* described "size := 1024 and a
byte-deterministic 1024-byte payload" but the same paragraph's
geometry note ("Total geometry: 6 × 512 = 3072 bytes (header+payload
for each of the two regular entries = 2 × 512 each, plus 1 header +
2 payload blocks for the middle entry = 3 × 512)") and *Verification*
("exactly 3072 bytes (verifiable via `wc -c`)") together imply a
512-byte payload — three entries × 2 blocks each = 6 × 512 = 3072.
A 1024-byte payload would yield 7 × 512 = 3584 bytes total.

Going with **size 512** (matching the verification spec) since:

1. Verification is the test pass/fail criterion — a 3584-byte file
   would not match `wc -c | grep 3072`.
2. The geometry breakdown in the parenthetical math sums to 6 blocks
   when the FIFO uses 1 header + 1 payload block (size 512).
3. Either size exercises the same code path
   (`skipEntryData input e.size` for non-zero `e.size`); padding
   arithmetic is not exercised either way (both 512 and 1024 are
   multiples of 512, so `paddingFor` returns 0).

The 1024-byte payload mention appears to be a planner typo
inconsistent with the verification spec.
