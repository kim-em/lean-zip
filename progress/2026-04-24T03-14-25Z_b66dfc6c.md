## What happened

Orientation and initial implementation went smoothly — added the
`Entry.versionNeeded` field, populated it from the CD in
`parseCentralDir`, and wired the strict-equality CD/LH check in
`readEntryData` between the method and flags checks (matching the
offset-order rationale the issue gave).

`lake build -R` passed clean. `lake exe test` then surfaced the core
problem:

    uncaught exception: zip: versionNeededToExtract mismatch between CD
    and local header for README (CD=45, LH=20)

at the existing `go-zip64.zip` interop fixture test. Checked the
fixture directly with a small Python inspector:

- LH: versionNeeded=20, sizes packed plainly (compSize=24,
  uncompSize=24), extra field empty.
- CD: versionNeeded=45, 32-bit size slots set to 0xFFFFFFFF, ZIP64
  extra block (tag 0x0001) carrying the actual u64 sizes.

Both headers are individually self-consistent — LH's own fields don't
use ZIP64 markers, CD's do. This is the standard Go-and-Python ZIP64
producer pattern: CPython `zipfile` computes LH-side and CD-side
`versionNeeded` independently from each side's own extras, and they
can legitimately differ when (as here) the payload sizes fit in 32
bits but the archive uses ZIP64 elsewhere.

Surveyed all interop fixtures: only `go-zip64.zip` has divergent LH/CD
versionNeeded; every other fixture has matching values. But that one
fixture is a real Go artefact and representative of a common producer
pattern.

## Decision

Reverted the source changes (`git checkout -- Zip/Archive.lean`) and
skipped the issue with a detailed comment explaining the interop
conflict and recommending three replan options:

1. **One-sided**: `localVersion ≤ entry.versionNeeded` — the
   minimum-scope correct check. Rejects the LH-capability-smuggle
   direction (LH claiming higher version than CD, e.g. LH has ZIP64
   extra but CD doesn't) while accepting Go's CD≥LH pattern.
2. **Per-side self-consistency**: each side's `versionNeeded ≥ 45` iff
   that side has a ZIP64 extra — more thorough but needs a "uses
   ZIP64" signal plumbed through both parsers.
3. **Informational-only**: treat the field as advisory (matching
   real-world consumer behaviour — Info-ZIP, 7zip, Go, and Python all
   feature-gate on actual ZIP64 extra blocks, not on this field) and
   check ZIP64 semantics at the feature level instead.

The issue's framing of `CD=45 / LH=20` as "a downgrade attack" is
inverted — that's the common case for non-ZIP64-in-LH entries in
ZIP64 archives, not an attack. A replan using option (1) would
collapse to a one-line source change (`==` → `<=`) and the cd-lh-version-mismatch.zip
fixture pair (LH=45, CD=20, with LH ZIP64 extra) to exercise it.
