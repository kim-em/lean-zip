## Analysis: Merge Conflict Patterns

### Quantitative findings

Of the last 30 merged PRs:
- **4 (13%)** were conflict-fix PRs (titles containing "conflict fix")
- **9 (30%)** of closed PRs were abandoned without merge (many due to conflicts)
- Every conflict was in `Zip/Spec/Zstd.lean` (6280 lines) or `Zip/Spec/ZstdFrame.lean` (2596 lines)

### Root cause

Multiple parallel agents add additive theorems to the same section markers in
the spec files. The theorems are logically independent but target the same
insertion point (end of a `/-! ## ... -/` section), causing git to see
overlapping changes.

### Key finding: cherry-pick always fails on spec files

Across 8+ sessions, the sequence is always:
1. Agent tries `git cherry-pick` or `git rebase`
2. Produces 6-12 conflict hunks
3. Agent aborts
4. Falls back to manual insertion on a fresh branch

No agent has ever successfully resolved these conflicts via git tooling.
The manual-insertion approach is the only working strategy.

### Structural assessment

The "fix conflict after the fact" approach is sustainable but expensive:
~13% of merged PRs are pure overhead. As the matrix grows (more block-type
pairs → more parallel agents → more spec file modifications), this rate
may increase. File splitting would be the structural fix, but it would
itself cause a conflict cascade for all in-flight PRs.

### Diminishing returns observation

Each compressed-first API-level theorem requires the corresponding frame-level
theorem, which in turn requires block-level — creating a 3-step lifting pipeline.
This means each new block-type combination generates 9+ issues (3 theorems × 3
levels + conflict fixes). The overhead ratio is manageable for the remaining
cells but would not scale to larger matrices.
