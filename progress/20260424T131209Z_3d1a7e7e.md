# Feature session 20260424T131209Z — CD-entry `versionMadeBy` lower-byte upper bound

- Branch: `agent/3d1a7e7e`
- Closes issue #1812.

## What was accomplished

Added a per-entry `versionMadeBy` lower-byte upper-bound guard at CD
parse time. Writer-side at `Zip/Archive.lean:116` emits
`3 * 256 + (if z64 then 45 else 20)`, so lean-zip-produced archives
carry lower byte `20` or `45`. APPNOTE §4.4.2.2 caps the lower byte at
`63` (spec version 6.3, APPNOTE 6.3.10); any higher value is beyond the
defined spec and a parser-differential smuggling vector against strict
readers. The new guard sits in `parseCentralDir` immediately after the
`sigCentral` check and before the `versionNeeded` read so CD+4
attribution precedes CD+6 attribution in error-message locality.

Only the lower byte is checked. Real archives vary widely in host-OS
code (upper byte: Info-ZIP `3`, Windows `11` NTFS, Mac `19`/`20`,
legacy DOS `0`), so a strict upper-byte check would risk rejecting
legitimate archives. The lower byte is a pure spec-version field with
a well-defined APPNOTE maximum.

## Decisions made

- Default `version_made_by=20` (not `0x0314` as the issue suggested).
  The issue's `0x0314` default would have changed every existing
  fixture's SHA-256, contradicting its own "preserving byte-identity"
  requirement. I chose byte-identity preservation since the existing
  SHA-256s are cited in progress entries and the new check treats both
  `0x0014` and `0x0314` as valid (both have lower byte `20 ≤ 63`).
- Cited `Zip/Archive.lean:498` in the inventory row (the throw line),
  matching the pattern for sibling bullets (`diskNumberStart` at
  :521, `entry local offset` at :620).
