extraction-bomb branches called out by `plans/track-e-current-audit-checklist.md`

1. `ZipTest/Archive.lean` — new block that builds an in-memory ZIP
   (single `bomb.txt` entry, ~6200 bytes from `mkTestData`), then
   confirms both `Archive.extract` and `Archive.extractFile` reject it
   with `"exceeds limit"` when `maxEntrySize := 10`. Exercises
   `Zip/Archive.lean:408-410`.

2. `ZipTest/Tar.lean` — new block that uses `Tar.create` to build a tar
   on disk containing one ≥100-byte entry, then confirms `Tar.extract`
   rejects it with `"exceeds limit"` under `maxEntrySize := 10`.
   Exercises `Zip/Tar.lean:565-566`.

3. `ZipTest/Tar.lean` — extension of the same block: gzips the bomb tar
   with `Gzip.compress`, then confirms `Tar.extractTarGzNative`
   (the function the issue calls "extractGz" — the only `Tar.extract*`
   that calls into `Zip.Native.GzipDecode.decompress`) rejects it with
   `"exceeds maximum size"` under `(maxEntrySize := 0)
   (maxOutputSize := 10)`. The substring is what `Inflate.inflateRaw`
   actually emits when its budget is exhausted, not the
   `"too large"` wording used by `GzipDecode.decompress` itself —
   `inflateRaw` runs first and fails first.

## Decisions

- Followed the in-file `.toBaseIO` + `match .ok / .error` style already
  used by the central-directory limit test in `ZipTest/Archive.lean`
  rather than `assertThrows`, per the issue's explicit instruction.
- No new fixtures under `testdata/`. All bomb inputs are generated at
  test time from `mkTestData` and `Tar.create` / `Gzip.compress`.
- No changes to default `maxEntrySize` / `maxOutputSize`. Those policy
  decisions remain for the sibling P2 inventory issue.
- Issue mentions `Tar.extractGz` and a hypothetical `(ByteArray, ...)`
  signature; the actual function is `Tar.extractTarGzNative` and it
  takes a `System.FilePath`. Wrote the gzipped tar to a temp file first
  to match the real API rather than introducing a new wrapper.
