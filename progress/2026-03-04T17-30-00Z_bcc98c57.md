# Progress: Treeless Literals (litType 3) Support

- **Date**: 2026-03-04 17:30 UTC
- **Session**: bcc98c57 (feature)
- **Issue**: #554

## Accomplished

- Implemented treeless literals (litType 3, RFC 8878 §3.1.1.3.1) in `parseLiteralsSection`
- Extracted `parseCompressedLiteralsHeader` and `decodeHuffmanLiterals` private helpers
  to share header parsing and stream decoding between litType 2 and 3
- Added `prevHuffTree : Option ZstdHuffmanTable` parameter to `parseLiteralsSection`;
  return type now includes the Huffman table used (`some` for type 2/3, `none` for raw/RLE)
- Threaded `prevHuffTree` through the `decompressBlocks` block loop so each compressed
  block's Huffman tree becomes the next block's previous tree
- Updated all callers in tests (ZstdNative.lean, FseNative.lean) for new return type
- Added roundtrip test: 256KB+ repetitive data compressed at level 3 → multi-block
  frame with treeless literals → native decompress → verify match
- Added error test: litType 3 in first block (no previous tree) → clear error message

## Decisions

- Used `Option ZstdHuffmanTable` (defaulting to `none`) rather than a separate function,
  keeping the API change minimal — callers that don't need treeless support use the default
- For litType 3, all `compressedSize` bytes are stream data (no tree descriptor to skip)
- Raw/RLE blocks don't produce or consume a Huffman tree (return `none`), preserving
  the previous tree for subsequent blocks
