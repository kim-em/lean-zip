## Findings per audit check

### A.1 Predicate correctness — OK

The guard at
[Zip/Archive.lean:600-603](/home/kim/lean-zip/Zip/Archive.lean:600):

```lean
let checkPath := if name.endsWith "/" then (name.dropEnd 1).toString else name
unless Binary.isPathSafe checkPath do
  throw (IO.userError
    s!"zip: CD entry has unsafe path {name.quote} at CD offset {pos}")
```

Operates on the decoded `name : String` (post-UTF-8/Latin-1 decode
at `:560-570`), not the raw `nameBytes : ByteArray`. This is the
correct layer: `Binary.isPathSafe` semantics live at the path /
component level (`..`, `.`, empty, drive-letter, absolute prefix),
all of which are defined after decode. Running on raw bytes would
(a) require duplicating the decode to split on `/` and (b) still
miss the component-level predicates.

The interpolated `{name.quote}` escapes control bytes via the
standard `String.quote` — any smuggled `\r`, `\n`, escape, or
control characters in a crafted name stay escaped on their way to
logs. Position is interpolated as decimal `Nat` (no attacker
control). The `at CD offset {pos}` trailer is the same
attribution-hook format used by the sibling CD-parse guards
(`:519` diskNumberStart, `:539` internalAttrs, `:559` NUL-byte,
`:622-623` versionNeeded).

Note: the `Binary.isPathSafe` implementation at
[.lake/packages/zipCommon/ZipCommon/Binary.lean:137](/home/kim/lean-zip/.lake/packages/zipCommon/ZipCommon/Binary.lean:137)
rejects: `startsWith "/"`, `any (· == '\\')`, any component
`== ".."` / `== "."` / `isEmpty`, and any length-2 component
ending in `":"`. This matches the guard's docstring comment and
the extract-time callers at `:1103` / `:1107`.

### A.3 Trailing-slash carve-out parity — OK

Extract-time carve-out at
[Zip/Archive.lean:1101-1108](/home/kim/lean-zip/Zip/Archive.lean:1101):

```lean
let checkPath := if entry.path.endsWith "/" then entry.path.dropEnd 1 |>.toString else entry.path
if entry.path.endsWith "/" then
  unless Binary.isPathSafe checkPath do
    throw (IO.userError s!"zip: unsafe path: {entry.path}")

  throw (IO.userError s!"zip: unsafe path: {entry.path}")
```

CD-parse carve-out at `:600`:

```lean
let checkPath := if name.endsWith "/" then (name.dropEnd 1).toString else name
```

Both strip exactly one trailing `/` and run `isPathSafe` on the
result. The extract-time version uses `entry.path.dropEnd 1 |>.toString`
(pipeline form); the CD-parse version uses `(name.dropEnd 1).toString`
(parens form). Both compile to the same expression shape. The
pair stays in lockstep under future edits to either side.

Behavioural equivalence on reference shapes:

| name / path    | endsWith "/" | checkPath  | isPathSafe(checkPath) |
| -------------- | ------------ | ---------- | --------------------- |
| `foo`          | no           | `foo`      | true                  |
| `foo/`         | yes          | `foo`      | true                  |
| `foo/bar`      | no           | `foo/bar`  | true                  |
| `foo/bar/`     | yes          | `foo/bar`  | true                  |
| `../evil.txt`  | no           | same       | **false**             |
| `/etc/passwd`  | no           | same       | **false**             |
| `C:\\evil.txt` | no           | same       | **false** (backslash) |
| `foo\\bar`     | no           | same       | **false** (backslash) |
| `foo/./bar`    | no           | same       | **false** (`.` comp)  |
| `foo/../bar`   | no           | same       | **false** (`..` comp) |
| `foo//bar`     | no           | same       | **false** (empty)     |
| `` (empty)     | no           | `` (empty) | **false** (empty)     |
| `/`            | yes          | `` (empty) | **false** (empty)     |

All five `isPathSafe`-rejected shapes trip the new guard.
All three directory-shape (`foo/`, `foo/bar/`, `dir/`) cases pass
the carve-out. Edge case: bare root `/` strips to empty string,
which `isPathSafe` correctly rejects (empty component).

## Recommendation for next-wave Track E cadence

PR #1840 closes the *path-shape* dimension of the filename
validation matrix, complementing PR #1831's *byte-content* (NUL)
dimension. Remaining content/shape dimensions at the CD-entry
name layer, in rough priority order:

1. **Control-character sweep in CD-entry `name`** — the NUL
   guard and path-shape guard close two dimensions; the next
   question is whether other ASCII control characters (`\r`,
   `\n`, `\x01`-`\x1F`, `\x7F`) should be rejected as well.
   Info-ZIP and Go's `archive/zip` vary; a future planner should
   survey the interop corpus before landing the guard. (This
   item was also recommended by the #1831 paired review.)
2. **CD-entry `name` length bounds** — APPNOTE caps filename
   length at the UInt16 max but real filesystems cap below that
   (Linux ext4: 255 bytes, Windows NTFS: 255 UTF-16 code units).
   Upper-bound reject at CD parse would close a DoS vector
   where an extract-path `IO.FS.writeBinFile` with a 64 KiB
   name might hit filesystem-specific errors with confusing
   messaging.
3. **EOCD `archiveComment` and CD-entry `fileComment`
   content-class** — APPNOTE §4.3.16 / §4.4.18 leave these
   byte-spaces unconstrained; NUL-in-comment is a smaller
   attack surface than NUL-in-name (comments don't route
   through `open`) but still a parser-differential vector for
   callers that surface `entry.fileComment` to users.
