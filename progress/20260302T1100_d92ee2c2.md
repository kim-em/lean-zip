## What was accomplished

Implemented Huffman-compressed literal decoding (litType 2) in
`parseLiteralsSection` within `Zip/Native/ZstdFrame.lean`.

### New functions added to ZstdFrame.lean
- `HuffmanEntry`, `ZstdHuffmanTable` — flat table structures
- `parseHuffmanWeightsDirect` — direct-representation weight parsing
- `weightsToMaxBits` — compute maxBits from weight array
- `buildZstdHuffmanTable` — construct flat lookup table from weights
- `parseHuffmanTreeDescriptor` — parse tree descriptor (direct mode)
- `decodeHuffmanSymbol` — decode one symbol via flat table lookup
- `decodeHuffmanStream` — decode N symbols from backward bitstream
- `decodeFourHuffmanStreams` — 4-stream decoding with 6-byte jump table

### Updated `parseLiteralsSection`
- Full compressed literals header parsing (sizeFormat 0-3)
- Routes to single-stream or 4-stream decoding based on sizeFormat
- Tree descriptor size is subtracted from compressedSize to get stream data size

### Tests added (ZstdNative.lean)
- Tests 46-54: weightsToMaxBits, buildZstdHuffmanTable, parseHuffmanTreeDescriptor, parseHuffmanWeightsDirect edge cases
- Tests 55-58: decodeHuffmanSymbol, decodeHuffmanStream, integration test with FFI-compressed data, decodeFourHuffmanStreams error case

## Key decisions
- Used "read maxBits, lookup, re-read numBits" pattern for Huffman decoding since BackwardBitReader doesn't support unreading bits
- Moved all Huffman infrastructure before `parseLiteralsSection` to resolve definition ordering
- `weightsToMaxBits` computes `Nat.log2 sum + 1` (smallest n where 2^n > sum of explicit weights)
- FSE-compressed Huffman weights return an error (future work)
