# Progress: Fast-L1 design — attribute the emit cost, then choose (#2726)

**Date**: 2026-07-02 UTC
**Session type**: Feature
**Issue**: #2726

## What was accomplished

Delivered the fast-L1 corner by *measuring the attribution first*, then
choosing the design the data supports — not the presumed fixed-only answer.

### Step 1 (gate): attribution — L1 is emit-bound, sizing is cheap

A tokens-held-constant microbench on Silesia (`ZipL1Attrib`, run on the
pre-change L1 tokens `lzMatchP data 1`). MB/s vs uncompressed bytes, median-of-5:
`(1)` `tokenFreqsP`, `(2)` freqs + dyn sizing/tree build (`dynamicCodeLengths` +
`dynHeaderCodes` + `dynBlockBytesWith`), `(3)` `fixedBlockBytes`, `(4)`
fixed-only emit (`deflateFixedBlockP`), `(5)` full base (`deflateRawBaseP`).

| file | (1) freqs | (2) freqs+dynSize | (3) fixSize | (4) fixEmit | (5) base | branch |
|------|-----------|-------------------|-------------|-------------|----------|--------|
| dickens | 1018 | 987 | 1022 | 137 | 121 | dynamic |
| mozilla | 976 | 961 | 976 | 164 | 143 | dynamic |
| mr | 1181 | 1085 | 1183 | 154 | 135 | dynamic |
| nci | 3639 | 3545 | 3645 | 535 | 455 | dynamic |
| ooffice | 664 | 619 | 667 | 121 | 99 | dynamic |
| osdb | 1483 | 1350 | 1489 | 185 | 159 | dynamic |
| reymont | 1432 | 1253 | 1442 | 174 | 152 | dynamic |
| samba | 1615 | 1529 | 1622 | 234 | 199 | dynamic |
| sao | 740 | 689 | 740 | 97 | 87 | dynamic |
| webster | 1147 | 1138 | 1148 | 177 | 151 | dynamic |
| x-ray | 565 | 538 | 565 | 96 | 79 | dynamic |
| xml | 2917 | 2493 | 2922 | 406 | 343 | dynamic |

**Reading (dickens, 9.7 MiB):** freqs 9.5 ms, freqs+dyn sizing+tree 9.85 ms (so
the tree build alone is ~0.3 ms — negligible), fixed emit 71 ms, base 81 ms. So:

- **Sizing (freqs + tree build + fixed sizing) is ~12% of base.** The dynamic
  tree build is essentially free.
- **The token walk + `BitWriter` emit is ~88% of base.** Fixed-only emit (137
  MB/s) is only ~13% faster than the full dynamic-arbitrated base (121 MB/s),
  **not** materially faster.
- Every Silesia file's base already emits the **dynamic** block; fixed-only
  gives up 6.5–25% ratio (dickens +24.6%, mr +24.9%).

**Decision rule (from the issue) fires against fixed-only (C):** sizing is not
a large fraction, and `deflateFixedBlockP` is not materially faster than the
base. The cost is the token walk / `BitWriter` (no trivial fix — already
UInt64-batch-packing). Fixed-only rejected. Fold-freqs (B) would only recover
the ~12% freqs walk and needs a new equality lemma — not worth it. The plan is
candidate **A**: a genuine `deflate_fast` matcher keeping dynamic arbitration.

### Candidate A: `deflate_fast` matcher policy (the shipped change)

`ZipL1Sweep` measures the matcher throughput (`lz77ChainIterP`) and the *true*
end-to-end throughput (match + emit fused via `deflateRawBaseP data
(lz77ChainIterP ...)`, so the token-count coupling a shallower matcher induces
is captured, not estimated). Measured on chungus, pinned to one quiet core,
median-of-5:

| config | e2e MB/s | Δspeed | geomean ratio | max-file ratio | worst per-file ratio regression |
|--------|----------|--------|---------------|----------------|---------------------------------|
| cd8/ic16 (today) | 56.4 | — | 0.3467 | 0.7657 | — |
| cd4/ic16 (chain 8→4 only) | 59.0 | +4.6% | 0.3525 | 0.7694 | — |
| cd8/ic4 (cap 16→4 only) | 61.1 | +8.3% | 0.3512 | 0.7669 | — |
| **cd4/ic4 (shipped)** | 63.9 | **+13.3%** | 0.3563 | 0.7704 | xml +6.3% |
| cd4/ic2 | 66.3 | +17.6% | 0.3614 | 0.7739 | reymont +8.6% |
| cd2/ic2 | 68.3 | +21.1% | 0.3692 | 0.7776 | xml +15.5% |

Chosen **cd=4, ic=4**: `cd=4` is exactly zlib `-1`'s `max_chain`; the light
insert cap is faithful to `deflate_fast`'s deferred insertion. +13.3% e2e
geomean, **every file same-or-faster** (nci +27%, xml +22%, osdb +18%; x-ray +1%
worst), ratio 0.356 (*below* the 0.38–0.40 fast-band ceiling the issue targets —
ratio headroom remains), worst single file +6.3% (xml, absolute 0.148→0.158 — no
blow-up). Dynamic arbitration and the stored fallback are kept. A measured more
aggressive point (`ic=2`: +17.6% e2e, ratio 0.361, worst file +8.6%) is recorded
in the `insertCap` docstring as the next step nearer the ceiling.

The baseline cd8/ic16 = 56.4 MB/s @ 0.347 matches the issue's stated
"57 MB/s @ ratio 0.347", confirming the measurement env is correct.

The `insertCap` docstring's old "cap below ~16 is counterproductive" claim was
re-measured and corrected: at chain depth 4, `cap = 4` is +8% e2e vs `cap = 16`
(cd4/ic16 59.0 → cd4/ic4 63.9) — the interior-insertion saving now outweighs the
higher token count the worse ratio produces.

### Measurement methodology note (stale `.lake` hazard)

An earlier round of cross-binary before/after measurements showed a spurious
~12% uniform slowdown on *unchanged* levels 2–8. Root cause: the main worktree's
`.lake` carried stale cached `moreLinkArgs` (from an early bare-shell build
before switching to `nix-shell`), so it linked a differently-sized binary
(5.88 MB vs the correct 6.34 MB) — the exact hazard the project CLAUDE.md warns
about ("plain `lake clean` is not sufficient"). After `rm -rf .lake` and a clean
rebuild, the main-worktree build of master's source became byte-identical to a
fresh-worktree build, and the phantom slowdown vanished. All numbers above are
from the clean env. The in-binary `l1-sweep` A/B (cd8/ic16 vs cd4/ic4 as runtime
args to the *same* compiled function) is immune to this and is the primary
evidence.

## Changes made

1. `chainDepth (level ≤ 1)`: 8 → 4; docstring rewritten with the attribution.
2. `insertCap (level ≤ 1)`: 16 → 4; docstring corrected with the re-measurement.
3. New attribution tooling: `ZipL1Attrib` (`l1-attrib` exe) and `ZipL1Sweep`
   (`l1-sweep` exe), registered in `lakefile.lean`.

## Proof / risk

Zero proof risk: `chainDepth`/`insertCap` are pure speed/ratio knobs — the
encoder contracts (`lz77ChainIter_resolves`, `lzMatch_{encodable,empty,resolves}`)
are ∀-quantified over the knob values, so no theorem statement or proof changed.
Roundtrip + native↔FFI decode interop remain byte-identical (`lake exe test`
green, incl. the FuzzInflate differential census).
