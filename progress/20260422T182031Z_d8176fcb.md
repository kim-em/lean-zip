## What this wave accomplished

### 1. Adler-32 characterizing-property ladder — closed

Three feature PRs (#1673, #1679, #1684) with three paired reviews
(#1680, #1687, #1691) extend the Adler-32 closed-form chain from
`n ∈ {0, 1}` (post-#1661) to arbitrary `n < 65521` with any byte
value:

| Rung | Feature PR | Review PR | Spec theorem |
|---|---|---|---|
| `_empty` | #1661 (pre-wave) | **#1674** (this wave) | `checksum [] = 1` |
| `_singleton` | #1661 (pre-wave) | **#1674** (this wave) | `checksum [b] = ofNat ((1 + b.toNat) * 65537)` |
| `_pair` | **#1673** | **#1680** | `checksum [b₁, b₂] = ofNat ((1 + b₁ + b₂) + (2 + 2b₁ + b₂) * 65536)` |
| `_replicate_zero` | **#1679** | **#1687** | `checksum (List.replicate n 0) = ofNat (1 + n * 65536)` (needs `hn : n < 65521`) |
| `_replicate` | **#1684** | **#1691** | `checksum (List.replicate n b) = ofNat ((n + T_n · b) · 65536 + (1 + n · b))` where `T_n = n(n+1)/2` (needs joint bounds `hA`, `hB`) |
| `_append` | #1641 (prior wave) | #1650 (prior wave) | Two-block composition identity |

After this wave the concrete closed-form rungs are complete. Both
`_replicate_zero` and `_replicate` use the strengthened-invariant
pattern: state the equality for a free starting `(a, bsum)` pair
(and for `_replicate`, include the `b · T_m` triangular term
where `T_m = m(m+1)/2`), induct on the replicated count, then
instantiate at `(1, 0)` and bridge to `UInt32` via the existing
`pack_toNat_of_bounds` helper. The triangular identity
`(m+1)(m+2)/2 = m(m+1)/2 + (m+1)` is proved locally via
`Nat.add_mul_div_right` — no Mathlib dependency needed (#1684
progress entry §"Proof shape").

`pack_toNat_of_bounds` kept `private` throughout. Both #1673's and
#1679's progress entries explicitly note that the visibility
ratchet rule from meditate #1651 §1 applies — no cross-file
consumer yet, so no promotion. #1691's review recorded the
full `private` discipline still holds across all five rungs.

**What is NOT on this ladder.** The speculative `checksum_combine`
(two-block combine formula; analog of zlib's `adler32_combine`)
remains unlanded — filed as issue #1690, still unclaimed at this
summarize's write time.

### 5. CRC32 ladder — first rung opens (#1692)

#1692 adds `Spec.checksum_singleton (b) : checksum [b] =
0xFF000000 ^^^ mkTable[0xFF ^^^ b.toNat]'h` — the CRC32 analog
of Adler-32 `_singleton`, as a single `mkTable` lookup. Three
supporting helpers ship *public* at the Spec level:

- `mkTable_size : mkTable.size = 256`
- `xor_ff_byte_lt_mkTable_size (b) : 0xFF ^^^ b.toNat <
  mkTable.size`
- `crcByteTable_mkTable_eq_crcByte : crcByteTable mkTable crc
  byte = crcByte crc byte`

Five further helpers (`crcBits8_split`,
`UInt32_ofNat_UInt8_toNat`, `and_0xFF_toNat_lt`, `xor_byte_shr8`,
`mkTable_getElem`) are `private` and *duplicated* from
`Zip/Native/Crc32.lean` rather than moved — to preserve the
`Spec → Native` import order. `Native.crc32_singleton` is a
straight wrapper via `updateBytes_eq_updateList`.

Two proof-engineering notes from #1692's progress entry worth
surfacing:

1. The `h`-proof inside `mkTable[i]'h` must use `by exact
   xor_ff_byte_lt_mkTable_size b`; direct term application
   triggers a *maximum recursion depth* elaboration error.
2. The final XOR identity is closed by `generalize
   mkTable[0xFF ^^^ b.toNat]'(hidx ▸ hlt) = t` then
   `bv_decide`. Without the `generalize`, `bv_decide` sees the
   two `mkTable` accesses as distinct opaque variables
   because their proof terms differ syntactically — the
   pitfall queued for codification in `lean-uint-bitvec`
   (§4 above).

**CRC32-ladder divergence boundary** — write it down before the
next planner assumes symmetry. The CRC32 ladder will *not*
follow Adler-32 past `_singleton` / `_pair` with concrete closed
forms: CRC-of-zeros has no clean closed form without GF(2)[x]
polynomial algebra, and the triangular-number identity that
powers Adler-32 `_replicate` has no analog in the CRC polynomial
ring. The planner has queued **#1694** (`_pair`) and **#1695**
(paired review of #1692) but *not* `_replicate_zero` /
`_replicate`. The next planner should not generate those
without a polynomial-algebra strategy. `checksum_combine`
(analog of zlib's `crc32_combine`, matrix-representation trick)
may be possible but is a larger dependency — a dedicated
meditate's call.
