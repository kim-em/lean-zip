## a. Tactic elegance — Spec.checksum_replicate

**Refactor applied.** Three avoidable line-cost spots were
collapsed; one dead `have` was removed.

### 1. Dead `hbT` (removed, `-3` LOC)

The inductive step carried a `have hbT : b.toNat ≤ b.toNat * ((k+1)*(k+2)/2)`
whose name never appeared in any subsequent tactic and whose value
was not needed implicitly by any `omega` call in the proof body.
The final `refine Prod.mk.injEq .. |>.mpr ⟨?_, ?_⟩ <;> omega`
discharges two *equalities*, not inequalities requiring the bound;
both `ih`-bound `by omega` calls use `hbtri`'s equation directly.
Verified by removal + clean build.

### 2. `hbyte` collapse (`-3` LOC net)

The original inductive step opened with:

```lean
have hbyte : updateByte (a, bsum) b = (a + b.toNat, bsum + (a + b.toNat)) := by
  show ((a + b.toNat) % MOD_ADLER,
        (bsum + (a + b.toNat) % MOD_ADLER) % MOD_ADLER) = _
  rw [Nat.mod_eq_of_lt (by simp only [MOD_ADLER]; omega),
      Nat.mod_eq_of_lt (by simp only [MOD_ADLER]; omega)]
rw [List.replicate_succ, updateList_cons, hbyte,
    ih (a + b.toNat) (bsum + (a + b.toNat)) (by omega) (by omega)]
```

Mirrored to the `_replicate_zero` precedent in #1687, this becomes:

```lean
rw [List.replicate_succ, updateList_cons]
simp only [updateByte, MOD_ADLER,
  Nat.mod_eq_of_lt (show a + b.toNat < 65521 by omega),
  Nat.mod_eq_of_lt (show bsum + (a + b.toNat) < 65521 by omega)]
rw [ih (a + b.toNat) (bsum + (a + b.toNat)) (by omega) (by omega)]
```

The `simp only` absorbs the `updateByte`/`MOD_ADLER` unfoldings and
both modular reductions in one pass, fixing the inner mod before
the outer via the iterated-rewrite behaviour of `simp only`. No
explicit `show` reformulation is needed.

### 3. Invariant multiplication order (`-2` LOC net)

The strengthened invariant originally stored the triangular·byte
product as `b.toNat * (m * (m + 1) / 2)`, while the caller's
hypothesis `hB` and theorem RHS both use `(m * (m + 1) / 2) * b.toNat`.
This mismatch required a named bridge

```lean
have hc : b.toNat * (n*(n+1)/2) = (n*(n+1)/2) * b.toNat := Nat.mul_comm ..
```

with two `rw [hc]` calls in the `hupdate` block. Swapping the
invariant order to match the theorem statement eliminates both.
The `hbtri` proof cost drops by one rewrite step too — no trailing
`Nat.mul_comm b.toNat k` needed, only `rw [htri k, Nat.add_mul,
Nat.succ_mul k b.toNat]`.

### 4. Zero-case `simp [updateList]` → `simp only` (bare-simp compliance)

The zero branch had `simp [updateList]`. Replaced with
`simp only [updateList, List.replicate, List.foldl_nil,
Nat.zero_mul, Nat.zero_div, Nat.add_zero]` to comply with the
no-bare-simp convention from the campaign summary in
`proof-review-checklist` SKILL.md. The `Nat.zero_div` entry is
needed because the RHS contains `(0 * (0+1) / 2) * b.toNat`
which reduces to `0 / 2 * b.toNat` via `Nat.zero_mul` before
`Nat.zero_div` collapses the residual division.

### Closing cadence

The final block

```lean
rw [← UInt32.toNat_inj]
simp only [checksum, hupdate,
  pack_toNat_of_bounds (show 1 + n * b.toNat < 65536 by omega)
    (show n + (n * (n + 1) / 2) * b.toNat < 65536 by omega),
  UInt32.toNat_ofNat_of_lt' hbnd]
omega
```

is the canonical three-step (`toNat_inj` + single `simp only` +
`omega`) used by every earlier ladder member (`_singleton`,
`_pair`, `_replicate_zero`). No further collapse available.

### Unchanged `have`s (`hka`, `hkb`, `hkab`, `hbtri`)

All four are implicitly consumed by `omega` via sub-term atoms:

- `hka : (k+1)*a = k*a + a` — needed for the B-component equality
  because `omega` treats `(k+1)*a` and `k*a` as distinct atoms.
- `hkb : (k+1)*b.toNat = k*b.toNat + b.toNat` — needed for the
  A-component equality.
- `hkab : k*(a+b.toNat) = k*a + k*b.toNat` — needed because
  `omega` treats `k*(a+b.toNat)` as an atom distinct from
  `k*a + k*b.toNat`.
- `hbtri` — needed to rewrite `(T_{k+1}) * b.toNat` into
  `(T_k) * b.toNat + k*b.toNat + b.toNat` under omega's atomic view.

Removal of any of these breaks the `<;> omega` closer (verified
mentally against the atomic decomposition; same pattern as
`hkb`-was-load-bearing verification in #1687 §a).

**Net delta:** `-6` LOC (`-23` removed, `+17` added in
`Zip/Spec/Adler32.lean`).

## a. Tactic elegance — Native.adler32_replicate

**No refactor.** Already at minimum cadence.

```lean
theorem adler32_replicate (n : Nat) (b : UInt8)
    (hA : 1 + n * b.toNat < 65521)
    (hB : n + (n * (n + 1) / 2) * b.toNat < 65521) :
    adler32 1 (ByteArray.mk (Array.replicate n b)) =
      UInt32.ofNat ((n + (n * (n + 1) / 2) * b.toNat) * 65536
                     + (1 + n * b.toNat)) := by
  have hunpack : Spec.unpack 1 = Spec.init := by decide
  simp only [adler32, updateBytes_eq_updateList, hunpack]
  exact Spec.checksum_replicate n b hA hB
```

The bridge uses the three-line template
(`unpack-equivalence-have` + `simp only [adler32, updateBytes_eq_updateList,
hunpack]` + `exact Spec.checksum_replicate ...`), identical to
`adler32_replicate_zero`'s `Array.replicate`-shaped form. The
`(ByteArray.mk (Array.replicate n b)).data.toList → List.replicate n b`
reduction is handled definitionally by `exact`; no
`Array.toList_replicate` rewrite needed, and none would be flagged
by the linter if added since it would be genuinely used. Confirmed
unchanged from #1684.

## d. Strengthen-then-instantiate

**Confirmed load-bearing.** `_replicate` is the second ladder
theorem to use the generalize-to-simplify pattern from
`.claude/CLAUDE.md` §*Dealing with Dependent Type Rewriting Issues
in Lean*; the first was `_replicate_zero`. For `_replicate`,
*both* state components vary because nonzero bytes shift the
A-component too, so the strengthened invariant quantifies over
both:

```lean
hstate : ∀ (m a bsum : Nat),
    a + m * b.toNat < 65521 →
    bsum + m * a + (m * (m + 1) / 2) * b.toNat < 65521 →
    updateList (a, bsum) (List.replicate m b) =
      (a + m * b.toNat, bsum + m * a + (m * (m + 1) / 2) * b.toNat)
```

Confirmed after refactor:

- Quantifies over `(a, bsum)` (both components free) ✅
- Hypothesis bounds exactly match what the inductive step needs (no
  strictly stronger bounds than required) ✅
- Uses fresh induction variable `m` (not `n`), avoiding the
  hypothesis-capture issue documented in #1679 ✅
- Triangular·byte product `(m * (m + 1) / 2) * b.toNat` order
  matches the caller-supplied `hB` shape, eliminating the `hc`
  bridge (new in this refactor) ✅

### Forward-looking note

With `_replicate` landed and audited, the ladder's concrete-shape
tier is complete. Subsequent Adler-32 characterizing-property work
should shift to one of two directions:

- **A zlib-style `adler32_combine` theorem**: compute
  `checksum (xs ++ ys)` from `checksum xs`, `checksum ys`, and
  `ys.length` without reprocessing data. Requires modular-arithmetic
  characterization of the A and B components and is the natural
  next step — RFC 1950 / zlib's `adler32_combine` function provides
  the reference algorithm.

- **A structural / algebraic property at the `ZMod MOD_ADLER` level**:
  out of reach without Mathlib. Record as blocked if attempted.

Per the review issue's instructions, no follow-up issues are
created here — the next planner handles routing.
