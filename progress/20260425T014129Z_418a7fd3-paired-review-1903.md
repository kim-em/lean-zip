## A. Guard summary

Existing guard, unchanged by this PR, at
[`Zip/Archive.lean:1080-1081`](/home/kim/lean-zip/Zip/Archive.lean:1080):

```lean
unless Binary.readUInt32LE localHdr 0 == sigLocal do
  throw (IO.userError s!"zip: bad local header signature for {label}")
```

`sigLocal` is the private constant at
[`Zip/Archive.lean:14`](/home/kim/lean-zip/Zip/Archive.lean:14)
‚Äî `0x04034b50`, the APPNOTE ¬ß4.3.7 LH magic. The guard reads exactly
the 4 magic bytes (the prior `readBoundedSpanFromHandle ‚Ä¶ 30`
ensures at least 30 bytes are available; the magic occupies bytes
0..3 of the LH span). The error substring
`"bad local header signature"` is unique in `Zip/Archive.lean`
(confirmed by `grep -n 'bad local header signature' Zip/Archive.lean`
‚Üí two hits: the comment block at `:764` and the throw at `:1081`).
Issue body cites the throw at `:1081` ‚Äî confirmed; no anchor drift
between PR #1903's merge commit `169c22c` and current branch HEAD.

The guard sits inside the helper `readEntryData`
([`Zip/Archive.lean:1078-1081`](/home/kim/lean-zip/Zip/Archive.lean:1078)),
which is invoked exclusively from `Archive.extract`'s per-entry
loop. `Archive.list` reads only the central directory and never
calls `readEntryData`, so the late LH-signature guard is structurally
unreachable from the listing path ‚Äî a property the new fixture
exploits to demonstrate the precedence story (see ¬ßB below).

### Minimality

The 122-byte size is the minimum for the stock `hello.txt`-payload
shape: 30 B LH + 9 B name + 6 B payload + 46 B CD + 9 B CD name +
22 B EOCD = 122 B. A byte-shorter fixture would require either a
shorter name or a shorter payload ‚Äî both of which the builder

`cd-*.zip` siblings. The size choice is family-uniformity-faithful,
not minimally-minimal. Justified by the cd-lh-mismatch.py module
docstring (the family standardization).

## H. Reviewer observations

- **H.1 Issue-body payload-size typo.** The issue body says "body
  5 B" and "stored-method `compSize == uncompSize` == 5". Current
  builder uses `PAYLOAD = b"hello\n"` = 6 B; the fixture's CD/LH
  compSize/uncompSize fields read `06 00 00 00` (= 6). Total
  archive size 122 B is correct because `30 + 9 + 6 + 55 + 22 =
  122` ‚Äî the `5`-vs-`6` discrepancy in the issue body is a 1-byte
  typo that does not affect any guard analysis (`compSize > 0
  ‚àß uncompSize > 0` clears the column-3 invariant equally for
  5 or 6, and the equality check `compSize == uncompSize` clears
  for any common value). Documented here for fidelity. Not a PR
  #1903 defect ‚Äî the originating session's progress entry
  (`progress/20260425T001721Z_bd47ae5d.md`) and the fixture itself
  use the correct 6-byte value; only the planning issue body
  carries the typo.

- **H.2 Builder kwarg signature surface area.** `make_lh` after
  PR #1903 has 9 keyword arguments
  (`method`, `comp_size`, `uncomp_size`, `crc`, `version`,
   `lh_mod_time`, `lh_mod_date`, `lh_flags`, `name_bytes`,
   `lh_signature`). Each kwarg's default is documented as
  load-bearing in the function's prologue comment (`:44-63`). At
  10 kwargs the function approaches a complexity threshold where
  a `dataclass`-style fixture-spec argument might be cleaner ‚Äî
  but every kwarg's default is precisely-tested via determinism,
  so the current design is robust. Flagged as future-watch
  rather than blocker; no action recommended for this PR.

- **H.3 Writer-side compliance.** `grep -n 'sigLocal\b'
  Zip/Archive.lean` shows three hits: the constant definition at
  `:14`, the writer's `writeUInt32LEAt buf 0 sigLocal` at `:89`
  (inside `writeLocalHeader`), and the reader's check at
  `:1080`. The writer's emission is unconditional: every LH
  produced by `lean-zip`'s `create` writes `sigLocal` at LH offset
  0. lean-zip's own `create` thus **cannot produce a fixture the
  late LH-signature guard would reject** ‚Äî writer-side hardening
  invariant holds, matching the sibling writer-side non-smuggle
  invariants flagged in paired-review-#1813 ¬ßA.7 (`writeLocalHeader`
  strictly precedes `writeCentralHeader`) and paired-review-#1889
  ¬ßH.2 (`buildRecord` emits 56-byte v1 EOCD64 only).

- **H.4 `Archive.list` vs. `Archive.extract` asymmetry.** The
  level-3 guard fires only inside `readEntryData`, which is
  invoked by `Archive.extract` but not by `Archive.list`.
  Consequence: `Archive.list cd-bad-lh-signature.zip` succeeds
  and reports a single 6-byte `hello.txt` entry ‚Äî only `Archive.
  extract` throws. This is an intentional invariant of the
  reader API (the listing path's only assertions are about CD
  bytes, not about backed-by-data files), but it is worth
  documenting that the late LH-signature guard is **not** a
  defense-in-depth check against malicious archive listing ‚Äî
  any further hardening of the listing path against
  LH-signature smuggling would require a CD-parse-time check
  or a `list`-side LH read. The current design is consistent
  with the rest of `Zip/Archive.lean`'s "list = CD-only,
  extract = full" split.
