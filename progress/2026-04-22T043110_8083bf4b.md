## What landed

- `ZipTest/FuzzInflate.lean` — deterministic xorshift64 PRNG + fuzz
  driver. `runFuzz seed iterations` and `runFuzzUntil seed
  deadlineMs` cycle through sizes `{0, 1, 16, 512, 8192, 65536}` and
  feed the same input through every FFI whole-buffer entry point
  (`Zlib.decompress`, `Gzip.decompress`, `RawDeflate.decompress`),
  every native whole-buffer entry point
  (`Zip.Native.Inflate.inflate`, `GzipDecode.decompress`,
  `ZlibDecode.decompress`, `decompressAuto`), and the streaming
  `Gzip.InflateState` path with random chunk sizes from
  `{1, 7, 31, 127}`. Any `IO.userError` is the handled case; an
  uncaught panic or sanitizer trap fails the run.
- `ZipTest.lean` — registered `ZipTest.FuzzInflate.tests`. The smoke
  path runs 100 iterations with seed `0xdeadbeef` as part of
  `lake exe test`. Measured cost: ≈ 10 ms added to the test run.
- `ZipFuzzInflate.lean` + `lean_exe fuzz_inflate` in `lakefile.lean`
  — stand-alone lake executable. Reads budget from
  `LEAN_ZIP_FUZZ_SECONDS` (default 30) or the first CLI arg; seed
  from `LEAN_ZIP_FUZZ_SEED` (default `0xdeadbeefcafef00d`).
  `scripts/fuzz-inflate.sh 3` completes ≈ 28 000 iterations in 3
  seconds on the current tree.
- `scripts/fuzz-inflate.sh` — wrapper with nix-shell re-exec fallback
  and docstring describing the sanitizer recipe (reuse
  `ZLIB_CFLAGS`/`ZLIB_LDFLAGS`/`LD_PRELOAD` from
  `scripts/sanitize-ffi.sh`).
- `plans/track-e-current-audit-checklist.md` — Priority 3 item 3
  ticked with a breadcrumb to the landed files.
- `SECURITY_INVENTORY.md` — bullet added to the zlib-FFI subsystem's
  *Current local guardrails* list describing the new harness and the
  sanitizer-integration recipe.

## Design notes

- **Deterministic PRNG inlined**: xorshift64 with a fixed fallback
  seed (`0xdeadbeefdeadbeef`) when `seed = 0`, so test runs are
  reproducible. No `IO.rand` or clock reads inside the inner loop.
- **Input size cap 64 KiB, output cap 4 MiB**: the issue specified a
  ≤ 1 MiB input cap, which I interpreted as an upper bound and then
  picked a tighter set that exercises the interesting edge cases
  (empty, 1-byte, header-sized, block-boundary, window-sized) while
  keeping smoke-test cost near zero. Output cap at 4 MiB is large
  enough that an accidentally-valid decode won't clip early but small
  enough that an accidental bomb terminates promptly.
- **`tryFFI` swallows any caught exception**: the issue specified
  that *any* `IO.userError` is the handled case. Catching at the
  `IO.Error` level (not pattern-matching on `.userError` only) keeps
  the driver simple — panics and sanitizer traps bypass the Lean
  catch anyway, so the distinction does not help. ASan/UBSan traps
  are process-level and the fuzz run will exit non-zero as intended.
- **Sanitizer recipe by documentation, not by modifying
  `sanitize-ffi.sh`**: the issue's in-scope diff list did not include
  `scripts/sanitize-ffi.sh`, so rather than teach it to accept a
  subcommand I documented the env-var reuse pattern in both the new
  script's docstring and the inventory entry. Anyone wanting
  sanitizer-covered fuzzing can source the sanitizer env block and
  run `scripts/fuzz-inflate.sh <seconds>`.

## Deviations from issue text

- The issue mentioned `Zip.Basic.zlibDecompress` as a call target —
  the actual API is `Zlib.decompress` (module `Zip.Basic`,
  namespace `Zlib`). Called by correct module-qualified name.
- Smoke test lives inside `ZipTest/FuzzInflate.lean` rather than a
  separate `FuzzInflateTest.lean` — there is no existing pattern of
  splitting a module's tests into a second file, and keeping them
  together matches the rest of `ZipTest/`.

## Follow-ups (not in this PR)

- Coverage-guided fuzzing (libFuzzer / AFL++) is intentionally out of
  scope for this first-cut harness — a future issue can seed a corpus
  from `testdata/` if empirically useful.
- The whole-buffer paths currently share the same input. A future
  refinement could fork the input after a `detectFormat` hint so each
  decoder sees inputs it has a nontrivial chance of parsing. For now
  the blanket approach is simplest and catches the most surprising
  failure modes (e.g. gzip header bytes accidentally parsed as raw
  DEFLATE).
