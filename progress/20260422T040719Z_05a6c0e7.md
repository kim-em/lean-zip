## 1. Doc-only PR cadence — is the review process appropriately weighted?

### Which PRs this covers

Pure-doc PRs in this batch (no `Zip/`, `ZipTest/`, or `c/` edits):

| PR | What changed | Progress entry | Review friction |
|---|---|---|---|
| #1573 | Limit-policy docstrings on FFI whole-buffer + streaming decompression | `e36d9897` | none recorded |
| #1585 | `Handle.read`/`Stream.read` local guard inventory row | `ce0c1732` | none recorded |
| #1586 | Archive + Tar public-API limit-policy docstrings | `062e8a92` | none recorded |
| #1589 | Upstream-runtime tracking sub-block | `c179a367` | none recorded |
| #1590 | Minimized Reproducer Corpus section | `a81c8e96` | rebase conflict → #1591/#1592 |
| #1594 | Native decompression API limit-policy docstrings | `78864c76` | none recorded |

Plus the two progress-entry-only ride-along PRs (#1584, #1592).
Eight doc-only PRs out of fifteen.

### Reading the progress entries

The `062e8a92` (PR #1586) entry's **"Verification"** section is
representative of the shape doc-only PRs settle into:

> `lake build` clean. `lake exe test` — all tests pass.
> `grep -rc sorry Zip/` — all zero. `grep -c "Decompression Limit
> Inventory" Zip/Archive.lean Zip/Tar.lean` → 3 each (one per public
> API touched). Diff limited to `Zip/Archive.lean` and `Zip/Tar.lean`.
> No `SECURITY_INVENTORY.md`, no `ZipTest/`, no `c/`, no fixtures, no
> signature/behaviour changes.

Every doc-only PR runs the same four-step verification — `lake build`
/ `lake exe test` / `sorry` count / scoped diff grep. For these PRs
`lake build` is a guaranteed no-op (the doc comment doesn't touch
any compiled surface), `lake exe test` is a guaranteed no-op, and
`sorry` count is guaranteed unchanged. Of the four verification
steps, only the scoped-diff grep ever actually detects anything.

### Is review catching bugs?

Review caught nothing in this batch. The only two rework events —
#1591 (rebase of #1590) and #1581 (rebase of #1577) — were both
mechanical checklist-file conflicts, not review findings. Neither
has a progress-entry friction note mentioning a reviewer flag.

### Recommendation

Add a `doc-only` label and a *shortened* verification checklist for
it. Do not remove review altogether — the three docstring waves in
Section 6 below did stay consistent precisely because the template
got pattern-matched each time, and review is one of the things
enforcing that pattern. But:

- **Shortened checklist** (in the PR description / coordination
  template):

      - [ ] `git diff --stat` shows only docstring/inventory/progress files
      - [ ] No error-substring change (or catalogue skill updated in same PR)
      - [ ] Cross-reference to `SECURITY_INVENTORY.md` section exists
            and resolves

- **Skip**: `lake build`, `lake exe test`, `sorry` count. None of
  these can change on a docstring-only PR. Doing them burns ~3 min
  per PR and caught nothing in the eight doc-only PRs this batch.

### Risks of a lighter path

1. **Error-substring drift.** The worst outcome of skipping `lake
   exe test` is a docstring that quotes a substring the source
   doesn't actually throw. The `error-wording-catalogue` skill
   partially defends against this but only if it's kept current —
   and the catalogue *was* stale this batch (missed
   `"exceeds maximum header size"` from #1597 until this meditate
   session updated it, see Section 2). So the lightweight checklist
   **must** include the "catalogue updated in same PR" bullet.
2. **Inventory-row rot.** A row in `SECURITY_INVENTORY.md` that
   says "PR #1543 added this fixture" stops being true if the
   fixture is renamed or deleted. The doc-only path cannot verify
   a linked file still exists without at least a `ls` check. Add a
   `scripts/check-inventory-links.sh` (one glob walk) and have the
   lightweight checklist run it. This is future planner work, not
   an action item for this PR.

## 3. Rebase-fix ride-along progress entries — healthy or overhead?

### What happened

Twice this batch a rebase fix on an auto-merge PR branch lost its
ride-along progress entry to an auto-merge race, and the progress
entry had to be submitted as its own tiny follow-up PR:

1. **#1581 → #1584** (session `0eaa9a61`). Rebased `agent/b87e52f8`
   onto master to resolve the `plans/track-e-current-audit-checklist.md`
   conflict after PR #1576 merged ahead of PR #1577. The rebased CI
   finished and auto-merged #1577 (squash → `1f72e0e`) before the
   progress entry could be tacked onto the same branch. #1584 was
   the one-file ride-along.
2. **#1590 → #1592** (session `452d633a`). Same shape, different PR.
   The `20260422T025200Z_a81c8e96.md` file from the #1583/#1590
   session plus the progress entry for the rebase fix went onto a
   separate tiny branch because the rebased #1590 auto-merged first.

### The two competing readings

- **(a) Healthy practice**: rebase fixes are work that deserves
  their own progress entry. One-file diffs land fast; the
  `progress/` directory stays complete.
- **(b) Overhead**: the rebase-fix session could have folded its
  one-paragraph note into the following session's progress entry.
  Two trivial PRs become zero.

### Evidence

Session `0eaa9a61`'s friction note already answered this:

> The auto-merge race meant my progress entry couldn't be added
> along with the rebased commits. Future fix-PR sessions should
> either (a) commit the progress entry *before* the rebased PR
> has time to auto-merge, or (b) accept the separate-PR pattern
> used here.

That framing is right. The real question isn't "ride-along vs.
separate-PR"; it's "progress-entry-first-commit vs.
progress-entry-tacked-on". When the entry is staged and committed
*before* the rebased push, it rides along cleanly; when it's
tacked on after, it races with auto-merge.

### Recommendation — call this healthy, with a small convention fix

One-file-progress PRs are *cheap*: they never touch code, never
run meaningful CI, and take ~2 minutes of dispatcher time each.
The `progress/` directory is the project's durable institutional
memory; missing entries would be a worse cost.

**Convention fix:** the `agent-pr-recovery` skill now documents
the "**progress-entry-first**" pattern (commit the entry, *then*
push the rebase, so a single PR closes both the fix issue and the
documentation). This session added that subsection to the skill
(+52 lines total), along with the "accept the split" fallback.

**What not to do**: do not try to defeat auto-merge once the race
has been lost. Force-pushing to amend the auto-merged commit is
useless (branch is deleted upstream), and reopening the PR under
a fresh branch creates churn for the same issue number. Either
progress-entry-first, or accept the one-file follow-up PR.

With this convention in place, ride-alongs stop being a question of
"healthy vs. overhead" and become a question of "commit order".
That's an answerable question, and the skill now documents the
answer.

### What does "Track E done" mean concretely?

Two plausible readings:

- **"Checklist-all-ticked"**: Track E is done the instant the
  last `- [ ]` becomes `- [x]`. Pro: unambiguous, mechanical.
  Con: doesn't buy the project any confidence beyond the
  specific hardening each item encoded.
- **"Checklist + fuzzer-clean operation"**: Track E is done when
  the checklist is fully ticked *and* the fuzz harness (P3.3) has
  run for ≥ some threshold period against mainline without
  finding new crashes.

### Recommendation — explicit exit criterion

Use the fuzzer-clean variant, and write it down:

> **Track E exit criterion.** Track E closes when:
>
> 1. Every box in `plans/track-e-current-audit-checklist.md` is
>    ticked.
> 2. The fuzz harness from P3.3 has run for **≥ 7 contiguous
>    days of CI-driven operation** (meaning: at least seven
>    successful CI runs on master, each invoking the fuzz harness
>    for its configured per-run time budget) without surfacing a
>    new crash or a new guard-bypass.
> 3. `SECURITY_INVENTORY.md` *"Missing work"* bullets under each
>    subsystem are either cleared or migrated to labelled GitHub
>    issues with a target milestone outside Track E.

The 7-day figure matches the shape of the libarchive and sqlite
community thresholds for "this target got long-running fuzz
exercise", and is long enough to catch time-dependent bugs (e.g.
allocation-growth over many decompression cycles) that
single-run coverage would miss.

### What this implies for planning

- #1595 (fuzz harness, P3.3) is the **gate**. Its landing
  starts the 7-day countdown. Closing Track E before that
  countdown finishes means closing it without buying the
  confidence signal the audit was meant to produce.
- P5.1/P5.2/P5.3 (proof-friendly guard lemmas) can land *during*
  the fuzzer-observation window. They don't need to wait for
  fuzzer-clean, because they're compile-time properties
  independent of runtime execution.
- If the fuzz harness surfaces a new crash within the 7-day
  window, the clock resets when the fix lands. That's a cost
  worth paying — silencing a found crash and then declaring
  Track E done would defeat the criterion.

This criterion should be added to `PLAN.md` by the *next planner
session* (it's an exit-criterion edit, which is not within scope
for this meditate — `PLAN.md` is off-limits to workers).

## 6. P2.4 docstring wave — consistency audit

The issue folded the P2.4 docstring-consistency review work into
this meditate. This section is the review.

### Methodology

Cross-file greps against the three landed PRs (#1573 FFI, #1586
Archive+Tar, #1594 Native) for the shared phrases the template was
supposed to produce:

1. The *cross-reference* to `SECURITY_INVENTORY.md`'s Decompression
   Limit Inventory.
2. The *error-substring quote* in the limit-policy paragraph.
3. The *`0` semantics* phrase (either "unlimited" for FFI/Archive
   callers or "rejects any non-empty output" for native callers).
4. The *default-value* number.

### Results

#### Cross-reference phrase

`grep -rn "SECURITY_INVENTORY.md.*Decompression Limit Inventory"` —
**17 hits** across `Zip/Basic.lean`, `Zip/Gzip.lean`,
`Zip/RawDeflate.lean`, `Zip/Archive.lean`, `Zip/Tar.lean`,
`Zip/Native/Inflate.lean`, `Zip/Native/Gzip.lean`. Every hit uses
the *identical* phrasing:

> See *Decompression Limit Inventory* in
> [`SECURITY_INVENTORY.md`](/SECURITY_INVENTORY.md) for the
> project-wide audit.

Zero drift. The template from #1573 was copied verbatim into #1586
and #1594.

#### Error-substring phrasings

- FFI side (#1573): quotes `"exceeds limit"` and
  `"decompressed size exceeds limit"` — matches
  `c/zlib_ffi.c:195` emission and `error-wording-catalogue` row 1.
- Archive side (#1586, `Archive.extract`): quotes
  `"zip: entry '…' uncompressed size (…) exceeds limit (…)"`
  — matches `Zip/Archive.lean:408-410` and catalogue row 2.
- Tar side (#1586, `Tar.extract`): quotes `"tar: entry <name>
  exceeds limit"` — matches `Zip/Tar.lean:565-566` and catalogue
  row 6.
- Native side (#1594): quotes `"Inflate: output exceeds maximum
  size"` (and for Gzip native: `"Gzip: total output exceeds
  maximum size"`) — matches `Zip/Native/Inflate.lean:269,285,321`
  and `Zip/Native/Gzip.lean:88`, i.e. catalogue rows 10-11.

Every quoted substring matches the live throw site *and* the
catalogue. The substrings are *different* (FFI `"exceeds limit"`
vs. native `"exceeds maximum size"`), but this is the intentional
FFI-vs-native split documented in the catalogue itself (the
"user-visible split" section). The docstrings are honest about
their subsystem — each one names the subsystem explicitly in the
same paragraph as the error quote.

#### `0`-semantics phrase

- FFI (`Archive.extract` with `maxEntrySize := 0`) and tar
  (`Tar.extract`): both docstrings say the string
  *"bomb-unsafe for untrusted input"* when describing `0`.
- Native (`Zip.Native.Inflate.inflate`,
  `Zip.Native.GzipDecode.decompress`,
  `Zip.Native.decompressAuto`): all three docstrings say the
  phrase *"rejects any non-empty output"* for `0`.

The asymmetry is real (FFI: `0 → unlimited`; native: `0 →
effectively zero budget`), it's documented consistently with one
phrasing per side, and it's called out in #1586's
`Archive.extract` docstring as a warning:

> Asymmetric `0` semantics: the FFI path treats `maxEntrySize = 0`
> as unlimited, while the native path silently caps at 256 MiB.

Zero drift. Three PRs, three authors, same four phrases.

#### Default-value numbers

- FFI: `maxDecompressedSize := 16 * 1024 * 1024` (16 MiB).
- Archive: `maxEntrySize := 1 * 1024 * 1024 * 1024` (1 GiB).
- Tar: `maxEntrySize := 1 * 1024 * 1024 * 1024` (1 GiB).
- Native Inflate: `maxOutputSize := 1 * 1024 * 1024 * 1024` (1 GiB).
- Native GzipDecode: `maxOutputSize := 256 * 1024 * 1024` (256 MiB).
- Native decompressAuto: `maxOutputSize := 256 * 1024 * 1024`
  (256 MiB).
- **New**: Tar `maxHeaderSize := 8 * 1024 * 1024` (8 MiB, from
  #1597).

Each docstring quotes its own default, and the numbers match the
source literals. The Native-Inflate-at-1-GiB-but-GzipDecode-at-256-MiB
asymmetry is intentional (documented in #1594's progress entry
`78864c76`) and called out in both docstrings.

### What made the sequential landings stay consistent?

The #1573 docstring was explicitly a **template**. Both #1586 and
#1594's progress entries refer back to it:

- `062e8a92` (#1586): *"following the PR #1573 template (parameter
  + default + behaviour at `0` + quoted error substring +
  cross-reference to `SECURITY_INVENTORY.md`)"*
- `78864c76` (#1594): *"mirrored the PR #1573 structure so that
  readers traversing FFI → Archive/Tar → Native hit the same
  four-bullet shape in every docstring"*

Two operational factors preserved the template:

1. **Single-author-per-PR for the docstring waves.** No PR in the
   three-wave sequence had multiple commits from different
   sessions. Template-preservation is much easier when one agent
   owns the wave.
2. **The template fits in one progress-entry bullet list.** Four
   phrases: default + 0-behaviour + error substring + cross-ref.
   That's small enough to hold in the author's head while writing
   the docstring, and small enough to be re-quoted in the progress
   entry.

### Drift risk going forward

The one consistency gap — **not drift, but a missed row** — is
`"exceeds maximum header size"` (#1597), which the
`error-wording-catalogue` skill was not updated to include in the
same PR. That's discovered and fixed by this meditate session's
skill update. It's the only drift event in the entire P2.4 wave.

### Recommendation

- **No consolidation issue needed.** The docstrings are already
  consistent; forcing a consolidation PR now would be make-work.
- **Preserve the two operational factors.** Future docstring waves
  (if any — the P2.4 checklist is ticked) should still assign
  each wave to a single session and the template-bullet-list
  should be quoted in the progress entry.

## 7. Parallelism ceiling — right call or idle dispatcher?

### What the dispatcher actually did

Throughout this batch the pod dispatcher held at **0–2 claimed
issues at any given time**. Peak concurrency was during the
P2.4 docstring wave (#1573 → #1586 → #1594), when at most two
sessions were running simultaneously.

### Why low concurrency was partly unavoidable

Several doc-only PRs genuinely serialise because they edit the
same file:

- #1585 (Handle/Stream inventory) and #1589 (Upstream tracking)
  both append to `SECURITY_INVENTORY.md`'s "Trusted Computing
  Base" section. Running them in parallel would conflict.
- #1586 (Archive+Tar docstrings) and #1594 (Native docstrings)
  both follow the #1573 template; running them in parallel means
  the template couldn't be quoted verbatim (PR #1573 has to land
  first).
- #1591 (rebase of #1590) and the Minimized Reproducer Corpus
  sub-block (#1590 itself) obviously can't run in parallel.

That accounts for maybe 60% of the wave.

### Where the dispatcher was idle

Harder to measure from `progress/` entries (they don't record
dispatcher-idle periods), but inferring from merge timestamps:

- Between #1586 merging (`20260422T022538Z`) and #1589 claiming
  (`20260422T024100Z`), ~16 minutes of dispatcher idleness. Both
  PRs touch `SECURITY_INVENTORY.md` but in independent sections
  (Archive docstrings vs. Upstream tracking) — they could have
  run concurrently with a little extra rebase work.
- Between #1590 merging and #1597 claiming, ~30 minutes. #1597
  is a code PR that does not conflict with any doc-only path.

### Was the thin queue the right call?

Partly. The planner kept the queue thin because the docstring
waves *did* serialise. But the queue being thin during
non-serialising windows (e.g. a proof-lemma issue could have been
running in parallel with doc work) is dispatcher idleness that
cost throughput without improving anything.

### Concrete queue-depth heuristic

Proposed (for the planner to encode in the next cycle):

- **Doc-only waves (issues touching only `SECURITY_INVENTORY.md`,
  `plans/`, public API docstrings, or `progress/`):** keep **4
  unclaimed** in the queue. Document-only PRs serialise on files,
  not on proof-state or on FFI binaries; having a short queue of
  them lets the dispatcher pick the next non-conflicting one
  without waiting for a planner round-trip.
- **Proof-heavy waves (issues touching `Zip/Spec/**` or requiring
  `try?`/`grind`):** keep **2 unclaimed**. Proof-heavy work
  often depends on a preceding lemma landing, and the cost of a
  mis-ordered claim is higher (a worker can burn a session on a
  proof that became trivially-solvable after the preceding PR
  landed).
- **FFI / native runtime waves (any issue touching `c/` or
  `Zip/Native/**`):** keep **3 unclaimed**. These tend to be
  middle-density: more parallelisable than proofs, less than
  docs.

### Tie to the Track E exit criterion

Once P3.3 lands and the fuzzer-observation window begins, all of
P5.1/P5.2/P5.3 is proof-heavy. The heuristic above predicts the
dispatcher should drop to ~2 concurrent sessions during that
stretch. That matches what #1596's sequencing needs (P5.2 and P5.3
depend on P5.1).

### Risks of the heuristic

- **Starvation of non-Track-E work.** If the dispatcher always
  drains the Track E queue to 4 first, non-Track-E work (e.g.
  future RFC 8878 Zstd extensions) stays starved. The heuristic
  should be per-track, not global — "keep 4 *Track E doc-only*
  unclaimed" is different from "keep 4 total unclaimed".
- **Rebase cost on long-running claimed issues.** If 4 doc-only
  issues are claimed simultaneously against the same
  `SECURITY_INVENTORY.md`, the second-to-land through
  fourth-to-land each pay rebase cost. Mitigation: the planner
  should label issues *"edits-inventory"* if they touch the
  inventory, and the queue-depth cap applies per-label, not
  globally. Only 2 *"edits-inventory"* issues in flight at once.
