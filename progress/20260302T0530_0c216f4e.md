## What changed

### 1. `lean-simp-tactics` skill — 5 bare-simp-resistant categories

Added a "Bare `simp` Resistant Patterns" section documenting the 5
categories discovered across 10+ review sessions:

1. **Nested `Option.bind` chains** (>3 levels): `simp only` needs the
   full bind database; accept bare simp for chains deeper than 3 levels
2. **`dite`/`if` with mismatched conditions**: goal has
   `(bits ++ suffix).length` but hypothesis proves about `bits.length`;
   `simp [hf]` bridges via `List.length_append` + arithmetic
3. **`Prod.mk.injEq` extraction**: documented as NOT bare-simp-resistant;
   `simp only [Except.ok.injEq, Prod.mk.injEq]` always works
4. **`readBitsLSB` as computation engine**: concrete bit evaluation that
   needs the full simp database; always accept bare simp
5. **BitVec/UInt16 normalization pipelines**: multi-type cast chains;
   accept bare simp when the `simp only` lemma set is unknown

Added a "When Bare simp Is Acceptable" table with comment templates
and a "categories that should always be converted" list.

### 2. `lean-monad-proofs` skill — linter-compliant patterns

- Updated "Closing `Except.error = Except.ok` Contradictions": replaced
  `simp at h` / `exact absurd h (by simp)` advice with
  `exact nomatch h` (canonical, no recursion depth issues, no lemma
  database needed)
- Updated `letFun` section: now recommends `dsimp only` instead of
  `simp only [letFun]`
- Added `Except.ok.injEq` + `Prod.mk.injEq` extraction chain pattern
  (35+ instances in the codebase, always `simp only`)
- Added `Option.bind` chain handling template: `cases + dsimp` for
  shallow chains (≤3 levels), bare simp for deep chains (>3 levels),
  guard + bind interleaving pattern

### 3. New `proof-review-checklist` skill

Created a 6-phase mechanical checklist for proof quality reviews:
- Phase 1: Record starting metrics (bare simp count)
- Phase 2: Mechanical cleanup (merge rw/simp, remove dead haves,
  extract lemmas, simplify obtain patterns)
- Phase 3: Bare simp decision tree (simp? → dsimp only → targeted
  replacement → accept with comment)
- Phase 4: Linter compliance (unused simp args, reduceIte, injEq)
- Phase 5: Verification (build, compare metrics, check no sorry)
- Phase 6: Commit format with metrics in body

Includes expected reduction targets by file size (50-95% depending
on file characteristics).
