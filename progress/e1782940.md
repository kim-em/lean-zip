## Decisions

- **Where to throw.** `findEndOfCentralDir` is still a pure
  `Option`-returning function. Issue body permitted either lifting it
  to `IO` or threading the raw values through; I chose the latter to
  keep the diff scoped. The check then lives in `listFromHandle`
  alongside the existing `cdOffset + cdSize <= fileSize` and
  `maxCentralDirSize` guards (parallel placement to where the
  standard-EOCD disk-number check lives, which is in `parseCentralDir`).
- **Sentinel defaults `(0, 1)`.** Returning the locator pair as raw
  values (not `Option`) requires defaults for the absent-locator case.
  `(0, 1)` matches the APPNOTE single-disk values, so the
  `unless ... == (0, 1)` check is trivially satisfied for non-ZIP64
  archives — no special-casing needed in `listFromHandle`.
- **Fixture genesis (option (a)).** `scripts/build-zip64-malformed-fixtures.py`
  patches `go-zip64.zip` rather than synthesizing a ZIP64 archive from
  scratch. Single-byte mutation, fully byte-reproducible from the
  script, and minimal diff vs. the canonical ZIP64 interop fixture.
  Issue B's PR (`scripts/build-zip64-malformed-fixtures.py`, per
  issue #1746's hint) was not yet landed when this PR opened, so this
  PR creates the file. If issue B lands a second fixture in the same
  family later, it can extend this generator.
- **Which field to flip.** `totalDisks = 2` rather than
  `diskWithZip64EOCD = 1`, because the canonical single-disk value of
  `totalDisks` is unambiguously `1` per APPNOTE §4.3.15 (whereas
  `diskWithZip64EOCD = 0` only allows mutation upward). Single-byte
  flip on the existing fixture either way; this side is more
  semantically meaningful as a "claims to be multi-disk" smuggling
  vector.
