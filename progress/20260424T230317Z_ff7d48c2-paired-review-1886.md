## A. Guard correctness

### A.1 Placement

The guard at
[Zip/Archive.lean:851-854](/home/kim/lean-zip/Zip/Archive.lean:851):

```lean
if uncompSize > 0 then
  unless compSize > 0 do
    throw (IO.userError
      s!"zip: CD entry has zero compressedSize with nonzero uncompressedSize \
         (compressedSize={compSize}, uncompressedSize={uncompSize}) for {name} at CD offset {pos}")
```

sits **immediately after** the column-2 empty-entry CRC invariant on
lines 817-820 (*"CD entry CRC must be zero when uncompressedSize is
zero"*) and **immediately before** the `entries.push` at line 855. The
three math-invariant checks now form a contiguous block in the order
col-1 (line 783-786) → col-2 (line 817-820) → col-3 (line 851-854),
post-ZIP64-resolution.

`uncompSize : UInt64` and `compSize : UInt64` are the **resolved**
sizes from the ZIP64 extra-data merge earlier in the loop body — not
the raw `stdCompSize` / `stdUncompSize : UInt32` values that could be
the `0xFFFFFFFF` sentinel. So:

- An archive with both fields `0xFFFFFFFF` and a ZIP64 extra resolving
  to `compSize = 0, uncompSize = 42` correctly **trips** the guard.
- An archive with `stdCompSize = 0xFFFFFFFF` and a ZIP64 extra
  resolving to any nonzero `compSize` **does not** trip the guard —
  as intended; `0xFFFFFFFF` is a sentinel, not an encoded zero.
- An archive with `stdUncompSize = 0` and any `compSize` skips the
  probe entirely — delegated to column 2 (empty-entry CRC).
- An archive with `stdCompSize = 0` and `stdUncompSize = 42` trips
  on the resolved values — the common case exercised by the fixture.

### A.2 Predicate correctness

Shape `if uncompSize > 0 then unless compSize > 0 do throw`: the
outer probe runs first so the `compSize` field is inspected only when
the non-empty-file premise holds; an entry with `uncompSize == 0`
skips the check entirely regardless of `compSize` value. This
delegates the `uncompSize == 0` case cleanly to column 2 and keeps
the two checks **ordering-disjoint**: col-2 fires only on
`uncompSize == 0`, col-3 fires only on `uncompSize > 0`. No entry
can trip both guards, and no legitimate empty entry (with a
compressed-form such as the `03 00` deflate empty-block marker
producing `compSize == 2, uncompSize == 0`) is spuriously rejected.

Note the deliberate asymmetry with col-1: col-1 uses
`method == 0 → compSize == uncompSize` (strict equality), which on
a stored empty entry collapses to `0 == 0` (vacuous), whereas
col-3's `uncompSize > 0 → compSize > 0` says nothing about
**equality** — it only requires *some* non-zero compressed payload.
On `method == 0` with `uncompSize > 0 ∧ compSize == 0`, col-1 fires
first (the equality mismatch), and col-3 remains silent; the two
are therefore redundant on the `method == 0` case but col-1's
stricter check wins by placement order. On `method == 8` with
`uncompSize > 0 ∧ compSize == 0`, col-1's `method == 0` gate skips
the entry and col-3 fires alone — closing the `method == 8`
residual the issue body described.

## E. Three-column family coherence

### E.1 Dimension-closure summary

Grepping the `throw (IO.userError s!"zip: CD entry ...`)` family
(`grep -rn 'zip: CD entry' Zip/Archive.lean`) confirms the
math-invariant block's contiguous layout and per-column error
wording:

| Col | Predicate                                          | Throw substring                                              |
|-----|----------------------------------------------------|--------------------------------------------------------------|
| 1   | `method == 0 ∧ compSize ≠ uncompSize`              | `"stored-method size mismatch"`                              |
| 2   | `uncompSize == 0 ∧ crc ≠ 0`                        | `"CRC must be zero when uncompressedSize is zero"`           |
| 3   | `uncompSize > 0 ∧ compSize == 0`                   | `"zero compressedSize with nonzero uncompressedSize"`        |

The three columns cover the reachable error shapes in the
`(method, compSize, uncompSize, crc)` writer-side-invariant space:

- **Method-0 stored** (`compSize == uncompSize`) — col-1 catches
  every violation.
- **Empty entry** (`uncompSize == 0`) — col-2 catches every
  `crc ≠ 0` violation (crc = 0 is legal on empty entries).
- **Non-empty entry, any method** (`uncompSize > 0 ∧ compSize == 0`)
  — col-3 catches; on method=0 this is *also* caught by col-1,
  so col-3 is redundant-but-safe on that case; on method=8 this
  is the sole guard.

**Uncovered region in this family**: `method == 8 ∧ compSize > 0
∧ uncompSize == 0 ∧ crc ≠ 0` — this is column 2's responsibility
(`uncompSize == 0 → crc == 0`), which covers it regardless of
method. Covered. ✓

**Uncovered region outside this family**: a correctly CRC'd,
correctly sized `method == 8` entry whose actual decompressed
bytes do not match `uncompSize` (e.g. `compSize = 20,
uncompSize = 42, crc` matches the decompressed payload's CRC,
but the decompressed payload is only 30 bytes long). This is
**not a CD-parse math invariant** — the CD carries metadata,
not the decoded payload — and is caught downstream at
`readEntryData` + inflate time by a different family of guards.
Out of scope for this column.

### E.2 Ordering-disjointness proof sketch

Two adjacent guards are **ordering-disjoint** if no single entry
trips both. Col-2 requires `uncompSize == 0`; col-3 requires
`uncompSize > 0`. These predicates are negations of each other on
`UInt64`, so at most one of col-2 / col-3 fires for any entry. ✓

Col-1 requires `method == 0`; col-3 requires (any method) ∧
`compSize == 0 ∧ uncompSize > 0`. On `method == 0 ∧ uncompSize > 0
∧ compSize == 0`, col-1's `compSize == uncompSize` check becomes
`0 == (positive)`, which is false — so col-1 fires first (line 786
< line 854), and col-3 is unreachable for that entry. Correct
behavior: col-1's error *"stored-method size mismatch"* pins the
attribution on the method-specific invariant rather than the
method-agnostic one. The redundancy is ordering-resolved; col-3
acts as the `method == 8` residual catcher.

## F. Writer-side invariant

### F.1 Writer non-emission

lean-zip's `Archive.create` at
[Zip/Archive.lean:195](/home/kim/lean-zip/Zip/Archive.lean:195)
computes `Checksum.crc32 0 fileData` over the input bytes and
populates `compressedSize` / `uncompressedSize` from the actual
compressed / raw data lengths. For method=8, the deflate encoder
always emits at least the 2-byte `03 00` empty-stored-block header
even on empty input, so `compSize ≥ 2` whenever `uncompSize > 0`
(strictly, whenever the deflate path runs at all). For method=0,
the writer sets `compSize = uncompSize` by construction. In both
cases, `uncompSize > 0 → compSize > 0` holds by construction.

The new CD-parse guard is therefore **read-side hardening only**;
it never fires on archives produced by `lean-zip`'s own
`Archive.create`. Writer-side callers of the public API who
hand-forge `Entry` records with crafted field values bypass
`Archive.create` entirely and do not exercise this invariant.
