## Decisions

- **Reject malformed LH ZIP64 rather than fall back to literal
  sentinels.**  When `stdLocalUncompSize == 0xFFFFFFFF` without a
  matching LH ZIP64 extra, the LH itself is malformed per APPNOTE.
  Treating the sentinel as a literal 4 GiB would silently misinterpret
  the archive; the new error is more truthful.
- **Did not add `Entry.flags`.**  The issue allowed skipping the
  flag-bit-11 (UTF-8) consistency check rather than thread `flags`
  through the entry struct in this PR; a code comment flags the gap so
  a follow-up issue can pick it up.
- **Reused `parseZip64Extra` rather than introducing
  `parseZip64LocalExtra`.**  The shapes match for any spec-conformant
  LH extra (conditional fields based on which std field equals
  `0xFFFFFFFF`); the LH-always-includes-both-sizes convention is not
  required for the consistency check to fire correctly, since the CD
  values are the source of truth.
