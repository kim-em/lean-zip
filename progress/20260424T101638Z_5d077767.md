## What landed

Track E ZIP64 (0x0001) extra-field `dataSize` exactness check
(APPNOTE §4.5.3): `parseZip64Extra` now rejects entries whose
ZIP64 extra-field `dataSize` is strictly greater than the
`8 * N` bytes consumed by the `N` sentinel-gated 32-bit standard
fields. Trailing slack is a parser-differential smuggling vector
— a lenient parser silently discards bytes beyond the consumed
prefix that can carry attacker-chosen data.

### Implementation

- `Zip/Archive.lean`: added `fpos := fpos + 8` in the `stdOffset`

  consumed) and a new `if fpos != fieldEnd then return none`
  guard before the `break`. Renamed the CD-side caller's error
  message from `"truncated ZIP64 extra field"` to
  `"malformed ZIP64 extra field"` to cover both the existing
  missing-field case and the new slack case. Net: +4 lines.

### Attack shape / parser-differential

Stricter parser (e.g. Info-ZIP, Go's `archive/zip`, CPython's
`zipfile`, 7-Zip) rejects any ZIP64 extra whose `dataSize` is
not exactly `8 * N`. Pre-PR lean-zip required only
`dataSize >= 8 * N` (via the `if fpos + 8 > fieldEnd then return none`
guards) so an oversized `dataSize` was silently accepted and the
trailing bytes were discarded after the `break`. Sibling of
PR #1761 (`zip64-eocd64-bad-recsize.zip`), which validates the
outer EOCD64 record-size field for the same attack class.
