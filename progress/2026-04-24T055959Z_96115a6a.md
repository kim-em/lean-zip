## Summary

Added an APPNOTE §4.3.14.1 validity check inside
`findEndOfCentralDir`'s ZIP64 EOCD64 parse branch: the EOCD64
`versionNeededToExtract` field at `bufPos + 14` is now read and
rejected when `< 45`. Closes a parser-differential smuggling vector —
a capability-gated extractor that feature-checks on this field
would decode the archive as plain ZIP (mis-reading
`cdOffset`/`cdSize` from the standard EOCD) while a permissive
reader silently processes the ZIP64 overrides.

The check is placed before the override field reads so the resulting
error does not reveal the smuggled values.

## Decisions

- **Option (a) for deliverable 2**: `findEndOfCentralDir` is *already*
  `IO (Option …)` as of the ZIP64 override sentinel work that landed
  in PR #1754. No tuple widening or additional plumbing was needed —
  the version check throws directly at the site.
- **Fixture via byte-patch (option (a) in deliverable 4)**: patched
  `testdata/zip/interop/go-zip64.zip` at absolute offset 158
  (= EOCD64 pos 144 + 14) from `\x2d\x00` (45) to `\x14\x00` (20).
  Byte-identical to the Go-generated source except for those two
  bytes. Deterministic, auditable, no new generator script needed.
  - Fixture SHA-256:
    `02bfb4213754855390b7e75cb77b42f301a3f327b6612b7d12f989bdd9f9d47b`
  - Fixture size: 242 B (same as source).
- **Strict `≥ 45`, no upper bound**: APPNOTE §4.3.14.1 is a lower
  bound only. Kept the comparison simple.
- **`versionMadeBy` at `bufPos + 12` intentionally not checked**:
  APPNOTE imposes no constraint on that field for EOCD64. Non-goal
  per the issue.
- **`set_option maxRecDepth 2048` added to `ZipTest/ZipFixtures.lean`**:
  the cleanup array literal at the tail of the file grew past the
  default `maxRecDepth` of 512 after adding the new fixture. Matches
  the pattern already used in `ZipTest/Helpers.lean` and
  `ZipTest/Gzip.lean`. Set-options do not propagate across imports
  in Lean 4, so the `Helpers.lean` setting was not enough.
- **SECURITY_INVENTORY.md line-anchor sweep**: my Archive.lean
  edits inserted 14 lines (docstring addendum + throw block) at the
  head of the ZIP64 branch, shifting every later citation by +19
  net (the full patch is +19 insertions overall). Ran a one-shot
  Python regex over the inventory corpus rows to shift
  `Zip/Archive.lean:N` → `Zip/Archive.lean:N+19` for all `N ≥ 297`
  and confirmed `scripts/check-inventory-links.sh` reports
  `errors=0, warnings=0`.
