# 2026-03-13 Feature: Proven-bounds data access in Binary.lean

**Session**: ae516c19 (feature)
**Issue**: #1430

## What was accomplished

Eliminated all `]!` runtime-bounds-checked array accesses in `Zip/Binary.lean`,
replacing them with statically verified proven-bounds `]` access.

### Approach

Used `if h :` guard pattern inside read functions to capture the bounds
hypothesis, then `data[i]'(by omega)` for proven access. Functions return
a sensible default (0 or "") for out-of-bounds, matching the previous
`getElem!` behavior. This preserves the external API — callers don't need
to supply proof arguments.

### Files changed

- **Zip/Binary.lean**: All 11 `]!` converted. Functions use `if h : offset + N ≤ data.size`
  guards with proven-bounds access in the then-branch.
- **Zip/Spec/BinaryCorrect.lean**: Proofs updated with `dif_pos` + `resolve_dite_getElem!`
  helper to resolve guards and convert back to `getElem!` for existing proof infrastructure.
- **Zip/Spec/ZstdTwoBlock.lean**: Four theorems needed `maxHeartbeats 400000` due to
  additional dite branches during case splitting (after Zstd.lean split).

### Key decisions

- **Internal guards vs preconditions**: Initially tried adding precondition parameters
  (`(_ : offset + N ≤ data.size := by omega)`), but this broke 30+ callers across
  Archive.lean, ZstdHuffman.lean, Gzip.lean, etc. Switched to internal `if h :` guards
  which preserve the external API while achieving proven-bounds internally.
- **`getElem!` in proofs**: Kept `getElem!`-based proof infrastructure via
  `resolve_dite_getElem!` helper that converts proven-bounds back to `getElem!`.
  This avoids rewriting all roundtrip proofs from scratch.
- **Heartbeat increase**: The `if h :` guard creates additional `dite` branches
  that slow case-splitting in Zstd proofs. Increased heartbeats from 200k to 400k
  for four affected theorems.

## Quality metrics

- `grep -c ']!' Zip/Binary.lean` = 0 (was 11)
- Sorry count unchanged: 4 (all in XxHash.lean, pre-existing)
- All Lean files build successfully (zstd_ffi.o failure is pre-existing C dependency)
