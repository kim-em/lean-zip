## Checksum-ladder-template conformance

Confirmed each rung matches the skill's three-part shape:

- `_empty`: `@[simp]`-tagged, `simp only [crc32, updateBytes, â€¦]; bv_decide` â€” fine.
- `_singleton`: Native bridge follows `simp only [crc32]; show _ ^^^ 0xFFFFFFFF = _; rw [<init-unpack>, updateBytes_eq_updateList, hdata]; exact Spec.checksum_singleton b` â€” matches template with CRC32's extra `show _ ^^^ 0xFFFFFFFF = _` step needed because the `crc32` wrapper folds the final XOR into its body.
- `_pair`: same shape.
- `_append`: uses `updateBytes_eq_updateList` + `Spec.updateList_append`, with a local `raw_eq` bv_decide helper to handle the `if init == 0 then â€¦` conditional.

The `show _ ^^^ 0xFFFFFFFF = _` type-narrow step in `_singleton` / `_pair`
is a CRC32-specific deviation (Adler-32's rungs don't need it because
`adler32` ends in `pack`, not XOR). This is a load-bearing deviation â€”
omitting it makes `rw [if_pos rfl, ...]` fail to find the match. Noted
here for future reviewers; not a bug.

## Decisions

1. **Deleted `crcBit_xor_high`**: Zero call sites. Introduced in the
   earliest CRC32 work (progress entry 2026-02-19) as a per-bit XOR
   distribution lemma, but was superseded within hours by
   `crcBits8_split` (the 8-fold variant actually needed at call sites).
   Re-deriving from `bv_decide` takes <1 second if a future proof ever
   needs it, so no loss.

2. **`table_size` privatized**: Per the `checksum-ladder-template`
   skill's visibility ratchet â€” "keep supporting helpers `private`
   until a second cross-file consumer materialises." Only consumer is
   in-file. The analogous Spec-side promotions
   (`Spec.mkTable_size`, `Spec.xor_ff_byte_lt_mkTable_size`) happened
   explicitly when `Native.crc32_singleton` added the second call site;
   `Native.table_size` has had no such trigger event. The Native-side
   `table` is kept `def` (not `private`) because `updateBytes` and
   `crc32` both reference it inside their public bodies, so the name
   surfaces through definitional unfolding anyway.

3. **`crcByteTable_eq_crcByte` kept public**: This is a real
   characterization (table-driven CRC byte step equals bit-by-bit CRC
   byte step), not just plumbing. Worth exposing even though the only
   current consumer is the in-file `updateBytes_eq_updateList`. Low
   cost to keep public; a future downstream algebra file may want it.

4. **`crc32_append` merge rejected**: Attempted to merge the two
   sequential `simp only` calls into one. Build failed with the goal
   state showing the `if init == 0 then 0xFFFFFFFF else init ^^^ 0xFFFFFFFF`
   conditional still present â€” confirming the `checksum-ladder-template`
   skill's caveat that ordering-sensitive merges break. The original
   two-phase structure is load-bearing: first phase normalizes the
   wrapper's init-xor with `raw_eq`, then the second phase applies
   the XOR-associativity identities and list-level updateList append.
   Leaving this as two `simp only` calls.
