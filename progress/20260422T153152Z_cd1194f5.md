## Findings

### 1. `Spec.checksum_pair` — cadence collapse to mirror `_singleton`

**Master** (pre-refactor) used a two-stage discharge:

1. `rw [← UInt32.toNat_inj]`
2. First `simp only [..., updateByte, init, MOD_ADLER]` — unfold only
3. Three `have` bridging facts (`ha₁`, `ha₂`, `hb₂`) — mod equalities
4. `rw [ha₁, Nat.zero_add, ha₁, ha₂, hb₂, pack_toNat_of_bounds (show …)
   (show …), UInt32.toNat_ofNat_of_lt' (show …)]` — `show … by omega`
   bound witnesses inlined into the `rw` arguments

**After refactor** matches the post-#1674 `_singleton` cadence at
`Zip/Spec/Adler32.lean:162-173`:

1. All eight bound witnesses upfront as named `have`s (`h1`, `h2`,
   `ha₁`, `ha₂`, `hb₂`, `hpack₁`, `hpack₂`, `hbnd`)
2. `rw [← UInt32.toNat_inj]`
3. Single `simp only […, ha₁, ha₂, hb₂, pack_toNat_of_bounds hpack₁
   hpack₂, UInt32.toNat_ofNat_of_lt' hbnd]` block
4. `omega`

**Why 0 LOC delta despite the collapse:** `_singleton` saved 4 LOC in
#1674 (commit `b4695e3`) because it had only 1 mod equality, 1 pack
bound (used twice as `pack_toNat_of_bounds hb hb`), and 1 UInt32
bound — three new `have`s replacing one `rw` line. `_pair` has 3 mod
equalities (different patterns: `1 + b₁`, `1 + b₁ + b₂`, sum), 2
distinct pack bounds (A and B differ), and 1 UInt32 bound — six new
`have`s replacing the `rw` line. The 5-line gain in setup exactly
cancels the 5-line drop in discharge.

**Why commit anyway:** structural uniformity with `_singleton`. Both
proofs in the closed-form chain now follow the identical four-step
template. The next agent extending to a `_triple` or higher arity
follows the same shape. No theorem statement changed.

**Tried and reverted:** inlining `pack_toNat_of_bounds (by omega)
(by omega)` directly in the `simp only` argument list (would have
saved 2 LOC). Lean's `simp only` cannot resolve the implicit `{a b}`
of `pack_toNat_of_bounds` when the bound proofs are anonymous
`(by omega)` — `omega` runs in an empty-ish elaboration context and
can't see `b₁.toNat < 256` from the surrounding `have`s. Named
`hpack₁` / `hpack₂` are required.

### 2. `Native.adler32_pair` — already at minimum

The native wrapper at `Zip/Native/Adler32.lean:58-66` is the canonical
three-line characterizing-property template:

```
have hdata : ((ByteArray.empty.push b₁).push b₂).data.toList = [b₁, b₂] := rfl
have hunpack : Spec.unpack 1 = Spec.init := by decide
simp only [adler32, updateBytes_eq_updateList, hdata, hunpack]
exact Spec.checksum_pair b₁ b₂
```

Identical shape to `adler32_singleton` at lines 47-53. No refactor
applies — the proof is the minimum viable bridge between the
`ByteArray`-level statement and `Spec.checksum_pair`. Confirms
meditate #1651 §3's "three-part native template" is intact.

### 3. `@[simp]` eligibility — defer to #1674's precedent (NO tag)

PR #1674 reviewed `_singleton` and chose **not** to attach `@[simp]`,
on the grounds that the LHS is a concrete two-element pattern that
simp would only fire on syntactic `[b]` matches. `_pair` is the same
class of pattern (concrete `[b₁, b₂]`), so the same judgement applies:
**no `@[simp]` tag for `_pair`**. This preserves the symmetry of the
chain (`_empty` is `@[simp]` because `[]` is a normalizing target;
`_singleton` and `_pair` are not because `[b]` and `[b₁, b₂]` are not
canonical normal forms simp would rewrite into).

Cross-check: `Zip/Spec/Adler32.lean:66` (`checksum_empty`) is
`@[simp]`; `:162` (`_singleton`) and `:180` (`_pair`) are not. Chain
is now coherent.

### 4. `pack_toNat_of_bounds` visibility — keep `private`

Master consumers (post-refactor):

| Site | Function |
|------|----------|
| `Zip/Spec/Adler32.lean:137` | `pack_unpack` |
| `Zip/Spec/Adler32.lean:151` | `unpack_pack_of_valid` |
| `Zip/Spec/Adler32.lean:172` | `checksum_singleton` |
| `Zip/Spec/Adler32.lean:200` | `checksum_pair` |

All four call sites are within `Zip/Spec/Adler32.lean` itself, so the
`private` keyword does not block any current usage (`private` only
affects cross-module visibility; in-file uses work regardless).
Promoting to `protected` would only matter if a downstream module
(e.g. a future RLE-Adler integration) wanted to reuse it.

Per the meditate #1651 visibility-ratchet pattern, the threshold for
ratchet is **the first cross-module consumer**, not the third in-file
consumer. No such consumer exists. Keep `private`.

The helper pattern is documented in `.claude/skills/lean-uint-bitvec/SKILL.md:84`
for anyone re-implementing it elsewhere; the pattern is portable
without needing the lean-zip helper itself to be exported.

This decision matches the prior reviews recorded in
`progress/20260422T123537Z_4ba91ca3.md` and
`progress/20260422T141331Z_48f0f22e.md`.
