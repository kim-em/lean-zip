## What was accomplished

Added three regression blocks to `ZipTest/Gzip.lean`:

1. **Concat-member limit** (between existing concat block and streaming
   compression): two 10-byte gzip members compressed with
   `Gzip.compress`, concatenated, then decompressed with
   `maxDecompressedSize := 5`. Must reject with `"exceeds limit"`. This
   pins the cross-member cumulative bomb check: `total` in
   `c/zlib_ffi.c:168-198` persists across `inflateReset` iterations, so
   even the first member's 10 bytes overshoot the 5-byte cap and the
   FFI must error — the test forbids silent truncation or
   member-boundary reset of the counter.
2. **Zero-length InflateState.push**: `Gzip.InflateState.new` →
   `push ByteArray.empty` → `push compressed` → `finish`. Asserts the
   concatenation of all three outputs equals `mkTestData`.
3. **Zero-length DeflateState.push**: `Gzip.DeflateState.new` →
   `push ByteArray.empty` → `push payload` → `finish`. Concatenated
   output round-trips through `Gzip.decompress` to `mkTestData`.

The existing basic-concat block at `ZipTest/Gzip.lean:32-39` was not
touched; the issue's suggested sanity check (temporarily weakening
`decoded.beq (part1 ++ part2)` to `decoded.beq part1`) was executed
and confirmed the assertion panics — proving the existing test does
exercise the second member. Reverted before committing.

## Decisions

- **Did not duplicate existing concat coverage.** `ZipTest/Gzip.lean`
  already had a happy-path concat-decompress test (lines 32-39 in the
  pre-change file). The new contribution is specifically the
  limit-across-member-boundary case, which is the real bomb vector.
  Per project CLAUDE.md ("Do NOT remove a working proof"), I kept the
  existing block intact and appended the limit block after it.
- **Error substring `"exceeds limit"`** matches the FFI wording
  emitted by `c/zlib_ffi.c:195`, consistent with `ZipTest/Zlib.lean:18`,
  `ZipTest/RawDeflate.lean:18`, and the existing gzip limit test at
  `ZipTest/Gzip.lean:19`.
- **`big := mkTestData`** was re-used for the zero-length-chunk tests
  rather than allocating a smaller fixture, since `mkTestData` is
  already in scope and keeps the payload non-trivial (100 × 62B =
  6200B) so the zero-length push is not accidentally equivalent to a
  one-byte payload.
- **No `testdata/` additions.** All payloads are generated in-memory
  from `.toUTF8` or `mkTestData` as specified in the issue.
