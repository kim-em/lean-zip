# Progress: Zstd block size and window size validation

**Date**: 2026-03-04 UTC
**Session**: cec01afd (feature)
**Issue**: #540

## Accomplished

1. **Block size validation** in `decompressBlocks`: rejects blocks with
   `blockSize > 131072` (128KB) per RFC 8878 §3.1.1.1.
2. **Window size threading**: `decompressBlocks` now takes `windowSize : UInt64`
   from the frame header, passed through from `decompressFrame`.
3. **Window size enforcement** in `executeSequences`: rejects sequence offsets
   exceeding `windowSize` per RFC 8878 §3.1.1.3.2.3. Uses optional check
   (windowSize=0 disables) for backward compatibility.
4. **Validation tests**: 4 new tests in `ZstdNative.lean`:
   - Crafted frame with oversized block header is rejected
   - Sequence with offset > windowSize is rejected
   - Valid executeSequences roundtrip produces correct output
   - Legitimate FFI-compressed data still decompresses

## Decisions

- **Skipped `blockSize ≤ windowSize` sub-check**: The issue suggested checking
  block size against window size, but this is incorrect per RFC 8878. Window
  size constrains back-reference offsets, not block sizes. A raw block can
  legitimately exceed the window size (e.g., 128KB raw block with 1KB window).
- **Used default parameter** for `windowSize` in `executeSequences` (default 0
  = disabled) to avoid breaking existing callers.
