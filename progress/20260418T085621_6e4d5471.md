## Why these 4 sites were skipped

Both `countMatch.go` and `mainLoop` (for the `hashTable`/`hashValid`
reads) interact badly with the existing iterator-equivalence proofs:

```lean
-- attempted internal-guard pattern in mainLoop:
let matchPos := if hh : h < hashTable.size then hashTable[h]'hh else default
```

In `mainLoop_eq` the proof does `unfold ... ; dsimp only [...] ; split`.
`dsimp only` performs let-zeta, so the freshly introduced
`if hh : h < ...` ends up at the top of the term tree. `split` then
greedily consumes that decidable expression instead of the intended
`if hlt : pos + 2 < data.size` outer guard, breaking the entire case
analysis. Same mechanism kills the `go` conversion via
`countMatch_eq` (a sibling `dsimp only` + `split` proof).

A clean fix exists — change `lz77GreedyIter.go` /
`lz77GreedyIter.mainLoop` to also use the internal-guard pattern, so
both sides of the equivalence have the same dite-shape — but that
modifies `lz77GreedyIter`, which the issue explicitly puts off-limits
("MUST NOT touch lz77GreedyIter / lz77Lazy / lz77LazyIter /
DeflateDynamic.lean"). Alternative restructurings (e.g. extracting a
helper like `loadByte`) would also touch the iterator side via the
equivalence proof.

## Skill notes

The `proven-bounds` skill correctly warns about the
"do-notation guard" pitfall and the `dite` simp-call updates needed
in spec proofs. A worth-adding pitfall: **internal-guard let-bindings
plus `dsimp only` + `split` is hostile**, because let-zeta exposes the
new dite and `split` consumes it greedily before reaching the intended
outer decidable. When the existing spec proof structure is
`dsimp only [funcName, ...]; split`, the proof author needs to either
change BOTH the function and any iterator equivalent (so dite-shapes
match), or refactor the spec to not rely on a top-level `split`.
