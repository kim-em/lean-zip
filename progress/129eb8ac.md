# Progress: Table-driven Huffman decode for inflate (Track D D-5)

- **Date**: 2026-06-02 UTC
- **Session**: 129eb8ac (feature, one-shot `/once`)
- **Issue**: #2484

## Accomplished

Added a `2^9` "fast-bits" lookup-table Huffman decoder to native inflate and
routed the runtime through it, keeping the bit-by-bit `HuffTree.decode` walk as
the canonical spec.

New code in `Zip/Native/Inflate.lean`:
- `HuffTree.tableEntry tree idx` — walk `tree` on the low bits of `idx`
  (LSB-first, matching `readBit`) for up to `fastBits = 9` steps, returning
  `(symbol, codeLen)` or the sentinel `(0, 0)` for long/empty.
- `HuffTree.buildTable tree` — `Array.ofFn` of all `2^9` entries.
- `HuffTree.peekFast` / `bitsAvail` — direct two-byte peek + remaining-bits count.
- `HuffTree.decodeWithTable tree table br` — peek 9 bits, index the table,
  consume `codeLen` bits in one step; fall back to `tree.decode br` for codes
  longer than 9 bits and when fewer than `codeLen` bits remain.
- `Inflate.decodeHuffmanFast` — copy of `decodeHuffman` that builds the
  literal/length + distance tables once per block and uses `decodeWithTable`.
- `attribute [implemented_by Inflate.decodeHuffmanFast] Inflate.decodeHuffman`.

Because `decodeHuffman` (the spec) is untouched, **every existing inflate
correctness proof holds verbatim** — `inflate_correct`, the `decode.goR`
correspondence, the loop-bound proofs: all unchanged. 0 sorries throughout.

## Validation

- `ZipTest/InflateTable.lean` (new): differential check of
  `decodeWithTable (buildTable tree) br` against `tree.decode br` — same symbol,
  same consumed `pos`/`bitOff`, same ok/error disposition — over trees spanning
  every code-length regime (complete depth 1/4/8/9/10/12, a mixed 1..15 spine,
  fixed lit/dist) from every bit offset, including short inputs that trip the
  not-enough-bits fallback. `HuffTree.decode` carries no `@[implemented_by]`, so
  this is a genuine head-to-head.
- Full `lake exe test` green (incl. the 1000-iteration inflate fuzz suite, which
  now exercises the fast path end-to-end).

## Benchmark

The four existing bench patterns are all copy- or stored-bound for *decode*
(`constant`/`cyclic`/`text` compress into a few very long matches; `prng` →
stored blocks), so none stresses Huffman symbol decode. Added a non-periodic
`words` pattern (PRNG-ordered English words → many short matches + literals) to
`ZipBenchReport.lean` + `bench/plot.py` (5-panel grid), and refreshed
`bench/results/latest.json` + `bench/graphs/*.svg`.

A/B (fast vs. `@[implemented_by]`-disabled tree walk, min-of-5 over the same
zlib-deflated `words` stream the dashboard times — the measurement must run the
decoder as a re-executed IO action per rep, or Lean memoises the loop-invariant
pure call and reports ~0):
- `words` 64 KiB: 20.5 → 28.3 MiB/s (**1.38×**)
- `words` 1 MiB: 21.5 → 28.1 MiB/s (**1.31×**)

Copy-bound patterns unchanged within noise. The speedup is modest — not the ~2×
a C table decoder gets — because in pure-Lean the per-byte output construction
(`ByteArray.push`, `copyLoop`) and the bounds-checked bit reader dominate decode
time; eliminating the per-bit tree descent shaves only part of the cost.

## Decisions

- **`@[implemented_by]` over proof-transfer.** The issue sketched proving
  `tableDecode = treeDecode` and transferring every inflate lemma by `rw`. The
  conservative route — keep `decodeHuffman` as the canonical, already-proven
  spec and attach the fast path via `@[implemented_by]` — touches **zero**
  existing proofs and is exactly how Lean stdlib ships fast implementations of
  proven specs. `@[implemented_by]` is trusted (unlike the banned
  `native_decide`, it never enters a *proof*), and the table decoder is
  validated by the differential test + the full roundtrip/fuzz suite.
- A formal symbol-level `decodeWithTable_eq` is a tractable follow-up — the
  `decodeBits` / `TreeHasLeaf` / `decode_go_complete` infrastructure in
  `Zip/Spec/HuffmanCorrect.lean` supports it (table entry = tree walk on the
  lookahead bits; `peekFast` bit `j` = `br.toBits` bit `j` for `j < bitsAvail`).
