## What was accomplished

Landed the CD-parse NUL-byte guard on entry name bytes end to end:

- **Guard** in `parseCentralDir` at
  [Zip/Archive.lean:559](/home/kim/lean-zip/Zip/Archive.lean:559):
  rejects CD entries whose raw `nameBytes : ByteArray` contains any
  `0x00` byte. Guard fires *before* the UTF-8 decode
  ([:562-572](/home/kim/lean-zip/Zip/Archive.lean:562)) so neither the
  UTF-8-flagged strict branch nor the Latin-1 fallback re-introduces
  NUL into logs. Error substring:
  `"CD entry name contains NUL byte"`. Since `ByteArray` has no
  `.any` in v4.29.1, the check is expressed as
  `(nameBytes.findIdx? (· == 0)).isSome`.
- **Fixture** `testdata/zip/malformed/cd-nul-in-name.zip` (118 B).
  Extended `scripts/build-cd-lh-mismatch.py` with a `name_bytes` kwarg
  on `make_lh` / `make_cd` / `write` (default `None` → derive from
  module constant `NAME`). The new builder emits `b"a\x00b.txt"`
  (7 bytes, NUL at index 1) on both the LH and CD name fields so the
  CD/LH name-bytes consistency invariant (issue #1722) stays intact
  once that guard lands.
- **Test assertion** in `ZipTest/ZipFixtures.lean` between
  `cd-entry-internal-attrs-reserved.zip` and `cd-patched-data-flag.zip`
  (assert-substring `"name contains NUL byte"`, distinct from
  `"invalid UTF-8 in entry name"` at old-line 550 / new-line 565).
- **Inventory update**: one *Recent wins* bullet under *ZIP Archive
  Reader/Extractor* and a new row in the *Minimized Reproducer Corpus*
  table, both citing PR #1831. Bumped every `Zip/Archive.lean:*` line
  citation whose original target was past the insertion point (>544)
  by +15 — pristine-master baseline was `errors=0, warnings=0`, which
  the post-change tree now also satisfies.
- **Error-wording catalogue** row added for the new substring.

## Decisions / key patterns

- **`ByteArray.findIdx?` vs `.any`**: v4.29.1's `ByteArray` has no
  `.any` method on the bare type (the issue's pseudocode assumed
  one). Swapped to `(nameBytes.findIdx? (· == 0)).isSome` — same
  structural semantics, already-in-stdlib API.
- **Line-citation shift**: my 15-line guard insertion drifted every
  `Zip/Archive.lean:*` citation in `SECURITY_INVENTORY.md` past
  original line 544 by +15. Fixed with a one-shot
  `re.sub(r'Zip/Archive\.lean:(\d+)', …)` pass that preserves the two
  citations I introduced (559, 562). Doc baseline returns to
  `warnings=0`. Net-new additive content in the inventory is
  ~34 lines (one *Recent wins* bullet + one corpus-table row); the
  rest of the 130/48 diff is the mechanical line-anchor bump.
- **Why on raw bytes, not decoded string**: matches the pattern of
  PR #1788's `validateExtraFieldStructure` — structural validation on
  the byte blob before higher-level parsing. Additional win: the
  error message never re-introduces NUL into log output, which a
  `String`-level check (emitted after Latin-1 fallback) could have
  accidentally done.
- **Tar side note (out of scope)**: the issue flags that `Zip/Tar.lean`
  likely has an analogous smuggling vector at tar-header parse time.
  Not addressed here — would need a separate Track E issue.
- **LH-side scope**: the LH name is read in `readEntryData` but never
  decoded to a `String` for downstream use; only the CD name reaches
  `Entry.path` and hence the public surface. A NUL in the LH name
  alone would trip the CD/LH name-bytes consistency check (#1722)
  once that lands, so the LH dimension is closed transitively.
