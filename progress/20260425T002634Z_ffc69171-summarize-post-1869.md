## Attack-class closures

### CD-parse mathematical-invariant family — column 3 (#1886)

The family now has three columns closed at CD parse:

| Column | Invariant | Closing PR | Method scope |
|---|---|---|---|
| 1 | `compSize == uncompSize` | #1773 | method = 0 (stored) only |
| 2 | `uncompSize == 0 → crc == 0` | #1857 | method-agnostic |
| 3 | `compSize == 0 → uncompSize == 0` | #1886 | method-agnostic |

Column 3's contribution beyond column 1: column 1 is method=0-specific
(`compSize == uncompSize` is only a tautology for the stored method);
column 3 covers all methods including method=8 (deflate). Crafted
deflate entries claiming `compSize = 0` with `uncompSize > 0` were
previously not rejected at CD parse — the parser-differential vector
was that a downstream consumer reading `Entry.uncompressedSize` for
allocation sizing would over-allocate for an entry whose compressed
payload could not produce that output (DEFLATE stream of size 0 cannot
emit any bytes; minimum non-empty deflate stream is 2 bytes for a stored
empty literal block).

### Cross-format NUL closure — UStar layer (#1880)

The post-#1843 progress entry's framing was *three code-guarded layers
+ one structurally-NUL-safe layer*:

- ZIP CD entry name (#1831, byte-content) — code-guarded
- Tar GNU long-name / long-link typeflags (#1865) — code-guarded
- Tar PAX extended-header key/value (#1866) — code-guarded
- Tar UStar header `name` / `linkname` (fixed-width 100-byte fields,
  NUL-terminated) — structurally immune (decoder stops at first NUL)

PR #1880 adds an explicit code guard on the raw 100-byte UStar `name`
and `linkname` fields plus the 155-byte `prefix` field, rejecting any
non-NUL data after the first NUL within the field width. This is
defense-in-depth — the structural construction of the
NUL-terminated-string decode already prevents smuggling — but it
catches a parser-differential corner case where an unaware consumer
processes the raw field bytes (not the decoded string) and would see
data past the first NUL. Now all four layers fail closed on raw NUL
content.

### EOCD64 v2-record explicit fixture coverage (#1889)

The existing PR #1761 (post-#1765 wave) implemented the EOCD64
record-size guard (`size of record == 44`) at `findEndOfCentralDir`,
rejecting any record-size value other than the APPNOTE §4.3.14 v1
constant. APPNOTE §4.3.14.2 documents a v2 strong-encryption EOCD64
extension that adds a `strongEncryptionCompositeSize` field, which
would manifest as a different record-size value. lean-zip does not
implement strong encryption, so v2 records must be rejected.

PR #1889 adds `zip64-eocd64-v2-record.zip` as a positive-rejection
fixture for that guard — the fixture sets the EOCD64 record-size to
56 (v2 minimum), which causes #1761's guard to fire. Closes the
coverage gap that the post-#1843 progress entry's *Follow-up planning
hooks* flagged.
