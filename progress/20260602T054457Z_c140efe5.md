### Method
Throwaway compiled driver (a temporary `lean_exe` + `ProfileDriver.lean`, both
removed before commit). Two laziness traps had to be defeated to get real
numbers:
1. **Thunk memoisation** — a loop-invariant pure result is computed once then the
   thunk is cached, so min-of-N reported ~0. Fixed by perturbing the input one
   byte per rep.
2. **Lazy `let`** — `let v := f x` in a `do`-block is *lazy*; the work happened at
   the later `sink + v` use, *outside* the timed region (everything read 0 µs).
   Fixed by forcing `v` (and each phase's input) through an IO-sequenced branch
   inside / before the timed region.

### Findings (1 MiB, single-threaded EPYC 9455, Lean v4.30.0-rc2)
- **Matcher `lz77GreedyIter` dominates**: 99% of one compress pass on `text`
  (87 471 / 87 994 µs), 85% on `prng` (104 011 / 123 105). Scales linearly
  (`Array.set!` is in-place — no quadratic blow-up); ~100 ns/byte ≈ 10 MB/s,
  which *is* the headline gap.
- **Double pass**: `deflateRaw` level ≥5 runs `deflateFixedIter` AND
  `deflateDynamic`, each recomputing the token stream. The dominant matcher runs
  twice; the second pass is ~42% of the full 244 964 µs (prng 1 MiB).
- Corrected the backlog's D-3 premise: the matcher is a single-probe hash, *not*
  a "linear scan"; a hash chain would help ratio, not speed. D-3 reframed as
  "shrink the matcher per-byte constant factor (boxed-`Nat` → `UInt32`/`USize`)".
