### A.3 Smuggled value choice: `diskWhereCDStarts = 99 (0x0063)`

`diskWhereCDStarts` is a `UInt16` field at standard-EOCD offset `+6`
(per APPNOTE §4.3.16 "Number of the disk where central directory
starts"). The APPNOTE §4.3.16 ZIP64-indirection sentinel is
`0xFFFF = 65535`.

The fixture's choice of `99` is well-defended:

- `99 ≠ 0xFFFF`: so the "relaxed sentinel" arm at :408 fails on its
  first disjunct (`stdDiskWhereCDStarts16 == val16Max`).
- `99 ≠ 0` (the actual single-disk archive's CD-disk number, carried
  in the ZIP64 EOCD64 `diskWhereCDStarts` at offset `+8`): so the
  second disjunct (`stdDiskWhereCDStarts16.toNat == diskWhereCDStarts`)
  also fails.
- `99` is also not numerically close to the ZIP64 override (`0`),
  avoiding any accidental off-by-one or sign-conversion coincidence.
- `99` is small enough to fit `UInt16` with no byte-order ambiguity
  (`0x0063`), matching the `cdSize=99` / `totalEntries=99` /
  `numEntriesThisDisk=99` sibling convention across the wave.

The value `99` is deliberately consistent across the per-slot wave
(`cdSize=99` in PR #1905, `totalEntries=99` in PR #1908,
`diskWhereCDStarts=99` here, `numEntriesThisDisk=99` in PR #1922,
`numberOfThisDisk=99` in in-flight PR #1909) — a uniform smuggling
convention that makes the family's fixture intent and inter-fixture
comparisons easy to read in `xxd` diffs.

## F. Reviewer observations

- **F.1 Cross-disk-dispatch smuggling-vector emphasis.** The feature-
  side PR body (commit message + test comment at :932-942 + inventory
  narrative at :1314) each note that `diskWhereCDStarts` is the
  cross-disk dispatch dual of the `numberOfThisDisk` smuggling vector
  — letting an attacker present "two different archives to two
  different parsers from the same byte sequence". This emphasis is
  appropriate: a relaxed reader that trusts a smuggled
  `diskWhereCDStarts=99` value would attempt to look up the CD on
  disk 99 (silently failing or applying disk-spanning logic that
  doesn't apply to the actual archive), while a strict reader using
  the ZIP64 override of `0` reads the same CD. The fact that lean-zip
  is a single-disk-only reader makes this less directly exploitable
  in lean-zip itself, but the fixture's value as a regression probe is
  highest precisely because it pins the slot most likely to interest
  multi-disk-aware producers / consumers (e.g. legacy WinZip PKZIP
  span-archive producers).

- **F.2 UInt16 overflow-convention consistency.** `diskWhereCDStarts`
  is UInt16 at standard-EOCD position `+6`; the ZIP64 override at
  EOCD64 position `+8` is UInt32. (Note: `diskWhereCDStarts` is *not*
  a UInt64 in the ZIP64 record — only the offset/size fields are; the
  disk-number fields are UInt32 in EOCD64 because APPNOTE allows up
  to ~4 billion disks, not because they need 64-bit values.) Per the
  per-slot-family convention, the four UInt16 standard-EOCD fields
  (`totalEntries`, `numberOfThisDisk`, `numEntriesThisDisk`,
  `diskWhereCDStarts`) use `val16Max = 0xFFFF` as the sentinel, while
  the two UInt32 fields (`cdSize`, `cdOffset`) use
  `val32Max = 0xFFFFFFFF`. The guard at :408 correctly uses
  `val16Max` for the UInt16 `diskWhereCDStarts` field. No
  off-by-16-bits hazard.

- **F.3 Writer-side compliance.** lean-zip's own `buildRecord` emits
  standard-EOCD `diskWhereCDStarts` with either the `val16Max`
  sentinel (ZIP64 path) or the actual disk-zero value (non-ZIP64
  path) — writer-side non-smuggle invariant holds (PR #1911 cannot
  fire on a lean-zip-produced archive). Sibling invariant to
  paired-review-#1880 §F.1 / paired-review-#1889 §H.2 / paired-
  review-#1908 §F.3.

- **F.4 Fixture non-bloat across the wave.** The five landed
  family fixtures (4 per-slot + 1 originating `nosentinel`) are all
  198 B byte-wise — the builder's additive extension preserves
  byte-budget (no incidental bloat). The total Track E fixture-corpus
  size in `testdata/zip/malformed/` grows by 198 B per per-slot
  fixture; the wave's 4 landed-so-far per-slot fixtures contribute
  792 B, the full wave of 5 per-slot fixtures (after PR #1909 lands)
  will contribute 990 B. Well within the corpus size budget flagged
  by the `malformed-fixture-builder` skill.

- **F.5 Guard-ordering sanity across the 6 per-slot sub-checks.** A
  future hardening that reordered the 6 sub-checks would change this
  fixture's attribution only if the reorder moved :408 after :411
  (highly unlikely given APPNOTE's field ordering). More realistically,
  a future hardening might **combine** the 6 sub-checks into a single
  loop over an array of slots — such a refactor would need to preserve
  the per-throw wording to keep this fixture's `assertThrows` substring
  valid. Not in scope for this PR, but worth flagging as a future-
  refactor-breakage risk for the per-slot fixtures. (Same observation
  as paired-review-#1916 §F.5 — applies to all per-slot wave members.)

- **F.6 `99 == diskWhereCDStarts` collision avoidance.** Unlike
  `cdSize` (UInt32, no practical constraint on `99`) or `totalEntries`
  (UInt16, `99` is plausible for a real archive), `diskWhereCDStarts
  = 99` is an in-range value that a real *multi-disk* archive could
  theoretically carry (PKZIP allows up to 65535 disks). In single-disk
  archives (lean-zip's only target) `99` is absurd, but the fixture's
  intent is to pin the override-mismatch arm — the fact that lean-zip
  rejects multi-disk archives downstream at :521 is *also* exercised
  by the sibling `eocd-disknum-mismatch.zip` fixture (PR #1742 at
  :1310 of the inventory). The two-stage layering (override-arm at
  :408 → single-disk sanity at :521) means this fixture pins the
  upstream guard specifically; a hypothetical "valid multi-disk
  archive on disk 99" probe would belong in a different fixture (and
  would trip :521 instead). The `99` choice is sound on this
  dimension.

- **F.7 Two-stage-layering note in fixture comment.** §C.1's

  correctly disambiguates which one this fixture pins. Worth
  preserving as a precedent for the `numberOfThisDisk` paired-review
