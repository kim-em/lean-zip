## What was done

Proof quality audit of `Zip/Spec/Fse.lean` (1991 lines, 47+ theorems).

### Duplicate pattern extraction

Five `decodeFseDistribution_*` theorems shared identical 15-20 line
unfolding/case-splitting boilerplate:
- `decodeFseDistribution_accuracyLog_ge`
- `decodeFseDistribution_accuracyLog_le`
- `decodeFseDistribution_sum_correct`
- `decodeFseDistribution_bitPos_ge`
- `decodeFseDistribution_pos_le_size`

Extracted `decodeFseDistribution_ok_decompose` private helper that
decomposes a successful call into its components (readBits result,
accuracy log relation, maxAccLog bound, decodeFseLoop result with
remaining=0). Each consumer reduced to 1-5 lines.

### Section header normalization

Fixed one `### ` → `## ` inconsistency (BitReader pos_le_size section).

### Comment cleanup

Removed 3 lines of redundant proof-strategy comments explaining the
`ForInStep.yield.inj (Except.ok.inj h)` pattern.

## Patterns discovered

- When using `cases hrd : expr` in Lean 4, `expr` is rewritten in the
  goal too. So if the goal mentions `expr`, it becomes the constructor
  value. Use `rfl` instead of `hrd` for proof obligations in that case.

## Other observations

- `readBits_go_data_eq` and `readBits_go_startPos_eq` have identical
  proof bodies (6 lines each). A generic field-preservation lemma
  could save ~4 lines but the generalization is tricky due to
  structure field access patterns. Not worth the complexity.
- The `buildFseTable_*` proofs (cells_size, numBits_le, newState_lt,
  symbol_lt) share 6-line decomposition headers but the types after
  decomposition vary, making extraction difficult. Left as-is.
- Pre-existing linker issue: `lake exe test` fails on this machine
  (`-lzstd` not found during final link). All Lean compilation succeeds.
