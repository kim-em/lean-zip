# Progress: Track E — Zstd multi-frame and skippable frame support

- **Date**: 2026-03-02T06:03Z
- **Session**: 712f3d96 (feature)
- **Issue**: #454

## What was accomplished

Added multi-frame and skippable frame support to the native Zstd decompressor
(RFC 8878 §3.1 and §3.1.2):

1. **`skipSkippableFrame`** — new function that validates skippable frame magic
   (0x184D2A50–0x184D2A5F), reads the 4-byte little-endian frame size, and
   returns the position after the frame data.

2. **Rewrote `decompressZstd`** from single-frame to multi-frame loop:
   - While data remains, checks the 4-byte magic number
   - Skippable frames: skipped via `skipSkippableFrame`
   - Zstd frames: decompressed and output concatenated
   - Unknown magic: error with offset information
   - Empty input returns empty ByteArray

3. **9 new tests** (tests 72–80 in ZstdNative.lean):
   - `skipSkippableFrame` valid frame, different magic values, truncated data
   - Multi-frame concatenation (two independent compressed frames)
   - Skippable frame before Zstd frame
   - Zstd frame followed by skippable frame
   - Trailing garbage detection
   - Empty input handling

4. **Updated test 22** from expecting an error on skippable frames to expecting
   empty output (skippable-only input).

## Decisions

- Skippable-only input returns empty ByteArray (not an error) — this matches
  RFC 8878 semantics and is the simplest correct behavior.
- Trailing bytes that don't form a valid frame magic are an error — prevents
  silent data loss from truncated or corrupted files.
