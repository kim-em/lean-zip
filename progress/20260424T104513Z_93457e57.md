## What landed

Track E ZIP64 (`headerId 0x0001`) duplicate-extra-block rejection
(APPNOTE §4.5): both the CD-side caller in `parseCentralDir` and the
LH-side caller in `readEntryData` now run a new
`hasDuplicateZip64Extra` guard before invoking `parseZip64Extra`.
Two well-formed `0x0001` blocks with different sentinel-resolved
payloads were a parser-differential smuggling vector — pre-fix
lean-zip used a `break` after the first match (a "first-wins"
policy) while a hypothetical "last-wins" parser would have used the
second block, yielding different decompression behaviour for
identical bytes.

### Attack shape / parser-differential

Stricter parsers (e.g. APPNOTE-conformant readers) reject any
extra-data containing duplicate registered header IDs — APPNOTE §4.5
("Extensible Data Fields") explicitly forbids it. lean-zip's pre-fix
loop processed the first `0x0001` block and broke out with no
post-loop scan for additional matches; trailing duplicate blocks
were silently discarded. A "first-wins" reader (lean-zip pre-fix)
and a "last-wins" reader would have produced different
`uncompressedSize`/`compressedSize`/`localOffset` triples for
identical bytes — a smuggling vector identical in shape to the
sub-field `dataSize` exactness vector closed by sibling PR #1785.

## Decisions

- **Helper outside `parseZip64Extra`, not inside.** Keeping the
  duplicate-detect scan as a separate `hasDuplicateZip64Extra`
  helper — invoked from both callers — preserves
  `parseZip64Extra`'s `Option` return type and thus avoids
  churning two callers' control flow. The cost is one extra
  pass over a buffer bounded by `UInt16.max` ≈ 64 KiB; trivial
  next to the surrounding `Handle.read` and `parseCentralDir`
  costs. Option B from the issue (replace `break` with a flag,
  re-check at end of loop) would have required either a return-
  type change to distinguish duplicate from missing/malformed,
  or an extra mutable carried out of the `Id.run` scope — both
  more invasive than a self-contained helper.
- **Distinct error wordings for CD and LH.** Mirrors the
  existing CD/LH split for the parse-failure family
  (`"malformed ZIP64 extra field"` vs
  `"truncated ZIP64 local extra field"`). The two layers fire
  at different times (CD at parse, LH at extract), so distinct
  wordings let regression tests for either layer fail
  unambiguously if the other layer's check ever subsumes it.
- **Two fixtures, not one combined.** The issue called this
  preferred. Each fixture exercises exactly one error path so
  a regression in attribution (e.g. a refactor that swaps the
  CD and LH guards) is loud rather than silent. The shared
  `write_dup_fixture(dup_in_cd, dup_in_lh)` keeps the script
  delta small (~85 LoC including helpers).
- **Gated on `sentinel == val32Max`.** The check fires only when
  `parseZip64Extra` would also fire — i.e. when at least one
  standard 32-bit field is the ZIP64 sentinel. This matches
  the existing call-site predicate and matches the issue's
  scope statement (the smuggling vector requires sentinel-set
  dependence). Duplicate `0x0001` blocks in the absence of any
  sentinel are still APPNOTE-violating but cannot affect
  resolved sizes/offsets in this codebase, so they're left to
  the unmodified behaviour pending a separate inventory item.
