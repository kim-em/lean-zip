# Meditate: WF recursion proof patterns from Track C2

**Issue**: #348
**Session**: 77e08376
**Date**: 2026-03-01

## Summary

Surveyed 20 progress entries (Feb 26 -- Mar 1) covering 16 merged PRs
to identify recurring proof patterns from the Track C2 fuel-to-WF
conversion work. Created a new skill and updated three existing skills.

## Changes Made

### New skill: `lean-wf-recursion`

Created `.claude/skills/lean-wf-recursion/SKILL.md` covering:

- **Unfolding WF functions**: `unfold f` vs `rw [f.eq_1]` vs `simp only [f]`
  (the latter is FORBIDDEN for WF functions -- loops). This was the single
  most frequently rediscovered mistake across progress entries (appeared
  in 3+ independent sessions).
- **Functional induction with `f.induct`**: Auto-generated induction
  principles matching a WF function's recursion structure. Was the biggest
  gap -- not documented anywhere despite being Lean 4's primary tool for
  reasoning about WF functions.
- **Termination measures**: Common patterns in this codebase (array size,
  list length, bit position arithmetic, bounded accumulator).
- **The `dataSize` parameter pattern**: Capturing `br.data.size` as an
  explicit parameter to avoid needing invariant lemmas in `omega`.
- **Dependent `if` guards**: `dif_pos`/`dif_neg` for resolving dependent
  `if` in proofs, `split at h` for case-analyzing guards in hypotheses.
- **`↓reduceIte` limitation**: Cannot reduce `if (false = true)` -- need
  explicit `if_neg` instead.
- **`do`/`guard` incompatibility**: Must use explicit `if`/`match` for WF
  compatibility.
- **Fuel-to-WF migration checklist**: Step-by-step guide for function and
  proof changes during conversion.

### Updated: `lean-fuel-induction`

Added cross-reference section pointing to the new `lean-wf-recursion`
skill for WF-specific patterns, keeping the fuel-induction skill focused
on fuel-based recursion.

### Updated: `lean-monad-proofs`

Added a prominent warning on the `dsimp only [bind, ...]` pattern,
noting it's the most commonly rediscovered pattern in the codebase
(appeared in 5+ progress entries independently).

### Updated: `lean-simp-tactics`

Added three new sections:
- `simp only []` for match reduction (empty argument list)
- Consolidated `↓reduceIte` decision guide table
- `decide` vs structural proofs for large finite types (avoids
  `maxRecDepth` requirements)

## Recurring Struggles Found

1. **`simp only [f]` on WF functions** (3+ rediscoveries): Agents try
   `simp only [wfFunction]`, Lean loops, then discover `unfold`. Now
   captured prominently in the new skill.

2. **`dsimp only [bind, ...]` vs `simp only [bind, ...]`** (5+ entries):
   The linter warning pattern. Now has a warning label in lean-monad-proofs.

3. **`↓reduceIte` with Bool-to-Prop** (2+ rediscoveries): `false = true`
   not reducible by `↓reduceIte`. Now in both lean-wf-recursion and
   lean-simp-tactics with a decision table.

4. **Race conditions on issue claims** (2 entries): Two agents independently
   reviewed the same file for issue #315. This is a coordination issue,
   not a skill gap.

## Impact

The 9+ remaining sorries from WF conversions (issues #342, #343, and
upcoming Deflate/DeflateSuffix/inflateLoop work) should benefit directly
from the new skill. The migration checklist and unfolding rules address
the most frequently encountered stumbling blocks.
