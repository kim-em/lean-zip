### A.2 Block geometry (1 536 B)

All four family fixtures are exactly **1 536 B** — 512 B of UStar
header + 1 024 B of zero padding (two trailing zero blocks forming a
well-formed end-of-archive). The zero-padding tail matches strict peer
parsers' (GNU tar / BSD tar / libarchive) accept-condition for
end-of-archive; the guard fires during header parse so the trailing
blocks are only exercised by the no-guard regression baseline.

The 1 536 B size is identical across the four sibling fixtures — the
only bytes that differ across the family are within the 512 B header
(the per-slot smuggled value plus the recomputed `chksum` octets).
This matches the `name` / `linkname` / `prefix` slot rows' geometry
exactly.

### A.3 Smuggled value choice: `uname = "trusted\x00rogue"` (12 B printable + interior NUL at byte 7)

Per the `buildUstarUnameNulInUname` helper added by PR #1944, the
fixture is built with `unameOverride := some "trusted\x00rogue"` so
the `uname` slot at offset 265 carries `"trusted\x00rogue"` — 12
meaningful bytes including the embedded NUL at byte 7 — while the
`name` slot at offset 0 carries `"safe"` (4 B printable, NUL-clean),
the `linkname` slot at offset 157 is all-NUL, and the `prefix` slot at
offset 345 is all-NUL. The 512 B header layout (only the differences
from the `prefix`-slot fixture's layout are noted; the rest is
identical):

| Slice                    | Offset       | Size  | Content |
|--------------------------|--------------|-------|---------|
| `name`                   | `0x000..0x063` | 100 B | `"safe"` (4 B printable) + 96 B NUL padding — **clean** |
| `mode` ... `mtime`       | `0x064..0x093` | 48 B  | octal numeric fields (writer-emitted) |
| `chksum`                 | `0x094..0x09B` | 8 B   | recomputed by writer to match the smuggled `uname` |
| `typeflag`               | `0x09C`        | 1 B   | `'0'` (REGTYPE — regular file) |
| `linkname`               | `0x09D..0x100` | 100 B | all-NUL — **clean** |
| `magic`                  | `0x101..0x106` | 6 B   | `"ustar\x00"` — UStar branch entered |
| `version`                | `0x107..0x108` | 2 B   | `"00"` |
| `uname`                  | `0x109..0x128` | 32 B  | **`"trusted\x00rogue"`** (12 B with interior NUL at byte 7) + 20 B NUL padding |
| `gname`                  | `0x129..0x148` | 32 B  | all-NUL — **clean** |
| `devmajor` ... `devminor`| `0x149..0x158` | 16 B  | NUL-padded (numeric fields) |
| `prefix`                 | `0x159..0x1F3` | 155 B | all-NUL — **clean** |
| `padding`                | `0x1F4..0x1FF` | 12 B  | NUL pad to 512 B block |

Header inspection confirms the four slot-pinning conditions in
source-position order (`name` → `linkname` → `prefix` → `uname`):

- **`name = "safe"` is NUL-free** — the `name`-arm sub-check at
  `Zip/Tar.lean:525` evaluates `false`, control passes to `linkname`.
- **`linkname` is all-NUL (clean)** — the `linkname`-arm sub-check at
  `Zip/Tar.lean:527` evaluates `false`, control passes to `prefix`.
- **`prefix` is all-NUL (clean)** — the `prefix`-arm sub-check at
  `Zip/Tar.lean:529` evaluates `false`, control passes to `uname`.
- **`uname = "trusted\x00rogue"` carries an interior NUL at byte 7**
  (between `'d'` and `'r'`, before the trailing NUL padding starts) —
  the `uname`-arm sub-check at `Zip/Tar.lean:531` evaluates `true`,
  the throw fires at `:532` with substring `"UStar uname contains NUL
  byte"`.

So the `uname` arm is the **uniquely** tripped sub-check for this
fixture — distinct from the three earlier arms by guard sequencing and
clean-slot construction. The choice of `"trusted\x00rogue"`:

- **12 printable-ASCII bytes** (`trusted`, `rogue` separated by an
  interior NUL) — symmetrically shaped to PR #1880's `name`-slot
  smuggle `"evil.txt\x00.tar"` (13 bytes; NUL at byte 8) /
  PR #1934's `linkname`-slot smuggle `"evil.lnk\x00.tar"` (13 bytes;
  NUL at byte 8) / PR #1937's `prefix`-slot smuggle
  `"badpfx\x00bad"` (10 bytes; NUL at byte 6). The interior NUL
  position is offset 7 inside the 12-byte payload — strictly
  *interior*, not a trailing padding NUL.
- **Encodes the trust-decision attack class in its content** — a
  caller routing on `entry.uname == "trusted"` (without parsing the
  full bytes) sees the truncated prefix while peer parsers preserve
  the full bytes; the contrast `trusted` → `rogue` makes the
  parser-differential threat model legible to a human reading the
  fixture.
- **Fits well within the 32-byte `uname` field** (12 of 32 bytes
  used; 20 B of NUL padding follows) — no field-overrun hazard. The
  `uname` slot is the smallest of the four guarded slots (32 B vs
  100 B for `name` / `linkname` and 155 B for `prefix`).
- **Distinct content from the three earlier slots' smuggles** — the
  `trusted`/`rogue` infix is recognisable in `xxd` diffs across the
  family and encodes the slot identity (matching the `evil.lnk`
  infix used by PR #1934's linkname slot and the `badpfx` infix used
  by PR #1937's prefix slot).

The smuggled value's `uname` slot location (offset 265 / 32 B) is the
4th-slot defense-in-depth extension; the 5-slot final closure waits
on the `gname` slot (offset 297 / 32 B, queued as unclaimed
issue #1946).

## B. Guard attribution

### B.1 4-slot guard enumeration (current master, post-#1944)

The 4-slot UStar interior-NUL guard at `Zip/Tar.lean:525-532`:

| # | Slot       | Guard line | Throw line | Field offset / width | Fixture PR | Reaches FS? |
|---|------------|------------|------------|----------------------|------------|-------------|
| 1 | `name`     | :525       | :526       | offset 0 / 100 B     | PR #1880   | yes         |
| 2 | `linkname` | :527       | :528       | offset 157 / 100 B   | PR #1934   | yes         |
| 3 | `prefix`   | :529       | :530       | offset 345 / 155 B   | PR #1937   | yes         |
| 4 | **`uname`** | **:531** | **:532**   | **offset 265 / 32 B**  | **PR #1944 (this review)** | **no — defense-in-depth** |

Each sub-check has the canonical form `if hasInteriorNul block
hdrSlot.1 hdrSlot.2 then throw (IO.userError "tar: UStar SLOT contains
NUL byte")`. The four throws share the prefix `"tar: UStar "` and
differ in the slot-name suffix
(`"name contains NUL byte"` /
`"linkname contains NUL byte"` /
`"prefix contains NUL byte"` /
`"uname contains NUL byte"`). Source-position order is `name` →
`linkname` → `prefix` → `uname`; a trip at the `uname` arm requires
all three earlier slots to be NUL-clean, which the fixture's
`name = "safe"` plus all-NUL `linkname` and `prefix` slots ensure.

The guards run **after** the checksum + magic checks (so header
integrity is confirmed first) and **before** any `Binary.readString`
call on the four string fields — the security-critical ordering.
Without the `uname`-arm guard, `Binary.readString` would silently
truncate the `uname` field at the NUL byte to `"trusted"`, and a
caller routing on `entry.uname` for a trust decision would see the
short prefix while strict peer parsers preserve the full bytes
(the parser-differential smuggle this fixture pins). Critically,
unlike the three earlier slots, the `uname` field does **not** flow
into `Tar.extract`'s filesystem operations — the guard is
defense-in-depth at the `Tar.list` layer.
