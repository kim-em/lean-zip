### B — Allocator-mismatch verification

The Rust shim allocates output via `compress_to_vec` /
`decompress_to_vec_with_limit`, normalises to `Box<[u8]>` via
`Vec::into_boxed_slice`, and returns the raw pointer + length through
`Box::into_raw` (Rust global allocator). `lean_miniz_oxide_free`
reconstructs `Box::from_raw(slice::from_raw_parts_mut(ptr, len))` so
the buffer is dropped through the Rust allocator — never libc `free`.

The Lean side at `c/miniz_oxide_ffi.c`:

- copies the bytes into a fresh `lean_alloc_sarray(1, out_len, out_len)`
  (Lean allocator) with `if (out_len > 0) memcpy(...)` so the empty
  case is well-defined,
- then calls `lean_miniz_oxide_free(out, out_len)` on every successful
  return path,
- never assigns the Rust pointer to anything Lean owns; the Lean
  `ByteArray` carries Lean-allocated bytes only.

`lean_alloc_sarray` panics on OOM rather than returning `NULL`, so
there is no failure window where the Rust buffer would leak between
allocation and free. The empty-output edge case (e.g. decompressing an
empty-data DEFLATE stream) returns a non-NULL but possibly-dangling
pointer from `Box::into_raw` for a `Box<[u8]>` of length 0; `Box::from_raw`
of a 0-length slice is well-defined and non-deallocating. Round-trip
is symmetric and exact.

### C — `panic = "abort"` invariant

`rust/miniz_oxide_shim/Cargo.toml` `[profile.release]` declares
`panic = "abort"` (alongside `lto = "thin"`, `codegen-units = 1`,
`opt-level = 3`). With abort-on-panic the Rust runtime cannot unwind
across the C ABI boundary. `src/lib.rs` declares
`#![deny(unsafe_op_in_unsafe_fn)]` so every unsafe op (slice from raw
parts, raw pointer writes) sits inside an explicit `unsafe { ... }`
block; the `unsafe extern "C"` entry points validate
`out_ptr.is_null()` / `out_len.is_null()` and the
`input.is_null() && input_len != 0` corner before constructing any
slice.

### D — Build-skip path

`lakefile.lean`'s `minizOxideEnabled` resolution order:

1. `MINIZ_OXIDE_DISABLE` set → `false` (opt-out).
2. `MINIZ_OXIDE_LDFLAGS` set → `true` (operator-supplied link).
3. Otherwise probe `cargo --version`; `true` iff exit 0.

Both the C-shim `-DHAVE_MINIZ_OXIDE` decision and the Rust-link
decision use the same probe, so they cannot drift apart.

When `HAVE_MINIZ_OXIDE` is undefined, both `*_ffi` entry points return
`mk_io_error(MINIZ_DISABLED_MSG)` containing the substring
`"miniz_oxide: not built with Rust support"`. The
`ZipTest/MinizOxide.lean` `withMiniz` helper matches that exact
substring and skips the body with a noisy console line, so CI on
minimal toolchains keeps passing. Verified locally on the cargo-enabled
toolchain by running `lake build` (199 jobs, no warnings) and
`lake exe test`; the test log shows `MinizOxide tests: OK`.
