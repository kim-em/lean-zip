# Inventory: TCB decision for miniz_oxide via Rust (closes #2362)

- Date: 2026-04-29T02:29Z
- Session: 7db2dbf3
- Issue: #2362 (audit follow-up from #2358; surfaced by PR #2356)

## Decision

**Option 1**: add a new `### miniz_oxide via Rust` subsection under
`## Trusted Computing Base`, status `guarded-locally`. The two
alternative options were (2) document under the existing
`### zlib via C FFI` row, and (3) leave out of scope as bench-only.

## Rationale

- **Public surface**: `MinizOxide.compress` / `MinizOxide.decompress`
  are public Lean APIs in `Zip/MinizOxide.lean`. Whether anyone wires
  them into a non-bench codepath today is a fact about the *current*
  call graph; the surface itself is part of the build's TCB whenever
  the comparator is enabled. Option 3 would create an undocumented
  invariant ("don't wire this into production") with no enforcement.
- **One-entry-per-FFI-helper precedent**: `### zlib via C FFI` is the
  existing structural sibling. Option 2 would mix two distinct trust
  domains (system zlib in C vs. an opt-in Rust crate via static lib)
  into one row, breaking the per-helper pattern.
- **Required Maintenance Rule alignment**: the rule already says
  *"whenever a new parser, extraction path, FFI helper, or streaming
  API is added, this file must be updated."* The `miniz_oxide_shim`
  is a new FFI helper; recording it now is exactly what the rule
  asks for, and avoids a special carve-out paragraph.
- **Status `guarded-locally`** matches the actual posture: opt-in
  build with stub-fallback path, `maxDecompressedSize` cap on
  decompress (default 1 GiB), `panic = "abort"` Rust shim, explicit
  `lean_miniz_oxide_free` finalizer to avoid Rust/libc allocator
  mismatch.

## Sanitizer / fuzz scope (per issue body's requirement)

The new subsection states explicitly: **the existing
`scripts/sanitize-ffi.sh` and `scripts/fuzz-inflate.sh` recipes do
not exercise `c/miniz_oxide_ffi.c` or `libminiz_oxide_shim.a`.** A
sibling recipe is needed before the public API leaves bench-only
scope; the subsection sketches the shape (Rust nightly +
`RUSTFLAGS="-Zsanitizer=address"` + link the sanitised static lib
into the existing fuzz-inflate driver). This is recorded as Missing
work; it is not blocking for the bench-only callers.

## Other Missing-work items recorded

- No upstream-tracking entry pinning the `miniz_oxide` v0.8 crate
  against a known-good audit (consumed transitively through `cargo`).
- `MinizOxide.compress`'s level argument is documented as 0–9 but
  not clamped at the Lean wrapper or in the C/Rust shim; the
  subsection notes this gap and recommends caller-side clamping for
  any non-bench use.
- A single trip-wire bullet: *"if a downstream caller wires these
  APIs into a non-bench codepath, the `guarded-locally` status must
  be re-evaluated alongside the sibling fuzz / sanitizer recipe."*

## Edits applied in this PR

## Items deferred (no new issues filed)

The Missing-work bullets in the new subsection are statements of
gap, not pre-filed issues. None of them are blocking for the
current bench-only scope; they will become actionable if and when a
downstream caller takes `MinizOxide.compress` /
`MinizOxide.decompress` into a non-bench codepath. Filing speculative
fuzz / clamp issues today would be over-eager — they are recorded
in the inventory as the canonical place for "known gap, not yet
scheduled."
