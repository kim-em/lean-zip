## Diagnosis correction (important for the next session)

The issue hypothesized the bottleneck was `ByteArray.push` "one byte at a
time" per bit. **That was wrong**: the old `writeHuffCode.go`/`writeBits.go`
already flushed *per byte* (pushed only when `bitCount+1 ≥ 8`). The real
cost is the **per-bit loop iteration** itself. A first attempt that merely
batched the byte flush but kept a per-bit reversal loop (`revBits`) gave
only **~1.05×** — confirming the iteration count, not pushes, dominates.

The win comes from making `writeHuffCode` **loop-free**: reverse the code's
16 bits with a branchless swap network (`reverse16`) + one down-shift,
instead of a `len`-iteration reversal. `writeHuffCode` is the hot path for
incompressible input (all literals). Result: **prng compress ~1.31×**
(1 MiB level 6 16.7 → 22.0 MB/s).

This is **short of the issue's ≈1.8× target**. Emit is now ~2× faster but is
only ~58% of the prng level-6 pass, so ~1.3× end-to-end is the ceiling from
emit alone. Reaching 1.8× needs the matcher constant-factor work too
(reframed D-3 follow-up). The List↔Array conversions originally tagged for
D-6 were confirmed negligible and removed from the backlog.

## Key proof patterns

- **`flushAcc_spec`** (strong induction on `total` via `Nat.strongRecOn`,
  *not* `flushAcc.induct` whose `if`-generated binders are awkward):
  flushing a `UInt64` holding `total` valid low bits yields
  `(List.range total).map (testBit acc)` and stays `wf`, given
  `acc.toNat < 2^total`. Bytes split off the low 8 bits each step
  (`Nat.testBit_div_two_pow`, `testBit_mod_two_pow`).
- **`reverse16_testBit`**: the swap network flips bit `j` ↔ `15-j`.
  `interval_cases` is Mathlib (unavailable) — enumerate `j ∈ 0..15` with an
  explicit `match j, hj with | 0,_ | … | 15,_ => bv_decide | _+16, h => omega`.
  Bridge `Nat.testBit (UInt16.toNat x) j = x.toBitVec.getLsbD j` via
  `BitVec.testBit_toNat` (rfl) before `bv_decide`.
- **`writeBits_spec`/`writeHuffCode_spec`**: characterize the accumulator
  `bitBuf ||| field <<< bitCount` with `testBit_or_shiftLeft_below` (low
  bits) and a new `testBit_or_shiftLeft_above` (high bits), then a
  `range_map_split` at the `bitCount`/field boundary aligns with
  `bw.toBits ++ writeBitsLSB`/`natToBits`.
- All the `UInt64.toNat_{and,or,shiftLeft,shiftRight,mod,toUInt8}` lemmas
  are `@[simp]`; `norm_num`/`interval_cases` are NOT available here.
- `writeBits` masks with `% (1 <<< n)` rather than `&&& ((1<<<n)-1)` — the
  modulo bridges cleanly via `toNat_mod`; it is a cold path (extra bits /
  headers), so the division cost is irrelevant.
