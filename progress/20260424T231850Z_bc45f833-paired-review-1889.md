## A. Guard summary

Existing guard (landed by PR #1761, no change in this PR), at
[`Zip/Archive.lean:343-345`](/home/kim/lean-zip/Zip/Archive.lean:343):

```lean
let eocd64RecordSize := Binary.readUInt64LE data (bufPos + 4)
unless eocd64RecordSize == 44 do
  throw (IO.userError
    s!"zip: ZIP64 EOCD64 record-size mismatch (size={eocd64RecordSize}, expected 44 for v1 EOCD64)")
```

Strict equality on `44` — the APPNOTE §4.3.14 formula value
`SizeOfFixedFields + SizeOfVariableData - 12 = 56 - 12 = 44`, not the
total 56-byte record length. The error substring
`"ZIP64 EOCD64 record-size mismatch"` carries both the attacker-
supplied actual value (`{eocd64RecordSize}`) and the expected `44` in
the s-string, so a reader of the test log can reconstruct which shape
was smuggled without additional inspection.

Issue body cites the throw at `:345` — confirmed post-grep. Anchor
drift: paired-review-#1761 recorded the throw at `:319`; current master
has shifted it by +26 lines (PR #1856 archive-layout invariant at

## B. Guard ordering (ZIP64-specific consistency sub-layer)

The `findEndOfCentralDir` ZIP64 branch now hosts five structurally
related checks (extending paired-review-#1761 §C's two-member table).
Ordering relative to `zip64-eocd64-v2-record.zip` at the trip point:

| # | Layer | Check | Throws at | Gate | Direction |
|---|-------|-------|-----------|------|-----------|
| 1 | Layout | EOCD64 record overlaps locator (`bufPos + 56 ≤ pos - 20`) | `Zip/Archive.lean:331` | `findEndOfCentralDir` ZIP64 branch | inequality |
| 2 | **Framing** | **EOCD64 record-size (v1 == 44)** | **`Zip/Archive.lean:345`** | **same** | **equality to 44** |
| 3 | Spec-version | EOCD64 `versionMadeBy` low byte ≤ 63 | `Zip/Archive.lean:363` | same | upper bound |
| 4 | Spec-version | EOCD64 `versionNeededToExtract` ≤ 63 | `Zip/Archive.lean:380` | same | upper bound |
| 5 | Override sentinel | 6-field override-vs-standard | `Zip/Archive.lean:398/401/404/407/410/413` | same | sentinel ∨ numeric match |

Fixture-specific ordering empirics: a reader walks the guards in the
listed order before tripping on guard #2.

- **Layout invariant (PR #1856) clears.** EOCD64 `bufPos = 100`,
  Locator at `pos - 20 = 172`: `100 + 56 = 156 ≤ 172` ✓. The layout
  guard is aware only of the v1 56-byte layout, not the 72-byte
  physical record; it asserts the record's nominal v1 footprint fits
  before the Locator. A v2 record whose physical footprint extends
  further (72 bytes here) can still clear the layout guard if the
  Locator is placed past the v2 tail — the fixture does exactly that,
  placing the Locator at `bufPos + 72 = 172`. If the v2 fixture had
  instead placed the Locator at `bufPos + 56` (contiguous with the v1
  nominal end but overlapping the v2 extensible sector), the layout
  guard would have fired instead of the record-size guard. The fixture
  is thus **correctly scoped** to the record-size arm rather than the
  layout arm.
- **Framing guard (PR #1761) trips.** `eocd64RecordSize = 60`,
  expected `44`. `unless 60 == 44 do throw …` — guard rejects with
  the `{eocd64RecordSize=60, expected=44}` interpolation.
- Guards #3-#5 are unreachable on this fixture (control-flow short-
  circuits on the `throw`). Hypothetically, if the guard-2 trip were
  removed, guards #3/#4 would pass (versionMadeBy low-byte `45 ≤ 63`,
  versionNeededToExtract `45 ≤ 63`) and the fixture would fall through
  to guard #5 — where the override sentinels also pass (cdSize = 55,
  cdOffset = 45, numEntries = 1 all match between standard EOCD and
  v1 EOCD64 body). The fixture is thus **sanity-correct on every axis
  except `recSize`**: any further hardening of the ZIP64 sub-layer
  would leave this fixture's attribution on `recSize`.

The ordering pattern (layout → framing → spec-version → override
sentinel) matches the general "validate frame shape before trusting
frame contents" invariant flagged by paired-review-#1761 §C — the
record-size guard validates the *frame* before any of the *fields
inside the frame* are read, so a v2-shape smuggle cannot execute its
per-field offsets.

### Minimality

The 214-byte size is near the theoretical floor for a v2-shape ZIP64
archive: 30 B LH + 9 B name + 6 B payload + 46 B CD base + 9 B CD name
+ 4 B signature + 8 B recSize + 44 B v1 body + 16 B extensible sector
(the minimum SES-shape width) + 20 B Locator + 22 B EOCD = 214 B. The
extensible sector could in principle shrink to 12 B (one
`(Header ID = 0x0000, Data Size = 4) → 4 zero bytes` record — 12 bytes
total), which would give `recSize = 56 + 12 − 12 = 56` and a 210 B
archive. The chosen 16 B width matches the APPNOTE-documented
minimal SES shape with `compositeSize` (8 B) + `encryptionAlgID`
(2 B) + 6 B of header-field slack, so the size choice is
APPNOTE-semantics-faithful rather than minimally-minimal. Justified in
the fixture docstring.
