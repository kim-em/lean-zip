## A. Guard correctness

- **A.1 Placement.** The three `hasInteriorNul` calls live in
  `parseHeader` at
  [Zip/Tar.lean:496-501](/home/kim/lean-zip/Zip/Tar.lean:496). The
  surrounding control flow runs **after** (a) the 512-byte block-
  size gate on line 454, (b) the all-zero end-of-archive detector on
  lines 457-466 (so NUL guards don't fire on legitimate end-of-
  archive blocks), (c) the checksum verification on lines 468-469
  (`throw "tar: header checksum mismatch"`), and (d) the UStar magic
  validation on lines 471-473 (`throw s!"tar: unsupported format
  (magic: {magic})"`); and **before** the `Binary.readString` calls
  that populate the returned `Entry` at lines 502 (`name`), 503
  (`prefix`), and 513 (`linkname`). Header integrity is therefore
  confirmed first — a garbage block whose `name` slot happens to
  contain a NUL would already fail the checksum or magic check, so
  the NUL guard never produces a false-positive error attribution
  on non-tar input. Simultaneously, the `Binary.readString` NUL-
  terminator truncation path cannot silently collapse a smuggled
  `"evil.txt\x00.tar"` payload to `"evil.txt"` before the guard
  runs. The 17-line comment block at lines 474-495 documenting the
  placement invariant and writer-side non-emission is consistent
  with the sibling guards' style (PR #1865 GNU and PR #1866 PAX).

- **A.2 Predicate correctness.** `hasInteriorNul` at
  [Zip/Tar.lean:442-449](/home/kim/lean-zip/Zip/Tar.lean:442) walks
  the `(offset, offset+len)` slice, setting `sawNul` on encountering
  `0x00` and returning `true` as soon as any non-NUL byte is seen
  *after* a NUL. Docstring on lines 437-441 captures intent:
  *"a NUL byte (0x00) followed somewhere later in the field by any
  non-NUL byte"*. Edge cases:
    - *Short name* (`"evil.txt\x00"` + 91 NUL padding): `sawNul`
      flips on byte 8; all subsequent bytes are NUL. Returns
      `false`. ✓
    - *Long name* (100 non-NUL bytes): `sawNul` never flips.
      Returns `false`. ✓
    - *All-NUL slice* (unused linkname/prefix shape): `sawNul`
      flips on byte 0; all subsequent bytes are NUL. Returns
      `false`. ✓
    - *Smuggled interior NUL* (`"evil.txt\x00.tar"` + padding):
      `sawNul` flips on byte 8; byte 9 is `0x2E` (`.`), non-NUL.
      Returns `true`. ✓
  One minor wording gap: issue #1883's A.2 says *"there exists a
  NUL byte 0x00 at any position strictly before the last position
  within the (offset, len) slice"* — strictly that would reject
  `"...\x00\x00"` (NUL at len-2 followed by NUL at len-1), whereas
  the landed helper accepts it. The landed "NUL followed by non-
  NUL" semantics match the docstring and are the correct definition
  of "interior NUL"; the issue wording was an over-approximation.
  Not a bug; flagged for audit traceability.

- **A.3 Error wording.** The three `throw (IO.userError "...")`
  calls on lines 497, 499, 501 carry exactly:
    - *"tar: UStar name contains NUL byte"*
    - *"tar: UStar linkname contains NUL byte"*
    - *"tar: UStar prefix contains NUL byte"*
  Distinct field attribution is preserved — a fused substring
  (`"tar: UStar header contains NUL byte"`) would have broken
  per-field attribution for paired-review, inventory links, and
  the `error-wording-catalogue` skill. The `"tar: "` namespacing
  prefix matches the existing Tar error style (cf. `"tar: header
  checksum mismatch"`, `"tar: unsupported format (magic: ...)"`,
  `"tar: path too long: ..."`).

      attribution.

    - Bytes 157..256: linkname — all NUL (unused field). ✓
    - Bytes 257..262: magic `"ustar\x00"`. Version at 263..264 =
      `"00"`. ✓

  a silent-skip branch for invalid-UTF-8, and the PR #1866 author
  chose to extend that skip branch rather than fork off a throw
  branch (documented in the PR #1866 merge commit message). All

## F. Writer-side invariant

- **F.1 Writer non-emission.** `buildHeader` at
  [Zip/Tar.lean:341-376](/home/kim/lean-zip/Zip/Tar.lean:341)
  populates the three UStar string fields via `Binary.writeString`:
    - Line 351: `hdr := writeField hdr hdrName.1 (Binary.writeString
      name hdrName.2)`
    - Line 363: `hdr := writeField hdr hdrLinkname.1
      (Binary.writeString entry.linkname hdrLinkname.2)`
    - Line 371: `hdr := writeField hdr hdrPrefix.1
      (Binary.writeString pfx hdrPrefix.2)`
  `Binary.writeString` (from `lean-zip-common`) copies UTF-8 bytes
  verbatim and NUL-pads the tail to the fixed field length — it
  does not filter or reject interior NULs. Net: if a caller hands
  `lean-zip`'s public `Tar.create` API a `String` containing
  `\x00`, the writer emits an archive the reader now rejects. The
  invariant is caller-held, matching the sibling writer-side
  policies at `Zip/Archive.lean:84` / `:110` (ZIP CD name-writer
  defers to caller `entry.path` sanitation). The new NUL guard is
  **read-side hardening only**; it does not fire on legitimate
  archives produced by `Tar.create` unless the caller planted the
  NUL in `entry.path` / `entry.linkname` themselves.

- **F.2 Interop corpus pre-flight.** Swept
  `testdata/tar/{interop,malformed,security}/*.tar` with a Python
  script that walks each archive block-by-block, checks for the
  UStar magic at offset 257, and scans the three string fields for
  interior NULs (NUL byte followed by a non-NUL byte later in the
  field). Only hit: `testdata/tar/malformed/ustar-name-nul-in-name.tar`
  at `block_offset=0` in the `name` field — the new fixture
  itself. Zero pre-existing tar fixtures carry interior NULs in
  UStar string slots, confirming the PR description's pre-flight
  claim and guaranteeing the new guard does not surface as a
  regression against any other fixture in the corpus.
