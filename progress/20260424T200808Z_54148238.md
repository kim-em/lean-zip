## Summary

Closes the NUL-byte smuggle vector at the Tar PAX extended-header
layer. `parsePaxRecords` in `Zip/Tar.lean` now drops any PAX record
whose raw `keyBytes` or `valueBytes` contains a NUL (0x00) in addition
to dropping records whose UTF-8 decode fails. The guard matches the
existing silent-skip precedent one line above — no new `IO` context is
introduced in `parsePaxRecords`.

Sibling to:
- ZIP CD entry name NUL guard (PR #1831, `cd-nul-in-name.zip`) — ZIP side.
- Tar GNU long-name / long-link NUL guard (PR #1865, in flight) — GNU side.

Together the trio closes the "smuggled NUL in any user-supplied
string field" attack class across ZIP and Tar.

## Decisions

- **Guard on raw bytes, not decoded strings.** The plan suggested
  `key.contains '\x00' || value.contains '\x00'` after the UTF-8
  decode. I used `(keyBytes.findIdx? (· == 0)).isNone &&
  (valueBytes.findIdx? (· == 0)).isNone` on the raw `ByteArray`
  instead. Rationale:
  1. Matches the sibling ZIP CD name NUL guard at
     `Zip/Archive.lean:630`, which also scans the raw
     `ByteArray` before decoding.
  2. Slightly cheaper (no UTF-8 traversal).
  3. Defense in depth — the check is correct even if a later
     refactor changed how the decoded `String` is represented.

- **Skip, not throw.** Chose the skip variant from the plan
  (the record is silently dropped). `parsePaxRecords` remains
  pure `Id.run do` with no `IO` lift; the follow-on
  regular-file entry's declared path wins. The test assertion
  is *positive regression* — like `hardlink-outside.tar` — rather
  than an `assertThrows`.
