## TODO-flagged residual warnings (3)

All three live in the *Decompression Limit Inventory* table's
"Bomb-limit regression test at [ZipTest/...]" column:

- Line 355 — `[ZipTest/Zlib.lean:17-22]`
- Line 356 — `[ZipTest/Gzip.lean:18-23]`
- Line 357 — `[ZipTest/RawDeflate.lean:17-22]`

Root cause: the drift detector extracts quotes from the full inventory
line, which in these rows includes declaration-style quotes like
`` `maxDecompressedSize : UInt64` `` and `` `1073741824` ``. Those
quotes refer to the `Zip/Basic.lean:15` API declaration on the same
row and pass the drift check against that file. But the script then
checks the *same* quotes against the `ZipTest/*.lean` cite, where
they never appear — test files use the keyword-argument form
(`maxDecompressedSize := 10`) and numeric expressions
(`1024 * 1024 * 1024`), not the typed declaration literal.

No line number in any of the three test files would satisfy the
quote check, so the cites are TODO-flagged with an inline comment
explaining that the ZipTest anchors are correct and the heuristic
mismatch is structural.

A fourth TODO on line 564 (`Tar.extract` regular-file short-read)
documents an inventory quote that captured an older metavariable
name (`{path}/{n}`) while the current source uses `{e.path}` and
`{remaining}`. The comment's explanatory backtick-spans
coincidentally include `{e.path}` and `{remaining}`, which appear
at line 686 of `Zip/Tar.lean` and make the detector pass — so this
row is *not* counted toward the residual-warning total. The TODO
still stands as a breadcrumb for a future quote refresh.

## Mechanical approach

1. Parsed `./scripts/check-inventory-links.sh` output with a small
   helper (`/tmp/resolve-drift.py`, not committed) that extracted
   `(inv_line, file, old_anchor, quotes)` tuples and ranked quote
   matches by (longest-first, nearest-to-old).
2. Reviewed every machine-pick against the inventory's semantic
   intent; for cases where the longest quote led to a semantically
   wrong line (e.g. `assertSpanInFile` matching a proof-internal
   occurrence), manually redirected to the `def` line where possible.
3. Applied 61 per-inventory-line edits via a second helper
   (`/tmp/apply-fixes.py`, not committed) that performed
   `line.replace(pattern, replacement)` for each `(inv_line, file,
   old)` tuple — safer than global `sed` because the same
   `(file:old)` token can appear on multiple inventory lines with
   different intents (e.g. `Zip/Tar.lean:610` on rows 371, 485, 486).
4. Iterated once: caught that `Zip/Archive.lean:388` and `:391`
   landed in swapped columns (line 478), swapped them via a direct
   `Edit`, re-ran the detector.
