## Findings

### Actionable (landed in this PR)

**`Adler32.Spec.checksum_singleton` proof tightened (-4 net LOC).**
The original proof body (14 lines) interleaved bound facts with
rewrites via `(show … by omega)` arguments to `pack_toNat_of_bounds`
and `UInt32.toNat_ofNat_of_lt'`. Refactor:

1. Hoist three supporting facts as named `have` bindings up-front:
   - `hb : 1 + b.toNat < 65536`
   - `ha : (1 + b.toNat) % 65521 = 1 + b.toNat`
   - `hbnd : (1 + b.toNat) * 65537 < UInt32.size`
2. Collapse the `rw [ha, Nat.zero_add, ha, pack_toNat_of_bounds …, UInt32.toNat_ofNat_of_lt' …]`
   chain into the preceding `simp only` set, with the named hypotheses
   passed inline.
3. Drop the intermediate `(1 + b.toNat) * 65537 ≤ 257 * 65537` step in
   the `hbnd` proof — `omega` handles the constant-multiplier linear
   bound directly given `b.toNat < 256`.

Body now reads as: collect facts → bridge to Nat → one simp call →
`omega`. Same template the `_append` proofs use (per #1650's review).

### Reviewed, no change

**`pack_toNat_of_bounds` visibility / naming.** Currently `private` in
`Adler32.Spec`. Has three callers, all in the same file
(`pack_unpack`, `unpack_pack_of_valid`, `checksum_singleton`). The
issue flagged this as a possible promotion candidate but no
out-of-file consumer exists — `private` correctly scopes it. Name
follows the project's `<op>_<view>_of_<precondition>` convention
(parallel to `Nat.mod_eq_of_lt`); calling it a "pack-Nat-view" lemma
would lose the `_of_bounds` precondition signal. Keep as-is. The
authoring session (`4ba91ca3`'s progress entry) already arrived at the
same conclusion.

**`unpack_pack_of_valid` and `pack_unpack`.** Pre-existing (#1641 /
earlier). Their bodies are not changed by #1661 and the `pack_toNat_of_bounds`
lemma they consume was introduced *with them*, not by #1661 — so
there is no "now subsumed by #1661" simplification to harvest.

**`@[simp]` annotations.** All four `_empty` theorems are `@[simp]`
(trivial rewrite targets). Both `_singleton` theorems are
deliberately *not* `@[simp]` (closed forms, not normal forms).
Distinction matches the `_append` precedent and is correct.

**Native bridging.** `adler32_empty` reduces via `Spec.pack_unpack`,
`adler32_singleton` bridges through `updateBytes_eq_updateList` to
`Spec.checksum_singleton`, and `crc32_empty` collapses via
`bv_decide` after the foldl-over-empty reduction. Each is a tight
`simp only` + one finishing tactic. No redundant intermediate
rewrites.

**Docstrings.** The Spec `_empty` docstrings are deliberately terse
(one line each: "*The CRC-32/Adler-32 checksum of the empty input is
0/1.*"). Adding the meditate #1651 §5 three-part structure (*what →
why → composes*) to a base case feels like restating the definition
— the "why" reduces to "by definition of `init` and `pack`". The
fuller-form `_append` and `_singleton` docstrings already use the
three-part structure where the explanation has substance. No change.

**Test surface (`ZipTest/NativeChecksum.lean`).** Two pre-existing
runtime literal-comparison `assert!`s (`Empty Adler32` lines 36–37,
`Empty CRC32` lines 67–68) overlap with the new `_empty` theorems but
are *not* `example` stanzas — they're `IO Unit` smoke checks against
a literal, surviving from before #1650. PR #1650's review explicitly
distinguished `example` stanzas (removed) from runtime conformance
tests (retained). Following that precedent, leave alone. The PR's
single-byte runtime checks (lines 39–43, 70–73) compare native vs FFI
and are *not* subsumed by `_singleton` (which proves a closed form,
not FFI agreement).

**CRC32 single-byte intentionally absent.** Confirmed correct per the
"Specifications" §1 antipattern in `.claude/CLAUDE.md` — the only
"closed form" available restates the implementation
(`crcByte 0xFFFFFFFF b ^^^ 0xFFFFFFFF`), so it would be tautological.
