## Accomplished

1. **Block size validation in `decompressBlocks`** (`Zip/Native/ZstdFrame.lean`):
   - Added `windowSize` parameter to `decompressBlocks`
   - After parsing each block header, validates `blockSize ≤ 131072` (128KB, RFC 8878 §3.1.1.2)
   - Validates `blockSize ≤ windowSize` when window size is available
   - Updated `decompressFrame` to pass frame header's `windowSize` through

2. **Window size enforcement in `executeSequences`** (`Zip/Native/ZstdSequence.lean`):
   - Added `windowSize` parameter to `executeSequences`
   - After resolving each sequence offset, validates `offset ≤ windowSize` when set
   - Existing `offset > output.size` check remains as the primary bounds check

3. **Validation tests** (`ZipTest/ZstdNative.lean`):
   - Oversized block (>128KB) is rejected with clear error
   - Block at exactly 128KB passes size validation (boundary check)
   - Block exceeding window size is rejected
   - Sequence offset exceeding window size is rejected
   - Legitimate FFI-compressed data still decompresses (no false positives)
   - Added `set_option maxRecDepth 2048` to handle growing test file

## Decisions

- Block size and window size validation are independent checks (both can trigger).
  The block size check fires first since it's in `decompressBlocks` before any
  sequence execution.
- Window size of 0 (default) skips the window-based checks, so callers that
  don't pass a window size maintain backward compatibility.
