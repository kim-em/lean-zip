The guard at
[Zip/Archive.lean:306-331](/home/kim/lean-zip/Zip/Archive.lean:306):

```lean

```

Point-by-point verification of the A.1 sub-items the issue body
raised:

1. **`pos ≥ 20` established by the outer guard**: `:298` —
   `if pos >= 20 then` encloses every subsequent read of `pos -
   20` and `pos - 12`, so the `Nat` subtraction in the unless
   guard's RHS (`pos - 20`) is well-defined. The `pos - 12` read
   for `eocd64Offset` is also under this guard.
2. **Well-defined `Nat` subtraction in `pos - 20`**: with
   `pos ≥ 20` as invariant, `pos - 20` equals `pos - 20` in
   integer arithmetic (no truncation). The LHS `bufPos + 56` is
   unconditionally a `Nat` sum — no truncation hazard there either.
   The guard compares `bufPos + 56 ≤ pos - 20`, equivalently
   `eocd64Offset + 56 - baseOffset ≤ pos - 20`, equivalently (after
   adding `baseOffset`) `eocd64Offset + 56 ≤ (baseOffset + pos) -
   20 = locatorPos`. Correct.
3. **The literal `56` is the EOCD64 v1 record size**: APPNOTE
   §4.3.14 specifies the v1 EOCD64 layout as a 56-byte fixed block
   (signature 4 + `size` 8 + versionMadeBy 2 + versionNeeded 2 +
   disk# 4 + diskWithCD 4 + entriesOnDisk 8 + totalEntries 8 +
   cdSize 8 + cdOffset 8 = 56). The adjacent record-size guard at
   `:343-345` confirms the self-declared `size` field equals `44`
   (56 − 12, per APPNOTE's "not counting signature + size field"
   convention). Both guards agree: lean-zip's consumer assumes the
   56-byte fixed layout.
4. **`eocd64Offset` decoded before the guard**: `:300` —
   `Binary.readUInt64LE data (pos - 12)` reads the Locator's +8
   `zip64EndOfCentralDirOffset` field (Locator layout: sig 4 +
   diskWithEOCD64 4 + offset 8 + totalDisks 4 = 20 bytes, so the
   offset field is at Locator+8 ≡ `(pos - 20) + 8 = pos - 12`).
   The decode precedes the guard site `:329`. The file-absolute
   `eocd64Offset` is then converted to buffer-relative `bufPos`
   at `:303` before the comparison.

Additional observation (not in the issue body but worth noting):
the `sigEOCD64` match at `:305` runs BEFORE the layout guard at
`:329`. This is correct precedence — if the buffer position
doesn't actually carry an EOCD64 signature, the Locator is
malformed in a different way (the EOCD64 offset is wrong), and
the outer fall-through (`else: ignore Locator, treat as
non-ZIP64`) is the right policy. The layout guard only fires for
archives that otherwise *look* like valid ZIP64.

- `(a)` fires on the buffer-relative layout equality

### E.1 Trio closure at ZIP's three record boundaries — OK

With PR #1856 landing, the ZIP archive-layout invariant trio is
complete:

| Granularity | PR | Invariant | Error substring |
|-------------|----|-----------|-----------------|
| Per-entry micro | #1813 | `localOffset + 30 ≤ cdOffset` | `"entry local offset overlaps central directory"` |
| ZIP64-trailer micro | **#1856** | `eocd64Offset + 56 ≤ locatorPos` | `"ZIP64 EOCD64 record overlaps locator"` |
| Archive-level macro | #1809 (in-flight) | `cdOffset + cdSize ≤ eocdPos` | `"central directory extends beyond EOCD"` (approximate, per PR #1809 draft) |

All three invariants reject "smuggled record overlap" attacks at
distinct ZIP record boundaries:

- **Per-entry (#1813)** closes the smuggling vector where a
  crafted CD entry's `localOffset` points into the CD region,
  aliasing the LH over the CD header bytes.
- **ZIP64-trailer (#1856, this PR)** closes the smuggling vector
  where a crafted ZIP64 Locator's `eocd64Offset` points into or
  past the Locator itself, aliasing the EOCD64 over the Locator
  bytes.

### E.2 Parser-differential vector — OK

The fixture exploits a classic parser-differential vector:

- **Strict readers** that perform the layout check (CPython
  `zipfile._EndRecData64`, now lean-zip ≥ #1856) reject the
  crafted fixture with an error substring that mentions the
  Locator–EOCD64 overlap.
- **Lenient readers** that skip the layout check (pre-#1856
  lean-zip, and plausibly some older implementations that
  predate the APPNOTE §4.3.6 pinning) accept the archive and
  read the trailing 8 bytes of the "EOCD64" from the Locator's
  first 8 bytes — conflating the Locator's `sigLocator64`
  (`0x07064b50`) + `diskWithEOCD64` UInt32 as the tail of an
  attacker-chosen EOCD64 structure.

Concretely: the 56-byte EOCD64 record layout maps the last 8
bytes to `cdOffset` (APPNOTE §4.3.14 +48 UInt64 field). When the
record overlaps the Locator by 8 bytes, those last 8 bytes of
the claimed EOCD64 correspond to the Locator's `sigLocator64`
(4 B) + `diskWithEOCD64` (4 B). A lenient reader that doesn't
check the layout reads those 8 bytes as `cdOffset`, deriving
`cdOffset ≈ 0x00000000_07064b50 = 118,055,760`, a
multi-gigabyte-looking offset that steers CD resolution to a
region entirely distinct from what the attacker controls
legitimately.

CPython's check-form `if reloff > offset: raise BadZipFile(...)`
transcribes to `if eocd64Offset > locatorPos - 56: raise ...`,
equivalently `eocd64Offset + 56 > locatorPos`, equivalently the
negation of the lean-zip invariant. The cited prose is accurate.
