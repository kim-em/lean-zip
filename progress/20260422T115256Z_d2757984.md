### Drift fixed: streaming-FFI catch shape

`#1647` narrowed `tryFFI`'s catch from blanket `_` to
`match e with | .userError _ => pure () | _ => throw e`, so panics
and runtime-level errors propagate instead of being silently
swallowed. The `tryStreaming` (low-level `InflateState.push/finish`)
probe was *deliberately* left untouched in `#1647` per its
scope-discipline list, and `#1653` then matched that blanket-catch
shape when adding `tryStreamingGzip` and `tryStreamingRawDeflate`.

The audit's directive — "`tryFFI`'s narrowed exception variants
apply to the three streaming probes as well" — explicitly broadens
the narrowing to all three streaming probes. Applied: all three
probes now follow the `.userError`-only catch with re-raise on
other variants. The 1000-iteration smoke still passes; no new IO
variants leak.

This is the principled answer because every streaming probe hits
the same FFI surface (`lean_gzip_inflate_push` /
`Gzip.decompressStream` / `RawDeflate.decompressStream`), which
emits only `.userError` (via `lean_mk_io_user_error`). Sanitizer
traps and panics are process-level and bypass IO catches anyway,
so narrowing strictly improves the fuzz harness's signal: a
spurious `.alreadyExists` or `.invalidArgument` reaching one of
these probes would now fail the run loudly instead of being
swallowed.

### PRNG threading

`oneIteration` (`ZipTest/FuzzInflate.lean:204-233`) threads state
contiguously: `s0 = xorshift state` → `s1` from `genBytes` → `s2`
from `splitIntoChunks` → `s3 = xorshift s2` → `pick maxClasses s3`
→ return `s3`. No probe re-seeds from a constant. Reproducibility
preserved.

### Smoke N consistency

All probes share the same iteration count (`runFuzz seed 1000`).
`#1647` raised the smoke from 100 → 1000; `#1653`'s new probes
inherit this without asymmetry. `runFuzzUntil` (used by the
wall-clock binary at `ZipFuzzInflate.lean`) is unbounded by class.

vs. `RawDeflate.decompressStream`). Below the 10-line threshold the
