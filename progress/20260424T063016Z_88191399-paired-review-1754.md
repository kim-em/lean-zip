## B. Interop smoke test (the critical regression)

The "sentinel ∨ numeric match" relaxation was introduced empirically
from `go-zip64.zip`; without the relaxation, Go-produced ZIP64
archives would be rejected. Tabulation below parses the six interop
fixtures directly via Python (`/tmp/paired-review-1754/interop-check.py`):
walk the tail for the EOCD signature, read the six standard-EOCD
fields at `pos+4/+6/+8/+10/+12/+16`, check for a Locator 20 bytes
before, and if present read the six ZIP64 EOCD64 fields at
`bufPos+16/+20/+24/+32/+40/+48`.

| Fixture                          | ZIP64 present | Sentinels | Numeric matches | Mismatches |
|---|---|---|---|---|
| `go-crc32-not-streamed.zip`      | no            | —           | all 6 N/A (no override) | 0 |
| `go-test.zip`                    | no            | —           | all 6 N/A (no override) | 0 |
| `go-unix.zip`                    | no            | —           | all 6 N/A (no override) | 0 |
| **`go-zip64.zip`** (sharp edge)  | **yes**       | 4 (cdSize, cdOffset, totalEntries, numEntriesThisDisk) | **2 (numberOfThisDisk, diskWhereCDStarts — both `0 == 0`)** | 0 |
| `latin1-name.zip`                | no            | —           | all 6 N/A (no override) | 0 |
| `utf8-flag.zip`                  | no            | —           | all 6 N/A (no override) | 0 |

Five of six interop fixtures carry no ZIP64 EOCD64 block, so the
sentinel check is structurally skipped (the guards live inside the
locator-success + EOCD64-signature branch at
`Zip/Archive.lean:304–305`). The **load-bearing** sixth fixture
`go-zip64.zip` exercises the ZIP64 branch and shows both
branches of the relaxation:

- **Four sentinel rows**: `cdSize=0xFFFFFFFF` with `z64=72`,
  `cdOffset=0xFFFFFFFF` with `z64=72`, `totalEntries=0xFFFF` with
  `z64=1`, `numEntriesThisDisk=0xFFFF` with `z64=1` — these are the
  APPNOTE §4.3.16-compliant overflow encodings.
- **Two numeric-match rows**: `numberOfThisDisk=0` with `z64=0`,
  `diskWhereCDStarts=0` with `z64=0` — these are the Go-specific
  zeros-instead-of-sentinels behaviour that empirically forced the
  relaxation. Without the `∨ stdXxx.toNat == zip64Val` disjunct, the
  `numberOfThisDisk` guard at `:344` and the `diskWhereCDStarts`
  guard at `:347` would both throw on `go-zip64.zip`.

All six fixtures parse cleanly; `lake exe test` green. The relaxation
is both necessary (Go emits real zeros) and sufficient (every
Go-interop fixture satisfies one of the two disjuncts per field). ✓

1. **Gate location** — the only dimension that fires inside

   catches the parser-differential vector — a strict-sentinel check
   would over-reject, and a sentinel-only-accepted-or-throw check

body holds: treating this as one row in the table rather than six
keeps the coverage matrix readable. ✓

**G.1 Issue-vs-PR citation drift (repeated) — worth filing.** This
is the **second** paired-review in a row (after #1756 §H.1) where
the PR under review introduces a `#<issue>` vs `#<PR>` citation
drift in `SECURITY_INVENTORY.md`. PR #1752 transcribed `#1744`
(issue) instead of `#1752` (PR); PR #1754 transcribed `#1745`
(issue) instead of `#1754` (PR). Both were self-corrected-forward
(PR #1754 fixed PR #1752's drift; PR #1754's drift is documented
here and a future PR can fix it).

Root cause: the planner template for the *Recent wins* bullet and
the *Minimized Reproducer Corpus* row is authored with the closing-
issue number (the only number the planner knows at plan-time), and
the worker transcribes it verbatim after merge without substituting
the PR number. `scripts/check-inventory-links.sh` cannot catch this
because both are valid GitHub cross-refs pointing at real objects.
Three possible mitigations — ranked by cost/benefit:

1. **Linter rule** (medium cost, permanent fix): extend
   `scripts/check-inventory-links.sh` to distinguish issue numbers
   from PR numbers (via `gh api repos/OWNER/REPO/issues/N` which
   returns a `pull_request` field for PRs only) and flag
   `issue → inventory` references where a `pull_request → inventory`
   reference was probably intended. Requires the linter to make
   online calls, which it currently does not — may want to restrict
   to a local cache.
2. **Planner-template convention** (low cost, human-dependent): when
   the planner authors the template for the inventory bullet, use a
   `#PR-TBD` placeholder that the worker substitutes with the actual
   PR number at inventory-write time. Relies on the worker not
   transcribing `#PR-TBD` verbatim — has the same failure mode as
   today.
3. **Post-merge self-correction convention** (what we're doing
   implicitly): the next feature PR sees the drift on master and
   fixes it as part of its own inventory update. This is what
   PR #1754 did for PR #1752's drift; whichever PR follows #1754
   into `SECURITY_INVENTORY.md` will likely fix #1754's drift.

**Recommendation**: file option 1 (linter rule) as a `/meditate`
candidate for the next self-improvement pass. Not a paired-review
blocker; flagged here for visibility.
