## Findings per audit check

### A. Guard shape and placement

- **A.1 predicate correctness — OK.** The guard at
  [Zip/Archive.lean:535](/home/kim/lean-zip/Zip/Archive.lean:535)
  reads literally `unless versionNeeded ≤ 45 do`. Not de-Morgan'd,
  not `< 46`, not bit-masked, not `.toNat`-narrowed. `versionNeeded`
  is the `UInt16` bound at
  [Zip/Archive.lean:479](/home/kim/lean-zip/Zip/Archive.lean:479)
  via `Binary.readUInt16LE data (pos + 6)` — comparison happens at
  `UInt16`, so `versionNeeded = 256` (a hypothetical low-byte
  collision under `.toUInt8` narrowing) still fails the `≤ 45`
  test. Boundary: `versionNeeded = 45` passes, `= 46` fails.

- **A.2 insertion site — OK.** Guard at
  [Zip/Archive.lean:535](/home/kim/lean-zip/Zip/Archive.lean:535)
  sits **before** the method allowlist at
  [Zip/Archive.lean:552](/home/kim/lean-zip/Zip/Archive.lean:552)
  (PR #1801), **before** `validateExtraFieldStructure` at
  [Zip/Archive.lean:567](/home/kim/lean-zip/Zip/Archive.lean:567)
  (PR #1788), **before** ZIP64 resolution at
  [Zip/Archive.lean:569-577](/home/kim/lean-zip/Zip/Archive.lean:569),
  and **before** the stored-method size invariant at
  [Zip/Archive.lean:612-615](/home/kim/lean-zip/Zip/Archive.lean:612)
  (PR #1773). A crafted archive with
  `versionNeeded = 51 ∧ method = 6 ∧ compSize ≠ uncompSize` trips
  the version guard first — which matches the plan's
  paired-review-distinct attribution for this fixture vs the method
  and size-invariant fixtures. Moving the guard after
  `parseZip64Extra` would burn the ZIP64 walk on un-extractable
  versions; the chosen site maximises fast-fail. Placement matches
  the 17-line comment block at `:518-:534` that justifies it.

- **A.3 error message shape — OK.** The thrown `IO.userError` at
  [Zip/Archive.lean:536-537](/home/kim/lean-zip/Zip/Archive.lean:536)
  is
  `s!"zip: unsupported versionNeededToExtract ({versionNeeded}) for {name} at CD offset {pos}; lean-zip supports values up to 45 (ZIP64)"`.
  It contains the target substring
  `"unsupported versionNeededToExtract"`, the offending version
  value, the entry name, and the CD offset. Cross-checks:
  - Pre-existing one-sided CD/LH downgrade throw at
    [Zip/Archive.lean:917-918](/home/kim/lean-zip/Zip/Archive.lean:917)
    is
    `s!"zip: LH versionNeededToExtract ({localVersion}) exceeds CD versionNeededToExtract ({entry.versionNeeded}) for {label}"`
    (PR #1736) — substring `"LH versionNeededToExtract"`. Both
    messages contain the bare substring
    `versionNeededToExtract`, so a `msg.contains "versionNeededToExtract"`
    assertion is ambiguous. Test-author discipline: match either
    the full `"unsupported versionNeededToExtract"` or the full
    `"LH versionNeededToExtract"` to disambiguate.
  - `.claude/skills/error-wording-catalogue/SKILL.md` grew two new
    rows in this PR (one for each substring, both with explicit
    grep-uniqueness prose calling out the shared
    `versionNeededToExtract` token and the required `"unsupported "`
    or `"LH "` prefix). The catalogue update makes the C.1
    test-author mistake hard to repeat.

- **A.4 writer-side compliance — OK.**
  `grep -n "entry\.versionNeeded\|versionNeeded :=\|z64 then 45" Zip/Archive.lean`
  returns:
  - `:90` — `writeUInt16LEAt buf 4 (if z64 then 45 else 20)`
    (LH versionNeeded)
  - `:116` — `writeUInt16LEAt buf 4 (3 * 256 + (if z64 then 45 else 20))`
    (CD versionMadeBy at CD+4; UNIX-origin flag `3` in the high byte,
    spec version in the low byte)
  - `:117` — `writeUInt16LEAt buf 6 (if z64 then 45 else 20)`
    (CD versionNeeded)
  - `:479` / `:624` — reader-side binding + `Entry` reconstruction
    from the parsed value (post-guard)
  - `:916`, `:918` — consumer-side reads of `entry.versionNeeded`
    (CD/LH downgrade check)

  Both writer-side `versionNeeded` writes (`:90`, `:117`) emit
  `if z64 then 45 else 20` — `UInt16` constant literals, not a
  computed expression. Writer never emits a value > 45, so the
  reader guard never fires on lean-zip-produced archives.

- **A.5 pre-ZIP64-resolution safety — OK.** APPNOTE §4.4.3.2
  specifies `versionNeededToExtract` as a plain `UInt16` with no
  sentinel. The ZIP64 sentinel `val16Max = 0xFFFF` is defined for
  entry-count fields in the standard EOCD, not for the
  `versionNeeded` field at CD+6. The guard can safely run before
  `parseZip64Extra` at `:573`; the ZIP64 resolution never rewrites
  `versionNeeded`.
