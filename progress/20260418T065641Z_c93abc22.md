## Work item

Issue #1494 ‚Äî *Track E: ZIP local-header span validation before `Handle.read`*.

## Summary

Landed the Priority-0 local-span validation called out in
`SECURITY_INVENTORY.md` and `plans/track-e-current-audit-checklist.md`.
Every attacker-controlled read in `Zip/Archive.lean` is now either
guarded by a known-bounded buffer (tail/CD) or preceded by an
`assertSpanInFile` check against the handle's actual file size.

## Changes

- `Zip/Archive.lean`
  - New `private def assertSpanInFile (fileSize offset length what)` ‚Äî
    rejects `offset > fileSize`, `offset + length` overflow, and
    `offset + length > fileSize` with a descriptive `IO.userError`.
  - `readEntryData` now reads the handle size once, then validates
    three spans before any `Handle.read`: the 30-byte local header,
    the `nameLen + extraLen` region, and the `compressedSize` payload.
    `nameLen + extraLen` is bounded by `2 * UInt16.max` so the
    intermediate `UInt64` sum cannot overflow.
- `ZipTest/ZipFixtures.lean`
  - Loads a new `oversized-compressed-size.zip` malformed fixture and
    asserts `Archive.extract` throws with substring `"local data span"`.
- `testdata/zip/malformed/oversized-compressed-size.zip`
  - 122-byte archive storing a 6-byte `hello.txt`; both the local
    header and the central directory claim `compressedSize = 0x200000`
    (2 MiB). The inline Python recipe in `ZipTest/ZipFixtures.lean`
    documents how to regenerate it.
- `SECURITY_INVENTORY.md` ‚Äî move "validate local-header data span"
  from *missing work* to *current guardrails* with a reference to the
  new fixture.
- `plans/track-e-current-audit-checklist.md` ‚Äî tick Priority-0 boxes 1,
  2, and 4; note that exabyte-scale ZIP64 variants remain open.
