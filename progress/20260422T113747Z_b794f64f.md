## Decisions

- **Wrap-detection vs explicit `UInt64.max - total`.** The plan offered
  both the checked-add form (`chunkSize > UInt64.max - total`) and a
  saturating alternative, and explicitly said "pick the one that reads
  clearly". I went with `next < total`: after the UInt64 add, `next <
  total` holds exactly when the add overflowed (since a non-overflowing
  add produces `next ≥ total`). This is shorter, uses no magic constant
  for `UInt64.max`, and composes cleanly with the existing cap check as
  a single `||` disjunct. Both disjuncts share the same `"exceeds
  limit"` throw, preserving the substring the regression tests
  (`ZipTest/Gzip.lean:180-195`, the `decompressFile` bomb test, and the
  `RawDeflate` analogue) grep for.

- **Shared error message.** The plan was explicit: overflow is a
  degenerate case of "exceeds limit" (cap ≤ UInt64.max, so crossing
  UInt64.max necessarily crosses the cap), and a separate error variant
  would break the `error-wording-catalogue` discipline. Kept a single
  throw per site.

- **Short-circuit.** The plan called out that the overflow disjunct
  should short-circuit before computing `total + chunkSize`. In this
  form, `next` is computed unconditionally (UInt64 addition is a pure
  wrapping op, so there is no semantic cost to computing the wrapped
  value — we simply observe it and reject). The short-circuit concern
  in the plan's canonical shape was aesthetic rather than semantic;
  this equivalent form avoids the magic constant entirely.
