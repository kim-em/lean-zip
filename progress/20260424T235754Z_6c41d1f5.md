## Decisions

- **Return `Except String` rather than `IO`.** `parsePaxRecords`
  has no IO dependencies; keeping it pure and lifting at the
  single IO call site preserves testability (cf. the
  `ZipTest/Tar.lean:199` unit test that calls `parsePaxRecords`
  directly). `IO.ofExcept` handles the lift transparently at
  the test call site.
- **Array scan `records.any` instead of `Std.HashSet`.** PAX
  headers typically carry a handful of records (Go's
  `archive/tar`, Python `tarfile`, libarchive all emit ≤10 in
  practice); an O(n²) scan is negligible at realistic record
  counts and avoids introducing a `Std.HashSet` import that is
  absent elsewhere in `Zip/` (confirmed by
  `grep -r HashSet Zip/` returning no hits).
- **Reject (hard error) rather than skip.** The existing
  invalid-UTF-8 and NUL-byte guards silently skip, but the
  issue explicitly asks for rejection — duplicate keys are a
  structural validity failure (as opposed to the individual
  record's byte-level corruption), and rejecting closes the
  parser-differential dimension cleanly. Silently skipping
  would still leave the first record in place and hide the
  attribution.
- **Error message wording.** Chose
  `tar: PAX extended header has duplicate {key.quote} record`
  to match the recent PAX error-wording family
  (`tar: PAX record path value contains NUL byte` from PR #1866),
  keep the `tar:` prefix consistent with the rest of
  `Zip/Tar.lean`, and surface the offending key via
  `String.quote` so logs attribute to the specific record
  without leaking non-printable bytes.
