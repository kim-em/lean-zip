# Review: Zstd native implementation quality and documentation

**Issue**: #657
**Date**: 2026-03-06
**Session**: 8ad37960

## Summary

Audited all five Zstd native implementation files: ZstdFrame.lean, ZstdHuffman.lean,
ZstdSequence.lean, Fse.lean, and XxHash.lean. The code is remarkably clean with zero
`sorry`s, zero bare `simp` calls, consistent error message formatting, and all public
functions documented with docstrings.

## Findings

### Code quality (all files)
- **No dead code** after the PR #599 file split — all imports are used, no orphaned functions
- **No unused imports** — each file imports exactly what it needs
- **Error messages** are consistent: "Zstd: ..." for Zstd-level errors, "FSE: ..." for
  FSE errors, "BackwardBitReader: ..." for bitstream errors
- **All `!` accesses** have adequate bounds guards (either explicit `if` checks or
  structural invariants from callers)
- **Edge cases handled**: empty input, zero-length blocks, boundary conditions all
  properly guarded

### Changes made

1. **ZstdHuffman.lean**: Cleaned up development-time "thinking aloud" comment in
   `buildZstdHuffmanTable` (lines 104-113 had "Wait —" and "Actually:" language from
   the development process). Replaced with concise 2-line summary.

2. **ZstdSequence.lean**: Added field docstrings to `SequenceCompressionModes` and
   `PrevFseTables` structures for consistency with other structures in the codebase
   (all other Zstd structures already had field docstrings).

### Observations (no changes needed)

- `decodeFseSymbols` in Fse.lean is only used in tests, not production code. This is
  fine — it's a useful public API for testing FSE decoding with a known count.
- `decodeFourHuffmanStreams` has a subtle edge case: if `regenSize < 4` in 4-stream mode,
  `s4Count` underflows (Nat). In practice this never occurs because encoders don't use
  4-stream mode for tiny inputs, and the result would be a bitstream error, not silent
  corruption.
- `decodeHuffmanSymbol` does two `readBits` calls and two table lookups (peek + actual
  decode). This works correctly but could be simplified if `BackwardBitReader` supported
  "putting back" bits. Not worth changing now.
- `XxHash.lean` uses namespace `XxHash64` instead of `Zip.Native.XxHash64`. This is
  consistent with how it's referenced from ZstdFrame.lean.
- `BitReader.lean` (not in audit scope) has inline comments instead of field docstrings
  on its structure — minor inconsistency with other files.
