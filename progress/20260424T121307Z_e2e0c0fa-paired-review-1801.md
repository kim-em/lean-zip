## Findings per audit check

### A. Guard shape and placement

- **A.1 predicate correctness — OK.** The guard at
  [Zip/Archive.lean:526](/home/kim/lean-zip/Zip/Archive.lean:526) reads
  literally `unless method == 0 ∨ method == 8 do`. Not de-Morgan'd,
  not `< 2`, not bit-masked. `method` is the `UInt16` bound at
  [Zip/Archive.lean:475](/home/kim/lean-zip/Zip/Archive.lean:475) via
  `Binary.readUInt16LE data (pos + 10)` — no `.toNat`, no narrowing,
  no sign-extension. Comparison happens at `UInt16`, so
  `method = 256` would fail both disjuncts and be rejected (it would
  have been `0` under a hypothetical `.toUInt8` narrowing). `method = 0`
  and `method = 8` are the only accepted values.

- **A.2 insertion site — OK.** Guard sits between the
  name-resolution block ending at
  [Zip/Archive.lean:511](/home/kim/lean-zip/Zip/Archive.lean:511) and
  the `validateExtraFieldStructure` call at
  [Zip/Archive.lean:541](/home/kim/lean-zip/Zip/Archive.lean:541),
  matching the comment block at `:512-:525`. Moving earlier (before
  name resolution) would drop the entry name from the error-message
  interpolation; moving later (after ZIP64 resolution at `:544-:551`)
  would preserve the name but burn the ZIP64 walk before rejecting an
  un-extractable method. The chosen site maximizes fast-fail while
  keeping the error message self-identifying. The comment-block claim
  that *"Archive.list also catches the anomaly and the error message
  attributes the fault to the CD-parse stage"* is about CD-parse vs
  late-`readEntryData` attribution — confirmed by E.1 below —
  independent of the within-CD placement.

- **A.3 error message shape — OK.** The thrown
  `IO.userError` at `:527-:528` is
  `s!"zip: unsupported compression method {method} for {name} (lean-zip supports method 0 (stored) and method 8 (deflate))"`.
  It contains the target substring `"unsupported compression method"`,
  the offending method value, and the resolved entry name. Cross-checks:
  - Late throw at
    [Zip/Archive.lean:906](/home/kim/lean-zip/Zip/Archive.lean:906)
    remains `s!"zip: unsupported method {entry.method} for {label}"`
    — bare `"unsupported method"` substring, no `"compression "`
    token, so the two wordings are disambiguable on the specific
    substring (the bare one is a substring of the new one, so
    test-author discipline matters; see D.3).
  - `.claude/skills/error-wording-catalogue/SKILL.md` row 27 documents
    the new substring with the interpolation shape; the narrative at
    lines 113-129 explicitly tells test authors to match the fuller
    form. OK.

- **A.4 pre-ZIP64-resolution safety — OK.** APPNOTE §4.4.5 specifies
  the compression method as a plain `UInt16` with no sentinel. The
  ZIP64 sentinel `val16Max = 0xFFFF` is defined for entry-count
  fields in the standard EOCD (see
  [Zip/Archive.lean](/home/kim/lean-zip/Zip/Archive.lean) search for
  `val16Max` / `val32Max`), not for the `method` field at CD+10. The
  guard can safely run before `parseZip64Extra` at `:547`; the ZIP64
  resolution never rewrites `method`.

- **A.5 writer-side compliance — OK.** `grep -n "entry\.method\|method :="
  Zip/Archive.lean` returns:
  - `:92` — `writeUInt16LEAt buf 8 entry.method` (LH write)
  - `:119` — `writeUInt16LEAt buf 10 entry.method` (CD write)
  - `:192` — `let method : UInt16 := if useDeflate then 8 else 0`
    (single writer-side assignment in `create`)
  - `:199` — `method := method` (`Entry` construction in `create`)
  - `:475` / `:572` — reader-side binding + `Entry` reconstruction
    from the parsed value (post-guard)
  - `:848`, `:895`, `:896`, `:906` — consumer-side reads of
    `entry.method` (CD/LH mismatch check + late dispatch)

  The only writer-side `entry.method` assignments are `:92` and `:119`,
  both sourced from `create`'s `let method` at `:192`, so the writer
  emits only `0` or `8`. Any round-trip via `create → list → extract`
  is trivially compliant.

### E. Precedence-shift soundness

- **E.1 late-throw reachability — OK (unreachable via public API).**
  Public API entry points in `Zip/Archive.lean`:
  - [`list` at `:921`](/home/kim/lean-zip/Zip/Archive.lean:921) →
    `listFromHandle` only (no `readEntryData`)
  - [`extract` at `:945`](/home/kim/lean-zip/Zip/Archive.lean:945) →
    `listFromHandle` at `:950`, then per-entry `readEntryData` at
    `:972`
  - [`extractFile` at `:990`](/home/kim/lean-zip/Zip/Archive.lean:990)
    → `listFromHandle` at `:994`, then `readEntryData` at `:997`

  `listFromHandle` ([`:757`](/home/kim/lean-zip/Zip/Archive.lean:757))
  unconditionally runs `parseCentralDir` at `:776`. If the CD
  carries a bad method, the `:526` guard throws before
  `listFromHandle` returns, and no `readEntryData` invocation is
  reached. So the `:906` late throw is **unreachable for
  CD-parseable archives via any public API**. Keeping it in place
  as defense-in-depth (matching PR #1801's stated intent) is the
  right call: `readEntryData` is `private`, but a future
  refactoring that lifts it (or adds a new public entry point that
  bypasses CD parsing) would silently lose the method guard
  otherwise.

- **E.2 CD-parse ordering vs ZIP64 resolution — OK.** CD-entry
  guards, in order, per `Zip/Archive.lean:470-:579`:
  1. `diskNumberStart == 0` at `:493`
  2. `entryEnd <= cdEnd` at `:497` (bounds)
  3. name-decode at `:501-:511` (UTF-8 flag branch)
  4. **method allowlist** `method == 0 ∨ method == 8` at `:526`
  5. `validateExtraFieldStructure extraData` at `:541`
  6. ZIP64 resolution (`hasDuplicateZip64Extra` + `parseZip64Extra`)
     at `:544-:549`
  7. stored-method size invariant `compSize == uncompSize` at
     `:564` (method=0 only, post-ZIP64-resolution — correctly
     placed so sentinels are not compared)

  No guard's predicate depends on a value resolved by a later
  guard:
  - `:526` reads `method` (bound at `:475`) and `name` (resolved at
    `:501-:511`); neither depends on ZIP64 resolution or on the
    extra-field structural check.
  - `:564` explicitly depends on the ZIP64-resolved `compSize` /
    `uncompSize` (see comment block at `:552-:562`).

  The issue's listed sequence
  `disk-number → method allowlist → extra-field structure →
  ZIP64 resolution → stored-method size invariant` matches the
  actual sequence with `entryEnd` and name-decode as additional
  upstream predecessors (both naturally self-contained reads).
