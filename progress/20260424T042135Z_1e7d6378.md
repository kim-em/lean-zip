## Decisions

- **Enforcement site — caller vs. function.** The issue
  suggested two options: lift `findEndOfCentralDir` to
  `IO (Option …)` and throw internally, or keep it pure and
  have `listFromHandle` enforce the postcondition. Chose the
  caller-side check. Rationale: `findEndOfCentralDir` has a
  single caller, so lifting the return type buys nothing;
  the caller already destructures `eocdPos` post-PR #1733 and
  has `tail` in scope for the `readUInt16LE` on `pos + 20`.
  The caller-side check is a 13-line diff (comment + let +
  unless + throw); the lift would have been 20+ lines
  touching the function's body AND the caller.
- **Strict `==` trailing-garbage enforcement.** Six interop
  fixtures all satisfy `pos + 22 + commentLength == fileSize`
  (including `go-test.zip` with a non-zero 26-byte comment), so
  the strict equality does not break any legitimate producer
  pattern observed in the corpus. Matches Go's `archive/zip`
  stance (rejects since 2021) and the project's *harden the
  parser* direction. If a future interop fixture surfaces with
  legitimate trailing bytes, the check can be relaxed to
  `!=` tolerance on specific signatures — but the evidence
  today argues for strict.
- **Single fixture, no `-overshoot` companion.** The issue
  flagged `eocd-comment-overshoot.zip` (pos + 22 +
  commentLength > data.size) as optional "if budget allows".
  Skipped. The single `!=` predicate covers both branches of
  the check, and the error message enumerates expected vs
  actual `fileSize` so either flavour is identifiable from the
  stack trace. Adding a separate fixture would add ~2 KB to
  the binary corpus and ~30 LOC of test wiring for zero
  incremental coverage of the code path.
- **Scan algorithm left unchanged.** The issue flagged
  "scan-then-find vs strict-match" as a potential hardening
  widening (verify `pos + 22 + commentLength == fileSize`
  *during* the backward scan, rejecting candidate positions
  that don't match — effectively moving to "unique EOCD at
  natural end of file" instead of "first EOCD from end").
  Kept scope tight: the strict postcondition at the chosen
  `pos` rejects every trailing-garbage attack, and the
  multiple-EOCD-candidate smuggling dimension is a distinct
  category that can file its own follow-up if desired.
- **PR number reference.** `SECURITY_INVENTORY.md` *Recent
  wins* and the new Corpus row use `#1742` as a best-guess
  placeholder (latest PR at session start was #1741 from log).
  If the PR lands as a different number, a follow-up doc
  tweak may be needed; the machine-readable anchor
  (`Zip/Archive.lean:554`) is stable regardless.

  - Change the EOCD scan algorithm (first-match-from-end) —
