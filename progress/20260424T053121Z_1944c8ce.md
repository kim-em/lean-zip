## Summary

Adds the per-entry counterpart to PR #1742's archive-level EOCD
disk-number consistency check. `parseCentralDir` now reads each CD
entry's `diskNumberStart` field at CD offset `pos + 34` (APPNOTE
§4.4.11) and rejects any nonzero value with the deterministic message
`zip: CD entry diskNumberStart mismatch (diskNumberStart=<n>) for entry
at CD offset <pos>; lean-zip supports single-disk archives only`.

The check fires immediately after the `commentLen` read, before the
`entryEnd > cdEnd` span check and before any name decode or ZIP64 extra
parsing — earliest-possible rejection on a metadata-only dimension.

## Decisions

- **Check placement**: option (a) from the issue body — immediately
  after the `pos + 34` read, before the `entryEnd > cdEnd` span check.
  Earliest-possible rejection on a metadata-only dimension; mirrors the
  archive-level EOCD disk-number check in `parseCentralDir`'s head.
- **Error message**: includes the field's actual value and the CD
  position (debuggability for multi-entry archives with one bad entry),
  but excludes the entry name — at the check-point the name has not
  yet been decoded.
- **Inventory PR-number placeholder**: used "this PR" in both the
  *Recent wins* bullet and the *Minimized Reproducer Corpus* row, to
  be filled in via a follow-up `chore:` commit after `coordination
  create-pr` assigns a number. Matches the pattern from PR #1742
  (commit `chore: fill in PR #1742 in SECURITY_INVENTORY.md`).
