Implemented the `scripts/check-inventory-links.sh` tool requested by

The script executes three passes over `SECURITY_INVENTORY.md`:

1. **Line-anchor existence** (hard failure, exit 1). Regex
   `(Zip|ZipTest)/[A-Za-z0-9_-]+(/[A-Za-z0-9_-]+)*\.lean:[0-9]+`
   tolerates both markdown-URL forms
   (`[Zip/Foo.lean:123](/home/kim/lean-zip/Zip/Foo.lean:123)`) and
   bare in-prose references. The worker-specific `/home/kim/lean-zip/`
   URL prefix is stripped implicitly because the pattern is not
   left-anchored. Each unique `file:lineno` pair is checked once
   for (a) file existence, (b) `wc`-independent line count via
   `awk 'END{print NR}'` (handles files lacking a final newline).

2. **Fixture-path existence** (hard failure, exit 1). Regex
   `testdata/[A-Za-z0-9_/.\-]+\.[A-Za-z]+` requires a `.<letters>`
   extension at the tail, which naturally excludes glob patterns
   like `testdata/tar/malformed/pax-*.tar` (the char class cannot
   match `*`, so backtracking fails) and bare directory references
   like `testdata/tar/security/`.

3. **Line-content sanity heuristic** (warning, not fatal). For each
   line anchor, gather quoted substrings (length 3..80 chars) from
   the same inventory source line using both `"..."` and `` `...` ``
   delimiters. If any such substring exists, check that at least one
   appears within the `<line> ± 2` window of the cited source file.
   Emit a warning listing the file, source line, quoted candidates,
   and the actual cited line if no substring matches.

## Decisions and patterns

- **Scope of the line-anchor regex.** The issue example used
  `Zip/Foo.lean:123`, but inspection of the current inventory turns
  up `ZipTest/Zlib.lean:14` at line 337 — a real drift vector that
  should not be silently skipped. Widened the pattern to accept
  both top-level directories while keeping the same path-character
  grammar. `Zip/Spec/...` and `Zip/Native/...` nested paths are
  handled by the `(/[A-Za-z0-9_-]+)*` suffix.
- **Fixture regex requires an extension.** Without `.<letters>` at
  the tail, a naive pattern matches fragments like
  `testdata/tar/malformed/gnu-longname-` truncated before a `*` in
  a glob, yielding spurious "missing fixture" errors. The extension
  requirement keeps the regex robust without an explicit
  allow-list of extensions, leaving the script extensible if
  `.gz`, `.bin`, etc. show up later.
- **Line-content heuristic intentionally narrow.** The `±2` window
  is the spec-specified tolerance. It surfaces the "line number
  cited but the quoted error string has moved" case, at the cost
  of false positives on citations to the *function-definition*
  line (where the throw is 5–20 lines later). On the current tree
  this produces ~55 warnings, all of the form *"cite points at the
  data-extraction step; throw is 20–40 lines later"*. Treating
  these as warnings (not errors) matches the issue's explicit
  framing.
- **Use `awk 'END{print NR}'` rather than `wc -l`.** `wc -l` counts
  newlines; a file ending mid-line is under-counted by one. `awk`
  counts records (including an unterminated last line), which is
  the line index space used by editors and grep.
- **Pass (c) dedupes on the full `grep -no` line via `sort -u`.**
  The same `(srcln, path, lineno)` triple appearing twice in one
  source line would otherwise emit duplicate warnings. Cross-line
  duplicates are preserved because each source line has its own
  prose context.
- **`set -euo pipefail` + `|| true` on grep invocations.** `grep`
  exits 1 when there are no matches, which under pipefail would
  abort the outer `while read` loop without executing — the loop
  simply doesn't iterate, which is the desired behavior. The
  `|| true` is defensive; on the current tree every pass finds
  matches.

## Warning noise level

The 55 warnings on the clean tree are not "known drift" — they are
false positives from citations that point at the `let x = ...`
extraction line rather than the `throw (IO.userError ...)` line.
For example, `Zip/Archive.lean:459` is cited for the
*"method mismatch"* error, but line 459 reads `localMethod` from
the header; the actual throw is at line 496, well outside the ±2
window.

A follow-up planner could consider widening the window (at the cost
of lower specificity) or adding a heuristic to skip citations whose
target line starts with `let`. For now, the spec-specified
threshold is preserved and the warnings are best read as
*"the cite could be more precise,"* not *"the code has moved."*
