## Accomplished

Added RFC 8878 infrastructure for converting FSE symbol codes to actual
sequence values, plus compression mode parsing for the sequences section header.

### ZstdFrame.lean (+135 lines)
- `litLenExtraBits`: 36-entry lookup table (RFC 8878 Table 14) mapping literal
  length codes 0-35 to (baseline, numExtraBits) pairs
- `matchLenExtraBits`: 53-entry lookup table (RFC 8878 Table 15) mapping match
  length codes 0-52 to (baseline, numExtraBits) pairs
- `decodeLitLenValue`, `decodeMatchLenValue`: look up code in table, return
  baseline + extraBits
- `decodeOffsetValue`: formula-based — `(1 <<< code) + extraBits` for code ≥ 1
- `SequenceCompressionMode` enum: predefined, rle, fseCompressed, repeat
- `SequenceCompressionModes` structure with fields for all three symbol types
- `parseSequencesHeaderWithModes`: extends header parsing to extract the 2-bit
  mode fields from the compression modes byte

### ZstdNative.lean (+138 lines)
- 19 new tests (tests 55-73) covering:
  - Literal length decode at boundary codes (0, 15, 16, 35) and out-of-range
  - Match length decode at boundary codes (0, 31, 32, 52) and out-of-range
  - Offset decode for codes 0, 1, 5, 20
  - Compression modes parsing: FSE+RLE+predefined, zero-sequence defaults, all-repeat
  - Table size verification (36 and 53 entries)

## Decisions
- Kept the original `parseSequencesHeader` intact since `decompressBlocks` doesn't

  a new function. The future interleaved decode loop issue will switch callers over.
