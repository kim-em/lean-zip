# Review: Fse.lean and XxHash.lean code quality audit

**Issue**: #574
**Session**: 461fb044-bde0-4144-be90-497bfc9744b2
**Date**: 2026-03-04

## Summary

Performed a focused code quality review of two foundational Track E modules:
Fse.lean (359 lines) and XxHash.lean (122 lines). Both are critical
infrastructure for Zstd decompression that had not been formally reviewed.

## Fse.lean Review Findings

**RFC conformance**: The implementation correctly follows RFC 8878 §4.1 for all
three components (distribution decoding, table construction, backward bitstream
reading). The variable-length encoding, stepping algorithm, and MSB-first
backward reading all match the specification.

**Finding 1 — Dead code (line 85-86)**: In `decodeFseDistribution`, the inner
`else` branch (`pure rawBits.toNat`) is unreachable. When `rawBits >=
lowThreshold`, `combined = rawBits * 2 + extraBit >= 2 * lowThreshold >=
lowThreshold`, so `combined >= lowThreshold` is always true. Harmless but could
be removed for clarity.

**Finding 2 — Silent fuel exhaustion in `decodeFseSymbolsAll`**: When the fuel
parameter (default 4096) runs out without the bitstream finishing, the function
returns success with a potentially truncated result rather than an error. In
practice, the default fuel is vastly larger than the max Huffman symbol count
(256), so this won't trigger on valid input. For malformed input, it silently
truncates rather than erroring.

**Finding 3 — No `readBits` bounds enforcement**: The `BackwardBitReader.readBits`
docstring says "n ≤ 25" but doesn't enforce this. Not a correctness bug (callers
are well-behaved), but the docstring creates an unchecked contract.

**Naming, docstrings, error messages**: All clean. Every public definition has
accurate docstrings. Error messages identify the specific failure with relevant
values. Function names follow Lean conventions consistently.

**Predefined distributions**: All three (LL, ML, OF) match RFC 8878 §6 Tables
15-17. Existing tests verify sums and sizes.

## XxHash.lean Review Findings

**Spec conformance**: Implementation matches the xxHash specification exactly.
All five prime constants verified against the spec. All algorithmic steps
(round, mergeAccumulator, processRemaining, avalanche, main hash) are correct.

**No issues found**. Clean, well-documented code with good inline annotations.

## Tests Added

5 new edge case tests:

1. **FSE: all -1 probability distribution** — 32 symbols each with prob -1,
   verifying each gets exactly 1 cell and all symbols appear exactly once
2. **FSE: BackwardBitReader multi-byte spanning read** — 12-bit read across
   byte boundaries verifying MSB-first ordering and cross-byte correctness
3. **XXH64: 45-byte input** — exercises the long path (1 stripe of 32 bytes)
   plus all three remaining-byte sub-paths (8-byte + 4-byte + 1-byte)
4. **XXH64: non-zero seed** — verified against independently computed reference
   value (seed=42, "seed test" → 0xBD5B2F29B94F97EE)
5. **XXH64: 4-byte input** — exercises only the 4-byte remaining path in the
   short (< 32 bytes) code path

## Quality Assessment

Both modules are well-written and correct. The code is clean, properly
documented, and follows project conventions. The findings are minor and none
affect correctness for well-formed input. The existing test suite (19 FSE tests,
11 XXH64 tests) was already solid; the 5 new tests close the remaining edge
case gaps.

## Refactoring Opportunities (noted, not implemented)

- Remove dead code branch in `decodeFseDistribution` (line 85-86)
- Add fuel exhaustion error in `decodeFseSymbolsAll`
- Add bounds check in `BackwardBitReader.readBits` for n > 25
