## Construction

- LH (30 B) + name `hello.txt` (9 B) + payload `hello\n` (6 B) = 45 B.
- CD (46 B) + name `hello.txt` (9 B) = 55 B; `cdSize = 55`,
  `cdOffset = 45`, `cdEnd = 100`. CD entry's `commentLen = 16` is
  declared but no comment bytes are physically present.
- EOCD (22 B); `eocdPos = 100`, total file size 122 B.
- At parse time `entryEnd = 45 + 46 + 9 + 0 + 16 = 116 > cdEnd = 100`,
  firing the :615 guard before any name decode.

All earlier CD-parse guards pass:

- EOCD `numberOfThisDisk == 0 && diskWhereCDStarts == 0` ✓
- EOCD `numEntriesThisDisk == totalEntries` (1 == 1) ✓
- Loop entry `pos + 46 ≤ cdEnd` (91 ≤ 100) ✓
- CD signature `0x02014b50` ✓
- `nameLen > 0` (9 > 0) ✓
- `diskNumberStart == 0` ✓
- `internalAttrs &&& 0xFFFE == 0` (0) ✓

So attribution pins to the per-entry footprint guard rather than a
sibling early-reject. The existing archive-level
`cdOffset + cdSize ≤ fileSize` guard at
[`Zip/Archive.lean:1057`](/home/kim/lean-zip/Zip/Archive.lean:1057)
also passes (100 ≤ 122). The macro-shape `cdOffset + cdSize ≤ eocdPos`
guard from in-flight PR #1809 is not in master yet, but my fixture
already satisfies it (100 == eocdPos == 100, `≤` not `<`).
