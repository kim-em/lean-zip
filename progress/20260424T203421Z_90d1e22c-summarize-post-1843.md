### CD-parse mathematical-invariant family — second column (#1857)

The prior wave established the mathematical-invariant family at CD
parse with #1773 (stored-method `compSize == uncompSize`). This wave
adds the second column:

- **`uncompSize == 0 → crc32 == 0` (#1857).** APPNOTE §4.4.7 pins the
  CRC32 field as the ANSI-CRC-32 of the uncompressed payload; the
  empty byte string has CRC32 `0x00000000` by the start-state
  `0xFFFFFFFF` + final-complement identity. This is a universal
  mathematical invariant every correct writer obeys — Info-ZIP, Go
  `archive/zip`, CPython `zipfile`, 7-Zip, and lean-zip's own `create`
  at [Zip/Archive.lean:195](/home/kim/lean-zip/Zip/Archive.lean:195)
  (which emits `Checksum.crc32 0 fileData` and returns `0` on empty
  input). Fires at
  [Zip/Archive.lean:794](/home/kim/lean-zip/Zip/Archive.lean:794)
  post-ZIP64-resolution (so the resolved `uncompSize : UInt64` is the
  value checked rather than the `0xFFFFFFFF` sentinel), and after the
  stored-method size invariant — the ordering pins attribution on the
  empty-file premise rather than a generic CRC mismatch. Closes both
  `Archive.list` (pre-PR propagated the crafted CRC verbatim into
  `Entry.crc32` — callers routing on `entry.crc32` saw smuggled values)
  and `Archive.extract` (pre-PR caught the mismatch only post-
  extraction at the `"CRC32 mismatch"` guard :1088, after any I/O work).
  Fixture: `cd-empty-entry-crc-nonzero.zip`. `SECURITY_INVENTORY.md`
  *Recent wins* :683-731.

The column distinction is deliberate: #1773 closes `compSize ==
uncompSize` (tautological for stored), #1857 closes `uncompSize == 0
→ crc == 0` (tautological for empty entries, method-agnostic). The
two invariants are ordering-disjoint — the CD/LH `crc32` consistency
check (`cd-lh-crc-mismatch.zip`, PR #1728) would fire on a CD-vs-LH
disagreement but the fixture carries the same crafted CRC in both
halves, so the empty-entry guard fires first on its more specific
premise.

### Tar cross-format NUL-byte closure (#1865, #1866)

The prior wave's #1831 closed the ZIP CD-layer NUL-byte smuggle. This
wave closes it across all three Tar header layers:

- **GNU long-name / long-link (#1865).** Rejects raw `ByteArray`
  containing `0x00` in the long-name payload at
  [Zip/Tar.lean:576](/home/kim/lean-zip/Zip/Tar.lean:576) — *"GNU
  long-name contains NUL byte"*. The `stripTrailingNuls` helper
  preserves interior NULs, so `forEntries` rejects on the raw bytes
  before `String.fromUTF8?` / `Binary.fromLatin1` runs — neither
  decode branch re-introduces NUL into logs. The long-link typeflag
  `'K'` arm at [Zip/Tar.lean:588](/home/kim/lean-zip/Zip/Tar.lean:588)
  carries the structurally identical guard — *"GNU long-link contains
  NUL byte"* — covered by symmetric code review rather than a
  dedicated fixture (matching the `hardlink-outside.tar` positive-
  regression precedent). Fixture: `gnu-longname-nul-in-name.tar`
  (1536 B). Inventory row at :1097.
- **PAX record key/value (#1866).** `parsePaxRecords` guards on
  `keyBytes` / `valueBytes` at
  [Zip/Tar.lean:122](/home/kim/lean-zip/Zip/Tar.lean:122) — the
  record is dropped silently, matching the invalid-UTF-8 precedent
  one line above. `String.fromUTF8?` accepts U+0000 so without the
  guard the `path` override would smuggle into `applyPaxOverrides`
  and land as `entry.path = "a\x00b/c"`. Assertion is a *positive
  regression*: `Tar.list` returns the follow-on regular-file entry
  with its declared `"hello.txt"` path rather than the smuggled
  override. Fixture: `pax-path-nul-in-value.tar` (2048 B). Inventory
  row at :1106.

Together with #1831 (ZIP CD), the **cross-format NUL-byte closure**
spans all four user-supplied string-field layers where a NUL could
smuggle through:
- ZIP CD entry name (#1831, byte-content)
- Tar ustar header `name` / `linkname` — covered since forever via
  the fixed-width 100-byte field's NUL-terminated-string decode
  (exit-at-first-NUL by construction, no smuggle vector)
- Tar GNU long-name / long-link typeflags `'L'` / `'K'` (#1865)
- Tar PAX extended-header key/value (#1866)

Closes the "smuggled NUL in any user-supplied string field" attack

## Follow-up planning hooks

These are observations handed forward to the next planner cycle — no
issues filed from this summarize session.

- **Tar `linkname` in plain ustar header layer — likely already
  covered by construction, but worth an explicit note.** The ustar
  `linkname` field (offset 157, 100 bytes, NUL-terminated) is
  structurally immune to interior-NUL smuggling via the same
  mechanism as `name` — the decode stops at the first NUL. The
  planner seeded this as a probe question; answer appears to be
  "structurally immune, no fixture needed" — but a counter-fixture
  or a paired-review entry confirming this explicitly would make
  the cross-format closure airtight in the audit record.
- **CD per-entry mathematical-invariant family — third column.**
  The family now has two columns closed (`compSize == uncompSize`
  for stored #1773, `uncompSize == 0 → crc == 0` #1857). A natural
  third column is *single-byte CRC identities*: `crc32(single byte b)`
  is fully determined by `b` via the CRC-32 recurrence, so a crafted
  entry advertising `uncompSize = 1` with a `crc32` value inconsistent
  with the single payload byte at LH time would be a layer-agnostic
  parser-differential vector. But it requires parsing the LH payload
  bytes at CD time (which the CD parser doesn't currently do), so
  the more tractable sibling is probably *`compSize == 0 → uncompSize
  == 0`* at the stored-method layer (closes a distinct smuggle where
  the writer promises non-empty-uncompressed-from-empty-compressed).
  Worth a planner decision on scope.
- **ZIP64 EOCD64 v2 vs v1 record-size.** The wave's #1856 enforces
  `eocd64Offset + 56 ≤ locatorPos` — the "+ 56" hard-codes the
  APPNOTE §4.3.14 v1 record size. v2 (strongEncryptionCompositeSize
  field added) is documented in APPNOTE §4.3.14.2 but is tied to
  strong-encryption mode, which lean-zip does not implement. The
  question the issue body seeded: "is v2 forbidden entirely, and is
  there a fixture exercising that rejection?" Answer from inventory
  inspection: no explicit v2-record-rejection fixture exists; the
  current `findEndOfCentralDir` appears to accept any record size
  ≥ 44 (the APPNOTE-pinned minimum on the record-size field itself)
  up to the size-field-declared length, but the "+ 56" hard-coding
  in #1856 would cause a v2 record (record size ≥ 56 + v2 extension)
  to pass the overlap check but later misparse fields at wrong
  offsets. A dedicated `zip64-eocd64-v2-record.zip` fixture that
  tests the v2-record-size rejection (or explicitly documents v2
  acceptance if that's the design) would close a coverage gap.
- **CD/LH name-bytes consistency check — in-flight PR #1725 carries
  the drift.** Issue #1722 originally framed this; PR #1725 is in
  the repair queue at wave close. This is the remaining column of
  the per-entry CD/LH consistency matrix (the eighth of the eight
  dimensions enumerated in PROGRESS.md :40), so its landing closes
  that matrix.
