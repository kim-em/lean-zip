## Sentinel check sharpness

The issue's "Decisions to make during implementation" block warned
that some real producers emit ZIP64 with real (non-sentinel) values
in the standard EOCD. The critical regression smoke test on
`testdata/zip/interop/go-zip64.zip` confirmed this — Go's
`archive/zip` emits `numberOfThisDisk=0` and `diskWhereCDStarts=0`
(real values, not `0xFFFF` sentinels) in the standard EOCD when
ZIP64 is used. Per the issue's fallback guidance, each guard was
relaxed to `stdValue == sentinel ∨ stdValue.toNat == zip64Value`:
the "numeric match" branch accommodates real producers, and
rejection fires only when the standard-EOCD value is both
non-sentinel **and** numerically disagrees with the ZIP64 override
— the parser-differential smuggling vector. The fixture's attack
bytes (`cdOffset=42` against ZIP64's `45`) satisfy both the
non-sentinel and the non-match conditions, so the guard fires.

## Decisions

- **`IO (Option …)` vs pure `Option`**: chose (a) from the issue —
  lifting the return type — rather than threading raw standard-EOCD
  values through the return tuple (b). The precedent from issue
  #1739 (PR #1743) nudged toward (b), but the checks there needed
  only two extra pieces of information (EOCD position + comment
  length) both read from the tail buffer in the caller. The ZIP64
  sentinel check needs to observe whether the ZIP64 override branch
  actually fired plus five raw std values; widening the return tuple
  with audit-only fields or duplicating the locator-detection logic
  in `listFromHandle` are both noticeably uglier than a single
  `IO` lift.
- **Fresh generator** rather than patching `go-zip64.zip`: a new
  script (`scripts/build-zip64-malformed-fixtures.py`) gives future
  ZIP64-malformed fixtures a clean home and avoids brittle byte-patching.
