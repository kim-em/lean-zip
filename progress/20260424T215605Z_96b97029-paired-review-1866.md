## A. Guard correctness

- **A.1 Placement.** Guard lives in `parsePaxRecords` at
  [Zip/Tar.lean:131-134](/home/kim/lean-zip/Zip/Tar.lean:131),
  *after* the pair-`String.fromUTF8?` decode on both `keyBytes` and
  `valueBytes` succeeds (line 131), and *before* the `records.push`
  at line 134. The existing invalid-UTF-8 silent-skip precedent is
  immediately above on line 131 (same `if let (some key, some value)
  := ...` scrutinee), so non-UTF-8 garbage still fails first and
  NUL-smuggling fails second — error attribution (or rather, silent-
  skip attribution) stays distinct.
- **A.2 Predicate correctness.** Guard is
  `(keyBytes.findIdx? (· == 0)).isNone && (valueBytes.findIdx? (· == 0)).isNone`
  — a byte-level scan on the *raw* `ByteArray` slices, matching the
  sibling ZIP CD name guard at
  [Zip/Archive.lean:630](/home/kim/lean-zip/Zip/Archive.lean:630)
  (not `String.contains '\x00'` on the decoded strings, as the issue
  body's A.2 suggested). The `ByteArray` slices cleanly exclude the
  length-prefix / `=` / trailing-`\n` parsing-layer bytes:
    - `keyBytes := data.extract lenEnd eqPos` — starts after the
      length prefix + space (`lenEnd`), ends before `=` (`eqPos`).
    - `valueBytes := data.extract (eqPos + 1) valueEnd` — starts
      after `=`, ends before trailing `\n` via the `valueEnd`
      adjustment at lines 116-119.
  Predicate catches NUL at any position in either slice; empty
  key / empty value (zero-length `ByteArray`) is correctly *not*
  flagged (`findIdx? (· == 0)` on an empty array returns `none`).
- **A.3 `applyPaxOverrides` reachability.** Control flow
  `forEntries → parsePaxRecords → paxOverrides → applyPaxOverrides`
  traced through
  [Zip/Tar.lean:606-609](/home/kim/lean-zip/Zip/Tar.lean:606)
  (PAX block parsed into `paxOverrides`) and
  [Zip/Tar.lean:139-143](/home/kim/lean-zip/Zip/Tar.lean:139)
  (match arm `"path" => e := { e with path := value }`). The new
  guard prevents the `(key, value)` pair from being pushed into
  `records` at all, so `applyPaxOverrides` never sees a NUL-
  containing value. The `"linkpath"` match arm at line 144 is
  similarly unreachable.
- **A.4 Error-wording uniqueness — deviation from issue body.** The
  landed guard **silently skips** the record rather than throwing:
  `grep -rn "PAX.*NUL\|NUL.*PAX" Zip/ ZipTest/` returns **zero**
  hits. There is no new error substring; the issue body's A.4
  grep target does not exist because the guard chose the skip
  branch (see A.5 below).
- **A.5 Per-record-skip vs per-archive-throw policy — skip chosen.**
  The landed code picks **skip** (the invalid-UTF-8 precedent on
  the line immediately above), not **throw** (the sibling ZIP /
  GNU NUL guards). The merge-commit message justifies this
  explicitly: *"Extends the existing silent-skip precedent in
  `parsePaxRecords` (invalid UTF-8)"* — the adjacent guard on
  line 131 already drops records whose bytes don't decode as
  UTF-8 without failing the whole parse (to support legacy
  interop), and the NUL guard re-uses that same
  `records := records.push ...` branch-to-skip machinery. The
  consequence is that the fixture assertion is a **positive
  regression** (see C.1 below), not `assertThrows` — the smuggled
  `path=a\x00b/c` record silently doesn't override `entry.path`,
  the follow-on regular-file entry keeps its declared `"hello.txt"`
  name, and `Tar.list` returns one entry with that path rather
  than failing with a thrown error. The commit message
  documents the reasoning; the deviation is deliberate and
  coherent with `parsePaxRecords`'s existing tolerance model.

## B. Fixture correctness

- **B.1 Size + determinism.** `wc -c` = 2048 B; SHA-256
  `171ee4fb499f5cc64862980a08448e3275bb42db2844563dca7b978fe1371bfa`.
  Ran `lake -R env lean --run scripts/build-pax-malformed-fixtures.lean`
  twice back-to-back; both runs produced byte-identical output
  matching the committed fixture's SHA-256 exactly. Build output:
  *"Built 6 malformed PAX fixtures under testdata/tar/malformed/."*
- **B.2 PAX record shape — issue-body length prefix was wrong; landed
  prefix is correct.** Hex-dumping the fixture shows the PAX data
  block at offset `0x200` carries `"14 path=a\x00b/c\n"` verbatim
  (13 bytes + trailing NUL padding to 512 per block geometry). The
  length prefix `14` matches the record's total byte length:
  2 digits + 1 space + 4 `"path"` + 1 `"="` + 5 `"a\x00b/c"`
  + 1 `"\n"` = 14 bytes. The builder comment at
  [scripts/build-pax-malformed-fixtures.lean:91-92](/home/kim/lean-zip/scripts/build-pax-malformed-fixtures.lean:91)
  records the same arithmetic. The issue body's shape `"17 path=a\x00b/c\n"`
  was a miscount; the landed fixture uses the correct `14`.
  Because the length prefix matches the actual record byte length,
  `parsePaxRecords` succeeds at the length-decode step (not caught
  by the `digitCount > 20` guard at line 95 nor the
  `recordEnd > data.size` guard at line 105), so the test exercises
  the new NUL guard — not one of the earlier bail-outs.
- **B.3 Follow-on regular-file entry.** Block 3 at offset `0x400`
  is a standard UStar regular-file header with `name = "hello.txt"`
  (`typeflag = '0'`), which `parseHeader` accepts and
  `applyPaxOverrides` leaves unchanged (because no PAX records
  survive to override `entry.path`). The fixture isolates the
  PAX path — no UStar short-name NUL or other tar anomaly.
- **B.4 Interop pre-flight.** Swept
  `testdata/tar/{interop,malformed,security}/*.tar` for any
  pre-existing fixture whose PAX record's decoded key or value
  contains NUL. One hit surfaced —
  `testdata/tar/malformed/pax-truncated-record.tar` — but on
  inspection the fixture's PAX data block declares `size = 50`
  (octal `"0000000062"` in the UStar header at offset 124)
  while the record itself claims length 100 (`"100 path=foo\n"`).
  `parsePaxRecords` therefore breaks at the `recordEnd > data.size`
  guard on line 105 — `recordEnd = 0 + 100 = 100 > 50 = data.size`
  — *before* reaching the new NUL guard. Pre-existing silent-skip
  behaviour preserved; no collateral regression from the new
  guard. All other PAX fixtures have zero NUL bytes in the
  declared record payload. Net: the new fixture is the only
  archive the NUL guard actually fires on.

  `Tar.create` API would produce an archive that the reader
  now silently drops. This is the intended boundary.
