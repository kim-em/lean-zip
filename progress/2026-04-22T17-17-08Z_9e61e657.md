## a. Tactic elegance — Spec.checksum_replicate_zero

**Refactor applied** (commit on this branch).

The original proof had three avoidable line-cost spots:

1. The inductive step opened with an explicit
   `show ((1 + (0:UInt8).toNat) % MOD_ADLER, …) = (1, k + 1)`
   followed by a two-step `rw [show … from by decide, show … from
   Nat.mod_eq_of_lt hk1]`. The `show` exists only because plain `rw`
   cannot reach into `updateByte`'s `let`-bindings without an
   explicit reformulation of the goal.

2. The `hbyte : updateByte (1, k) 0 = (1, k + 1)` was a separate
   `have` whose only consumer was the next `rw`. Once the proof
   form is re-styled as `simp only [updateByte, …]`, the
   intermediate `have` becomes unnecessary — the simp can fire
   directly on the post-`updateList_cons` goal.

3. The instantiation step
   `have hupdate : updateList init (List.replicate n 0) = (1, n)`
   used a three-line `have h := … ; show … ; rw [h, Nat.zero_add]`
   ladder. The `show` is needed because `init` doesn't unfold under
   `rw`; `simpa [init]` does both the unfolding and the `0 + n` cleanup
   in one step.

**Refactored form** (proof body lines 211–231; was 211–236):

```lean
| succ m ih =>
  intro k hkm
  have hk1 : k + 1 < 65521 := by omega
  rw [List.replicate_succ, updateList_cons]
  simp only [updateByte, MOD_ADLER, show (0:UInt8).toNat = 0 from rfl,
    Nat.add_zero, show (1:Nat) % 65521 = 1 from rfl, Nat.mod_eq_of_lt hk1]
  rw [ih (k + 1) (by omega)]
  congr 1; omega
…
have hupdate : updateList init (List.replicate n 0) = (1, n) := by
  simpa [init] using hstate n 0 (by omega)
```

The `simp only` arg list looks longer than the old `rw [show … from
by decide, show … from Nat.mod_eq_of_lt hk1]`, but it absorbs the
three `let`-binding unfolds (`updateByte`, `MOD_ADLER`,
`(0:UInt8).toNat = 0`) plus both modular reductions in one block,
where the old form needed an explicit `show` reformulation to reach
the same unfolded shape.

The closing `rw [ih (k + 1) (by omega)] / congr 1; omega` cadence is
unchanged — the IH still consumes the strengthened form and a final
`omega` discharges `(k + 1) + m = k + (m + 1)`.

`pack_toNat_of_bounds` + `UInt32.toNat_ofNat_of_lt'` cadence in the
final block is identical to `_singleton` / `_pair` and already at the
minimum the helper supports; no further collapse is available there
(and #1650's three-`simp only`-arg collapse does not apply because
this proof has only one bridging step rather than three).

**Net delta:** `-5` LOC (`-10` removed, `+5` added).

## a. Tactic elegance — Native.adler32_replicate_zero

**No refactor.** Already at minimum cadence.

The Native bridge is three lines:

```lean
have hunpack : Spec.unpack 1 = Spec.init := by decide
simp only [adler32, updateBytes_eq_updateList, hunpack]
exact Spec.checksum_replicate_zero n hn
```

It is one `have`-line shorter than `adler32_singleton` /
`adler32_pair`, which need an extra `hdata :
(ByteArray.push … ).data.toList = [b]` step. Here
`(ByteArray.mk (Array.replicate n 0)).data.toList` reduces to
`List.replicate n 0` definitionally — so the bridge can drop
straight into `Spec.checksum_replicate_zero` after the unpack/init
collapse, with no list-shape rewrite needed.

This is the minimum the three-part template supports for an
`Array.replicate`-shaped input. No change recommended.

## d. Strengthen-then-instantiate

**Confirmed load-bearing.** A direct induction on
`checksum (List.replicate n 0) = …` cannot succeed because the
fold's start point is fixed at `init = (1, 0)`; the IH would need
to fire on `updateList (1, 1) (List.replicate m 0)`, but the
non-strengthened induction only yields the IH at `updateList init`.

The current strengthening
`∀ (m k : Nat), k + m < 65521 → updateList (1, k) (List.replicate m 0) = (1, k + m)`
is the minimal generalization: it frees the second component (`k`)
so the IH can be reapplied at `(1, k + 1)`. The first component
(`a = 1`) stays fixed because zero bytes never move it.

**Forward-looking note.** PR #1684 (general
`checksum_replicate`, merged `ec3ba96`) needed a *more* strongly
strengthened invariant — both components vary because nonzero
bytes shift the `A` component too:

```lean
hstate : ∀ (m a bsum : Nat),
    a + m * b.toNat < 65521 →
    bsum + m * a + b.toNat * (m * (m + 1) / 2) < 65521 →
    updateList (a, bsum) (List.replicate m b) =
      (a + m * b.toNat, bsum + m * a + b.toNat * (m * (m + 1) / 2))
```

The pattern transferred cleanly: free both components, prove a
two-bound invariant, instantiate at `(1, 0)` with `n` and the
caller-supplied bounds. Future `_replicate`-flavored ladder
extensions (e.g. mixed-byte windows, periodic patterns) should
reach for the same shape — generalize over both components and
expose the bound predicates explicitly to the instantiator.
