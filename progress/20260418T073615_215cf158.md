## Summary

Fixed master CI by combining remedy 1 (`-R` flag) with remedy 3 (pin
floating `zipCommon` ref). Neither remedy alone was sufficient because
zipCommon `main` had drifted past the last commit compatible with
master's lemma references.

## What changed

- `.github/workflows/ci.yml`: `lake build` → `lake -R build`,
  `lake exe test` → `lake -R exe test`. Safety net (`rm -rf .lake` on
  cache miss) kept in place as directed by the issue for remedies 1–2.
- `lakefile.lean`: `require zipCommon ... @ "main"` pinned to commit
  `f53e8866e9c8427aa887dc9329d43912968c45f9` — the exact commit that
  zipCommon resolved to in the last successful `lake build` on master
  CI (seen in run 24599212775, timestamp 06:51:11Z).

No Lean code changes. No changes to `run_io linkFlags` (remedy 4
deferred as the issue instructed).

## Why both remedies were required

- **Just `-R`**: master code uses `Array.getElem!_le_set!_incr`,
  `Nat.or_two_pow_eq_add`, `List.foldl_set_getElem_mem`, etc. All four
  were renamed or dropped in zipCommon commits after `f53e886`. A fresh
  CI `lake build` against `main = 76d8428` fails with unknown-constant
  errors before ever reaching the trace-validation step.
- **Just pin**: PR #1502 established that `lake -R build` succeeds but
  `lake exe test` (without `-R`) still raises
  `compiled configuration is invalid` on CI. The test step must also
  reconfigure.
