Session type: feature (autonomous pod agent, UUID prefix `ce0c1732`).
Issue: [#1575](https://github.com/kim-em/lean-zip/issues/1575) — Track E P4.3
record locally-guarded `Handle.read` / `Stream.read` callsites.

## What landed

- New subsection **"Local guard inventory for `Handle.read` and
  `Stream.read`"** in `SECURITY_INVENTORY.md`, placed at the end of the
  existing *"Decompression Limit Inventory"*. Twelve rows cover every
  read reachable from untrusted archive bytes in `Zip/Archive.lean`
  and `Zip/Tar.lean`:
  - Archive.lean: 6 rows (readExact for EOCD tail, central directory,
    local header, local name+extra, compressed data; plus the
    `readExactStream` helper documented as test-only today).
  - Tar.lean: 6 rows (readExact for 512-byte header blocks;
    readEntryData grouped over GNU long-name / long-link / PAX
    extended / PAX global; skipEntryData grouped over directory /
    symlink / unsupported typeflags / `Tar.list`; inline
    regular-file and padding reads in `Tar.extract`; fixed-chunk
    gzip outer read in `extractTarGz`).
  Each row's *"Local guard"* column names a concrete symbol
  (`assertSpanInFile`, `maxCentralDirSize`, CD-vs-LH consistency,
  `UInt16` type bound, `Nat → USize` roundtrip, etc.) or honestly
  says `none` and explains why.
- Added a sub-bullet under the Lean Runtime entry's "Missing work"
  section pointing at the new inventory.
- Ticked Priority 4 item 3 in `plans/track-e-current-audit-checklist.md`
  with a pointer to the new subsection.

## Decisions / honest calls

- **Excluded `Zip/Tar.lean:442`** (`create`'s `h.read` on local files
  chosen by the archive author) — not attacker-controlled, documented
  in the prose preamble.
- **Grouped four-caller rows** for `readEntryData` and `skipEntryData`
  rather than one-per-caller, to keep the table near the "~10 rows"
  target the issue requested.
- **Two residual gaps flagged** for follow-up (not fixed here, per the
  doc-only scope):
  1. `Tar.readEntryData` at the four GNU long-name / PAX callsites
     accumulates `entry.size` into memory without a local cap.
  2. `Tar.extract` relies on `maxEntrySize` defaulting to `0` (no
     limit). Both are already covered by the *"Decompression Limit
     Inventory"* recommendations but had not been explicitly tied to
     the read-site audit before.
