## Findings per audit check

### A.1 Predicate correctness — OK

The guard at
[Zip/Archive.lean:557-559](/home/kim/lean-zip/Zip/Archive.lean:557):

```lean
let nameBytes := data.extract (pos + 46) (pos + 46 + nameLen)
-- ... (14 lines of comment)
if (nameBytes.findIdx? (· == 0)).isSome then
  throw (IO.userError
    s!"zip: CD entry name contains NUL byte at CD offset {pos}")
```

`nameBytes : ByteArray` is extracted from the raw archive bytes
at CD offset `pos + 46` (APPNOTE §4.4.7 — name payload
immediately after the 46-byte CD header), and the predicate is
the literal `(nameBytes.findIdx? (· == 0)).isSome` over the
raw `ByteArray` — not a post-decode `String.contains '\x00'`
form. The check runs on the raw bytes before the subsequent
`let name ←` decode block (line 560), so:
- the error message can never re-introduce attacker-controlled
  NUL bytes into logs (the `pos` interpolation is a `Nat`);
- both the UTF-8-flagged and Latin-1-fallback decode branches
  at `:561-570` are closed uniformly (the guard fires
  regardless of the bit-11 UTF-8 flag);
- the `Binary.fromLatin1` fallback, which would otherwise
  preserve NUL as U+0000 in the decoded `String`, cannot
  produce an `Entry` whose `path` embeds NUL.

### A.2 Insertion site — OK

The guard sits in `parseCentralDir`'s per-entry loop,
immediately after the `nameBytes` extraction (line 544) and
before the `let name ←` decode block (line 560). Full pre-NUL
check chain in current master (`parseCentralDir` lines
491-620):

1. Signature check `sig == 0x02014b50` (`:495-496`; break on
   non-match, no throw)
2. CD-header field reads (`versionNeeded`/`flags`/`method`/
   `crc`/`stdCompSize`/`stdUncompSize`/`nameLen`/`extraLen`/
   `commentLen`, `:497-507`)
3. `diskNumberStart == 0` (`:516-519`, PR #1759)
4. `internalAttrs &&& 0xFFFE == 0` (`:536-539`, PR #1819)
5. `entryEnd <= cdEnd` bounds (`:540-542`)
6. `stdOffset` read (`:543`, UInt32 at pos+42)
7. `nameBytes` extraction (`:544`)
8. **NUL-byte guard (`:557-559`, PR #1831)**
9. UTF-8 / Latin-1 decode (`:560-570`)
10. `versionNeeded <= 45` upper bound (`:588-590`, PR #1807)
11. `method == 0 ∨ method == 8` allowlist (`:605-607`,
    PR #1801)
12. `flags &&& 0x0020 == 0` patched-data bit-5 (`:619+`,
    PR #1824)

Running the NUL guard pre-decode is the correct placement:
the alternative (post-decode `String.contains '\x00'`) would
(a) let the error-message `{name}` interpolation re-inject
NUL into logs if the downstream throw references it, and
(b) force the Latin-1 fallback to carry NUL through
`Binary.fromLatin1` unconditionally (no way to distinguish a
legitimately-encoded `\x00` from the attacker-supplied raw
byte). The raw-bytes guard closes both concerns uniformly.

Guard 7 (`entryEnd <= cdEnd`) correctly precedes the NUL
guard: a malformed archive with a `nameLen` that overflows
`cdEnd` must be caught by the bounds check before `nameBytes`
is extracted, otherwise `data.extract` would silently truncate
to the tail of `data` and the NUL scan would see irrelevant
bytes from beyond the CD region.

### E.2 Attack-class closure — OK

The NUL-byte vector closes the *content* dimension at the
CD-name layer for `Archive.list` (which previously returned
an `Entry` with the NUL-containing `path` verbatim — both
`String.fromUTF8?` and `Binary.fromLatin1` preserve U+0000).
It simultaneously closes the `Archive.extract` path: the
extract-time POSIX `open`/`stat` layer would have truncated
at NUL and deposited the extracted file at the short-form
prefix (`evil.txt` rather than the smuggled `evil.txt\x00.zip`),
bypassing peer readers' strict NUL handling.

The in-flight CD/LH name-bytes consistency check (#1722)
does **not** close the symmetric LH-side gap on its own:
#1722 rejects *CD-vs-LH divergence* but does not inspect
per-byte contents. A crafted archive with matching NUL in
both CD and LH names (like the new fixture) would pass
#1722's CD-vs-LH equality check and reach the UTF-8 decode
with a NUL-bearing `String`. The NUL-byte guard closes this
residual gap by rejecting at the CD side before either
decode branch or the in-flight #1722 consistency check
fires. No residual gap identified.

Writer-side at
[Zip/Archive.lean:84](/home/kim/lean-zip/Zip/Archive.lean:84)
and `:110` inherits the invariant from caller-supplied
`entry.path` — lean-zip's writer does not enforce
NUL-freeness at emit time, but legitimate callers never
embed NUL in filenames (Unix/macOS/Windows filesystems all
reject NUL in paths, so no caller could obtain a NUL-bearing
path from a real filesystem walk). The guard is defence-
in-depth on the reader side.

The next content-class columns to close, in rough
priority order:

1. **CD-entry `fileComment` NUL byte** — APPNOTE §4.4.18
   leaves the comment byte-space unconstrained; a NUL in
   the comment is a smaller attack surface than the name
   (comments don't route through filesystem `open`) but
   still a parser-differential vector for callers that
   surface `entry.fileComment` to users.
2. **EOCD `archiveComment` NUL byte** — APPNOTE §4.3.16
   comment field; same rationale as (1) but at the
   archive level.
3. **CD/LH extra-data content-class** — the extra-data
   sub-field `dataBytes` payloads are APPNOTE-opaque but
   may embed NUL or other problematic bytes; low-priority
   since the existing structural checks (#1788, #1793,
   #1785) already close most smuggling vectors.
4. **Control-character sweep in CD-entry `name`** — the
   NUL guard closes the NUL dimension; the next question
   is whether other ASCII control characters (`\r`, `\n`,
   `\x01`-`\x1F`, `\x7F`) should also be rejected. Info-ZIP
   and Go's `archive/zip` vary on this — a future planner
   should survey before adding the guard.

Follow-up 4 is the most likely next issue in the sequence;
a planner should cross-reference the interop corpus to
confirm no legitimate archive carries control characters
in names before landing the guard.
