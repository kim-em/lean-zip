## Decisions

- **Single row per family, not per slot**. The issue body asked for
  "at least four new catalogue rows" with the family-substring approach
  for the UStar and GNU rows. I followed that and used the recommended-match
  column to call out the per-slot distinguishing prefixes — same pattern as
  the existing CD entry NUL byte row (line 38 of SKILL.md).
- **Included the `uname` slot in the UStar row**. The issue body listed
  three substrings (`name` / `linkname` / `prefix`) but PR #1944 (commit
  28ef13c — landed since the issue was opened) extended the family to
  `uname` as the 4th defense-in-depth slot. The issue's verification
  checklist says "Recommended-match substrings each match the actual error
  message thrown by current Zip/Tar.lean", so the row reflects the 4-slot
  shape at HEAD. The 5th-slot `gname` is explicitly noted as deferred to
  follow-up issue #1946.
- **Included the optional PAX silent-skip row**. The issue body marked
  it optional, but the row fits naturally as the silent-skip dual to the
  PAX duplicate-key throw on the same `parsePaxRecords` surface. Schema-fit:
  the *Full message* column carries `(silent skip — no IO.userError)` and
  the *Recommended match substring* column carries
  `(no assertThrows substring — attribution is fixture-only)` plus the
  attribution-by-recovered-value documentation. Both columns parse as
  italic prose rather than substring values, but the row remains in the
  same 4-column shape as every other row.
- **Placed the new rows before the existing tar bomb-limit rows**, per
  the issue's "before the existing tar bomb-limit rows because they are
  parse-time guards (logically earlier in the pipeline)" instruction.
