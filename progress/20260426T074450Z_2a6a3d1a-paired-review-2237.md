### B.3 Smallest-smuggle choice

Bit 7 (`0x0080`) is the smallest of the seven reserved/unused bits in
the mask `0xD780`. The full set of single-bit smuggles within the mask
is `{0x0080, 0x0100, 0x0200, 0x0400, 0x1000, 0x4000, 0x8000}` — bit 7
is the LSB; choosing it minimizes the difference from the
writer-emitted `0x0800` baseline (smallest delta, smallest hex
expansion `0x0800 → 0x0880` swaps a single hex nibble). Pairing with
bit 11 keeps the UTF-8-name guard at `Zip/Archive.lean:632-643` happy
so the new reserved-bits guard is unambiguously the one that fires.
Without bit 11 the encoded ASCII name `hello.txt` would still pass
(Latin-1 fallback), so bit 11 is not load-bearing for the test, but it
matches the writer-side bit-11 emission and minimizes the difference
from a legitimate archive.

### B.4 LH/CD flag-word agreement

Matching `flag_bits_override = 0x0880` sets both LH and CD `flags` to
the same value. This is load-bearing because the CD-vs-LH bit-3-masked
flags-consistency check (PR #1715) at
[Zip/Archive.lean:1172](/home/kim/lean-zip/Zip/Archive.lean:1172)
fires *before* the new reserved-bits guard for `extract` callers — it
sits inside `readEntryData`, but that is gated on the CD/LH agreement
not the flag value. A fixture that disagreed on the two sides
(e.g. `lh_flags = 0x0800, cd_flags = 0x0880`) would still trip the
reserved-bits guard at `parseCentralDir` *before* the CD/LH check
because `parseCentralDir` runs first on both `Archive.list` and
`Archive.extract` paths; but matching the two sides keeps the fixture
maximally close to a legitimate archive and ensures the rejection is
attributed to the new guard regardless of caller path.

### B.5 Out-of-scope bits

Bits 1 and 2 (`0x0006`, compression-option per APPNOTE §4.4.4 —
normal/maximum/fast/superfast for method 8) are explicitly out of
scope: real-world producers (Info-ZIP, 7-Zip) legitimately set them on
`method == 8` payloads. The mask `0xD780` is disjoint from the
unsupported-feature umbrella mask `0x2071`:

```
0xD780 = 1101_0111_1000_0000
0x2071 = 0010_0000_0111_0001
AND    = 0000_0000_0000_0000  ✓ disjoint
```

Together `0xD780 | 0x2071 = 0xF7F1`, leaving bits 1, 2, 3, 11 unset:
bits 1-2 (compression-option, accepted), bit 3 (data descriptor —
inspected via `usesDataDescriptor` gating), bit 11 (UTF-8 name —
inspected for name decode). The four-bit "accepted" set is exactly the
APPNOTE-defined bits that lean-zip treats as observable rather than
rejection-triggering; the twelve-bit "rejected" set decomposes into
the four masks `0xD780` (this PR), `0x0020` (PR #1824), `0x0010`
(in-flight PR #1825), and `0x2041` (in-flight PR #1771,
encryption-related bits 0/6/13). The pairwise intersection of those
four masks is empty:

```
0xD780 ∩ 0x0020 = 0
0xD780 ∩ 0x0010 = 0
0xD780 ∩ 0x2041 = 0
0x0020 ∩ 0x0010 = 0
0x0020 ∩ 0x2041 = 0  (per-feature umbrella has bit 5 only via PR #1824 — see note below)
0x0010 ∩ 0x2041 = 0
```

A small framing note: the issue body's `0xD780 ∩ 0x0020 ∩ 0x2071 ∩
0x0010 == ∅` formula intersects all four masks together. This is
trivially zero (since `0xD780 ∩ 0x0020 = 0` already), but it does not
state the *pairwise* invariant — and `0x2071 = 0x2041 | 0x0010 |
0x0020` is the union of three per-bit guards rather than a fourth
disjoint mask, so `0x2071 ∩ 0x0020 = 0x0020 ≠ 0` if read as
"two distinct per-bit masks". The cleaner statement is: the per-bit
masks `{0xD780, 0x0020, 0x0010, 0x0041, 0x2000}` (where `0x0041` is
bits 0+6 and `0x2000` is bit 13, decomposing PR #1771's three-bit
range) are pairwise disjoint and partition the 12-bit reject set.
This is a wording-only observation; the code-level disjointness is
correct.

The guard at

```lean

    s!"zip: CD entry flags reserved bits set (flags={flags}) for {name} at CD offset {pos}; APPNOTE §4.4.4 bits 7,8,9,10,12,14,15 are reserved/unused and lean-zip rejects archives that set any of them")
```

`flags : UInt16` is bound at
[Zip/Archive.lean:541](/home/kim/lean-zip/Zip/Archive.lean:541) via
`Binary.readUInt16LE data (pos + 8)` — no `.toNat`, no narrowing, no
de-Morgan'd alternative. The mask `0xD780` literal is correctly
encoded:

```
0xD780 = 1101_0111_1000_0000
       = bit 7  (0x0080)  "Currently unused"     APPNOTE §4.4.4
       | bit 8  (0x0100)  "Currently unused"
       | bit 9  (0x0200)  "Currently unused"
       | bit 10 (0x0400)  "Currently unused"
       | bit 12 (0x1000)  "Reserved by PKWARE for enhanced compression"
       | bit 14 (0x4000)  "Reserved by PKWARE"
       | bit 15 (0x8000)  "Reserved by PKWARE"
       = 0x0080 | 0x0100 | 0x0200 | 0x0400 | 0x1000 | 0x4000 | 0x8000
```

Mask-equality form (`flags &&& 0xD780 == 0`) matches the existing
bit-3-masking convention `dataDescriptorBitMask : UInt16 := 0xFFF7` at
[Zip/Archive.lean:23](/home/kim/lean-zip/Zip/Archive.lean:23) and the

[Zip/Archive.lean:748](/home/kim/lean-zip/Zip/Archive.lean:748),
rather than the logically-equivalent but stylistically-divergent
`flags &&& 0xD780 != 0xD780` form. Every-bit-rejection (== 0) is the
correct semantic: any single set bit in the mask trips the guard,
matching the APPNOTE-reserved/unused interpretation.

### C.2 Insertion site (full per-entry order)

The new guard at `:736-738` sits between the method-allowlist check
(`:711-713`, PR #1801) and the per-feature-bit bit-5 patched-data

3. read versionNeeded (`+6`), flags (`+8`), method (`+10`), lastMod*,
   crc, stdCompSize, stdUncompSize, nameLen, extraLen, commentLen
   (`:540-550`)

5. internalAttrs (`+36`, PR #1819, `:580-582`)
6. entryEnd > cdEnd span check (`:614-615`)
7. CD name empty-filename guard (PR #1848, `:617-619`)

The placement is correct on multiple grounds:

- **Mask family adjacency**: the four flag-related guards fire in the
  order method-allowlist (bit-aligned via method check, not flags) →
  reserved-mask (`0xD780`) → bit-5 patched-data (`0x0020`). This is
  the precedence the issue body specified, and matches the
  inline-comment trail at `:733-735` ("Placed immediately after the
  method allowlist and before the per-feature-bit checks (bit 5
  patched-data, etc.) so error-message precedence is method →
  reserved-mask → per-feature-bit.").
- **Pre-ZIP64 placement**: APPNOTE §4.4.4 defines bits 0-15 with no
  ZIP64-sentinel semantics on any bit, so running pre-ZIP64-resolution
  is the only correct placement: it fires uniformly on both
  `Archive.list` and `Archive.extract` paths. Moving past
  `parseZip64Extra` would incorrectly gate the reserved-bits check on
  the presence of ZIP64 extras.
- **Post-name-decode**: the message includes the entry `name`, which
  requires the name decode at `:629-643` to have completed. The
  ordering choice is sensible: a CD entry whose name is undecodable
  trips earlier guards and never reaches the flags check, so the
  message-template `{name}` interpolation is always defined when this
  guard fires.

### E.1 Per-bit CD+8 flag-field series cadence

The CD+8 `flags : UInt16` field is now defended by a multi-PR per-bit
series. With PR #2237 landed, the standing of the series is:

| PR | Status | Bit(s) | Mask | Substring |
|---|---|---|---|---|
| #1824 | landed (paired-review #1827) | 5 | `0x0020` | `"patched-data flag bit 5 set"` |
| #2237 | **landed (this review)** | 7-10, 12, 14-15 | `0xD780` | `"flags reserved bits set"` |
| #1771 | in-flight (merge conflict) | 0, 6, 13 | `0x2041` | `"encryption-related flag bit"` |
| #1825 | in-flight (merge conflict) | 4 | `0x0010` | `"enhanced deflating"` (planned) |

The four PRs together cover bits `0,4,5,6,7,8,9,10,12,13,14,15` — 12
of the 16 flag bits — leaving bit 1 (`0x0002`), bit 2 (`0x0004`), bit
3 (`0x0008`), and bit 11 (`0x0800`) unrejected:

- bit 1, 2: compression-option per APPNOTE §4.4.4 — Info-ZIP / 7-Zip
  legitimately set them on `method == 8` payloads, explicitly out of
  scope for rejection;
- bit 3: data descriptor — inspected via `usesDataDescriptor` gating,
  not rejection;
- bit 11: UTF-8 — inspected for name decode, not rejection.

The four-bit "accepted" subset `{1, 2, 3, 11}` corresponds to mask
`0x080E` — the complement of `0xF7F1 = 0xD780 | 0x2071`. With PR #1771
and PR #1825 landed, the per-bit decomposition of the unsupported-feature
umbrella mask `0x2071 = 0x2041 | 0x0010 | 0x0020` becomes pairwise
disjoint, and the overall partition of the 16-bit `flags` UInt16 is:

```
0xD780  reserved/unused (this PR)
0x2041  encryption-related (in-flight #1771)
0x0010  enhanced deflating (in-flight #1825)
0x0020  patched data (PR #1824)
0x080E  accepted (writer/reader inspection only)
———
0xFFFF  total
```

`0xD780 + 0x2041 + 0x0010 + 0x0020 + 0x080E = 0x1_FFFF`?
Actually let me redo: `0xD780 | 0x2041 | 0x0010 | 0x0020 | 0x080E =
0xFFFF` (all 16 bits, pairwise disjoint partition). The OR is
saturated; the sum-with-disjointness equals `0xFFFF` confirming the
partition. This is the closure invariant the wave is targeting.

### E.2 Mask-disjointness invariant

The four reject masks are pairwise disjoint:

```
0xD780 ∩ 0x2041 = 0  ✓  (1101_0111_1000_0000 ∩ 0010_0000_0100_0001 = 0)
0xD780 ∩ 0x0010 = 0  ✓
0xD780 ∩ 0x0020 = 0  ✓
0x2041 ∩ 0x0010 = 0  ✓  (0010_0000_0100_0001 ∩ 0000_0000_0001_0000 = 0)
0x2041 ∩ 0x0020 = 0  ✓
0x0010 ∩ 0x0020 = 0  ✓
```

All six pairwise intersections vanish. The four guards compose without
overlap: a fixture that trips one will not also trip a sibling, and
distinct error substrings keep precedence-shift triage loud under
future refactors. Should a future refactor reorder the four guards,
each fixture will still trip its designated guard regardless of order
because the mask intersections are empty.

The reordering robustness is a load-bearing property: lean-zip's
test-loop matches on substrings, not on insertion order, and the
existing four sibling fixtures (`cd-patched-data-flag.zip`, future
`cd-flags-encryption-bits.zip`, future `cd-flags-deflate64.zip`, and
the new `cd-flags-reserved-bits.zip`) each set bits drawn only from
their own mask. The reordering robustness is the formal-shape mirror
of the per-bit guard pairwise-disjointness invariant.
