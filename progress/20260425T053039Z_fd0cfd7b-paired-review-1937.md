### A.2 Block geometry (1 536 B)

All four family fixtures are exactly **1 536 B**: 512 B of UStar header
+ 1 024 B of zero padding (two trailing zero blocks forming a
well-formed end-of-archive). The zero-padding tail matches strict peer
parsers' (GNU tar / BSD tar / libarchive) accept-condition for
end-of-archive; the guard fires during header parse so the trailing
blocks are only exercised by the no-guard regression baseline.

The 1 536 B size is identical across the four sibling fixtures — the
only bytes that differ across the family are within the 512 B header
(the per-slot smuggled value plus the recomputed `chksum` octets). This
matches the `name`-slot and `linkname`-slot rows' geometry exactly.

### A.3 Smuggled value choice: `prefix = "badpfx\x00bad"` (10 B printable + interior NUL)

Per the `buildUstarPrefixNulInName` helper added by PR #1937, the
fixture is built with `pathOverride := some ("badpfx\x00bad",
"name.txt")` — the writer's `pathOverride` hook splits the input pair
into `prefix` and `name` slots directly (rather than running the
default `splitPath` heuristic on `entry.path`), so the `prefix` slot at
offset 345 carries `"badpfx\x00bad"` while the `name` slot at offset 0
carries the clean `"name.txt"`. The 512 B header layout:

| Slice                    | Offset       | Size  | Content |
|--------------------------|--------------|-------|---------|
| `name`                   | `0x000..0x063` | 100 B | `"name.txt"` (8 B printable) + 92 B NUL padding — **clean** |
| `mode` ... `mtime`       | `0x064..0x093` | 48 B  | octal numeric fields (writer-emitted) |
| `chksum`                 | `0x094..0x09B` | 8 B   | recomputed by writer to match the smuggled `prefix` |
| `typeflag`               | `0x09C`        | 1 B   | `'0'` (REGTYPE — regular file) |
| `linkname`               | `0x09D..0x100` | 100 B | all-NUL — **clean** |
| `magic`                  | `0x101..0x106` | 6 B   | `"ustar\x00"` — UStar branch entered |
| `version`                | `0x107..0x108` | 2 B   | `"00"` |
| `uname` ... `devminor`   | `0x109..0x158` | 80 B  | NUL-padded — **clean** (uname/gname slots NUL-free) |
| `prefix`                 | `0x159..0x1F3` | 155 B | **`"badpfx\x00bad"`** (10 B with interior NUL at byte 6) + 145 B NUL padding |
| `padding`                | `0x1F4..0x1FF` | 12 B  | NUL pad to 512 B block |

Header inspection confirms the three slot-pinning conditions:

- **`name = "name.txt"` is NUL-free** (the smuggled NUL is in
  `prefix`, not `name`) — so the `name`-arm sub-check in
  `parseHeader` evaluates `false`, the guard does NOT fire on the
  `name` arm, and control reaches the `linkname` sub-check.
- **`linkname` is all-NUL (clean)** — the `linkname`-arm sub-check
  evaluates `false`, control passes through.
- **`prefix = "badpfx\x00bad"` carries an interior NUL at byte 6**
  (between `'x'` and `'b'`, before the trailing NUL padding starts) —
  the `prefix`-arm sub-check evaluates `true`, the throw fires with
  substring `"UStar prefix contains NUL byte"`, and (post-PR #1944)
  control never reaches the `uname` sub-check.

So the `prefix` arm is the **uniquely** tripped sub-check for this
fixture. The choice of `"badpfx\x00bad"` is well-defended:

- **10 printable-ASCII bytes** (`badpfx`, `bad` separated by an
  interior NUL) — symmetrically shaped to PR #1880's `name`-slot
  smuggle `"evil\x00.tar"` (12 bytes; NUL at byte 4) and to PR #1934's
  `linkname`-slot smuggle `"evil.lnk\x00.tar"` (13 bytes; NUL at
  byte 8). The interior NUL position is offset 6 inside the 10-byte
  payload — strictly *interior*, not a trailing padding NUL.
- **Distinct from the `name` and `linkname` slots' smuggles** — the
  `badpfx` infix is recognisable in `xxd` diffs across the family and
  encodes the slot identity in its content (matching the `evil.lnk`
  infix used by PR #1934 to mark the linkname slot).
- **Fits well within the 155-byte `prefix` field** (10 of 155 bytes
  used; 145 B of NUL padding follows) — no field-overrun hazard.
- **Exercises the `pathOverride` writer hook** (the smuggle vector
  PR #1937 explicitly opens up at `Tar.UStar.buildHeader`) — the
  fixture is exactly the regression for the writer-side `pathOverride
  := some (..., ...)` smuggle path. Without the guard, a malicious
  caller passing `pathOverride := some ("evil/path\x00bad",
  "innocent.txt")` would emit a header that round-trips through
  `Tar.list` as `entry.path = "evil/path/innocent.txt"` truncated to
  `"evil/path"` while a strict peer parser preserves the full bytes —
  a parser-differential smuggle.

The smuggled value's `prefix` slot location (offset 345 / 155 B) is
the third and last slot of the 3-slot filesystem-reaching arm; sibling
slots are `name` at offset 0 / 100 B (PR #1880) and `linkname` at
offset 157 / 100 B (PR #1934).

Each sub-check has the canonical form `if hasInteriorNul block

(`"name contains NUL byte"` /

the `name` and `linkname` slots must be NUL-clean (and this fixture's
`name = "name.txt"` and all-NUL `linkname` slot ensure that).

- **F.1 Per-slot-distinct substring discipline at the test level.**

- **F.2 `name = "name.txt"` discipline for the prefix-slot fixture.**
  The fixture's `name = "name.txt"` is deliberately a 8-byte
  printable-ASCII payload with no smuggling potential. This
  discipline ensures the `name`-arm guard cannot fire first under
  any header-parse interpretation — even an attacker who somehow
  controlled the `name` field would see this fixture decline the
  smuggle on that arm. The `"name.txt"` choice is symmetric to
  PR #1934's `name = "safe"` for the linkname-slot fixture (4-byte
  printable; no smuggle); both demonstrate the test-side analog of
  the writer-side invariant ("`Binary.writeString` is NUL-padding-only
  unless the caller passes a literal `\x00` codepoint or a
  `pathOverride`"). The discipline is uniform across the per-slot
  family.

- **F.3 Writer-side compliance with the `pathOverride` caveat.**
  lean-zip's own `Tar.UStar.buildHeader` emits `prefix` via
  `Binary.writeString`, which is NUL-padding-only — the writer
  cannot emit an interior NUL in the `prefix` slot unless either
  (a) the caller passes `entry.path` with a literal `\x00`
  codepoint that survives the default `splitPath` heuristic (very
  unlikely in practice since `splitPath` operates on
  forward-slash-delimited substrings), or (b) the caller passes
  `pathOverride := some ("...\x00...", "...")` — the explicit
  writer-side smuggle vector that this fixture exercises
  directly. The writer-side non-smuggle invariant holds for
  unintentional smuggles (PR #1937's guard cannot fire on a
  legitimately-produced lean-zip archive), but the `pathOverride`
  vector demonstrates a **deliberate-smuggle case** — and the
  reader-side guard at `Zip/Tar.lean:530` is the regression's
  primary defense. The fixture's `pathOverride := some
  ("badpfx\x00bad", "name.txt")` injection exercises that
  deliberate-smuggle path end-to-end.
