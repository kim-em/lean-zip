## Key decisions and patterns discovered
- **`MProd` vs `Prod`**: Desugared `do`-notation uses `MProd` for tuple state,
  NOT `Prod`. Type annotations `(s : A × B)` create `Prod` which causes
  unification failure. Fix: omit the annotation so Lean infers `MProd`.
- **`Array.getElem_setIfInBounds` is `@[grind =]` not `@[simp]`**: After
  `unfold Array.getD` + `simp [Array.size_setIfInBounds]`, the bound proof
  gets transported and `simp only [Array.getElem_setIfInBounds]` can't match.
  Solution: prove `getD_set!` helper using `Array.getD_eq_getD_getElem?` +
  `Array.getElem?_setIfInBounds`.
- **`get!` = `getD _ 0` for Nat**: `Array.getElem!_eq_getD` gives the bridge.
  For Nat, `default = 0` so they're definitionally equal. Use `show` or `exact`
  directly without conversion.
- **forIn counter vs element**: In `forIn_range_preserves_indexed`, the counter
  `k` and element `a` are distinct variables (even though equal for `[:n]`).
  Must name the element and use it (not the counter) when indexing into arrays
  from the loop body, so omega can connect bounds to goal terms.
