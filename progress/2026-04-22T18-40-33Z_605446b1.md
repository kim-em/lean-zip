## Tactic elegance findings

### `Spec.checksum_singleton`

The issue flagged the closing `generalize` + `bv_decide` dance as
load-bearing. Confirmed empirically by removing the `generalize` line
and rebuilding: `bv_decide` reports

    It abstracted the following unsupported expressions as opaque
    variables: [mkTable[255 ^^^ b.toNat], mkTable[255 ^^^ b.toNat]]

i.e. the two table accesses share an index but carry **different**
proof terms (`hidx ▸ hlt` on the LHS, `xor_ff_byte_lt_mkTable_size b`
on the RHS). `bv_decide` abstracts non-BitVec subterms syntactically,
so it sees them as distinct opaque variables and fails. The
`generalize` step makes them syntactically identical by factoring out
the shared `mkTable[0xFF ^^^ b.toNat]'(hidx ▸ hlt)` expression as a
single variable `t`. The next consumer of this idiom should keep it.

Also tried dropping the `by exact` wrapper on line 161
(`mkTable[...]'(by exact xor_ff_byte_lt_mkTable_size b)`). Confirmed
still required: replacing it with a bare term triggers
`maximum recursion depth has been reached` during elaboration. The
wrapper forces elaboration to defer type-class / unification work
until after the proof body begins, breaking a loop. Keep it.

**Small wins applied** (−2 LOC, one bare simp eliminated):

- `hb32 : b.toNat < UInt32.size` was proved via
  `by have := b.toNat_lt; simp [UInt32.size]; omega` — a bare `simp`
  used only to unfold `UInt32.size`. Replaced with
  `Nat.lt_trans b.toNat_lt (by decide)`, which avoids `simp` entirely
  and makes the witness mechanical.
- The `Nat.and_two_pow_sub_one_of_lt_two_pow (n := 8) (by have ...)`
  argument can be replaced by `b.toNat_lt` directly, because
  `b.toNat_lt : b.toNat < 2 ^ 8` (the statement shape Lean needs).
  Same simplification applied inside `xor_ff_byte_lt_mkTable_size`.

Tried consolidating the `hidx` proof (rewriting 5 `rw` lemmas + 3
`show ... from rfl` into a single `simp only` + `rfl`). Does **not**
work: `simp only [UInt32.toNat_and, UInt32.toNat_xor, ...]` leaves
residual `UInt32.toNat 4294967295 &&& UInt32.toNat 255` that is not
definitionally reducible to `0xFFFFFFFF &&& 0xFF` in the simp context.
The explicit `show ((0xFFFFFFFF : UInt32).toNat) = 0xFFFFFFFF from rfl`
rewrites are needed to normalize those numerals. Kept the 7-line
proof.

### `Spec.crcByteTable_mkTable_eq_crcByte`

New public bridge lemma (3-line proof body via
`simp only [crcByteTable]` + `rw [dif_pos, mkTable_getElem,
UInt32.ofNat_toNat]` + `simp only [crcByte]` + final
`rw [crcBits8_split _, xor_byte_shr8]`). Structurally mirrors the
Native-side `crcByteTable_eq_crcByte` (`Zip/Native/Crc32.lean:72`)
and is already as tight as the helper stack allows. No refactor
applied.

The lemma is stated at the Spec layer (with `mkTable` baked in)
rather than for an arbitrary `table : Array UInt32`, which is the
right call: future ladder rungs in the Spec namespace can reuse it
directly without re-proving table specialisation.

### `Native.crc32_singleton`

Three-line bridge via `updateBytes_eq_updateList` +
`Spec.checksum_singleton`, matching the `adler32_singleton` template
(`Zip/Native/Adler32.lean:48`) in spirit but with one extra line
because `crc32`'s wrapper contains an `if init == 0` guard that
`adler32`'s does not.

Tried tightening the `show _ ^^^ (0xFFFFFFFF : UInt32) = _` +
`rw [show (0 == 0) = true from rfl, if_pos rfl, ...]` sequence to a
single `show (if _ then _ else _) ^^^ _ = _` + `rw [if_pos rfl, ...]`.
**Does not work**: causes `maximum recursion depth has been reached`
on elaboration, the same infinite-loop symptom as the
`mkTable[...]'(by exact ...)` pattern. The existing two-step `show`
+ `rw` is the local minimum.

The `by exact Spec.xor_ff_byte_lt_mkTable_size b` inside the
`getElem` proof term on line 104 is the same elaboration trick as in
the Spec counterpart. Not touched.

private helpers duplicated from `Zip/Native/Crc32.lean` into

No semantic drift. The duplication is the **intentional choice**
recorded in the issue ("keep Spec independent of Native"). If these
ever drift — e.g. a new Native-side micro-optimisation specialises
one of these helpers — that's a signal for the next meditate agent
to factor them out into `Zip/Spec/Crc32/Bits.lean` or similar. Not
needed today.

## Next rung: `Spec.checksum_pair` (for the next planner)

Public helpers #1692 landed that `checksum_pair` will consume:

1. `Spec.mkTable_size : mkTable.size = 256`
2. `Spec.xor_ff_byte_lt_mkTable_size (b : UInt8) : 0xFF ^^^ b.toNat < mkTable.size`
3. `Spec.crcByteTable_mkTable_eq_crcByte (crc : UInt32) (byte : UInt8) : crcByteTable mkTable crc byte = crcByte crc byte`

All three already promoted to non-private in #1692 per the
visibility-ratchet convention (meditate #1651 §4: promote on first
cross-rung consumer).

Expected `checksum_pair` LOC budget: ≤ 120 LOC across Spec + Native.
The Adler-32 analog (#1673) came in at ~70 LOC, but CRC32 is harder
because the two-byte closed form involves **two** `mkTable` lookups
plus a shift, so the final equation is
`((0xFFFFFFFF ^^^ mkTable[i₁]) >>> 8) ^^^ mkTable[i₂] ^^^ 0xFFFFFFFF`
with nested index-rewriting — expect ~1.5× the Adler-32 proof size.

**Do not** extract a common `crcByteTable_compose_eq_crcByte` yet.
Visibility-ratchet rule: wait for a third consumer (likely
`checksum_replicate`-style). The single
`crcByteTable_mkTable_eq_crcByte` helper is enough for `pair`.
