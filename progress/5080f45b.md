## Key Finding: Two Different Approaches

The two merged PRs represent **different stages** of the same pipeline:

1. **Stage 1** (PR #1414): Add type-level precondition, document invariant, `]!` stays
2. **Stage 2** (PR #1417): Convert guards to `if h :`, add `@[simp]` lemmas, eliminate `]!`

PR #1414 stopped at Stage 1. This is fine as an incremental step, but the `findTableCode.go`
function is now in an inconsistent state: it has the precondition but doesn't use it for
proven bounds. A follow-up should complete Stage 2 for this function.

## Recommendations for Remaining Conversions

1. **Use PR #1417 as the gold standard pattern**: Guard capture, `@[simp]` size lemmas,
   `'(by simp [...]; omega)` proof term, `getElem!_pos` for spec bridging
2. **Always document skipped patterns**: PR #1417's commit message explains why
   `codeLengthOrder[i]!` wasn't converted — this should be standard practice
3. **Complete findTableCode.go** (PR #1414 follow-up): Convert the 2 guards to `if h :`,
   add proven bounds for all 5 accesses. The `hsize` precondition is already in place.
4. **DeflateDynamic.lean is high-difficulty**: All 11 patterns have correctness proofs
   that `unfold` the functions. Converting requires updating 4+ spec files each time.
   Recommend batching into a single focused PR, not per-pattern.
5. **Inflate.lean remaining 4**: `codeLengthOrder[i]!` needs `numCodeLen ≤ 19` threading
   (documented in PR #1417). `insertLoop`'s `nextCode[len.toNat]!` needs a size invariant
   on `nextCode`. `decodeCLSymbols`'s `codeLengths[idx-1]!` needs `idx > 0 ∧ idx < codeLengths.size`.

## No Low-Hanging Fixes Found

The inconsistency between PR #1414 and #1417 is real but fixing it requires
converting `findTableCode.go` guards and updating 2 proof files — this is a
feature PR, not a quick review fix. All other patterns in the merged PRs are
internally consistent.
