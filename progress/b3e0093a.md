# Session b3e0093a — Fix Huffman decode content mismatch in Zstd conformance tests

**Issue**: #703
**Branch**: `agent/b3e0093a`
**Status**: Complete

## Summary

Fixed two bugs in the native Zstd decompressor that caused 12/48 conformance
test failures (all text-pattern tests):

### Bug 1: Huffman weight trailing zero trimming

`parseHuffmanTreeDescriptor` trimmed trailing zero weights from the weight
array before building the Huffman table. Zero weights are significant — the
implicit last symbol's index equals `weights.size`, so removing trailing
zeros shifts which symbol gets the implicit weight.

Example: FSE decoder produces 121 weights with weight[120]=0. Trimming
reduces to 120 weights, shifting the implicit symbol from 121 (0x79='y')
to 120 (0x78='x'). Since 'y' appears in text but 'x' does not, this
caused single-byte substitution errors.

Fix: Remove both trimming blocks (direct and FSE paths).

### Bug 2: Offset code 0 special case in `decodeOffsetValue`

`decodeOffsetValue` had `if code == 0 then extraBits.toNat` returning 0
instead of applying the uniform RFC 8878 formula `(1 << code) + extraBits`.
For code 0 (0 extra bits), the correct result is 1 (repeat offset 1 =
reuse previous offset from history), not 0.

With raw offset 0, `resolveOffset` fell through to the unreachable default
case returning offset=1 (always copy last byte). This caused massive
corruption in text inputs ≥500B: all wrong bytes were 0x0A (newline).

Fix: Remove the special case; the formula works uniformly for all codes.
