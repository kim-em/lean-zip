### Macro visibility note

The `unfold_except` and `frame_from_blocks` macros in Zstd.lean are `local macro` (hygiene false). After the split, they'll only be visible in the foundation file. Options:
1. Change to `scoped macro` in the `Zstd.Spec` namespace (then `open Zstd.Spec` makes them available)
2. Duplicate in each file that needs them
3. Move to a shared macro file

Option 1 is cleanest.

## ZstdFrame.lean Assessment (3413 lines)

ZstdFrame.lean is 3.4x the 1000-line max. It wraps Zstd.lean's frame-level results into API-level (`decompressZstd`) theorems.

### Proposed split (3ŌĆō5 files)

| File | Lines | Content |
|------|-------|---------|
| `ZstdFrame.lean` (keep) | ~375 | Core multi-frame composition, base cases |
| `ZstdFrameContent.lean` (new) | ~1081 | API-level content characterization |
| `ZstdFrameCompleteness.lean` (new) | ~1957 | API-level two-block completeness matrix |

The 1957-line completeness file should be further split by first-block type into 4 files of ~400-500 lines each. Defer this split until after Zstd.lean is split, since ZstdFrame.lean imports from it.

## Decisions

- 8-file split for strict <1250 compliance (except ZstdTwoBlockSimple at 1210 and ZstdContentCompressed at 1245)
- Linear import chain avoids cycles
- Execution gated on active PR merges
- Macro visibility needs `scoped macro` conversion
