### A.1 Predicate correctness — OK

The guard at
[Zip/Archive.lean:817-820](/home/kim/lean-zip/Zip/Archive.lean:817):

```lean
if uncompSize == 0 then
  unless crc == 0 do
    throw (IO.userError
      s!"zip: CD entry CRC must be zero when uncompressedSize is zero \
         (crc={crc}, uncompressedSize={uncompSize}) for {name} at CD offset {pos}")
```

`uncompSize : UInt64` is the **resolved** size from the ZIP64
extra-data merge at
[Zip/Archive.lean:740-748](/home/kim/lean-zip/Zip/Archive.lean:740)
— not the raw `stdUncompSize : UInt32` from CD +24 that could be
the `0xFFFFFFFF` sentinel. So:

- An archive with `stdUncompSize = 0xFFFFFFFF` and a ZIP64 extra
  resolving to `0` correctly **trips** the guard.
- An archive with `stdUncompSize = 0xFFFFFFFF` and a ZIP64 extra
  resolving to any nonzero value **does not** trip the guard — as
  intended.
- An archive with `stdUncompSize = 0` and no ZIP64 extra trips on
  any nonzero `crc` — the common case exercised by the fixture.

Shape `if ... then unless ... do throw`: the `uncompSize == 0` probe
runs first so the CRC field is inspected only when the empty-file
premise holds; an entry with `uncompSize > 0` skips the check
entirely regardless of `crc` value. Matches the PR rationale
comment at
[Zip/Archive.lean:811-812](/home/kim/lean-zip/Zip/Archive.lean:811):
*"Check `uncompSize == 0` first so the CRC field is inspected only
when the empty-file premise holds."*

Error message interpolates `crc`, `uncompSize`, `name`, `pos` —
standard CD-parse-guard shape. `uncompSize` is always `0` when the
guard fires but is included for symmetry with sibling errors.
`crc : UInt32` prints as its decimal value (Lean default `toString`
on `UInt32`); `0xDEADBEEF` decodes to `3735928559`. A grep on the
error substring pinpoints the guard uniquely (see A.3 below).

**Universality of the invariant.** APPNOTE §4.4.7 defines the
CRC32 field as the ANSI-CRC-32 of the uncompressed payload. The
CRC-32 algorithm (RFC 1952 §8, or equivalently IEEE 802.3) starts
from state `0xFFFFFFFF` and final-complements against `0xFFFFFFFF`;
on the empty byte string there is no input to process, so the
complemented start-state is returned directly:
`~0xFFFFFFFF = 0x00000000`. Hence `uncompSize == 0 → crc == 0` is
a universal mathematical invariant independent of any writer
convention — every correct writer (Info-ZIP, Go `archive/zip`,
CPython `zipfile`, 7-Zip) and lean-zip's own `create` at
[Zip/Archive.lean:189](/home/kim/lean-zip/Zip/Archive.lean:189)
(which emits `Checksum.crc32 0 fileData` and returns `0` on an
empty `fileData`) satisfies it.

**Method-agnosticism.** The invariant holds regardless of
compression method, since it is a statement about the **decoded**
payload. For `method=0`, an empty entry has `compSize == uncompSize
== 0`. For `method=8` (deflate), an empty payload encodes to the
2-byte `03 00` empty-block (BFINAL=1, BTYPE=00, LEN=0, NLEN=0xFFFF
— APPNOTE §5 / RFC 1951 §3.2.4), so `compSize=2` with
`uncompSize=0`; the guard still fires because it inspects only
`uncompSize`. The fixture exercises `method=0`; no `method=8`
fixture exists for this column at present, but the guard text
explicitly asserts method-agnosticism.

### E.1 Mathematical-invariant column survey

**Closed columns (2).**

- **PR #1773** — `method=0 → compSize == uncompSize` (stored-method
  tautology, single-dimensional). Fires on `cd-stored-size-mismatch.zip`
  and on the ZIP64-resolved `oversized-zip64-{compressed,uncompressed}-size.zip`
  pair.
- **PR #1857** — `uncompSize == 0 → crc == 0` (empty-file CRC-32
  tautology, method-agnostic). Fires on `cd-empty-entry-crc-nonzero.zip`.

**Candidate third column (not closed, filed here as a
documentation seed).**

- **`method=8 ∧ uncompSize == 0 → compSize == 2 ∧ compData == 03 00`**
  (deflate-encoded empty-stream tautology). RFC 1951 §3.2.3-4
  fixes the minimum DEFLATE encoding of a zero-byte stream as the
  2-byte `03 00` block (BFINAL=1, BTYPE=00 stored-no-len, but
  since the `BTYPE=00` branch requires byte-alignment, the actual
  minimum is technically `0x03 0x00` a BFINAL+BTYPE=01 fixed-huffman
  block terminating immediately on the EOB code). Pre-extract, the
  CD-parse layer can check the `compSize == 2` invariant; the
  `compData == 03 00` payload check requires LH+data reads and so
  belongs to the extract path. Not a near-term priority: no known
  CVE family exploits this column and the two-byte Hamming-distance
  from the invariant is small.

- **`CRC-of-known-byte` identities** (e.g. `crc(<single byte b>)
  == crc_table[b]`) — this is **not closable** at CD-parse time
  because it requires computing the CRC from the compressed (or
  decompressed) payload. Belongs to the post-extraction layer and
  is already served by the existing `"CRC32 mismatch"` check at
  `:1165`.

So the CD-parse mathematical-invariant matrix effectively has
**2 closed columns + 1 near-term open column** (the method-8
empty-stream column). Not a regression; recording it here so the
next planning sweep can decide whether to file an issue.

### E.2 Parser-differential vector — OK

Pre-PR, `parseCentralDir` propagated the crafted `crc=0xDEADBEEF`
into `Entry.crc32` verbatim and returned. Lenient callers routing
on `entry.crc32` before any I/O — typical patterns include
logging, deduplication (`Map Crc32 → Entry`), deflate-cache lookup,
or CRC-cross-checking with an external manifest — saw the smuggled
value.

**Strict reader alignment.** Strict parsers (Go `archive/zip` —
specifically `archive/zip/reader.go`'s CRC check, CPython
`zipfile.ZipInfo.CRC` consumers, Info-ZIP's `unzip -t`) reject
archives failing the CRC check at extraction; a subset
(`zipfile.ZipFile.testzip`, `unzip -T`) reject pre-extract via CRC
computation. None of the major strict tools accept
`uncompSize=0 ∧ crc != 0` as a legitimate entry.

**Parser-differential narrative.** Pre-PR, lean-zip was in the
"lenient" column on `Archive.list` (no gate) and only caught the
anomaly via the `"CRC32 mismatch"` post-extraction guard on
`Archive.extract`. PR #1857 moves lean-zip into the "strict" column
on both APIs for this specific invariant. Additional strictness:
attribution now pins on the invariant itself rather than on the
generic CRC mismatch (the post-extract guard at `:1165` would
**not** fire on this fixture because `fileData` is the empty
ByteArray and `crc32 0 empty == 0`, which **equals** the stored
CRC… wait, actually the stored CRC is `0xDEADBEEF ≠ 0`, so the
post-extract guard **would** fire; but only after an attacker
gets I/O execution — see E.3).

### E.3 Post-extraction fallback — OK

`"CRC32 mismatch"` at
[Zip/Archive.lean:1165](/home/kim/lean-zip/Zip/Archive.lean:1165)
remains in place as defense-in-depth. On the fixture, the
pre-PR `Archive.extract` execution trace was:

1. `parseCentralDir` returned `Entry { ..., crc32 = 0xDEADBEEF,
   uncompressedSize = 0, ... }`.
2. `readEntryData` re-read the LH, decoded the 0-byte payload
   (`method=0, compSize=0`, so the result is `ByteArray.empty`).
3. `actualCrc = Checksum.crc32 0 ByteArray.empty = 0`.
4. `0 ≠ 0xDEADBEEF`, so the post-extract guard fired with
   `"zip: CRC32 mismatch for hello.txt: expected 3735928559, got 0"`.

**Post-PR.** `parseCentralDir` fails at step 1 with the CD-parse
guard; steps 2-4 are unreachable. The `:1165` post-extract guard
is unreachable for CD-parseable archives via the public
`Archive.extract` API, but it stays in place as defense-in-depth
for:
- non-empty archives where a nonzero CRC is correct for a wrong
  payload (the "normal" CRC-mismatch case, covered by
  `bad-crc.zip` at `:1115`),
- direct `readEntryData` callers that bypass `parseCentralDir`
  (none exist in the public API; hypothetical internal callers).

Cross-check: `bad-crc.zip` continues to fire on the post-extract
guard (non-empty payload + crafted CRC), confirming the
precedence-shift story. `cd-empty-entry-crc-nonzero.zip` fires on
the CD-parse guard. Two fixtures, two distinct precedences, no
overlap.
