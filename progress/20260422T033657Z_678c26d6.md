Closed the residual gap in `Tar.readEntryData` flagged by row 8 of the
*Local guard inventory* in `SECURITY_INVENTORY.md`. The four GNU
long-name / GNU long-link / PAX extended / PAX global header

### Implementation

- `Zip/Tar.lean`
  - Added `defaultMaxHeaderSize : Nat := 8 * 1024 * 1024` (8 MiB) with
    a docstring justifying the value (GNU `PATH_MAX` тЙИ 4 KiB, libarchive
    PAX records тЙИ 1 MiB; 8 MiB is three orders of magnitude above
    realistic header sizes).
  - Threaded `maxHeaderSize : Nat := defaultMaxHeaderSize` through:
    - `readEntryData` тАФ throws `IO.userError` containing
      `"exceeds maximum header size"` before any byte is buffered when
      `size > maxHeaderSize`.
    - `forEntries` тАФ passes the cap to all four header callsites.
    - Public APIs: `Tar.extract`, `Tar.list`, `Tar.extractTarGz`,
      `Tar.extractTarGzNative`. `maxEntrySize` defaults are unchanged.
  - Confirmed the regular-file payload path does NOT use
    `readEntryData` тАФ it uses an open-coded loop at the regular-file
    branch of `extract`. Cap therefore does not affect payload reads.

## Decisions

- Chose a separate `maxHeaderSize` knob (not reuse of `maxEntrySize`)
  to keep the existing `maxEntrySize := 0 тЗТ unlimited` payload
  semantics intact while still giving header pseudo-entries a
  sensible default cap. Rationale captured in the issue body and now
  in the docstrings.
- Picked `8 MiB` as the default for the same reasons enumerated in
  the issue body (PATH_MAX, libarchive convention, room to spare).
- Picked `0o77777777777` (~8 GiB, max representable in 11-digit octal)
  for the fixture's size field so the writer's existing octal path
  produces a clean header without needing GNU base-256.
- Error string `"exceeds maximum header size"` chosen for the
  `error-wording-catalogue` requirement: short, unambiguous, used
  verbatim in both `IO.userError` and `assertThrows` substring.
