```
:673       checkPath := name with trailing '/' stripped

:677-693   APPNOTE §4.4.3.2 versionNeeded rationale
:694-696   versionNeeded ≤ 45 upper-bound (PR #1807)  ← (d)
```

Order `(a) → (b) → (c) → (d)` holds exactly as the issue body
prescribed. Critical observations:

- `(a)` fires on the raw `UInt16` field — no field-reads beyond
  `pos + 28`, so even a `cd-empty-name` fixture that is otherwise
  truncated or malformed (shorter than the full 46-byte header)
  would still trip `(a)` before the `entryEnd > cdEnd` overrun check
  at `:613-615` can examine `extraLen` / `commentLen`. Note: the
  outer `while pos + 46 <= cdEnd` guard at `:537` already ensures
  the 46-byte fixed header is inside `cdEnd`; the empty-filename
  guard does not depend on later variable-length reads being
  in-bounds.
- `(a) → (b)`: if a future attacker crafts a fixture with both
  `nameLen == 0` AND (impossibly) a NUL byte somewhere — `(a)` wins.
- `(a) → (c)`: the empty-filename guard short-circuits before the
  decode + `isPathSafe` check. `Binary.isPathSafe` would otherwise
  catch `""` via its empty-component rejection, but only if the
  UTF-8 / Latin-1 decode succeeded first. Deliberate precedence —
  failure attribution pins to the structural anomaly, not the
  path-safety predicate.
- `(a) → (d)`: empty-filename is independent of the
  `versionNeededToExtract` check; `(d)` would fire for a crafted
  `versionNeeded = 51` (AES) with `nameLen = 1` (single-byte name),
  while `(a)` fires first for `versionNeeded = 51` with
  `nameLen = 0`.
