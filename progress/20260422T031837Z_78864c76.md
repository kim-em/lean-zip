## Summary

Doc-only PR. Brought the four public Zip.Native decompression entry
points up to the PR #1573 / #1586 docstring template (default,
zero-policy, error substring, inventory cross-reference). With this
landing, every layer of the public decompression API — FFI
(PR #1573), Archive + Tar (PR #1586), and now native — carries the
limit policy in its public docstring, closing Priority 2 item 4 of
`plans/track-e-current-audit-checklist.md`.

## Changes

- `Zip/Native/Inflate.lean`
  - `Zip.Native.Inflate.inflate`: docstring now states the 1 GiB
    default, the asymmetric `0`-policy (here `0` rejects any
    non-empty output, the opposite of the FFI convention), quotes
    `"Inflate: output exceeds maximum size"`, and cross-refs the
    inventory.

- `Zip/Native/Gzip.lean`
  - `Zip.Native.GzipDecode.decompress`: 256 MiB default, total-
    across-members semantics, asymmetric `0`-policy, both error
    substrings (`"Gzip: total output exceeds maximum size"` from the
    outer guard and `"Inflate: output exceeds maximum size"` from
    the inner inflate guard), inventory cross-ref.
  - `Zip.Native.ZlibDecode.decompress`: 256 MiB default, notes that
    the bound is forwarded to the inner `Inflate.inflateRaw` with no
    separate guard at this layer, asymmetric `0`-policy, error
    substring `"Inflate: output exceeds maximum size"`, inventory
    cross-ref.
  - `Zip.Native.decompressAuto`: 256 MiB default, names all three
    possible dispatch targets and the corresponding error substrings,
    asymmetric `0`-policy, inventory cross-ref.

- `plans/track-e-current-audit-checklist.md`
  - Priority 2 item 3 (bomb tests): ticked. Verified by
    `grep -l "bomb\|exceeds limit\|exceeds maximum size"` returning
    all six test files (`RawDeflate`, `Gzip`, `Zlib`, `Archive`,
    `Tar`, `NativeGzip`).
  - Priority 2 item 4 (limit policy in docstrings): ticked. References
    PRs #1573, #1586, and this PR.

## Decisions

- Followed PR #1586's pattern for the gzip docstring (one-paragraph
  block prepended to the existing one-line summary) rather than
  PR #1573's inline-into-summary style. The Native docstrings already
  had a separate "Returns the decompressed data." trailing sentence;
  appending a paragraph after that read more cleanly than collapsing
  it.
- Documented both halves of the FFI/native `0`-policy contradiction
  explicitly in every native docstring. The issue called this out as
  the whole point of the PR; future readers comparing FFI and native
  call-sites need to see this asymmetry without chasing implementation.
- Did NOT touch `SECURITY_INVENTORY.md`, signatures, defaults, error
  wording, or any test files. The native `decompress` `0` semantics
  are documented as-is — the issue was explicit that this was a
  doc-only PR, and changing the policy is a separate decision.
