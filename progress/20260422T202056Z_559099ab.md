## A. Separation-formula design

**Name.** `updateList_separation` — clear and descriptive, not cryptic.
Matches the issue's suggested spelling.

**Induction shape.** Single left-induction on `xs` with `| nil | cons`
branches, using the `ih` hypothesis once in the cons case. No reverse
induction, no double induction. The session entry's claim "simple
left-induction (no reverse induction needed)" holds.

**Reuse of `sumBytes` / `triSum`.** Both are `private def`s with
`_cons` rewrite lemmas. Used in:

- `updateList_separation` (both, base + step)
- `updateList_init_fst` (sumBytes) / `updateList_init_snd` (both)
- `updateList_combine` (both, through the init-fst/snd bridges)
- `checksum_combine` (transitively, via `updateList_combine`)

Multiple internal call sites justify the named definitions over
inline expressions.

**Accidental duplication check vs `_replicate` (#1684/#1691).** The
earlier `checksum_replicate` proof used an inline *closed-form*
triangular number `n * (n + 1) / 2` (valid because all bytes equal,
so the triangular-weighted sum collapses). `checksum_combine` uses
the *byte-weighted* variant `triSum` for a general byte list. These
are genuinely different invariants — collapse `triSum (List.replicate
n b) = (n * (n + 1) / 2) * b.toNat` is a derived fact, not
duplication. No refactor opportunity without proving the
`triSum_replicate` identity and re-routing `checksum_replicate` through
the separation formula — which would introduce machinery in a
closed-form ladder rung where the inline derivation reads clearly.
Keep as-is.

## B. Private mod helpers

`mod_subst_mul`, `mod_subst_mul_middle`, `mod_absorb_leading` — all
three are `private theorem`, not `@[simp]`, not exported. Proofs use
only generic `Nat.add_mod`, `Nat.mul_mod_mod`, `Nat.add_assoc`,
`Nat.mod_add_mod` — they would work for any positive modulus, not
just `MOD_ADLER`.

**Visibility-ratchet decision (per meditate #1651 §8).** Each helper
has exactly one in-repo call site today (`mod_subst_mul` used inside
`mod_subst_mul_middle`; `mod_subst_mul_middle` and `mod_absorb_leading`
each used once in `updateList_separation`). One consumer each means
keep `private`; defer hoisting to a shared mod-helper section until a
second Adler-adjacent consumer materialises (e.g., if/when a polynomial
GF(p)[x] characterisation of Adler-32 lands). Record here that they are
*candidates* for hoisting, not present ones.

**Stdlib subsumption check.** Each helper is 1–3 `rw` steps over
existing stdlib lemmas. The named wrappers trade a tiny inline saving
for proof-body readability in `updateList_separation`'s cons case
(which cites "extract the leading (b + (a + x)) % M, then substitute
the inner (a + x.toNat) % M"). The named invocation keeps the
arithmetic intent legible. No inline refactor warranted.

## C. Dropped `len2 % MOD_ADLER` optimisation

**No overflow risk.** `adler32_combine` takes `len2 : Nat` and
immediately unpacks `a` / `b` into `Spec.State = Nat × Nat` via
`Spec.unpack`. All arithmetic in the body is `Nat` (unbounded
precision); `len2 * Spec.MOD_ADLER` cannot overflow `Nat`. The final
`Spec.pack` re-narrows into `UInt32`, bounded by the `Valid` invariant
proved in the correctness theorem. The issue's overflow concern
("`n · a1` where `n` can be UInt32-sized") does not apply — the
implementation never does UInt32 multiplication.

**Runtime cost.** `adler32_combine` is a single O(1) call per merge
point; there is no hot loop to optimise. Even N-way parallel hashing
across N workers calls `adler32_combine` only N−1 times. Dropping the
`len2 % MOD_ADLER` reduction trades a negligible runtime cost for one
fewer proof obligation. Correctly dropped.

## D. Statement shape (`.1`/`.2` vs `let ⟨a1, a2⟩`)

`Spec.checksum_combine` uses five `let`-bindings at the theorem-
statement level (`a1`, `a2`, `b1`, `b2`, `n`). The resulting equation
body fits on two lines. Readable at the API surface; the `let`-shape is
at least as readable as the issue-body's `let ⟨a1, a2⟩` destructuring
form, and avoids the dependent-match dance that destructuring would
introduce in downstream users.

`Native.adler32_combine_eq_concat` is the user-facing form — no
projections at all:

    adler32_combine (adler32 1 xs) (adler32 1 ys) ys.size
      = adler32 1 (xs ++ ys)

This is the zlib-compatible API. The `.1`/`.2` stays confined to the
Spec side, as it should.

## F. Test coverage

Four `(xs, ys)` pairs in `ZipTest/NativeChecksum.lean`:

| xs length | ys length | notes |
|-----------|-----------|-------|
| 3         | 0         | empty-ys boundary |
| 13        | 1         | single-byte ys |
| 1         | 10        | moderate ys, tiny xs |
| 13        | 10        | moderate on both sides |

**Off-by-one in `n = ys.length`.** Two cases have `ys.length > 1` (10
each). Any shift `n → n + 1` or `n → n − 1` would be caught by cases
3 and 4 because the formula's linear term `n * a1` and the saturating
term `n * MOD_ADLER - n` both scale with `n`. Even the `n = 0` vs
`n = 1` split (cases 1 and 2) would catch a constant-offset error. ✓

**Empty-`xs` boundary.** Not covered. If `xs = []`, `adler32 1 xs = 1 =
pack (1, 0)`, so `a1 = 1`, `a2 = 0`, and the formula should reduce to
`adler32 1 ys` exactly. The math is clean (the `+ MOD_ADLER - 1` shift
cancels `a1 = 1` trivially), but empirical coverage is missing. This is
a minor gap — the correctness theorem `adler32_combine_eq_concat`
already covers it symbolically, and the theorem-over-tests precedent
says not to pad tests when a theorem closes the boundary. **Flag as a
future test extension, no change in this review.**

**Large-random split.** A 1000-byte split-at-500 case would add
differential coverage against silent overflow on the Native side. Same
theorem-over-tests logic applies: the correctness theorem already
guarantees the identity for all `ByteArray` pairs. **Flag as future
work, no change this review.**
