# 2026-04-21T23:30:27Z — feature — bomb-limit regression tests for Gzip / RawDeflate / Native Gzip

## Issue

#1552 — Track E Priority 2: add decompression-bomb regression tests
covering the streaming / whole-buffer decompression paths
(`Gzip.decompress`, `RawDeflate.decompress`,
`Zip.Native.GzipDecode.decompress`).

## What was done

Additive test-only change. No `Zip/` sources touched.

- `ZipTest/Gzip.lean` — right after the roundtrip: compress `big`,
  call `Gzip.decompress gzipped (maxDecompressedSize := 10)` via
  `.toBaseIO`, require `.error` whose message contains
  `"exceeds limit"`. Mirrors the Zlib block verbatim.
- `ZipTest/RawDeflate.lean` — same pattern for
  `RawDeflate.decompress rawCompressed (maxDecompressedSize := 10)`.
- `ZipTest/NativeGzip.lean` — added right before the
  `Error cases` block. Calls
  `Zip.Native.GzipDecode.decompress gzippedBomb (maxOutputSize := 10)`
  directly (native returns `Except String _`, not `IO`) and asserts
  `.error msg` where `msg.contains "exceeds maximum size"`.
  A one-line comment flags that the native substring diverges from
  the FFI's `"exceeds limit"`.

## Key findings / decisions

- FFI error substring (`"exceeds limit"`) is emitted by
  `c/zlib_ffi.c:195`.
- Native-side overrun error originates in
  `Zip/Native/Inflate.lean:269,285,321` with literal string
  `"Inflate: output exceeds maximum size"`. Picking the substring
  `"exceeds maximum size"` keeps the assertion stable even if the
  overrun is detected at the outer `Zip/Native/Gzip.lean:88`
  (`"Gzip: total output exceeds maximum size"`) instead of the
  inflate loop — both contain the chosen substring.
- `mkTestData` is ~6200 bytes, so a cap of 10 is guaranteed to
  overrun.
