**A.2 Check placement, ungating, and combined-conjunction form.** The
LH-side reads and the consistency check sit at
[`Zip/Archive.lean:722-723`](/home/kim/lean-zip/Zip/Archive.lean:722)
(reads) and
[`:787-790`](/home/kim/lean-zip/Zip/Archive.lean:787) (throw):

```lean
let localModTime := Binary.readUInt16LE localHdr 10
let localModDate := Binary.readUInt16LE localHdr 12
...
-- DOS last-modified time/date (APPNOTE §4.4.6) must agree between CD
-- and LH.  Unlike crc/compSize/uncompSize, these fields are metadata
-- and are NOT legitimately zeroed by the data-descriptor bit — keep
-- the check ungated.  A single combined `unless` because the two
-- fields are physically adjacent (LH +10/+12, CD +12/+14) and
-- semantically a single DOS timestamp pair.
unless localModTime == entry.lastModTime ∧
       localModDate == entry.lastModDate do
  throw (IO.userError
    s!"zip: lastModTime/Date mismatch between CD and local header for {label} (CD time={entry.lastModTime}, date={entry.lastModDate}; LH time={localModTime}, date={localModDate})")
```

Verified:

- **Placement inside `readEntryData`, pre-payload-read.** The check
  sits at `:787-790`, well above the `readBoundedSpanFromHandle` for
  the compressed payload at
  [`:820`](/home/kim/lean-zip/Zip/Archive.lean:820). A malformed
  fixture is rejected before `maxEntrySize` is charged against any
  I/O.
- **Not nested under `usesDataDescriptor`.** The
  `let usesDataDescriptor := (localFlags &&& 0x0008) != 0` at
  [`:791`](/home/kim/lean-zip/Zip/Archive.lean:791) appears **after**
  the modtime/date check, not before — so the check is unconditional
  and fires regardless of bit 3. The comment at `:781-786` explicitly
  captures the rationale: APPNOTE §4.4.6 gates bit 3 on
  crc/compSize/uncompSize only, so the timestamp pair is always
  carried in the LH. The crc/compSize/uncompSize checks that follow
  at `:793-801` are correctly nested inside
  `unless usesDataDescriptor do`, contrasting with the ungated
  modtime/date above them.
- **Combined `∧`, not two separate `unless`es.** Single
  `unless ... ∧ ... do` with one throw. The `∧` is short-circuiting
  but both reads are unconditional above it, so that's a performance
  non-concern; structurally it means a single diagnostic carries all
  four values when either field disagrees.
- **Error message reports all four values.** `CD time={entry.lastModTime}`,
  `date={entry.lastModDate}`, `LH time={localModTime}`,
  `date={localModDate}` — four interpolations per APPNOTE
  attribution. A fixture that mismatches only on `time` still
  surfaces both `date` values in the message, so operators can
  diagnose which field diverged at a glance without re-running with
  instrumentation. ✓

## B. Interop smoke test (the critical regression)

The strict-equality modtime/date check is the riskiest aspect of PR
#1769 — unlike `versionNeededToExtract` (PR #1736, one-sided to
tolerate Go-style CD > LH) it has no tolerance for producer drift.
Issue #1774 §2 explicitly calls out that any interop fixture
disagreement would warrant replanning.

I parsed every CD-entry `+12/+14` pair against its corresponding LH
`+10/+12` pair across all six `testdata/zip/interop/` fixtures,
resolving the ZIP64 local-header offset sentinel in `go-zip64.zip`.
Result:

| Fixture                               | Entries | Mismatches |
|---|---:|---:|
| `go-crc32-not-streamed.zip`           | 2       | 0          |
| `go-test.zip`                         | 2       | 0          |
| `go-unix.zip`                         | 4       | 0          |
| `go-zip64.zip`                        | 1       | 0          |
| `latin1-name.zip`                     | 1       | 0          |
| `utf8-flag.zip`                       | 1       | 0          |
| **Total**                             | **11**  | **0**      |

All 11 entries have CD and LH timestamp pairs agreeing on both the
time `UInt16` and the date `UInt16` — no real-world producer
disagreement on these two slots. The strict-equality choice is the
right one for the observed producer corpus. The feature-side progress
entry's §2 had already done this check; I re-ran independently and
confirm the same result.

`lake exe test` is green including all interop-parsing blocks
(`ZipTest/ZipFixtures.lean` §§1–4 for go-test/go-zip64/go-unix/
go-crc32-not-streamed and `ZipTest/Utf8Fixtures.lean` for latin1-name/
utf8-flag). No replan needed. ✓

Contrast with PR #1736's one-sided `versionNeededToExtract` check:
that check has `LH ≤ CD` direction because Go routinely emits
`CD=45, LH=20` for ZIP64 archives whose LH sizes fit in 32 bits. The
modtime/date fields have no analogous producer asymmetry — Go's
`archive/zip` copies the DOS time/date pair verbatim from the LH into
the CD header, and every other observed producer does likewise.
Strict equality is therefore sound, and the two-sided direction
(modtime/date) vs. one-sided (version) asymmetry reflects real
producer behaviour rather than arbitrary design choice. ✓
