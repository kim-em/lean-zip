PAX-extended-header pseudo-entry header + a 512 B PAX records block
(carrying the smuggled `"18 linkpath=a\x00b/c\n"` record + 494 B of

the trailing-zero-block end-of-archive marker.

This 4-block / 2 048 B geometry matches the sibling
`pax-path-nul-in-value.tar` (PR #1866) and the broader malformed-PAX
family (`pax-oversized-length.tar`, `pax-truncated-record.tar`,

(1 536 B, three blocks) and the GNU long-name / long-link family
(1 536 B, three blocks); the difference is the additional 512 B
PAX-records block that sits between the pseudo-entry header and the

### §B.3 Smuggled record value: `"18 linkpath=a\x00b/c\n"` (18 B with interior NUL at byte 13)

Per `fixtureLinkpathNulInValue` at
[`scripts/build-pax-malformed-fixtures.lean:108-109`](/home/kim/lean-zip/scripts/build-pax-malformed-fixtures.lean:108):

```lean
def fixtureLinkpathNulInValue : ByteArray :=
  "18 linkpath=a".toUTF8 ++ ByteArray.mk #[0x00] ++ "b/c\n".toUTF8
```

The 18-byte record decomposes as: 2-byte digit-prefix `"18"` + 1-byte
space + 8-byte key `"linkpath"` + 1-byte `'='` + 5-byte value
`"a\x00b/c"` + 1-byte `'\n'` terminator. The declared length 18
matches the actual byte length including the terminator (POSIX SUSv4
§pax record-format: `"<len> <key>=<value>\n"`). The interior NUL
sits at record byte 13, value byte 1 (between `'a'` and `'b'`).

The `String.fromUTF8?` decode at
[`Zip/Tar.lean:144`](/home/kim/lean-zip/Zip/Tar.lean:144) returns
`some` for both `keyBytes = "linkpath"` (NUL-clean 8-byte UTF-8)
and `valueBytes = "a\x00b/c"` (5-byte ASCII; U+0000 is valid UTF-8
per RFC 3629). So the UTF-8 gate is not the discriminator — the
**`(valueBytes.findIdx? (· == 0)).isNone`** sub-check at
[`Zip/Tar.lean:146`](/home/kim/lean-zip/Zip/Tar.lean:146) is the
guard the fixture trips: `valueBytes` contains `0x00` at value byte
1 of 5, so `findIdx?` returns `some 1`, `isNone` returns `false`,
the conjunction at lines 145-146 evaluates `false`, and the record
is silently dropped (the `records.push (key, value)` at
[`Zip/Tar.lean:150`](/home/kim/lean-zip/Zip/Tar.lean:150) is
skipped). The companion `(keyBytes.findIdx? (· == 0)).isNone` at
line 145 evaluates `true` (key is NUL-clean), so the discriminator
is uniquely the `valueBytes` arm.

### §B.4 Builder mechanics (no override hook needed)

The fixture builder uses the existing `buildPaxMalformedFixture`
helper at
[`scripts/build-pax-malformed-fixtures.lean:38`](/home/kim/lean-zip/scripts/build-pax-malformed-fixtures.lean:38)
without any override hook. The helper takes a raw `ByteArray` of PAX
records, prepends a synthesised pseudo-entry header carrying the
length, computes the UStar checksum, and appends the trailing
zero-byte `hello.txt` entry — all driven from a single `ByteArray`
input.

This contrasts with the UStar fixture-builder family
(`scripts/build-ustar-malformed-fixtures.lean`) which threads a
`pathOverride` hook for the `prefix` slot's `splitPath` transform.
For PAX no override hook is needed because:

- The smuggled PAX record is appended verbatim into the records
  block with no writer-side transform (no `splitPath`-style
  splitting between the `path` and `prefix` slots).
- The pseudo-entry header is writer-emitted with default
  `path = "PaxHeader/hello.txt"`, `typeflag = 'x'`,
  `size = 18` (matches the records block), and a recomputed UStar
  checksum (`011554` octal — see §B.5).
- The trailing regular entry's header carries `path = "hello.txt"`
  with all metadata defaults.

The builder pattern matches the
`malformed-fixture-builder` skill's "writer-side override hooks"
section at
[`.claude/skills/malformed-fixture-builder/SKILL.md:181-203`](/home/kim/lean-zip/.claude/skills/malformed-fixture-builder/SKILL.md:181):
because the PAX records arm does not transform the input bytes
between the builder caller and the on-disk record block, the
verbatim-write pattern applies — no `Option`-typed override is
needed.

### §B.5 `xxd` diff against the sibling `pax-path-nul-in-value.tar`

A `diff <(xxd pax-linkpath-nul-in-value.tar) <(xxd pax-path-nul-in-value.tar)`
shows exactly three byte-cluster differences:

```
9,10c9,10
< 00000080: 3030 3030 3032 3200 …  0000022.…   ← linkpath: size=022₈ (18)
< 00000090: 3030 3000 3031 3135 3534 …                  chksum=011554₈
---
> 00000080: 3030 3030 3031 3600 …  0000016.…   ← path: size=016₈ (14)
> 00000090: 3030 3000 3031 3135 3537 …                  chksum=011557₈
33,34c33,34
< 00000200: 3138 206c 696e 6b70 6174 683d 6100 622f  18 linkpath=a.b/
< 00000210: 630a 0000 …                              c.…
---
> 00000200: 3134 2070 6174 683d 6100 622f 630a 0000  14 path=a.b/c...
> 00000210: 0000 0000 …                              .…
```

Three differences, all expected for a 4-byte-longer key:

1. **Header `size` field at offset `0x07C` (12 B)** — `pax-linkpath`
   carries `"00000000022\0"` (octal 22 = decimal 18), `pax-path`
   carries `"00000000016\0"` (octal 16 = decimal 14). The size is
   the byte-length of the records block; `linkpath` is 4 B longer
   than `path` (key length 8 vs 4 B; digit-prefix stretches from
   `14 ` to `18 `).
2. **Header `chksum` field at offset `0x094` (8 B)** — recomputed
   `"011554\0 "` (octal 4 972) vs `"011557\0 "` (octal 4 975); the
   3-unit delta absorbs the changed `size` field plus the
   typeflag/path bytes the writer touches when computing the
   header. Independent arithmetic recomputation is unnecessary for
   this paired-review — the byte-deterministic builder rerun
   (§B.1) confirms the on-disk chksum matches what the builder
   produced.
3. **Records-block content at offset `0x200` (18 vs 14 B)** — the
   smuggled record itself: `pax-linkpath`'s 18-byte
   `"18 linkpath=a\x00b/c\n"` vs `pax-path`'s 14-byte
   `"14 path=a\x00b/c\n"`. The interior NUL sits at offset `0x20D`
   (`pax-linkpath`) vs `0x209` (`pax-path`); both well inside the
   512 B records block.

All other header bytes (offsets `0x000-0x07B`, `0x09C-0x1FF`) and
all bytes of blocks 2-3 (the trailing regular entry header and body)
are byte-identical between the two fixtures. The diff is **minimal
and in-family** — exactly the three slots that should differ
between two per-slot siblings of the value-side override family.

### §B.6 Per-slot printable-prefix discipline (printable-`'a'` prefix)

The smuggled value `"a\x00b/c"` carries a 1-byte printable-ASCII
prefix `'a'` before the interior NUL — what a parser-differential
caller would see if `Binary.fromLatin1` (or its reader equivalent)
truncated at the NUL. PR #1979 deliberately re-uses the **same**
printable prefix as the sibling PR #1866's `path`-slot fixture
(both use `'a'` and suffix `"b/c"`), in contrast to the post-#1928
wave's UStar / GNU long-name family fixtures where each slot got a
distinct prefix per the `malformed-fixture-builder` skill's per-slot
printable-prefix discipline at
[`.claude/skills/malformed-fixture-builder/SKILL.md:143-179`](/home/kim/lean-zip/.claude/skills/malformed-fixture-builder/SKILL.md:143)
(extended by PR #1973).

The PAX family is the only post-#1928-wave per-slot family where
the two siblings share a printable prefix. The choice is sound: the
PAX guard fires on `valueBytes` regardless of the key, so per-slot
attribution at the fixture-construction level comes from the *key*
(`path` vs `linkpath`), not the value. Re-using the value across
both slots emphasises that the *key* is the slot discriminator,
while the value is just the smuggle payload. The trailing-NUL
invariant (rule 2 of the skill — the smuggled value's last byte
must be non-NUL or `stripTrailingNuls`/the guard's `findIdx?` input
invariant would erase the interior NUL) is satisfied: the value's
last byte is `'c'`.

## §C. Guard-side code review

### §C.1 Guard at `parsePaxRecords`, lines 145-150

The guard is unchanged on master post-#1979 (PR #1979 does not touch
`Zip/Tar.lean`). The current source at
[`Zip/Tar.lean:144-150`](/home/kim/lean-zip/Zip/Tar.lean:144):

```lean
if let (some key, some value) := (String.fromUTF8? keyBytes, String.fromUTF8? valueBytes) then
  if (keyBytes.findIdx? (· == 0)).isNone
      && (valueBytes.findIdx? (· == 0)).isNone then
    if records.any (·.1 == key) then
      err := some s!"tar: PAX extended header has duplicate {key.quote} record"
      break
    records := records.push (key, value)
```

The control-flow ordering inside `parsePaxRecords` is documented in
the doc-comment block at
[`Zip/Tar.lean:134-143`](/home/kim/lean-zip/Zip/Tar.lean:134):

1. **UTF-8 decode gate** (line 144) — `String.fromUTF8?` on both
   `keyBytes` and `valueBytes`. Drops any record whose raw bytes
   are not valid UTF-8. Note: `String.fromUTF8?` accepts U+0000 as
   valid UTF-8, so this gate does **not** filter NUL-bearing values.
   This is the explicit motivation for the second gate.
2. **NUL-byte silent-skip gate** (lines 145-146) — both `keyBytes`
   and `valueBytes` must be NUL-free under `findIdx? (· == 0)`. The
   conjunction guards against either arm carrying a NUL: a NUL-bearing
   key is dropped (defense-in-depth — `applyPaxOverrides`'s case match
   at lines 161-191 has no NUL-bearing key in its match arms, so a
   NUL-bearing key would already be silently ignored downstream), and
   a NUL-bearing value is dropped (the on-record arm — without this
   guard the value would smuggle into the per-key field assignment in
   `applyPaxOverrides`, which is what the fixture exercises).
3. **Duplicate-key hard-reject gate** (lines 147-149) — if the key
   is already in `records`, set `err` and `break` (introduced by
   PR #1899; tested by `pax-duplicate-path.tar`).
4. **Push gate** (line 150) — only records that pass all three
   prior gates are pushed.

The PR #1979 fixture trips gate 2 specifically (the `valueBytes`
arm of the conjunction), at line 146.

### §C.2 Silent-skip semantics (record dropped, parsing continues)

Unlike PR #1899's duplicate-key hard-reject (gate 3) which sets
`err := some msg` and `break`s the record loop with a downstream
`throw (IO.userError msg)` in `forEntries`'s `typePaxExtended`
branch, the NUL-byte silent-skip at gates 1-2 simply **omits** the
record from the returned `records` array. `parsePaxRecords` returns
`.ok records` with the smuggled record absent; the next-iteration
control-flow advances `pos := recordEnd` and continues parsing
remaining records.

The silent-skip vs hard-reject distinction is documented in the
`error-wording-catalogue` skill's "Tar PAX extended-header NUL-byte
silent-skip" row at
[`.claude/skills/error-wording-catalogue/SKILL.md:54`](/home/kim/lean-zip/.claude/skills/error-wording-catalogue/SKILL.md:54)
and explicitly contrasted with the throw-side "Tar PAX
extended-header duplicate-key" row at line 53. The PR #1979 fixture
extends the silent-skip family with a per-slot probe but does not
change the semantic distinction.

### §C.3 `applyPaxOverrides` `"linkpath"` case-arm

The `applyPaxOverrides` definition at
[`Zip/Tar.lean:157-192`](/home/kim/lean-zip/Zip/Tar.lean:157)
case-matches each key against a fixed enumeration of well-known
PAX overrides:

```lean
def applyPaxOverrides (entry : Entry) (records : Array (String × String)) : Entry := Id.run do
  let mut e := entry
  for (key, value) in records do
    match key with
    | "path" => e := { e with path := value }
    | "linkpath" => e := { e with linkname := value }
    | "size" => ...
    | "mtime" => ...
    | "uid" => ...
    | "gid" => ...
    | "uname" => e := { e with uname := value }
    | "gname" => e := { e with gname := value }
    | _ => pure ()  -- ignore unknown keys
  return e
```

The **`"linkpath"` case-arm at [`Zip/Tar.lean:162`](/home/kim/lean-zip/Zip/Tar.lean:162)**
is the per-slot attribution target for PR #1979. Without the gate-2
silent-skip in `parsePaxRecords`, a smuggled `linkpath=a\x00b/c`
would survive into this case-arm with `value = "a\x00b/c"`, and the
arm would set `entry.linkname := "a\x00b/c"`. Subsequent
`Tar.list` consumers routing on `entry.linkname` for trust decisions
would see the full 5-byte string with the embedded NUL, while
`Tar.extract`'s symlink path at the eventual `Handle.createSymlink`
call would feed `"a\x00b/c"` to `symlink(2)`, which POSIX truncates
at the first NUL to `"a"` — a classic POSIX-truncation smuggle on
the symlink target.

The companion `"path"` arm at line 161 is the sibling slot covered
by PR #1866's `pax-path-nul-in-value.tar` — that arm's smuggle
target is `entry.path` (which subsequently flows into
`Binary.isPathSafe` and `IO.FS.writeBinFile` via `Tar.extract`'s
regular-file path).

The two case-arms (`"path"` and `"linkpath"`) are the **only** two
arms in `applyPaxOverrides` whose payload is a free-form string that
flows into a filesystem-reaching field on `Tar.Entry`
(`entry.path` and `entry.linkname` respectively). The other case-arms
either parse the value as numeric (`"size"`, `"mtime"`, `"uid"`,
`"gid"` — where a NUL in the value would terminate the digit-parsing
loop early but not produce a smuggle) or write to non-filesystem-
reaching attestation fields (`"uname"`, `"gname"`). So the 2-slot
value-side override family is exactly the set of `applyPaxOverrides`
case-arms that need per-slot fixture coverage to pin attack-class
coverage at this layer; PR #1866 + PR #1979 between them cover the
full family.

### §C.4 Positive-regression assertion semantics

The fixture's assertion is a **positive regression** — the test
asserts that `Tar.list` does not throw and that the smuggled value
is **absent** from the recovered metadata. This matches the
sibling `pax-path-nul-in-value.tar` row's documented convention in
the `error-wording-catalogue` skill at line 54
("attribution-by-recovered-value: assert that extraction does not
throw and that the smuggled value is absent from the recovered
metadata").

Specifically the assertion pair on PR #1979:

- **Loop body** (line 156, shared with all PAX malformed fixtures):
  `entries[0]!.path == "hello.txt"` — the trailing regular-file
  entry's path stays at the writer-emitted default, confirming
  the smuggled override did not reach `applyPaxOverrides`'s
  `"path"` arm. (This shared check would *also* trip on a guard
  failure at the `path`-slot fixture — but a `linkpath` smuggle
  would not affect `entry.path`, so the shared check alone does
  not pin the `linkpath` arm.)
- **Per-slot follow-up** (lines 167-172, new with PR #1979):
  `linkpathEntries[0]!.linkname == ""` — the trailing regular-file
  entry's `linkname` stays at the `typeRegular` default of `""`
  (the entry has no symlink target because it is a regular file),
  confirming the smuggled `linkpath` override did not reach
  `applyPaxOverrides`'s `"linkpath"` arm. **This is the per-slot
  attribution check that distinguishes PR #1979's coverage from
  PR #1866's `path`-slot coverage** — the two checks are
  complementary, not redundant.

The `entries[0]!.linkname == ""` check is the canonical positive-
regression form for the `linkpath` slot — the alternative would be
`entries[0]!.linkname != "a\x00b/c"`, but checking against the
expected default (`""` for `typeRegular`) is a stronger invariant
because it also catches the failure mode where the override was
applied with a different value (e.g. truncated at NUL to `"a"`)
rather than the full smuggle.
