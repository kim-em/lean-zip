# Progress — 2026-04-24 06:21 UTC — feature session 6198786e

## Summary

Closes issue #1762 — Track E hardening for CD general-purpose flag
word. Adds early rejection at `parseCentralDir` for the three APPNOTE
§4.4.4 encryption-related bits that lean-zip cannot honour: bit 0
(traditional ZipCrypto), bit 6 (strong encryption, SES/AES, Annex H),
and bit 13 (encrypted central directory). Combined mask `0x2041`.

Prior behaviour: a crafted archive with any of these bits set would be
accepted at CD-parse time and proceed to `readEntryData`, which would
either run DEFLATE over pre-encrypted bytes (yielding garbage or a
native-inflate error deep in the pipeline) or deliver raw stored
plaintext (for method 0). The only guard that fired was the late CRC32
mismatch in `Zip/Archive.lean:758`. New behaviour: immediate rejection
with error substring `"encryption-related general-purpose flag set"`.

Orthogonal to the CD/LH flags-consistency check (PR #1715) — that
catches CD-vs-LH *divergence*; this catches *absolute bit presence*
even when CD and LH agree. The fixture sets bit 0 on both sides,
making the divergence check pass so the new guard is the one that
fires.

## Decisions

- **Single combined mask vs. three separate checks**: combined (mask
  `0x2041`). Reports the raw `flags` value so the offending bit is
  identifiable by inspection. Matches the sibling `diskNumberStart`
  check pattern.
- **Fixture bit**: bit 0 (traditional ZipCrypto). Textbook case,
  single fixture covers the full combined mask.
- **Placement**: `parseCentralDir` (CD-parse time), matching the
  `diskNumberStart` per-entry pattern. Fires before any LH read —
  defence in depth.
- **Reserved bits (7, 8, 9, 14, 15) not checked**: APPNOTE leaves them
  unassigned. Rejecting unassigned bits would break on future
  producers; out of scope for this issue which targets the three
  concretely-documented encryption bits.
