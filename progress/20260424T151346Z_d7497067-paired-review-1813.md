### A.1 Guard shape and predicate — OK

The guard at
[Zip/Archive.lean:648](/home/kim/lean-zip/Zip/Archive.lean:648)
reads literally:

```lean

unless localOff ≤ cdFileOff ∧ 30 ≤ cdFileOff - localOff do
  throw (IO.userError
    s!"zip: entry local offset overlaps central directory for {name} (localOffset={localOff}, cdOffset={cdFileOffset})")
```

### A.6 `UInt64` wrap-safety — OK

Lean 4 `UInt64.sub` wraps modulo 2⁶⁴ (not saturating to 0, *pace*
the issue body's "underflow to 0" phrasing). The guard is
correct regardless because the conjunction short-circuits on the
left:

- `localOff = UInt64.max - 29, cdFileOff = UInt64.max - 30`:
  left conjunct `(max-29) ≤ (max-30)` is `false` → `P ∧ Q`
  evaluates to `false` regardless of Q → guard throws ✓
- `localOff = UInt64.max - 30, cdFileOff = UInt64.max - 30`:
  `max-30 ≤ max-30` is `true`; `(max-30) - (max-30) = 0`;
  `30 ≤ 0` is `false` → `P ∧ Q` is `false` → throws ✓
- `localOff = UInt64.max - 30, cdFileOff = UInt64.max`:
  `max-30 ≤ max` true; `max - (max-30) = 30`; `30 ≤ 30` true →
  passes. Mathematically legitimate (LH occupies
  `[max-30, max)`), but in practice `assertSpanInFile` on the
  extract path enforces `localOff + 30 ≤ fileSize`, so any real
  archive large enough to pass here would be caught downstream.

For the fixture case `localOff = 50, cdFileOff = 45`:
`50 ≤ 45` is `false`; subtraction wraps to `UInt64.max`, but
the conjunction is `false ∧ true = false` → throws ✓.

Whether Lean's `And` short-circuits or evaluates strictly is
immaterial: `UInt64` subtraction is total (no UB), wraps
modulo 2⁶⁴, and any wrapped value remains a valid `UInt64`. The
asymmetric form also matches the `SpanInFile` lemma-corpus
convention used elsewhere in `Zip/Archive.lean` (`assertSpanInFile`
at `:698`, the CD-bounds check at `:868`), keeping the
wrap-safety reasoning uniform.

### A.7 Writer-side compliance — OK

`grep -n "writeLocalHeader\|writeCentralHeader" Zip/Archive.lean`
returns:

- `:83` — `private def writeLocalHeader`
- `:109` — `private def writeCentralHeader`
- `:203` — `let localHdr := writeLocalHeader entry` (inside the
  per-entry loop in `create`)
- `:211` — `let hdr := writeCentralHeader entry` (inside the
  CD-emission loop in `create`)

The per-entry loop at `:190-206` writes each entry's LH + data
sequentially, updating `offset := offset + localHdr.size.toUInt64
+ compData.size.toUInt64` at `:206`. After the loop completes at
`:207`, `cdOffset := offset` at `:208` captures the file position
at the end of all LH-and-data bytes. Then the CD loop at
`:210-213` iterates `entries` and calls `writeCentralHeader` at
`:211`. Every `writeLocalHeader` call strictly precedes every
`writeCentralHeader` call because the two loops are sequential
with `cdOffset := offset` in between. Each `entry.localOffset` is
captured from the running `offset` at `:200` before its
`writeLocalHeader` call, so for every entry:

```
entry.localOffset + localHdr.size + compData.size ≤ cdOffset
```

and in particular `entry.localOffset + 30 ≤ cdOffset` is a
writer-side invariant. The reader guard therefore never fires on
lean-zip-produced archives.

## Recommendation for next-wave Track E cadence

The CD-parse archive-layout invariant dimension is now closed at
both granularities: the per-entry micro-shape (PR #1813, this
review) and the archive-level macro-shape (in-flight PR #1809
against issue #1799). Between them, the legitimate APPNOTE
§4.3.6 layout `[LH+data]* [CD] [EOCD]` is fully characterized at
CD parse time — no LH can claim a position inside or past the
CD, no CD can extend past the EOCD. The `Archive.list`
parser-differential window (which previously lacked any gate for
either dimension) is now closed on both axes at the `list` entry
point, not just at `extract`.

The next natural Track E target in this problem space is the
symmetric invariant at the EOCD layer: *per-entry `diskStart == 0`
and `localOffset` values must also satisfy any ZIP64-EOCD64
sibling fields* — specifically, the ZIP64 EOCD64 `cdOffset` and
`cdSize` should have the same `cdOffset + cdSize ≤ zip64EocdPos`
relation enforced. (Issue #1758 closed the EOCD64
`versionNeededToExtract ≥ 45` sanity check; the sibling layout
invariant at the ZIP64 layer may still have a gap.)

A second candidate: the CD-parse `cd-nul-in-name.zip` dimension
(issue #1829, unclaimed) closes the CD-name-bytes dimension, a
different axis from layout. That dimension and the
`cd-lh-name-mismatch.zip` dimension (issue #1722) together would
close the CD-entry name-bytes invariant surface.
