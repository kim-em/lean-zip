### 1. Merge Conflict Cascade Analysis

**The pattern**: PRs #565 and #549 developed merge conflicts. Recovery
issues #568 and #558 were created. Recovery PRs #577 and #561 were
produced — and both developed conflicts again. Three PRs (#561, #577,
#591) are currently conflicted.

**Root cause**: `Zip/Native/ZstdFrame.lean` (1059 lines) is a single
large file modified by multiple concurrent agents. When any PR touching
it merges, all other outstanding PRs on that file become stale. With
55-64 commits of master divergence, cherry-pick recovery is fragile.

**The recovery-of-recovery anti-pattern**: Creating a recovery issue for
a conflicted recovery PR produces a cascade that compounds rather than
resolves. Each recovery attempt is done against an increasingly diverged
master, making conflicts worse.

**Concrete recommendations** (codified in agent-pr-recovery skill):
1. **Never chain recoveries** — if a recovery PR develops conflicts,
   redo from scratch on current master. Two levels of recovery means
   the approach is wrong.
2. **Rebase immediately before pushing** — even if you just created the
   branch, another PR may have merged in the interim.
3. **Check for pending PRs on the same files** before starting recovery.
   If another PR modifies the same hot file and is closer to merging,
   let it land first.
4. **Flag hot files for splitting** — files over 500 lines causing
   repeated conflicts should be prioritized for splitting over further
   feature work.
5. **Planners: serialize issues on the same file** using `depends-on:`
   rather than allowing concurrent modification.

**Recommended merge order for current conflicted PRs**:
1. PR #591 (new spec files only, independent) — merge first
2. PR #561 (validation) — provides foundations
3. PR #577 (wiring, uses validation) — merge last

### 2. Spec-vs-Implementation Sequencing Analysis

**Question**: Should Zstd spec work (#582, #585) wait until the
implementation (#552, compressed block wiring) stabilizes?

**Answer**: Continue specs in parallel. Rationale:
- Specs target stable individual functions (`buildFseTable`,
  `resolveSequenceFseTables`, `executeSequences`) whose signatures
  are settled, not the integration wiring that #552 adds.
- DEFLATE precedent: specs were created in parallel with implementation
  (spec formalization started 2026-02-19, while Track C2 native WF
  recursion wasn't completed until 2026-03-02). Many theorems remain
  `sorry` without blocking development.
- The spec predicates (valid Huffman table, valid FSE distribution,
  valid sequence state) are characterizing properties independent of
  how the decompressor wires them together.
- Specs actually help implementation: they clarify what invariants
  the wiring code (#552) should check.

**Guidance for future planners**: Spec issues should target stable
component functions, not integration paths. Leave correctness theorems
about end-to-end decompression as `sorry` until the pipeline is
complete.
