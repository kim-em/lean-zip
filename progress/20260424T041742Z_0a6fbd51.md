## Decisions

- **Check lives in `parseCentralDir` head, not in
  `findEndOfCentralDir`.** The disk-number check has no
  CD-side counterpart — it is an EOCD-internal sanity check — so
  it could live in `findEndOfCentralDir` before the return and
  avoid widening the return tuple. I kept the check in
  `parseCentralDir` anyway, so all CD/EOCD consistency assertions
  sit next to each other for auditability and so the pattern
  established by the `totalEntries` check (PR #1733) stays
  consistent. The threaded tuple slots are cheap.
- **Single combined check, both values reported.** The two
  disk-number fields could fire independent checks with distinct
  substrings, but the smuggling vector is the same (cross-disk
  claim) and a single throw is more concise. Both values are
  interpolated into the message for deterministic attribution
  regardless of which field triggered.
- **Post-ZIP64-override.** APPNOTE allows the standard EOCD to
  use the `0xFFFF` sentinel in either disk-number slot to mean
  "consult ZIP64"; when ZIP64 is present, the 32-bit EOCD64
  value overrides, and that is the value the check consumes. This
  matches the `totalEntries` pattern already in the codebase.
- **Strict `== 0` is safe for all existing interop fixtures.**
  Regression smoke-checked all six of `testdata/zip/interop/*.zip`
  (Go, CPython, info-zip, ZIP64): none set nonzero
  `numberOfThisDisk` or `diskWhereCDStarts` on either the
  standard EOCD or the ZIP64 EOCD64. The check does not break
  any legitimate producer we track.

## Patterns discovered / reinforced

- **The `check-inventory-links.sh` script catches line drift.**
  After adding ~20 lines to `Zip/Archive.lean`, 32 anchors went
  stale. A Python sub-regex pass keyed on the exact
  `Zip/Archive.lean:NNN` marker (terminator `)` or `]`) shifted
  them all atomically. The warnings-vs-errors split on the script
  makes pre-land verification easy: `errors=0, warnings=0` is
  green.
- **Byte-identity-preserving generator defaults.** Adding kwargs
  to a shared fixture-builder helper must default to the
  zero-preserving values so the SHA-256s of the existing fixtures
  don't move. Verified by regenerating and diffing — only the new
  fixture appeared in `git status`.
