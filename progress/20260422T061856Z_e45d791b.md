### Deliverable 2 — Counter-state mechanism

- **Order of operations is correct.** The overflow check at
  [`Zip/Gzip.lean:91-93`](/home/kim/lean-zip/Zip/Gzip.lean:91) and
  [`Zip/RawDeflate.lean:64-66`](/home/kim/lean-zip/Zip/RawDeflate.lean:64)
  fires *before* `output.write chunk` at lines 95 and 68 respectively.
  The sink-state invariant ("partial output ≤ `maxDecompressedSize`") is
  preserved end-to-end. The PR's regression tests assert this invariant
  directly at
  [`ZipTest/Gzip.lean:186`](/home/kim/lean-zip/ZipTest/Gzip.lean:186)
  and
  [`ZipTest/RawDeflate.lean:95-96`](/home/kim/lean-zip/ZipTest/RawDeflate.lean:95).
- **`0 = unlimited` contract preserved.** The short-circuit
  `maxDecompressedSize ≠ 0 && next > maxDecompressedSize` at
  [`Zip/Gzip.lean:91`](/home/kim/lean-zip/Zip/Gzip.lean:91) and
  [`Zip/RawDeflate.lean:64`](/home/kim/lean-zip/Zip/RawDeflate.lean:64)
  skips the throw when `maxDecompressedSize = 0`. The counter still
  increments on every chunk but never trips; the default-parameter path
  is exercised indirectly by
  [`ZipTest/Gzip.lean:141-144`](/home/kim/lean-zip/ZipTest/Gzip.lean:141)
  (`decompressFile` with no explicit cap) and
  [`ZipTest/Gzip.lean:199-206`](/home/kim/lean-zip/ZipTest/Gzip.lean:199)
  (large-data file roundtrip, default `0`). No direct
  "`maxDecompressedSize := 0` through `decompressStream`" regression is
  present, but the short-circuit is visibly correct in the source and
  the `decompressFile` default exercises the same code path.
- **`UInt64` arithmetic.** `total + chunk.size.toUInt64` at
  [`Zip/Gzip.lean:90`](/home/kim/lean-zip/Zip/Gzip.lean:90) and
  [`Zip/RawDeflate.lean:63`](/home/kim/lean-zip/Zip/RawDeflate.lean:63)
  is a raw add. `UInt64.max ≈ 16 EiB` makes a wrap practically
  unreachable (the output sink would be exhausted first), but a
  saturating or overflow-checked add would make the defence explicit.
  Flagged as a low-severity hardening item; not a correctness bug
  — see recommendation **F-d** below.

- **F-c (medium)** — Extend `ZipTest/FuzzInflate.lean` to drive
  `Gzip.decompressStream` and `RawDeflate.decompressStream` (not just
  the low-level `InflateState.push/finish` surface at
  [`ZipTest/FuzzInflate.lean:99-106`](/home/kim/lean-zip/ZipTest/FuzzInflate.lean:99))
  with a PRNG-picked `maxDecompressedSize` cap per iteration. This
  exercises the new `IO.Ref UInt64` counter under randomised
  cross-chunk state — exactly the invariant the fuzz harness is
  shaped to surface.

- **F-d (≤ 50 lines, hardening)** — Replace the raw
  `total + chunk.size.toUInt64` add at
  [`Zip/Gzip.lean:90`](/home/kim/lean-zip/Zip/Gzip.lean:90) and
  [`Zip/RawDeflate.lean:63`](/home/kim/lean-zip/Zip/RawDeflate.lean:63)
  with a saturating or checked add (e.g., `if chunk.size.toUInt64 >
  UInt64.max - total then throw ... else total + chunk.size.toUInt64`).
  Severity low — `UInt64.max ≈ 16 EiB` is unreachable in practice —
  but the current code silently wraps past `UInt64.max`. A saturating
  add makes the defence-in-depth explicit and matches the C-side
  overflow-guard style used in `c/zlib_ffi.c`.
