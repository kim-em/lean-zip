## Findings

### A.1 — `validateExtraFieldStructure` shape

- Declared `private def validateExtraFieldStructure : ByteArray → Bool`
  at `Zip/Archive.lean:371`. Purely functional, no `IO`/`Except`
  reshape — matches Option C of the issue's LoC preference.
- Body at `Zip/Archive.lean:371-377` uses `Id.run do` with a `while`
  loop and a single early `return false` on sub-field overrun. Not a
  fuel-based recursion.
- Termination: each iteration advances `epos` by `4 + dataSize ≥ 4 > 0`
  (since `dataSize.toNat ≥ 0`), so the loop guard `epos + 4 ≤ extraData.size`
  eventually fails. No fuel parameter, no risk of fuel-shortfall bug.
- Failure cases handled:
  - **(a) Overrunning sub-field**: `if epos + 4 + dataSize > extraData.size
    then return false` at line 375. ✓
  - **(b) Trailing 1–3 byte partial header**: loop exits when
    `epos + 4 > extraData.size`; the final `return epos == extraData.size`
    at line 377 is `false` whenever `epos < extraData.size`. ✓
- Success cases:
  - Empty blob: `epos = 0`, loop body skipped, `0 == 0 = true`. ✓
  - Blob ending exactly on sub-field boundary: `epos` ends at size;
    `true`. ✓
  - Well-formed multi-sub-field: each iteration advances past one
    complete sub-field; terminates with `epos == size`. ✓
- **Minor divergence from issue preference**: issue text suggested "an
  explicit loop (not via `Id.run do` with early returns that could hide
  a fuel bug)", but the implementation uses `Id.run do` + `while` with
  an early return. Not a defect — there is no fuel here, and the loop
  terminates by strict advancement of `epos`. Noted for the record.

### A.5 — Precedence vs. PR #1785

The fixture `cd-extra-overrun-datasize.zip` carries extra-data bytes
`[headerId=0x5455][dataSize=0xFF][4 payload bytes]` (8 bytes total), and
its CD entry has **no ZIP64 sentinel** (`stdCompSize=6, stdUncompSize=6,
stdOffset=0`; none equal `0xFFFFFFFF`). The CD-side trace is:

1. `Zip/Archive.lean:524` — `validateExtraFieldStructure` runs:
   `epos=0`, loop reads `dataSize=0xFF`, `0 + 4 + 255 > 8` → `return false`.
2. `Zip/Archive.lean:525` — throws `"zip: malformed extra field for hello.txt"`.
3. `parseZip64Extra` is **never reached** — the sentinel branch at
   line 527 is gated by `||` of three sentinel comparisons, all `false`
   for this fixture. Confirmed by fixture construction in
   `scripts/build-zip64-malformed-fixtures.py:197-203` (`make_cd_extra_overrun`)
   which explicitly emits non-sentinel size/offset fields.

Contrast with `zip64-extra-oversized-datasize.zip` (PR #1785's
fixture):

1. Outer validator: blob is `[headerId=0x0001][dataSize=16][16 bytes]` =
   20 bytes total, and the CD entry declares `extraLen=20`. So
   `epos=0 → 20`, `epos == extraData.size`, `return true`. Outer layer
   passes.
2. `parseZip64Extra` fires (sentinel is set on `stdCompSize`), and its
   inner exactness check rejects with
   `"malformed ZIP64 extra field for <name>"` because
   `fpos != fieldEnd` after consuming the single `8 * N = 8` bytes.

Neither throw substring accidentally matches the sibling's fixture: a
test matching `"malformed extra field"` would not match the message
`"zip: malformed ZIP64 extra field for hello.txt"` (the intervening
"ZIP64" breaks the substring), and vice versa. No cross-contamination.
