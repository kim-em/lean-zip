## What was accomplished

- Added a new *"Decompression Limit Inventory"* section to
  `SECURITY_INVENTORY.md` under *"Known Immediate Audit Targets"*. The
  section inventories every public decompression or extraction API
  (FFI + native + archive + tar), with file:line-linked entries and
  columns *Entry point / Parameter / Default / Semantics of 0 /
  Notes*.
- Flagged three known inconsistencies:
  - `Archive.readEntryData` silently upgrades `maxEntrySize = 0`
    to a 256 MiB cap on the native backend while the FFI backend
    stays unlimited — same caller argument, different bomb-rejection
    behaviour depending on `useNative`.
  - `Tar.extractTarGzNative.maxOutputSize` defaults to 256 MiB while
    the whole-buffer FFI decoders (`Zlib.decompress`,
    `Gzip.decompress`, `RawDeflate.decompress`) default to
    unlimited.
  - Native decoder defaults disagree with each other —
    `Zip.Native.Inflate.inflate` defaults to 1 GiB while
    `Zip.Native.GzipDecode.decompress`,
    `Zip.Native.ZlibDecode.decompress`, and
    `Zip.Native.decompressAuto` default to 256 MiB.
- Called out that `Archive.{list,extract,extractFile}` mixes a
  finite default `maxCentralDirSize` (64 MiB) with an unlimited
  default `maxEntrySize` (0), which is easy to misread.
- Noted that `Gzip.decompressStream`, `Gzip.decompressFile`, and
  `RawDeflate.decompressStream` have no output-size parameter at
  all — bounded only by the caller's sink. Proposed recommendation 2
  adds one.
- Added a *"Recommended policy"* subsection with six proposals (FFI
  whole-buffer defaults → 256 MiB; streaming FFI gains a cap;
  archive per-entry defaults → 1 GiB; remove the silent native
  upgrade; optional whole-archive cap; native defaults
  normalised). Numbers are placeholders for discussion, not final.
- Called out caller-impact: no Lean-level caller passes `0`
  explicitly today, so migration is local and grep-detectable.
- Ticked Priority 2 items 1–2 in
  `plans/track-e-current-audit-checklist.md` with pointers to the
  new section. Items 3 (bomb-limit regression tests) and 4
  (docstring policy updates) remain unchecked — they are the
  sibling issues #1552 / #1553 and a follow-up.

## Decisions made

- **Scope discipline.** The issue explicitly forbade changing any
  signatures, defaults, or docstrings in `Zip/` — inventory +
  policy proposal only. No `Zip/*.lean` files are touched; the
  commit is doc-only.
- **Final numbers deferred.** The *"Recommended policy"*
  subsection proposes concrete defaults (256 MiB whole-buffer,
  1 GiB per-entry, 1 GiB for all natives) but marks them as
  seed values for the follow-up implementation issues, not
  decisions made in this PR. The inventory stays stable even
  if the follow-ups pick different numbers.
- **Checklist cross-link target.** The
  `track-e-current-audit-checklist.md` entries link to
  `SECURITY_INVENTORY.md:151` — the `## Decompression Limit
  Inventory` heading — rather than to section 3 (which is an
  older short paragraph that now carries a *See below* pointer).
