## Findings per audit check

### A.1 Guard placement — OK

Both arms execute the guard between `stripTrailingNuls`
(which consumes the 512-byte NUL padding that tar
writers append to fill the block) and the decode
cascade (`String.fromUTF8?` then `Binary.fromLatin1`
fallback). For GNU long-name
([Zip/Tar.lean:583-591](/home/kim/lean-zip/Zip/Tar.lean:583)):

```lean
let nameData ← readBoundedEntryData input entry.size.toNat maxHeaderSize
  "GNU long name"
let nameBytes := stripTrailingNuls nameData            -- :585
if (nameBytes.findIdx? (· == 0)).isSome then           -- :586
  throw (IO.userError "tar: GNU long-name contains NUL byte")
let name := match String.fromUTF8? nameBytes with
  | some s => s
  | none => Binary.fromLatin1 nameBytes
```

Crucially, the predicate runs on the *decoded raw bytes*,
so the guard is independent of whether the decode would
have succeeded via UTF-8 or fallen back through Latin-1
— both paths produce Lean `String`s that silently accept
U+0000 (Lean `Char.ofNat 0` is a valid `Char`), so
neither decode branch re-introduces NUL into the Lean
string abstraction. The long-link arm at `:595-602` is
structurally identical.

### A.2 Predicate correctness — OK

`(nameBytes.findIdx? (· == 0)).isSome` fires iff any
byte of `nameBytes` equals `0x00`. After
`stripTrailingNuls`, the returned `ByteArray` is exactly
`data.extract 0 n` where `n` is the largest index such
that every byte in the range `[n, data.size)` is `0x00`
— i.e. trailing NULs are removed, and any NUL remaining
at position `< n` is by construction surrounded on its
right by at least one non-NUL byte, which is the
definition of an *interior* NUL. No off-by-one: the
scan starts at index `0` (the first byte), not `1`.

Boundary sanity-check on the all-zero input edge:
`stripTrailingNuls` on a 14-byte all-zero payload
returns `ByteArray.empty` (`n` decrements to 0,
`data.extract 0 0 = ByteArray.empty`); `findIdx?` over
an empty array returns `none`, so the guard does not
fire. But an all-zero long-name payload is unreachable:
the long-name pseudo-entry declares the intended
payload length in its `size` field, and `readNumeric`
on an all-zero size field yields `0`, in which case
`readBoundedEntryData input 0 maxHeaderSize` returns
`ByteArray.empty`, `stripTrailingNuls` is the identity
on empty, and `String.fromUTF8? ByteArray.empty` yields
`some ""` — a legitimate zero-byte long-name that both
pre- and post-PR treat as "apply `""` as the next
entry's path override". No regression.

The interior-NUL detection is also correct in the
asymmetric case where trailing NULs are absent (already
covered by the non-regressing `gnu-longname-no-terminator.tar`
assertion): if the full 100-byte payload is non-NUL,
`stripTrailingNuls` returns it unchanged, `findIdx?`
returns `none`, and the guard does not fire.

### B.2 Payload decoding — OK

The long-name payload starts at file offset `0x200`
(block index 1). First 14 bytes, per `xxd`:

```
00000200: 6576 696c 2e74 7874 002e 7461 7200 0000
           e v i l . t x t \x00 . t a r \x00
```

= `"evil.txt" (8) ++ 0x00 ++ ".tar" (4) ++ 0x00` = 14
bytes. Matches the pseudo-header's `size = 0000000016`
(octal) = 14 (decimal) at the size-field offset of the
preceding long-name header (verified via `xxd` at file
offset `0x7C`: `3030 3030 3030 3030 3031 3600` =
`"00000000016\x00"`).

After `readBoundedEntryData input 14 maxHeaderSize`,

`stripTrailingNuls` removes the single trailing NUL at
position 13, yielding a 13-byte `nameBytes` =
`"evil.txt\x00.tar"`. `findIdx? (· == 0)` on these 13
bytes returns `some 8` (the NUL between `"evil.txt"`
and `".tar"`), `.isSome` is `true`, and the guard
throws. The 14-byte single-trailing-NUL construction is
minimal: it forces `stripTrailingNuls` to strip exactly
one byte, which means the interior-NUL + trailing-NUL
interaction is exercised distinctly (a multi-trailing-NUL
payload would still work, but would not add coverage).

### D.1 PAX record guard — OK

PR #1866 landed as merge commit `2dddfa8`
(2026-04-24T20:13:45Z), 9 minutes after PR #1865. The
PAX-side guard lives *inside* `parsePaxRecords` at
[Zip/Tar.lean:122-133](/home/kim/lean-zip/Zip/Tar.lean:122)
rather than as a throwing guard in the caller:

```lean
-- Skip records with invalid UTF-8 rather than panicking.
-- Also skip records whose raw key or value bytes contain NUL (0x00):
-- `String.fromUTF8?` accepts U+0000 as valid UTF-8, so without this
-- guard an attacker-controlled PAX record `path=a\x00b/c` would
-- decode to a `"path"` override whose value smuggles a NUL past
-- `Binary.isPathSafe` into `Tar.extract`, where POSIX `open`
-- truncates at NUL and the file lands at the short prefix while
-- `Tar.list` callers see the full embedded string — a classic
-- parser-differential / filesystem-truncation smuggle vector,
-- sibling to the ZIP CD name NUL guard at Zip/Archive.lean:630.
if let (some key, some value) := (String.fromUTF8? keyBytes, String.fromUTF8? valueBytes) then
  if (keyBytes.findIdx? (· == 0)).isNone
      && (valueBytes.findIdx? (· == 0)).isNone then
    records := records.push (key, value)
```

Two confirmations:

1. **Guard fires before `applyPaxOverrides`.** The
   record is only pushed into `records` if *both* the
   key bytes and value bytes pass the NUL predicate.
   `forEntries` stores the resulting `records` array in
   `paxOverrides`
   ([Zip/Tar.lean:609](/home/kim/lean-zip/Zip/Tar.lean:609))
   and only later feeds it to `applyPaxOverrides` when
   the *next* regular entry is parsed. Because any
   crafted PAX record with `path=a\x00b/c` never enters
   the array, the `"path"` / `"linkpath"` match arms
   in `applyPaxOverrides`
   ([:143-144](/home/kim/lean-zip/Zip/Tar.lean:143))
   never see the smuggled value.
2. **Silent-skip pattern, not throw.** Matches the
   pre-existing `parsePaxRecords` failure mode for
   invalid UTF-8, inconsistent-length records, and
   length-parse-error records. The fixture assertion
   at
   [ZipTest/TarFixtures.lean:139-150](/home/kim/lean-zip/ZipTest/TarFixtures.lean:139)
   is correspondingly a *positive* regression
   (like `hardlink-outside.tar`): the test checks that
   `Tar.list` returns exactly the follow-on regular
   `"hello.txt"` entry with its declared path, rather
   than the smuggled `"a\x00b/c"` override. The
   in-source comment at `:135-138` explicitly calls
   this out.

### E.2 Writer-side confirmation — OK

`buildPaxEntry` and `create` in the same file emit
entry paths via `entry.path.toUTF8` / `entry.linkname.toUTF8`.
A NUL byte can reach the output stream only if the Lean
`String` input already contains the U+0000 codepoint —
which requires a caller to explicitly construct such a
string (Lean's `String.Pos` indexing tolerates it; there
is no sanitisation at the `String` level). In practice:

- `buildPaxEntry` writes `"path=<utf-8>\n"` records
  where `<utf-8>` is `entry.path.toUTF8`. A NUL-embedded
  path would round-trip through the same guard at
  `parsePaxRecords:132-133` on the read side — the
  reader silently drops the crafted override.
- `create` at `:478-533` builds UStar headers via
  `Binary.writeBytes` on the first 100 / 155 bytes of
  the UTF-8 encoding. A NUL byte in those bytes would
  C-string-terminate the field when read back, which
  on a non-truncated round-trip would surface as a
  short path rather than as a filesystem write at the
  NUL-truncated destination. Low-priority (not a
  writer-side smuggle), but noted.

No changes recommended to the writer side for this PR.
