### A.2 Block geometry (1 536 B)

All five family fixtures are exactly **1 536 B** — 512 B of UStar
header + 1 024 B of zero padding (two trailing zero blocks forming a
well-formed end-of-archive). The zero-padding tail matches strict peer
parsers' (GNU tar / BSD tar / libarchive) accept-condition for
end-of-archive; the guard fires during header parse so the trailing
blocks are only exercised by the no-guard regression baseline.

The 1 536 B size is identical across all five sibling fixtures — the
only bytes that differ across the family are within the 512 B header
(the per-slot smuggled value plus the recomputed `chksum` octets).

fixture is built with a direct `entry.gname := "trusted\x00rogue"`

- **`gname = "trusted\x00rogue"` carries an interior NUL at byte 7**
  (between `'d'` and `'r'`, before the trailing NUL padding starts) —
  the `gname`-arm sub-check at `Zip/Tar.lean:538` evaluates `true`,

So the `gname` arm is the **uniquely** tripped sub-check for this
fixture — distinct from the four earlier arms by guard sequencing and
clean-slot construction. The choice of `"trusted\x00rogue"`:

- **13 printable-ASCII bytes** (`trusted`, `rogue` separated by an
  interior NUL) — symmetrically shaped to PR #1944's `uname`-slot
  smuggle `"trusted\x00rogue"` (same 13-byte payload; same NUL
  position at byte 7). Choosing identical content for the two
  defense-in-depth slots emphasises that they share a single attack
  class (parser-differential `Tar.list` smuggling for trust
  decisions). The interior NUL position is offset 7 inside the 13-byte
  payload — strictly *interior*, not a trailing padding NUL.
- **Encodes the trust-decision attack class in its content** — a
  caller routing on `entry.gname == "trusted"` (without parsing the
  full bytes) sees the truncated prefix while peer parsers preserve
  the full bytes; the contrast `trusted` → `rogue` makes the
  parser-differential threat model legible to a human reading the
  fixture.
- **Fits within the 32-byte `gname` field** (13 of 32 bytes used; 19 B
  of NUL padding follows) — no field-overrun hazard. The `gname` slot
  shares the 32 B width with the sibling `uname` slot (smallest of
  the five guarded slots; 100 B for `name`/`linkname` and 155 B for
  `prefix`).
- **Distinct slot location from the four earlier slots** — though the
  payload is identical to PR #1944's, the slot offset (297 vs 265) and
  the per-slot attribution discriminator (test substring `"gname"` vs
  `"uname"`) differ. The `xxd` diff between the gname-slot and
  uname-slot fixtures shifts the 13-byte payload from bytes 265-277 to
  bytes 297-309 (a clean +32 B offset shift), with the recomputed
  `chksum` octets and the per-slot byte sums adjusted accordingly.

### A.4 Checksum verification (offset 148, 8 B)

Independent recomputation of the UStar header checksum (sum of all
512 header bytes treating the chksum field as 8 ASCII spaces):

- name slice (`"safe"` + 96 NUL): 115 + 97 + 102 + 101 = **415**
- mode (`"0000644\0"`, 4·`'0'` + `'6'` + `'4'` + `'4'` + NUL):
  4·48 + 54 + 52 + 52 + 0 = **350**
- uid (`"0000000\0"`, 7·`'0'` + NUL): 7·48 = **336**
- gid (`"0000000\0"`, 7·`'0'` + NUL): 7·48 = **336**
- size (`"00000000000\0"`, 11·`'0'` + NUL): 11·48 = **528**
- mtime (`"00000000000\0"`, 11·`'0'` + NUL): 11·48 = **528**
- chksum (8 spaces during compute): 8·32 = **256**
- typeflag (`'0'`): **48**
- linkname (100 NUL): **0**
- magic (`"ustar\x00"`): 117 + 115 + 116 + 97 + 114 + 0 = **559**
- version (`"00"`): 2·48 = **96**
- uname (32 NUL): **0**
- gname (`"trusted\x00rogue"` + 19 NUL):
  - `trusted` = 116 + 114 + 117 + 115 + 116 + 101 + 100 = **779**
  - NUL = 0
  - `rogue` = 114 + 111 + 103 + 117 + 101 = **546**
  - 19 NUL = 0
  - subtotal: **1 325**
- devmajor + devminor + prefix + padding: **0**

Total = 415 + 350 + 336 + 336 + 528 + 528 + 256 + 48 + 0 + 559 + 96
+ 0 + 1 325 + 0 = **4 777**

In octal: 4 777 = **011251**₈. The on-disk chksum field at offset 148
reads `"011251\0 "` (6 octal digits + NUL + space — the canonical
UStar form per POSIX.1-1988). The independent recomputation matches
the writer's value exactly — checksum integrity is preserved, so the
header passes the `validateUStarChecksum` gate and reaches the
interior-NUL guards.

Each sub-check has the canonical form `if hasInteriorNul block

NUL byte")`. The five throws share the prefix `"tar: UStar "` and
differ in the slot-name suffix
(`"name contains NUL byte"` /
`"linkname contains NUL byte"` /
`"prefix contains NUL byte"` /
`"uname contains NUL byte"` /
`"gname contains NUL byte"`). Source-position order is `name` →
`linkname` → `prefix` → `uname` → `gname`; a trip at the `gname` arm
requires all four earlier slots to be NUL-clean, which the fixture's
`name = "safe"` plus all-NUL `linkname`, `prefix`, and `uname` slots
ensure.

The guards run **after** the checksum + magic checks (so header
integrity is confirmed first) and **before** any `Binary.readString`
call on the five string fields — the security-critical ordering.
Without the `gname`-arm guard, `Binary.readString` would silently
truncate the `gname` field at the NUL byte to `"trusted"`, and a
caller routing on `entry.gname` for a trust decision would see the
short prefix while strict peer parsers preserve the full bytes
(the parser-differential smuggle this fixture pins). Like `uname`
(and unlike the three filesystem-reaching slots), the `gname` field
does **not** flow into `Tar.extract`'s filesystem operations — the
guard is defense-in-depth at the `Tar.list` layer.

data-flow structure of `Tar.extract`. The five UStar string fields
split cleanly along this dimension, with no fields straddling the
boundary. PR #1957's terminal-closure narrative in the *Recent wins*
