# Progress: Initial Zstd Frame Specification Types

- **Date**: 2026-03-04 23:50 UTC
- **Session type**: feature
- **Issue**: #578

## What was accomplished

Created `Zip/Spec/Zstd.lean` — the first Zstd specification file in the project,
establishing foundational types and predicates for RFC 8878 frame and block headers.

### Specification predicates (all with `Decidable` instances):
- `validMagic`: exact match on `0xFD2FB528`
- `isSkippableMagic`: range `0x184D2A50`–`0x184D2A5F`
- `validFrameHeaderDescriptor`: reserved bit (bit 3) must be 0
- `validWindowSizeExponent`: exponent ≤ 41
- `validBlockType`: block type ≠ reserved
- `validBlockSize`: block size ≤ 131072 (128KB)
- `ValidBlockHeader`: combines type and size validity

### Correctness theorems:
- `parseBlockHeader_type_ne_reserved` — **proven**: when `parseBlockHeader` succeeds,
  the block type is not reserved. Uses monadic unfolding with `obtain ⟨rfl, rfl⟩` pattern.
- `parseFrameHeader_magic` — **sorry**: `parseFrameHeader` uses `do`-notation with
  mutable state (`let mut`), making direct unfolding hit heartbeat limits.
  Needs a helper lemma isolating the initial magic-check fragment.
- `parseBlockHeader_blockSize_le` — **sorry**: the implementation doesn't enforce
  the 128KB limit (21-bit field allows up to 2^21-1). Needs #540's validation.

### Infrastructure:
- Added `DecidableEq` instance for `ZstdBlockType` (needed for `validBlockType`)
- Registered in `Zip.lean` imports

## Decisions made

- Followed `Zip/Spec/Deflate.lean` pattern for module structure and docstrings
- Used `Zstd.Spec` namespace (paralleling `Deflate.Spec`)
- Deferred `parseFrameHeader_magic` proof rather than fighting monadic state unfolding —
  a helper lemma about the first 8 lines of `parseFrameHeader` would make this trivial
