# Progress: Track E — FSE-compressed Huffman tree descriptor for Zstd

**Date**: 2026-03-02 10:30 UTC
**Session**: a341e8a3 (feature)
**Issue**: #468

## Accomplished

Implemented FSE-compressed Huffman tree descriptor decoding (RFC 8878 §4.2.1),
the common case in real-world Zstd files at default compression levels.

### Changes

1. **`Zip/Native/Fse.lean`**: Added `decodeFseSymbolsAll` — like `decodeFseSymbols`
   but loops until `BackwardBitReader.isFinished` instead of a fixed count. Uses
   a fuel parameter (4096) for termination.

2. **`Zip/Native/ZstdFrame.lean`**:
   - Added `import Zip.Native.Fse`
   - Added `parseHuffmanWeightsFse`: creates a BitReader over the compressed range,
     calls `decodeFseDistribution` (maxSymbols=256, maxAccLog=6), builds the FSE
     table, creates a BackwardBitReader for remaining bytes, decodes all weights
   - Updated `parseHuffmanTreeDescriptor`: header byte >= 128 now dispatches to
     `parseHuffmanWeightsFse` instead of returning an error

3. **`ZipTest/ZstdNative.lean`**:
   - Updated test 51: FSE mode with invalid data still errors (but no longer
     checks for specific "FSE-compressed" error message)
   - Added test 55: Integration test with `zstd -3` compressed data — verifies
     `parseHuffmanTreeDescriptor` succeeds on real FSE-compressed tree descriptors
     and produces reasonable maxBits
   - Added test 56: Unit test for `decodeFseSymbolsAll` with a manually-built
     2-symbol FSE table

## Key decisions

- Used `BitReader.alignToByte` to find where the FSE distribution encoding ends,
  since the backward bitstream starts at the next byte boundary after the
  distribution data
- The integration test (test 55) gracefully handles cases where zstd doesn't use
  compressed literals at the tested compression level, since block format depends
  on data characteristics
