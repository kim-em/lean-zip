## What was accomplished

Added the all-zero closed-form theorem to the Adler-32
characterizing-property ladder, plus its `Native` bridge:

- `Zip/Spec/Adler32.lean`:
  - `checksum_replicate_zero (n : Nat) (hn : n < 65521)` —
    `checksum (List.replicate n 0) = UInt32.ofNat (1 + n * 65536)`.
    Placed between `checksum_pair` and `checksum_append` per the
    issue's slot.
- `Zip/Native/Adler32.lean`:
  - `adler32_replicate_zero (n : Nat) (hn : n < 65521)` —
    `adler32 1 (ByteArray.mk (Array.replicate n 0)) =
       UInt32.ofNat (1 + n * 65536)`.
    Placed between `adler32_pair` and `adler32_append`.

The `Spec` proof strengthens the goal to a generalized invariant
`updateList (1, k) (List.replicate m 0) = (1, k + m)` (for any
starting B-component `k` with `k + m < 65521`), proven by induction
on `m`. The inductive step uses `List.replicate_succ`, the fact that
`updateByte (1, k) 0 = (1, k + 1)` (no modular reduction fires), and
recursion at `(1, k + 1)`. Finishing applies the existing
`pack_toNat_of_bounds` helper and `UInt32.toNat_inj`.

The `Native` proof reuses the same three-line bridge cadence that
`adler32_singleton` and `adler32_pair` use: unpack the default
init, drop into `Spec` via `updateBytes_eq_updateList`, then apply
`Spec.checksum_replicate_zero`. The `(ByteArray.mk (Array.replicate
n 0)).data.toList → List.replicate n 0` step is handled
definitionally by `exact`, so no explicit `Array.toList_replicate`
rewrite is needed (a draft using one tripped the `unusedSimpArgs`
linter, confirming reduction happens for free here).

## Decisions / patterns

- **Strengthen-then-instantiate** for the induction: stating the
  invariant for a free starting `k` avoids the awkward
  `updateByte (1, k) 0 → (1, k+1)` step having to also re-prove
  the constant first component each time. This is the same
  generalize-to-simplify mental model already codified in the
  project CLAUDE.md.
- **Fresh induction variable** (`m`, not `n`): doing
  `induction n` after `hn : n < 65521` was in scope swept the
  hypothesis into the induction motive, leaving `ih` with an
  unwanted `Nat → Prop` first argument. Pulling the strengthened
  statement out as `∀ (m k : Nat), …` and inducting on the bound
  `m` keeps the IH clean.
- **No new `@[simp]` tags**: matches the existing `_pair` /
  `_singleton` precedent; this theorem is hypothesis-bearing so
  `@[simp]` would be even less appropriate.
- **`pack_toNat_of_bounds` left `private`**: still no
  out-of-file consumer, so the visibility discussion noted in the
  issue stays at the same conclusion as the `_pair` issue.
