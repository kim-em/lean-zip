## Issue
https://github.com/kim-em/lean-zip/issues/1538 — Track E Priority 1:
malformed PAX extended-header regression fixtures.

## What was accomplished
- Added `scripts/build-pax-malformed-fixtures.lean` — a deterministic
  Lean builder that emits five minimal UStar tar archives whose first
  entry is a malformed PAX extended header (typeflag `'x'`) followed by
  a regular-file entry carrying `"hello\n"`. Each fixture is 2048 bytes,
  matching the geometry of the existing `bad-checksum.tar` /
  `no-magic.tar` fixtures (no end-of-archive zero blocks — `readExact`
  already breaks on short read).
- Emitted and committed five fixtures under
  `testdata/tar/malformed/`:
  - `pax-oversized-length.tar` — PAX record with a 29-digit decimal
    length prefix; exceeds the `digitCount > 20` guard.
  - `pax-truncated-record.tar` — declares `length = 100` but the
    PAX data block is only 50 bytes; trips `recordEnd > data.size`.
  - `pax-invalid-utf8-key.tar` — key bytes are a lone continuation
    byte `0xC3`; `String.fromUTF8?` returns `none`.
  - `pax-invalid-utf8-value.tar` — value bytes encode a UTF-16
    surrogate `ED A0 80` (U+D800); rejected by `String.fromUTF8?`.
  - `pax-inconsistent-length.tar` — declares `length = 10` with no
    `=` separator within; skipped by the `eqPos < recordEnd` guard.
- Added a test block to `ZipTest/TarFixtures.lean` that asserts each
  fixture lists to exactly one entry with `path == "hello.txt"`
  (PAX silently skipped, regular entry still visible). Registered the
  five new filenames in the temp-file cleanup loop.
- Updated `plans/track-e-current-audit-checklist.md`: moved the
  malformed-PAX-fixtures bullet from `- [ ]` to `- [x]` with a
  parenthetical listing the five new fixtures.

## Key findings
- No guard gap surfaced: all five malformations were silently skipped
  by existing `parsePaxRecords` / `String.fromUTF8?` guards, matching
  the issue's expectation. No new hardening needed.
- Lean's `String.fromUTF8?` rejects both lone continuation bytes and
  UTF-16 surrogates, as expected per RFC 3629.
