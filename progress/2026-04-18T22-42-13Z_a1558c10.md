## What landed

New malformed fixture
`testdata/zip/malformed/oversized-zip64-uncompressed-size.zip` (134
bytes) plus a regression test in `ZipTest/ZipFixtures.lean` matching
on `"size mismatch"`. Sibling of #1531 (oversized-`compressedSize`
ZIP64 variant, PR #1543).

Fixture shape:
- Valid ZIP64 archive storing `hello\n` (6 bytes, method = stored).
- Local header: 32-bit `compressedSize = 6`, `uncompressedSize =
  0xFFFFFFFF` (ZIP64 marker, but `readEntryData` never re-parses
  these fields from the local header — it only reads `nameLen` and
  `extraLen`).
- Central-directory entry: 32-bit `compressedSize = 6`,
  `uncompressedSize = 0xFFFFFFFF`, `localOffset = 0`.
- Central-directory ZIP64 extra: header id `0x0001`, data size 8,
  u64 `uncompressedSize = 0x1000000000000000` (≈ 1.15 EiB). No
  `compressedSize`/`localOffset` fields in the extra (they fit in
  32 bits), matching `parseZip64Extra`'s conditional parse order.

## Which check fired

Not a span check — the `"size mismatch for {label}"` check in
`readEntryData` at
[Zip/Archive.lean:448-449](../Zip/Archive.lean):

```lean
unless fileData.size.toUInt64 == entry.uncompressedSize do
  throw (IO.userError s!"zip: size mismatch for {label}")
```

Trace through on this fixture:
- `entry.uncompressedSize = 0x1000000000000000`,
  `entry.compressedSize = 6`, `fileSize = 134`.
- All three `assertSpanInFile` calls pass — the small
  `compressedSize` keeps the `local data span` check happy.
- Local 6-byte payload is read; method = 0 stored, so `fileData =
  compData` (size 6).
- CRC check passes (we wrote the correct CRC).
- Final size check: `fileData.size.toUInt64 = 6`,
  `entry.uncompressedSize = 0x1000000000000000` → mismatch →
  `IO.userError "zip: size mismatch for hello.txt"`.

## Is this a guard gap?

**No.** The issue's case 3 ("oversized `uncompressedSize` claim is
silently ignored") does NOT apply: `readEntryData` validates
`fileData.size == entry.uncompressedSize` after every read. The
oversized `uncompressedSize` is caught cleanly without panic or OOM —
no upstream `extractToFile` allocation is driven by it (the actual
allocation comes from the small `compressedSize` payload read; for
stored method that's also the decompressed size).

Worth noting: the fixture uses method 0 (stored), so it never reaches
`Native.Inflate.inflate` or `RawDeflate.decompress`. For method 8
(deflated) with an oversized `uncompressedSize`, `readEntryData`
applies the `maxEntrySize` check at line 409-410 *before* reading,
which would reject the oversized claim earlier with a different
substring (`"exceeds limit"`). That path is exercised by other
existing tests (the `maxEntrySize` codepath is a separate guard).

## Exabyte value written

`0x1000000000000000` (= 2^60 bytes, ≈ 1.152 × 10^18 bytes,
≈ 1.15 EiB). Mirrors #1531/PR #1543 exactly so the two fixtures use
the same canonical "exabyte-scale" value at hex-diff time.

## Build recipe

Inlined in `ZipTest/ZipFixtures.lean` adjacent to the
`oversized-compressed-size.zip` block, matching the precedent set by
the 32-bit fixture and #1531/PR #1543 (no committed generator
script). Single `python3 -c '...'` invocation using only `struct`
and `zlib` from stdlib. Differs from #1543's recipe only in:

- the `lh` `compressedSize` and `uncompressedSize` fields are swapped
  (`len(p)` and `0xFFFFFFFF` exchange positions),
- the `cd` `compressedSize` and `uncompressedSize` fields are
  swapped likewise,
- the `ex` u64 value applies to `uncompressedSize` (it's the same
  `<HHQ` packing — the meaning shifts because `parseZip64Extra`
  reads ZIP64-extra u64s in order based on which 32-bit slots are
  `0xFFFFFFFF`).
