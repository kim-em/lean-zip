## What created friction

- **Two fix attempts for the Lake/toolchain regression (#1495 →
  #1506).** The `Lean 4.29.1 + zipCommon drift` breakage compounded
  two independent causes:
  1. `lake exe` validating the compiled lakefile against a newer
     manifest written during `lake build`, producing the
     `compiled configuration is invalid` error.
  2. Floating `zipCommon` at `main` drifted past the last commit
     compatible with master's lemma names.
  #1495 addressed only (1) by swapping `lake exe test` → `lake test`.
  That didn't actually add `-R` to the test step, so CI still failed.
  #1506 finally combined `lake -R build` / `lake -R exe test` with a
  hard pin of `zipCommon` to `f53e886`. Retrospectively, the cheap
  diagnostic (*does a fresh `.lake` against `main` build at all?*)
  would have surfaced (2) on the first attempt. The three diag PRs
  (#1496, #1503, #1507) ended up being closed without merging once
  the dual root cause was understood — they did useful diagnostic
  work but are now dead.
- **Codex PR #1499** was an alternative approach to the same drift
  (downstream rename + pin to `89204d6`). It landed alongside #1506
  rather than being superseded — the two branches were independent
  enough that the merges composed, but this was by luck. During the
  several hours that both branches were live, it wasn't obvious which
  approach was canonical.
- **Stale issue-body metrics.** At least three issue bodies had
  metrics that didn't match `HEAD` at claim time:
  - #1484 Gzip: claimed "4 sorries (all XxHash)" — actual 0
    (XxHash had moved to `lean-zip-common`).
  - #1261 Inflate review: claimed "62/28 bare simp" — actual 0/0
    from the already-completed bare-simp campaign.
  - Issue body numbers lagged the summarize cycle. Workers handled
    this correctly (adjusted and noted in the progress entry), but
    it's a small tax every time.
- **Spec cascade cost when runtime functions change shape.** Session
  `bcee9302` found that converting three `]!` sites in
  `emitTokensWithCodes` requires rewriting ~300 lines across
  `DeflateDynamicEmit.lean`, `DeflateDynamicFreqs.lean`, and
  `DeflateDynamicCorrect.lean`. This is the underlying reason
  clusters A/B of #1505 are still open.
- **`List.pmap` inlining trips `rfl`.** Session `709ef861` hit
  `maximum recursion depth` inside `deflateDynamic_spec.hdef` when
  `List.pmap` was inlined in a function whose spec is established by
  `rfl`. Hiding the `pmap` behind a named helper (`freqsToPairs`)
  fixed it — `List.pmap` is `@[expose]` so inline uses are eagerly
  reduced during elaboration. This is a new pattern the current
  proven-bounds SKILL doesn't mention.
- **`mut`-rebinding subtlety in Gzip.** Session `5ac76a23` found that
  `unless` bodies in do-notation rebind `mut` variables and drop the
  proof-bound names, not just `if … then throw` (the form already in
  the skill's pitfall #5). Fix: read every byte you need from the
  array **before** any control-flow branch that could rebind `pos`.
