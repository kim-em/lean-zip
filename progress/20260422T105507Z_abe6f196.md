## 1. One-PR-per-`SECURITY_INVENTORY.md`-bullet cadence — now stable

### What happened

The *Recommended policy* block lists five items; the wave executed
them as five separate PRs, in strict numerical-rec order, all
within a 103-minute window:

| Rec. | PR | Merge | Docstring wording change |
|---|---|---|---|
| Rec. 5 | #1617 | 06:31Z | Native-side `maxOutputSize` default → `1 * 1024 * 1024 * 1024` |
| Rec. 3 | #1618 | 06:48Z | Archive/Tar `maxEntrySize` default → `1 * 1024 * 1024 * 1024` |
| Rec. 1 | #1623 | 07:35Z | FFI whole-buffer `maxDecompressedSize` default → `1 * 1024 * 1024 * 1024` |
| Rec. 4 | #1630 | 08:10Z | New `maxTotalSize` parameter on whole-archive extraction |
| Rec. 2 | #1631 | 08:28Z | Streaming-FFI default flip (param added by #1610) |

Each PR's commit message names its rec. number
(`3bbf32e0`: *"Recommended policy item 5 rewritten as an Executed
past-tense one-liner"*; `d536c981`: *"Recommended policy item 3 —
rewritten as the Executed past-tense one-liner"*; `bebc249a`:
*"Recommended policy item 4 as a one-line Executed entry"*). The
inventory bullet mutates from future-tense (*"Recommend raising
default to 1 GiB…"*) to past-tense (*"Executed — default raised to
1 GiB in #1623. See that PR for caller impact."*).

### Why this is now the stable shape

Comparing against the prior wave's three-PR P2.4 split
(#1573 / #1586 / #1594):

- Both waves use **one PR per bullet**, not a multi-bullet rollup.
- Both waves **assign each PR to a single session** (no
  cross-session commits on any of the ten PRs across the two
  waves).
- Both waves **quote the template phrasing in the progress entry**
  so the next author can grep-verify their own phrasing.

Two independent waves producing the same three operational
invariants is enough to call this shape **load-bearing**, not
contingent. It is worth formalising.

### What to codify

A new inventory-reconciliation skill (draft in §8 below) should
document:

1. The *Executed past-tense one-liner* phrasing ("Executed — …
   See PR #N.")
2. The *one PR per bullet* rule, with its cost-of-violation
   framing (multi-bullet PRs would make `scripts/check-inventory-links.sh`
   warnings harder to attribute).
3. The *single-author-per-wave* invariant preserved from the #1569
   meditate (prior wave) as a carry-over.

### Caveats

The cadence *is* contingent on the inventory still having
numbered *Recommended policy* items. Once that block empties
(which this wave largely did), the cadence does not apply to
e.g. *Missing work* bullets, which have no numbering and are
individually smaller. The skill should note this scope.

## 2. Half-closed two-step pattern — recurring, formally codify

### What happened

#1610 added `maxDecompressedSize : UInt64 := 0` to the streaming
FFI entry points, with `0 = unlimited` semantics (behaviour
unchanged for existing callers). #1631 flipped the default to
1 GiB eighteen minutes later. Between them, two PRs (#1616's
review, #1622's rebase-fix ride-along) and one unrelated feature
PR (#1623) landed.

The prior wave's analogue is #1573 → #1586 / #1594 (template
docstring added first, then propagated). The earlier #1487-era
`lean-zstd` extraction had the same structure (add the parameter
with a safe default, flip the default later once callers are
known safe).

### Why a two-step

The author of #1610 documented the rationale inline:

> Leaving the default at `0` means *"no behaviour change for any
> existing caller on this PR's merge commit"*. Flipping the
> default is a separate concern — it needs caller-impact analysis
> and possibly a `ZipTest/` migration, which are independent of
> the parameter-plumbing work.

`da8fb649` also specifically flags that `#1625` (the follow-up
issue for the flip) was filed at the same time as the PR body
went up, so the planner and reviewers could see the half-closed
shape at review time.

### When the two-step pays for itself

Two-step when:

- The parameter addition is mechanical (threading `maxDecompressedSize`
  through an already-existing function signature).
- The default flip needs independent review — caller migration,
  test cost analysis, fuzz exposure.
- The param-add touches multiple subsystems but the flip is
  localised (Rec. 2's flip only affects the streaming-FFI
  module's three entry points).

One-step when:

- The parameter has no existing callers (no backwards-compat
  burden).
- The flip's caller impact is trivially bounded at review time
  (e.g., single-call-site).

#1630 (Rec. 4) is *one-step* — the parameter was new, with no
existing callers. The author considered and rejected a two-step
approach in `bebc249a`:

> `maxTotalSize` is a new parameter with no callers outside the
> test suite and the docstring default landed at 1 GiB directly.
> A two-step here would mean an unused `0` default for one PR
> cycle, which looks like accidental zero-cap exposure.

Good meta-judgement. The two-step is a tool, not a rule.

### What to codify

Adding a subsection to the new inventory-reconciliation skill
(§8) headed *"When to use the half-closed two-step"*, with the
three-bullet *when* and the two-bullet *when not* above.

## 3. Ride-along progress-entry rebase-fix PRs — now routine, needs explicit skill

### What happened

**Three** ride-along rebase-fix PRs in this wave alone
(#1622, #1624, #1632), versus two in the prior wave (#1584, #1592).
All five are shape-identical:

1. A feature PR's auto-merge queue contention produces a
   `SECURITY_INVENTORY.md` conflict (usually in *Known
   inconsistencies* or *Missing work*).
2. Another agent rebases the feature branch.
3. The rebased feature PR auto-merges before the rebase author's
   progress entry can ride along.
4. The progress entry lands as a tiny one-file follow-up PR.

The prior meditate (#1600 §3) called this pattern *"healthy
practice"* and documented the convention fix in `agent-pr-recovery`
(commit the progress entry first, so one PR closes both fix and
doc). That pattern is explicitly referenced in `dac1a047`,
`8e74071c`, and `eded8629`:

- `dac1a047` (#1622): *"attempted progress-entry-first per the
  agent-pr-recovery skill, but auto-merge won the race on #1618"*
- `8e74071c` (#1624): *"follows the 'progress-entry-first'
  pattern documented in the agent-pr-recovery skill ('Auto-merge
  race: ride-along progress entries')"*
- `eded8629` (#1632): *"the rebased #1631 auto-merged before this
  session could amend the progress entry on the rebased branch"*

Three wins for the *accept-the-split* fallback, zero wins for the
*progress-entry-first* primary path. That ratio is evidence
enough to re-evaluate the convention.

### The real cost

Each ride-along costs:

- ~1 session of dispatcher attention (~3-5 minutes of agent
  wall-time, plus CI).
- ~1 additional PR against the issue counter (counted as
  infrastructure in the summarize).

At three per wave, that's non-trivial but not catastrophic. The
alternative (losing the progress entry) would be worse because
`progress/` is the project's institutional memory.

### What changed since #1600

The prior meditate assumed the *progress-entry-first* commit
ordering would resolve the race. In practice, auto-merge happens
on CI completion, not on push, so an author who rebases → pushes
→ waits-for-CI → sees-auto-merge-fire has no opportunity to tack
on a progress entry even with perfect commit ordering. The
*"push rebase and progress entry together in one push"* is the
only winning ordering, and that requires the author to write the
progress entry *before* pushing the rebase, which defeats the
point of a progress entry (it's written after the rebase is
verified).

### Recommendation — update the skill

The `agent-pr-recovery` skill's *Auto-merge race* section should
be rewritten to:

- **Declare the split PR the primary path, not the fallback.**
  Document the two-file shape: rebase PR (feature + conflict
  resolution) and ride-along PR (progress entry + no code).
- **Keep a short "if you're really fast you can ride along"
  note** with a 30-second window estimate, but not as the primary
  recommendation.
- **Name the convention** — call it a *"ride-along ladder"* so
  planners know to expect one ride-along PR per non-trivial
  rebase fix, and can budget dispatcher attention accordingly.

## 5. Characterizing-property emergence — the first two proofs establish a template

### What happened

Two adjacent ~90-minute sessions produced the first two
`_append` characterizing-property theorems:

- `505fda1c` (#1640, CRC32): `Spec.Crc32.crc32_append` and
  `Native.Crc32.updateBytes_append`. Top-level statement:
  `crc32 (a ++ b) = crc32Combine (crc32 a) (crc32 b) b.size`.
- `0c234bba` (#1641, Adler-32): `Spec.Adler32.adler32_append` and
  `Native.Adler32.updateBytes_append`. Top-level statement uses a
  `pack_toNat_of_bounds` helper to collapse the bitwise OR.

Both proofs share a three-part structure:

1. **Spec-level append identity** — prove the compositional
   property at the `Spec/` level where the function is `foldl` or
   equivalent over a `List UInt8`.
2. **Native-level append theorem** — drop to `updateBytes_append`
   by rewriting through `updateBytes_eq_updateList`, which is the
   standard Native↔Spec bridge lemma.
3. **Top-level entry-point wrapper** — peel the user-facing
   `crc32` / `adler32` down to the native call and apply the
   Native append theorem.

The first proof was slower (`505fda1c` reported ~90 min for the
`crc32_append` + Native bridge); the second was faster
(`0c234bba` reported ~70 min, citing the #1640 structure as a
template).

### The two divergence points

CRC32 and Adler-32 differ in their bitwise-collapse automation:

- **CRC32** — uses `bv_decide` to collapse `UInt32` equations
  involving XOR. `505fda1c` noted *"bv_decide handles the final
  case-split on `x == 0` directly; no manual `by_cases` needed"*.
- **Adler-32** — uses `Nat.two_pow_add_eq_or_of_lt` plus `omega`
  to collapse the bitwise-OR pack/unpack. `0c234bba` factored this
  into a reusable `pack_toNat_of_bounds` helper, saving ~30 lines
  of duplicated proof compared to an inline approach.

Both automation choices fit the project's `lean-no-mathlib`
constraint and match `.claude/CLAUDE.md`'s preference for
`omega` / `decide` / `bv_decide` over manual arithmetic.

### What this means for future hash work

Any future hash family (XxHash, FNV-1a, further fluid checksum
variants) can follow this template directly:

- Prove `Spec._append` as an algebraic identity over `List UInt8`.
- Prove `Native._updateBytes_append` via the `updateBytes_eq_updateList`
  bridge.
- Wrap in the user-facing entry-point theorem.

The inline skill draft in §8 calls this out as a candidate
characterizing-property skill, but I think it's too thin with
only two examples. Wait for a third (likely XxHash once
`lean-zstd` stabilises upstream) before promoting.

### `pack_toNat_of_bounds` is more widely useful

`0c234bba`'s extracted helper for collapsing bitwise-OR into
`Nat`-addition when each summand fits in its nibble is general
enough to be worth a row in `lean-uint-bitvec`. §8 contains the
skill-diff.

## 6. Compile-time `rfl` signature probes — emerged organically, worth naming

### What happened

Three PRs in the wave (#1617, #1623, #1631) changed default
literal values in function signatures (all `maxOutputSize` /
`maxDecompressedSize` from an older value to `1 * 1024 * 1024 *
1024`). Each PR needed a regression check: is the default
literal still what the docstring claims?

All three converged on the same pattern: add a one-line
compile-time probe of the shape

```lean
/-- Regression probe: `maxDecompressedSize` defaults to 1 GiB. -/
example : @Zlib.decompress = @Zlib.decompress
          (maxDecompressedSize := 1024 * 1024 * 1024) := rfl
```

This compiles iff the default is literally `1024 * 1024 * 1024`.
If the default drifts (e.g., a future PR bumps it to 2 GiB), the
`rfl` fails and the probe fires as a compile error.

`3bbf32e0` documented the design choice:

> A bare `#check` only type-checks the signature, not the default
> value. A `#eval` would print on every test build and be noisy.
> An `example ... rfl` is silent on success and loud on failure —
> exactly what a regression probe should be.

`0aefbf8e` (#1623) and `54e1961a` (#1631) adopt this pattern
verbatim, including the docstring comment shape.

### Why this matters

The project's `scripts/check-inventory-links.sh` can verify that
the docstring *mentions* a number, but it cannot verify the
number equals the actual default. The `rfl` probe closes that
loop at compile time, without any script.

### What to codify

One short row in the **project CLAUDE.md's *Tests* subsection**
could note this pattern, but project CLAUDE.md is off-limits to
agents. An alternative is a new short skill or a row in
`proof-review-checklist`. I lean toward the latter because the
pattern is a review hygiene item, not a proof technique.

§8 contains the draft.

- **The review's triage labels become load-bearing.** F-a / F-b /

  recommendations in one of three buckets: *"landed in the
  reviewed PR"*, *"already addressed upstream"*, *"filed as

  the next planner doesn't have to re-read the review to decide
  what to queue.

A small `/review` command improvement: when a review produces
≥ 3 recommendations, require F-series tags and an explicit
triage bucket per recommendation. Flag for the next meditate or
planner; no skill or command edit proposed here because one more
wave of evidence would solidify it.

### 8.3 — `lean-uint-bitvec` — `pack_toNat_of_bounds` helper pattern

Proposed new subsection after the existing *"bv_decide on UInt
equations"* block:

> ### Packing small nibbles via bitwise OR
>
> When a `Nat` (or `UInt*`) is built by `x ||| (y <<< n)` with
> `x < 2^n` and `y < 2^m`, its `toNat` (or value) equals
> `x + y * 2^n`. The library lemma for this is
> `Nat.two_pow_add_eq_or_of_lt` (or, bitwidth-aware, the
> `BitVec` / `UInt*` equivalent).
>
> A reusable helper to apply this to two-nibble packs:
>
> ```lean
> lemma pack_toNat_of_bounds {x y : Nat} {n : Nat}
>     (hx : x < 2 ^ n) :
>     (y <<< n ||| x) = y * 2 ^ n + x := by
>   rw [Nat.two_pow_add_eq_or_of_lt hx]
>   ring
> ```
>
> Use this whenever a checksum, packed header, or bitfield adds
> lower bits into a higher-bit container. Adler-32 (PR #1641) is
> the project's first user; CRC32 could adopt it on its next
> refactor.

Also add a short row to the "when to prefer `bv_decide`" table:

| Goal shape | Prefer | Why |
|---|---|---|
| `if x == 0 then a else b = c` on `UInt32` | `bv_decide` | Handles the case-split inline without manual `by_cases` |

```markdown
# Inventory Reconciliation (lean-zip)

Patterns for executing `SECURITY_INVENTORY.md` *Recommended
policy* and *Missing work* bullets as individual PRs.

## One PR per bullet

**Rule**: each numbered *Recommended policy* item (and each
bulleted *Missing work* item) lands as its own PR. Do not
batch two items into one PR.

**Cost of violation**: `scripts/check-inventory-links.sh`
warnings cannot be attributed to a single executing PR, and the
post-merge line-anchor drift compounds. Plus the planner loses
the bullet-to-PR-number 1:1 mapping in PROGRESS.md.

**Precedent**: Rec. 1/2/3/4/5 landed as five PRs in the
2026-04-22 audit-completion wave (#1617, #1618, #1623, #1630,
#1631). The prior wave's P2.4 docstring split (#1573, #1586,
#1594) is the earlier analogue.

## The *Executed — …* past-tense one-liner

When a PR closes a numbered bullet, rewrite that bullet (in
the same PR) from future-tense to past-tense:

- **Before**: `- [ ] Recommend raising native-side default to
  1 GiB to match FFI whole-buffer (Rec. 5).`
- **After**: `- [x] **Rec. 5 executed**: native-side default
  raised to 1 GiB in #1617; see that PR for caller impact.`

Keep the `- [x]` checkbox (for `scripts/check-inventory-links.sh`
compatibility). Keep the rec. number (`**Rec. 5 executed**`) so
the planner can cross-reference. One sentence on caller impact
with a PR-number link.

## Half-closed two-step

Use a *two-step* (parameter-add PR + default-flip PR) when:

- The parameter is *new to a public API* but the existing
  callers are numerous.
- The default flip needs independent caller-impact analysis.
- The param-add touches multiple subsystems but the flip is
  localised.

Use a *one-step* when:

- The parameter has no existing callers (no backwards-compat
  burden).
- The flip's caller impact is trivially bounded.

**Between-step discipline**: during the half-closed window,
the bullet is re-phrased as *"half-closed by #N (parameter
added, default still 0). Flip tracked as #M."* — the
`#M` is the follow-up issue, filed at the same time as the
param-add PR goes up.

**Precedent**: #1610 (half-closed, default `0`) → #1631
(default-flip) was the 2026-04-22 Rec. 2 execution. #1630 is
the counter-example (new param, no callers, one-step).

## Line-anchor drift

Every inventory-touching PR should run
`scripts/check-inventory-links.sh` before PR creation. If the
warning count increases, cite the new warnings in the PR body
(they may be legitimate — e.g., if the PR moves an `IO.userError`
emission site).

The drift detector's warning baseline is *"non-zero and
growing slowly"*. Do not block on it unless the warnings point
to a genuine `grep`-to-line mismatch.

## Single author per wave

If two inventory-reconciliation bullets are claimed
concurrently, the second-to-merge pays rebase cost on
`SECURITY_INVENTORY.md`. Planner should throttle to ≤ 2
*edits-inventory* concurrent (per the #1600 meditate's §7
heuristic) and prefer *sequential* bullets over parallel ones.

## Scope — what this skill does not cover

- Does not cover *new* top-level inventory sections (that's a
  planner-design call, not a reconciliation shape).
- Does not cover the prose of the docstring changes themselves —
  those follow the #1573 template, documented in
  `error-wording-catalogue`.
```

## 9. Friction points

### F1 — Issue-template snapshot staleness in fast waves

**Symptom**: the summarize (#1646 / session `92f77207`) found
the issue body's *"remaining gaps"* list listed four items
already landed before the summarize session started. The sorry
baseline (*"sorry count 4 → 4, all in `Zip/Spec/XxHash.lean`"*)

(per #1487, 2026-03-27).

**Cause**: the planner's issue-template snapshot captures the
state at issue-creation time. In a six-hour wave, state moves
faster than planner snapshots.

**Evidence**: `92f77207` (summarize) §Scope boundary:

> **Sorry-count baseline in the issue body is also stale.** The
> issue claims "sorry count 4 → 4 (unchanged, all in
> `Zip/Spec/XxHash.lean`)". The XxHash spec file does not exist
> in this repository.

**Mitigation**: the summarize handled this in-narrative (honest
framing rather than silent propagation). That works but costs
one paragraph per affected field. A structural fix:

- Add a *"verify-at-claim"* step to `/feature`, `/review`,
  `/summarize`, and `/meditate` commands: re-run the key
  queries from the issue body at claim time (e.g.,
  `grep -rc sorry Zip/`, `gh pr list --state merged`, the
  `plans/track-e-current-audit-checklist.md` checkbox counts)

**Where to codify**: the `agent-worker-flow` skill's Step 4

with a *Quality-metric re-verification* bullet citing the exact
commands for the common baselines (sorry count, PR-list delta,
checklist-tick delta).

### F2 — `SECURITY_INVENTORY.md` merge-conflict shape

**Symptom**: three of the wave's 21 PRs (#1622, #1624, #1632)
are ride-along rebase fixes. All three resolve conflicts in the
same region: *Known inconsistencies* or *Missing work* bullet
lists where two parallel feature PRs each removed a (different)
bullet.

**Cause**: `SECURITY_INVENTORY.md` is a *shared mutable list*.
Two agents removing bullets in parallel produces a Git conflict
whose only sensible resolution is *"delete both of our bullets,
keep the rest"*. Git can't infer this structurally.

**Evidence**:

- `dac1a047` (#1622): *"The conflict block had #1618's removed
  bullet on one side, #1617's removed bullet on the other.
  Kept neither, added no new content — this is the canonical
  parallel-bullet-removal resolution shape."*
- `8e74071c` (#1624): *"Same shape as #1622: parallel removal
  of two different *Missing work* bullets by #1617 and #1623."*
- `eded8629` (#1632): *"#1630 and #1631 both deleted from
  *Recommended policy*; union-deletion resolution."*

**Mitigation**:

- *Short-term*: the resolution shape is now formulaic. The
  `agent-pr-recovery` skill's *Auto-merge race* section should
  name it explicitly — *union-deletion resolution* — with a
  two-line recipe.
- *Medium-term*: structure the inventory so bullets are in
  sorted sections, each with its own `- [ ]` / `- [x]` row.
  When two agents mutate rows independently, Git's line-level
  diff handles them cleanly (no conflict). This is a
  structural change to the inventory, not a process change.
  Flag for a planner session; the #1600 meditate already
  recommended a TOC which is a necessary precursor.

### F3 — `maxRecDepth` ceiling in long `do`-block test files

**Symptom**: two PRs in the wave (#1630 and #1642) hit the Lean
elaborator's `maxRecDepth` ceiling on `ZipTest/Tar.lean` /
`ZipTest/Gzip.lean` when adding a new test block.

**Evidence**:

- `bebc249a` (#1630): *"The single `do` block of
  `ZipTest.Tar.tests` was already close to the elaborator's
  `maxRecDepth` ceiling. Rather than raise the ceiling (which
  would mask the real issue), I extracted
  `runMaxTotalSizeBombTest` as a separate top-level helper."*
- `6302c83d` (#1642): *"Adding the 25-line bomb-limit test
  block pushed `ZipTest.Gzip` over. Raised `maxRecDepth` to
  2048 locally (`set_option maxRecDepth 2048` at the top of
  the file) because the file's `do` blocks are already
  heterogeneous enough that extracting a helper would not be
  proportionate."*

**Two alternative fixes in the same wave.** The extract-helper
path (#1630) is preferable in principle but costs more
refactoring attention. The `set_option` raise-the-ceiling path
(#1642) is cheaper in the moment but *does* mask file-size
rot.

**Cause**: each `do`-block statement in a test file elaborates
to an `HAppend.hAppend` on `TestM Unit`, which stacks linearly
in the elaborator. A 40-50 statement `do` block saturates the
default ceiling.

**Mitigation**:

- *Current rule-of-thumb* (implicit in `bebc249a`): when adding
  a test block that would push the file over, extract a
  top-level helper and call it from the `do` block.
- *Add to the project's* `lean-simp-tactics` or a new
  `lean-test-module-layout` skill: flag that `ZipTest/*.lean`
  files with > ~40 `do`-statement-level test cases should use
  extracted helpers. Document the two acceptable mitigations
  (extract vs. `set_option`) with the cost framing.

This didn't appear in the prior meditate, and it only showed
up twice in this wave — borderline for a skill. Flag for
observation; if a third wave hits the same ceiling, promote to
a skill.

### F4 — Fresh-worktree `lake -R build` reconfigure gotcha

**Symptom**: sessions `f0395b5d` and `8e74071c` both lost time

**Evidence**:

**Recommendation**: add a `coordination worktree-build` one-shot
