### 1. Merge Conflict Cascade Analysis

**The pattern**: PR develops conflicts → recovery issue created → recovery PR
develops conflicts → another recovery issue... (the cascade).

**Concrete instances**:
- PR #565 (compressed block wiring) → recovery #568 → PR #577 (conflicting)
- PR #549 (block/window validation) → recovery #558 → PR #561 (conflicting)
- PRs #591 and #593 (spec files) conflict on `Zip.lean` import manifest

**Root causes identified**:

1. **Large shared file**: ZstdFrame.lean (1059 lines) is modified by multiple
   concurrent agents. 17 commits touched it since 2026-03-01. Every merge to
   master invalidates outstanding PRs on this file.

2. **Manifest file**: `Zip.lean` is a root import list. Every new module adds
   a line. Two PRs creating separate spec files both modify `Zip.lean`, causing
   conflicts even though their actual content files don't overlap. These conflicts
   are trivial (just add the import in the right place) but trigger the same
   recovery workflow as non-trivial conflicts.

3. **Recovery creates more work**: Creating a recovery issue for a conflicted PR
   means another agent claims it, builds a branch, pushes — but by then more PRs
   have merged, and the recovery PR itself is stale.

**Recommendations for planners**:
- Split ZstdFrame.lean before scheduling more concurrent work on it (review #583
  already identified 3 extraction targets: ZstdHuffman, ZstdSequence, ZstdFrame-core)
- Serialize modifications to hot files — don't schedule 3 agents on one file
- Do NOT create recovery issues for Zip.lean-only conflicts
- Prioritize the bottleneck: #552 (compressed block wiring) blocks other work

**Recommendations for workers**:
- Always `git fetch origin && git rebase origin/master` before pushing
- If your recovery PR develops conflicts, rebase it yourself — don't create
  another recovery issue (the cascade stops with you)
- Check for other open PRs on the same files before starting work

### 2. Spec-vs-Implementation Sequencing Analysis

**Current approach**: Spec files (Zip/Spec/Zstd.lean, Fse.lean, ZstdHuffman.lean,
ZstdSequence.lean) are being created in parallel with ongoing implementation work
on ZstdFrame.lean.

**Assessment**: This is working and should continue. Reasons:

- Spec files define predicates (what SHOULD be true) that are independent of
  implementation details. They import types from `Zip.Native.ZstdFrame` but only
  reference stable public types, not internal implementation.

- Proofs that depend on implementation details (monadic unfolding of parsers) are
  explicitly deferred as sorrys with documented rationale — they'll be filled in
  after the implementation stabilizes.

- Concrete proofs that CAN be done now (predefined distribution validity, basic
  property checking) are being completed, providing immediate value.

- The DEFLATE track did specs after implementation, but that was a different
  situation — DEFLATE was already feature-complete. Zstd is still evolving, and
  writing specs now helps validate the design before it solidifies.

**Risk**: ZstdFrame.lean changes could break spec imports if types are renamed or
restructured. This risk is manageable because specs reference stable public types.

**Recommendation**: Continue parallel spec work. The file-splitting of ZstdFrame.lean
(when it happens) will actually help — spec files can import from more specific
modules (ZstdHuffman, ZstdSequence) instead of the monolithic ZstdFrame.
