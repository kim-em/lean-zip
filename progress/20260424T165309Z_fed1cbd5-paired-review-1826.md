## Findings per audit check

### A.1 Predicate correctness — OK

The guard at
[Zip/Archive.lean:334-337](/home/kim/lean-zip/Zip/Archive.lean:334):

```lean
let eocd64VersionMadeBy := Binary.readUInt16LE data (bufPos + 12)
unless (eocd64VersionMadeBy &&& 0xFF) ≤ 63 do
  throw (IO.userError
    s!"zip: ZIP64 EOCD64 versionMadeBy spec-version byte too high (versionMadeBy={eocd64VersionMadeBy}, spec-version={eocd64VersionMadeBy &&& 0xFF}, max supported 63)")
```

`eocd64VersionMadeBy : UInt16` is read with `Binary.readUInt16LE`
at `bufPos + 12` — the APPNOTE §4.4.2 field offset for the EOCD64
`versionMadeBy`. The mask `0xFF` isolates the low byte
(`versionMadeBy &&& 0xFF`), and the `≤ 63` comparison is on
`UInt16` after masking (no `.toNat` / `.toUInt8` narrowing, no
de-Morgan'd alternative such as `> 63` in the throwing branch).
The high byte (host OS code per APPNOTE §4.4.2.2) is explicitly
out of scope and cannot influence the guard: the mask zeroes it
before the comparison. Boundary check: `0x0000`, `0x002D`
(`go-zip64.zip`), `0x003F` (`63`), `0xFF3F` (`63` with Windows
host OS byte) all pass; `0x0040` (`64`, fixture value), `0x0340`
(fixture value with Unix host OS), `0x00FF`, `0x03FF` all reject.

### A.2 Insertion site — OK

The guard sits in `findEndOfCentralDir`'s ZIP64 branch,
immediately after the existing EOCD64 record-size check
(`:316-319`, PR #1761) and before the ZIP64 field reads
(`cdSize`/`cdOffset`/`totalEntries`/... at `:338-343`) and the
ZIP64-override sentinel checks (`:353-370`). Full EOCD64-branch
order in current master:

1. EOCD signature backscan (`:274-280`)
2. Standard-EOCD field reads (`stdNumberOfThisDisk16` etc.,
   `:285-296`)
3. ZIP64 Locator signature check (`:299`)
4. EOCD64 offset dereference + buffer-bounds check (`:300-304`)
5. EOCD64 signature check (`:305`)
6. EOCD64 record-size check `== 44` (`:316-319`, PR #1761)
7. **EOCD64 versionMadeBy low-byte ≤ 63 (`:334-337`, PR #1826)**
8. EOCD64 ZIP64 field reads (`cdSize`/`cdOffset`/... `:338-343`)
9. ZIP64-override sentinel checks (`:353-370`)

Running the `versionMadeBy` guard before the ZIP64 field reads is
the correct placement: APPNOTE §4.4.2.2 defines `versionMadeBy` as
an unconditional producer-identity field with no ZIP64-sentinel
semantics, so running it pre-override-resolution guarantees it
fires on every ZIP64 archive regardless of whether the standard
EOCD carries sentinels. Placed immediately after record-size
preserves the *"trust record-size first, then per-field shape"*
cadence established by PR #1761: if the record-size is wrong,
`versionMadeBy` at `+12` may not mean what we think it does.
Moving the guard later (past the `cdSize`/`cdOffset` reads) would
leak attacker-controlled `UInt64` values from a not-yet-validated
EOCD64 record into downstream state before the spec-version check
closes the parser-differential vector.
