### A.3 Smuggled value choice: `numEntriesThisDisk = 99 (0x0063)`

`numEntriesThisDisk` is a `UInt16` field at standard-EOCD offset `+8`
(per APPNOTE §4.3.16 "total number of entries in the central
directory on this disk"). The APPNOTE §4.3.16 ZIP64-indirection
sentinel is `0xFFFF = 65535`.

The fixture's choice of `99` is well-defended:

- `99 ≠ 0xFFFF`: so the "relaxed sentinel" arm at :411 fails on its
  first disjunct (`stdNumEntriesThisDisk16 == val16Max`).
- `99 ≠ 1` (the actual single-disk archive's per-disk entry count,
  carried in the ZIP64 EOCD64 `numEntriesThisDisk` at offset `+24`):
  so the second disjunct
  (`stdNumEntriesThisDisk16.toNat == numEntriesThisDisk`) also fails.
- `99` is also not numerically close to the ZIP64 override (`1`),
  avoiding any accidental off-by-one or sign-conversion coincidence.
- `99` is small enough to fit `UInt16` with no byte-order ambiguity
  (`0x0063`), matching the `cdSize=99` / `totalEntries=99` /
  `diskWhereCDStarts=99` sibling convention across the wave.

The value `99` is deliberately consistent across the per-slot wave
(`cdSize=99` in PR #1905, `totalEntries=99` in PR #1908,
`diskWhereCDStarts=99` in PR #1911, `numEntriesThisDisk=99` here,
`numberOfThisDisk=99` in in-flight PR #1909) — a uniform smuggling
convention that makes the family's fixture intent and inter-fixture
comparisons easy to read in `xxd` diffs.

- **F.1 Two-distinct-guards-sharing-a-field-name attribution.** The

- **F.2 UInt16 overflow-convention consistency.** `numEntriesThisDisk`
  is UInt16 at standard-EOCD position `+8`; the ZIP64 override at
  EOCD64 position `+24` is UInt64. Per the per-slot-family
  convention, the four UInt16 standard-EOCD fields (`totalEntries`,
  `numberOfThisDisk`, `numEntriesThisDisk`, `diskWhereCDStarts`)
  use `val16Max = 0xFFFF` as the sentinel, while the two UInt32
  fields (`cdSize`, `cdOffset`) use `val32Max = 0xFFFFFFFF`. The
  guard at :411 correctly uses `val16Max` for the UInt16
  `numEntriesThisDisk` field. No off-by-16-bits hazard. The
  implementation reads
  `stdNumEntriesThisDisk16.toNat == numEntriesThisDisk` where
  `numEntriesThisDisk : Nat` is sourced from
  `(Binary.readUInt64LE data (bufPos + 24)).toNat` — the UInt16
  → Nat conversion on the LHS and UInt64 → Nat conversion on the
  RHS guarantee a precise comparison without sign / overflow
  ambiguity (sibling invariant to paired-review-#1908 §F.2 for
  `totalEntries`).
