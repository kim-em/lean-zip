## Decisions and non-obvious bits

Two surprises during implementation, both captured in the script
header and in the commit message:

1. **Lean's bundled clang has no compiler-rt asan runtime.**
   Passing `-fsanitize=address,undefined` at link time fails with
   `cannot open libclang_rt.asan.a`. Workaround: do *not* put
   `-fsanitize=...` in `ZLIB_LDFLAGS`; instead link GCC's libasan and
   libubsan directly (`-lasan -lubsan`) with rpath so the runtimes
   are discoverable.
2. **ASan requires "first in initial library list".** When libasan
   loads as an ordinary `DT_NEEDED` entry it normally comes after the
   Lean shared libraries, and ASan aborts with "ASan runtime does not
   come first". Workaround: `LD_PRELOAD` libasan.so + libubsan.so for
   the single test invocation.

Build-system gotcha confirmed via the per-object trace: Lake's
`buildO` only hashes `hardArgs` (just `-fPIC` for us), not
`weakArgs` (where `ZLIB_CFLAGS` lives). That means toggling sanitizer
flags alone does not invalidate the cached `.o` ŌĆö `lake -R clean`
(which also forces config re-elaboration so the new `ZLIB_*` env vars
are re-read) is mandatory and runs unconditionally before the
sanitizer build.

The script also:

- Re-execs under `nix-shell --run` when `pkg-config` cannot find
  zlib and `shell.nix` is present (so the script works both inside
  and outside nix-shell).
- Sets `ASAN_OPTIONS=detect_leaks=0:abort_on_error=1` because Lean's
  runtime intentionally leaks bootstrap state; and
  `UBSAN_OPTIONS=print_stacktrace=1:halt_on_error=1` so UBSan
  findings are both visible and fatal.
- Has a `--help` flag that documents the requirements, env vars,
  and recovery for the likely failure modes.
