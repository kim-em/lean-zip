## Correctness audit findings

The de-boxing/packing equivalence theorems are **genuine equivalences, not
tautological or vacuous** (CLAUDE.md taxonomy level 3, algorithmic
correspondence — appropriate here since the de-boxed walk *is* a re-encoding of
the reference walk):

- `chainWalkFast_eq` / `chainWalkGuarded_eq`: proven-bounds (`[..]'h`) walk =
  panic-checked reference walk, bridged by `getElem!_pos`. Real (control flow
  identical, access method differs).
- `chainWalkPacked_eq`: the packed accumulator `bestPos*512 + bestLen` equals
  the reference walk's `(bestPos, bestLen)` re-encoded at *every* step. The
  packing is injective only because `bestLen < 512`, which is independently
  proven by `chainWalk_fst_le` (`bestLen ≤ maxLen`, from `chainWalk_spec`'s
  invariant `Q`) combined with the `min 258 _ ≤ 511` clamp at every call site
  (`min258_le_511`). The `_mod` / `_div` decode lemmas then recover the
  components exactly — a real characterization, not a restatement.
- **On-path check:** `mainLoop_eq_chain` (and the lazy `mainLoop_eq_chainLazy`)
  rewrite the iterative loop through `chainWalkGuardedPacked_mod` /
  `chainWalkGuardedPacked_div`, confirming the de-boxed packed return is the one
  actually consumed on the path the equivalence theorem covers. Not vacuous.
- `chainWalk_spec`'s invariant `Q` is a substantive in-window backward-match
  property (`bestPos < pos`, `pos - bestPos ≤ windowSize`, byte-equality over
  the match, `bestLen ≤ maxLen`) — the encodability/validity contracts the
  encoders consume rest on it, not on `prev`/`hashTable` contents.

No tautological or weakened statements found; no statement was changed.
